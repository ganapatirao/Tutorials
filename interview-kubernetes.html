1. Kubernetes Fundamentals

Q1: What is Kubernetes and why is it used?

Kubernetes (K8s) is an open-source container orchestration platform for automating deployment, scaling, and management of containerized applications.

Key Features:

  • Automated scaling: Scale applications up and down based on demand
  • Self-healing: Automatically restart failed containers
  • Load balancing: Distribute traffic across containers
  • Rolling updates: Deploy updates without downtime
  • Storage orchestration: Automate storage provisioning
  • Batch execution: Run batch jobs and cron jobs
  • Service discovery: Automatic service registration and discovery
Q2: What are the key components of Kubernetes architecture?

Control Plane Components:

  • API Server: REST API endpoint for Kubernetes operations
  • etcd: Key-value store for cluster configuration data
  • Scheduler: Assigns pods to nodes based on resource requirements
  • Controller Manager: Runs controller processes (replication, endpoints, namespace)
  • Cloud Controller Manager: Interfaces with cloud providers

Worker Node Components:

  • Kubelet: Agent that runs on each node and communicates with API server
  • Kube-proxy: Network proxy and load balancer
  • Container Runtime: Runs containers (Docker, containerd, CRI-O)

2. Pods and Containers

Q4: What is a Pod in Kubernetes?

Pod is the smallest deployable unit in Kubernetes. It represents a single instance of a running process in the cluster.

Key Characteristics:

  • Contains one or more containers
  • Shares network namespace and storage
  • Has a unique IP address
  • Ephemeral - can be created and destroyed
  • Containers in a pod can communicate via localhost
 Pod example
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
spec:
  containers:
  - name: nginx
    image: nginx:1.21
    ports:
    - containerPort: 80
Q5: What is the difference between a Pod and a Container?
  • Container: A lightweight, standalone package that includes everything needed to run an application (code, runtime, libraries, settings). Created from a Docker image.
  • Pod: A wrapper around one or more containers that share the same network namespace and storage. Pods are the smallest deployable unit in Kubernetes.
Q6: What are the different types of Pods?
  • Single-container pod: Most common type. Contains one container.
  • Multi-container pod: Contains multiple containers that work together (sidecar pattern, ambassador pattern, adapter pattern).
  • Static pod: Managed by kubelet, not by API server. Used for system components.
  • Init container: Specialized containers that run before app containers. Used for setup tasks.

3. Deployments and ReplicaSets

Q7: What is a Deployment in Kubernetes?

Deployment provides declarative updates for Pods and ReplicaSets. It manages the lifecycle of Pods and ensures the desired state is maintained.

Key Features:

  • Manages ReplicaSets
  • Provides rolling updates
  • Supports rollbacks
  • Self-healing capabilities
  • Declarative configuration
 Deployment example
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80
Q8: What is a ReplicaSet?

ReplicaSet ensures that a specified number of Pod replicas are running at all times.

Key Features:

  • Maintains a stable set of replica Pods
  • Creates or deletes Pods as needed
  • Typically managed by Deployments
  • Selector-based Pod management
Q9: What is the difference between Deployment and ReplicaSet?
Aspect Deployment ReplicaSet
Purpose Manages updates and rollbacks Ensures replica count
Rolling Updates Supported Not supported
Rollback Supported Not supported
Usage Production deployments Internal use by Deployment

4. Services and Networking

Q10: What is a Service in Kubernetes?

Service is an abstraction that defines a logical set of Pods and a policy to access them.

Service Types:

  • ClusterIP: Exposes service on cluster-internal IP. Default type.
  • NodePort: Exposes service on each node's IP at a static port.
  • LoadBalancer: Exposes service externally using cloud provider's load balancer.
  • ExternalName: Maps service to external DNS name.
 Service example
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  selector:
    app: nginx
  ports:
  - protocol: TCP
    port: 80
    targetPort: 80
  type: LoadBalancer
Q11: What is the difference between ClusterIP, NodePort, and LoadBalancer?
Type Access Use Case
ClusterIP Cluster internal only Internal services, databases
NodePort Node IP + Port Development, testing
LoadBalancer External IP Production, external access
Q12: What is Ingress in Kubernetes?

Ingress is an API object that manages external access to services in a cluster, typically HTTP/HTTPS.

Key Features:

  • Load balancing
  • SSL/TLS termination
  • Name-based virtual hosting
  • Path-based routing
 Ingress example
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: nginx-ingress
spec:
  rules:
  - host: example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: nginx-service
            port:
              number: 80

5. ConfigMaps and Secrets

Q13: What is a ConfigMap in Kubernetes?

ConfigMap is an API object used to store non-confidential data in key-value pairs. Pods can consume ConfigMaps as environment variables, command-line arguments, or configuration files.

 ConfigMap example
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  database_host: "mysql"
  database_port: "3306"
  app_settings: |
    debug=true
    log_level=info
Q14: What is a Secret in Kubernetes?

Secret is an object that contains sensitive data such as passwords, OAuth tokens, and SSH keys. Secrets are encoded in base64.

 Secret example
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
data:
  username: YWRtaW4=  # base64 encoded
  password: cGFzc3dvcmQ=  # base64 encoded
Q15: What is the difference between ConfigMap and Secret?
Aspect ConfigMap Secret
Data Type Non-sensitive data Sensitive data
Encoding Plain text Base64 encoded
Use Case Configuration files, environment variables Passwords, API keys, certificates
Etcd Storage Plain text Encrypted (if configured)

6. Storage and Volumes

Q16: What is a PersistentVolume (PV)?

PersistentVolume (PV) is a storage resource in the cluster that has been provisioned by an administrator or dynamically provisioned using Storage Classes.

PV Types:

  • HostPath: Storage from host node's filesystem
  • NFS: Network File System
  • AWS EBS: Amazon Elastic Block Store
  • Azure Disk: Azure managed disk
  • GCE Persistent Disk: Google Cloud persistent disk
Q17: What is a PersistentVolumeClaim (PVC)?

PersistentVolumeClaim (PVC) is a request for storage by a user. It is similar to a Pod consuming node resources. PVCs consume PV resources.

 PVC example
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: pvc-claim
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi
  storageClassName: standard
Q18: What is the difference between PV and PVC?
Aspect PersistentVolume (PV) PersistentVolumeClaim (PVC)
Purpose Storage resource Request for storage
Created By Administrator or dynamically User/Developer
Binding Binds to PVC Binds to PV
Lifecycle Independent of Pods Bound to Pod lifecycle

7. Namespaces and Resource Quotas

Q19: What is a Namespace in Kubernetes?

Namespace is a way to divide cluster resources between multiple users or teams. It provides a scope for names.

Key Features:

  • Virtual cluster within a physical cluster
  • Resource isolation
  • Resource quotas per namespace
  • Network policies per namespace
Q20: What is a ResourceQuota?

ResourceQuota provides constraints that limit aggregate resource consumption per namespace.

 ResourceQuota example
apiVersion: v1
kind: ResourceQuota
metadata:
  name: compute-resources
spec:
  hard:
    requests.cpu: "4"
    requests.memory: 8Gi
    limits.cpu: "8"
    limits.memory: 16Gi

8. StatefulSets and DaemonSets

Q21: What is a StatefulSet?

StatefulSet is the workload API object used to manage stateful applications. It provides guarantees about the ordering and uniqueness of Pods.

Key Features:

  • Stable, unique network identifiers
  • Stable, persistent storage
  • Ordered, graceful deployment and scaling
  • Ordered, graceful deletion and termination
Q22: What is a DaemonSet?

DaemonSet ensures that all (or some) nodes run a copy of a Pod. As nodes are added to the cluster, Pods are added to them.

Use Cases:

  • Cluster storage daemon (glusterd, ceph)
  • Log collection daemon (fluentd, logstash)
  • Node monitoring daemon (Node Exporter)

9. Security and RBAC

Q23: What is RBAC in Kubernetes?

Role-Based Access Control (RBAC) is a method of regulating access to computer or network resources based on the roles of individual users within an enterprise.

Key Components:

  • Role: Defines permissions within a namespace
  • ClusterRole: Defines permissions cluster-wide
  • RoleBinding: Binds a role to users/groups in a namespace
  • ClusterRoleBinding: Binds a cluster role cluster-wide
Q24: What is a ServiceAccount?

ServiceAccount provides an identity for processes that run in a Pod. Processes in containers contact the API server to authenticate.

Key Features:

  • Provides identity for Pods
  • Used for API server authentication
  • Automatically created for each namespace
  • Can be associated with RBAC policies

10. Monitoring and Logging

Q25: What are the key monitoring tools for Kubernetes?
  • Prometheus: Monitoring system and time series database
  • Grafana: Visualization and dashboarding
  • Kubelet: Provides metrics about nodes
  • cAdvisor: Container metrics
  • Metrics Server: Resource usage metrics
Q26: What are the key logging tools for Kubernetes?
  • Fluentd: Log collector and forwarder
  • Fluent Bit: Lightweight log processor
  • ELK Stack: Elasticsearch, Logstash, Kibana
  • Loki: Grafana's logging system

11. Helm Package Manager

Q27: What is Helm?

Helm is the package manager for Kubernetes. It helps you manage Kubernetes applications using Helm Charts.

Key Concepts:

  • Chart: A collection of files that describe a related set of Kubernetes resources
  • Repository: A place where charts are stored and shared
  • Release: An instance of a chart running in a Kubernetes cluster
Q28: What is a Helm Chart?

Helm Chart is a package of pre-configured Kubernetes resources. It contains templates, values, and metadata.

Chart Structure:

  • Chart.yaml: Chart metadata
  • values.yaml: Default configuration values
  • templates/: Kubernetes resource templates
  • templates/NOTES.txt: Usage notes

12. Advanced Topics

Q29: What is a Custom Resource Definition (CRD)?

Custom Resource Definition (CRD) extends the Kubernetes API by adding custom resources. It allows you to define your own resource types.

 CRD example
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: crontabs.stable.example.com
spec:
  group: stable.example.com
  versions:
    - name: v1
      served: true
      storage: true
  scope: Namespaced
  names:
    plural: crontabs
    singular: crontab
    kind: CronTab
Q30: What is an Operator in Kubernetes?

Operator is a method of packaging, deploying, and managing a Kubernetes application. It uses custom resources to manage applications and their components.

Key Concepts:

  • Uses CRDs to define application state
  • Implements control loop logic
  • Automates operational tasks
  • Examples: Prometheus Operator, Etcd Operator

  • kubectl create: Imperative approach. Creates resources directly. Doesn't maintain state. Not idempotent.

Q36: What is the difference between headless service and normal service?
  • Headless Service: No cluster IP assigned. DNS returns Pod IPs directly. Used for StatefulSets, custom load balancing.
  • Normal Service: Has cluster IP. DNS returns service IP. Used for load balancing across Pods.
Q37: What is the difference between init container and sidecar container?
  • Init Container: Runs before app containers. Runs to completion. Used for setup tasks. One-time execution.
  • Sidecar Container: Runs alongside app containers. Runs continuously. Used for logging, monitoring. Continuous execution.
Q38: What is the difference between pod disruption budget and resource quota?
  • Pod Disruption Budget: Limits number of Pods that can be voluntarily disrupted. Ensures availability during maintenance.
  • Resource Quota: Limits total resources in namespace. Prevents resource exhaustion.
Q39: What is the difference between network policy and service mesh?
  • Network Policy: Kubernetes native. Controls traffic at IP/port level. L3/L4 network segmentation.
  • Service Mesh: Additional layer (Istio, Linkerd). Controls traffic at application level. L7 traffic management, mTLS, observability.
Q40: What is the difference between static pod and normal pod?
  • Static Pod: Managed by kubelet. Not managed by API server. Used for system components. Configured on node.
  • Normal Pod: Managed by API server. Managed by controllers. Used for applications. Configured via manifests.
Q41: What are the best practices for Kubernetes security?
  • Use RBAC: Implement role-based access control
  • Use Secrets: Store sensitive data in Secrets, not ConfigMaps
  • Network Policies: Restrict pod-to-pod communication
  • Image Scanning: Scan container images for vulnerabilities
  • Pod Security Policies: Restrict pod capabilities
  • Use ServiceAccounts: Assign appropriate service accounts
  • Enable Audit Logging: Log all API server requests
  • Use TLS: Encrypt all communication
  • Regular Updates: Keep Kubernetes and components updated
  • Use Admission Controllers: Validate and mutate requests
Q42: What are the best practices for Kubernetes performance?
  • Resource Limits: Set appropriate CPU and memory limits
  • Resource Requests: Set appropriate CPU and memory requests
  • Horizontal Pod Autoscaler: Scale based on CPU/memory
  • Vertical Pod Autoscaler: Adjust resource requests
  • Cluster Autoscaler: Scale cluster nodes
  • Use Probes: Implement liveness and readiness probes
  • Optimize Images: Use small, optimized container images
  • Use Node Affinity: Schedule pods appropriately
  • Monitor Resources: Use monitoring tools
  • Use Caching: Implement caching where appropriate
Q43: What are the best practices for Kubernetes cost optimization?
  • Right-size Resources: Set appropriate resource limits and requests
  • Use Spot Instances: Use preemptible VMs for non-critical workloads
  • Cluster Autoscaler: Scale down unused nodes
  • Resource Quotas: Limit resource usage per namespace
  • Use Efficient Images: Use smaller images to reduce storage costs
  • Monitor Costs: Use cost monitoring tools
  • Use Shared Services: Share services across namespaces
  • Cleanup Unused Resources: Remove unused PVs, PVCs, and images
  • Use Reserved Instances: Commit to reserved instances for savings
  • Optimize Storage: Use appropriate storage classes

Q50: What are the common Kubernetes troubleshooting scenarios?
  • Pod not starting: Check image, resources, node capacity
  • Pod crash looping: Check logs, probes, application errors
  • Service not accessible: Check endpoints, selectors, network policies
  • PVC pending: Check storage class, PV availability
  • Node not ready: Check kubelet, node resources, network
  • Deployment not updating: Check replica sets, pod status
  • High resource usage: Check resource limits, requests, autoscaling
  • Network issues: Check DNS, network policies, CNI plugin
  • Authentication issues: Check RBAC, service accounts, tokens
  • Scheduling issues: Check taints, tolerations, node affinity

Q53: How do you implement multi-cluster Kubernetes management?

  name: myapp
  namespace: default

    - name: cluster1

  overrides:
  - clusterName: cluster2
    clusterOverrides:

Q54: How do you implement Kubernetes autoscaling strategies?

Autoscaling automatically adjusts resources based on workload.

# Horizontal Pod Autoscaler (HPA)
apiVersion: autoscaling/v2
kind: HorizontalPodAutcaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Pods
    pods:
      metric:
        name: http_requests_per_second
      target:
        type: AverageValue
        averageValue: "1000"
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 50
        periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 0
      policies:
      - type: Percent
        value: 100
        periodSeconds: 15

# Vertical Pod Autoscaler (VPA)
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutcaler
metadata:
  name: myapp-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  updatePolicy:
    updateMode: Auto
  resourcePolicy:
    containerPolicies:
    - containerName: myapp
      minAllowed:
        cpu: 100m
        memory: 100Mi
      maxAllowed:
        cpu: 1
        memory: 1Gi
      controlledResources: ["cpu", "memory"]

# Cluster Autoscaler
# Enable via cloud provider or Helm
helm install cluster-autoscaler stable/cluster-autoscaler \
  --set cloudProvider=aws \
  --set autoDiscovery.clusterName=my-cluster \
  --set awsRegion=us-west-2

apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8srequiredlabels
spec:
  crd:
    spec:
      names:
        kind: K8sRequiredLabels
  targets:
    - target: admission.k8s.gatekeeper.sh/v1
      rego: |
        package k8srequiredlabels
        violation[{"msg": msg, "details": {"missing_labels": missing}}] {
          required := input.parameters.labels
          provided := input.review.object.metadata.labels
          missing := [label | label := required[label] | not provided[label]]
          count(missing) > 0
          msg := sprintf("missing required labels: %v", [missing])
        }

apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sRequiredLabels
metadata:
  name: require-labels
spec:
  match:
    kinds:
    - apiGroups: [""]
      kinds: ["Pod"]
  parameters:
    labels:
      - "app"
      - "environment"

Network policies control traffic flow between pods.

# Deny all traffic
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress

# Allow traffic from specific namespace
apiVersion: networking.k8s.io/v1

)

func admit(w http.ResponseWriter, r *http.Request) {
    var admissionReview v1.AdmissionReview
    body, _ := ioutil.ReadAll(r.Body)
    json.Unmarshal(body, &admissionReview)

        Allowed: true,
    }
    
    resp, _ := json.Marshal(admissionReview)
    w.Write(resp)
}
