1. Microservices Fundamentals

Q1: What is Microservices Architecture?

Microservices Architecture is an architectural style that structures an application as a collection of small, autonomous services modeled around a business domain.

Key Characteristics:

  • Small Services: Each service is small and focused
  • Autonomous: Services are independently deployable
  • Decentralized: No central database or governance
  • Technology Agnostic: Different services can use different technologies
  • Resilient: Failure of one service doesn't bring down the system
Q2: What are the benefits of Microservices?
  • Scalability: Scale individual services independently
  • Flexibility: Use different technologies for different services
  • Faster Development: Small teams can work independently
  • Better Fault Isolation: Failure in one service doesn't affect others
  • Easier Maintenance: Smaller codebases are easier to understand
  • Continuous Deployment: Deploy services independently
Q3: What are the challenges of Microservices?
  • Complexity: More complex to design and manage
  • Network Latency: Inter-service communication adds latency
  • Data Consistency: Distributed transactions are complex
  • Testing: End-to-end testing is more complex
  • Deployment: More complex deployment infrastructure
  • Monitoring: Need distributed tracing and monitoring

  • Large Applications: Complex applications with many features
  • Multiple Teams: Multiple teams working on different features
  • Independent Scaling: Different services have different scaling needs
  • Technology Diversity: Need different technologies for different features
  • Frequent Deployments: Need to deploy different features independently

3. Service Communication

Q6: What are the types of service communication?
  • Synchronous: Request-response pattern. Services wait for response. Examples: REST, gRPC.
  • Asynchronous: Event-driven pattern. Services don't wait for response. Examples: Message queues, event streams.

Q8: What is message queuing?

Message Queuing is a method of asynchronous communication between services using a message broker. Services send messages to a queue, and other services consume them.

Benefits:

  • Decoupling: Services are decoupled
  • Asynchronous: Non-blocking communication
  • Reliability: Messages are persisted
  • Scalability: Consumers can scale independently

Popular Message Brokers:

  • RabbitMQ, Apache Kafka, AWS SQS, Azure Service Bus

4. API Gateway

Q9: What is an API Gateway?

API Gateway is a server that acts as an API front-end, receiving API requests, enforcing throttling and security policies, passing requests to the appropriate service, and returning the response.

Key Functions:

  • Request Routing: Route requests to appropriate services
  • Load Balancing: Distribute load across services
  • Authentication: Centralized authentication
  • Rate Limiting: Protect services from overload
  • Caching: Cache responses to improve performance
  • Request/Response Transformation: Transform requests and responses
Q10: What are the patterns for API Gateway?
  • Single Gateway: One gateway for all services. Simple but potential bottleneck.
  • Per-Service Gateway: Gateway for each service. More complex but no single point of failure.
  • Backend for Frontend (BFF): Gateway per client type. Optimized for specific clients.

5. Service Discovery

Q11: What is service discovery?

Service Discovery is the process of automatically detecting devices and services on a computer network. In microservices, it allows services to find each other without hardcoding addresses.

Types:

  • Client-Side Discovery: Client queries service registry. Example: Netflix Eureka.
  • Server-Side Discovery: Client calls load balancer. Load balancer queries registry. Example: AWS ALB.
Q12: What is a service registry?

Service Registry is a database of available service instances. Services register themselves on startup and deregister on shutdown.

Popular Service Registries:

  • Netflix Eureka: Java-based, REST-based
  • Consul: Go-based, DNS-based
  • Zookeeper: Java-based, consensus-based
  • Kubernetes: Built-in service discovery

6. Data Management

Q13: What is database per service pattern?

Database per Service Pattern is where each microservice has its own database. This ensures loose coupling and independent scaling.

Benefits:

  • Loose Coupling: Services are independent
  • Data Isolation: Each service owns its data
  • Technology Freedom: Use appropriate database for each service
  • Independent Scaling: Scale databases independently
Q14: How do you handle distributed transactions?

Distributed Transactions involve multiple services and databases. Traditional ACID transactions don't work well in distributed systems.

Patterns:

  • 2-Phase Commit (2PC): Coordinated commit. Strong consistency but blocking.
  • Saga Pattern: Sequence of local transactions. Eventual consistency.
  • TCC (Try-Confirm-Cancel): Business-level compensation.
Q15: What is the Saga pattern?

Saga Pattern is a sequence of local transactions where each transaction updates data within a single service. If a transaction fails, compensating transactions undo the changes.

Types:

  • Choreography: Services communicate via events. Decentralized coordination.
  • Orchestration: Central coordinator manages saga. Centralized coordination.

7. Resilience Patterns

Q16: What is the Circuit Breaker pattern?

Circuit Breaker Pattern prevents cascading failures by stopping requests to a failing service after a threshold of failures is reached.

States:

  • Closed: Normal operation, requests pass through
  • Open: Failing state, requests are blocked
  • Half-Open: Testing if service has recovered
Q17: What is the Retry pattern?

Retry Pattern automatically retries failed requests with exponential backoff to handle transient failures.

Best Practices:

  • Use exponential backoff
  • Limit number of retries
  • Retry only idempotent operations
  • Handle non-transient failures appropriately
Q18: What is the Bulkhead pattern?

Bulkhead Pattern isolates resources so that a failure in one part of the system doesn't affect other parts. It's inspired by ship bulkheads that prevent flooding from spreading.

Implementation:

  • Thread pool isolation
  • Semaphore isolation
  • Connection pool isolation

8. Security in Microservices

Q19: How do you secure microservices?
  • Authentication: Use OAuth 2.0, OpenID Connect
  • Authorization: Implement RBAC, ABAC
  • Service-to-Service: Use mTLS, service tokens
  • API Gateway: Centralized security at gateway
  • Network Security: Service mesh, network policies
  • Secrets Management: Use vault, Kubernetes secrets
Q20: What is mTLS?

mTLS (Mutual TLS) is an extension of TLS where both client and server authenticate each other using certificates. It's used for secure service-to-service communication.

Benefits:

  • Authentication: Both parties authenticate
  • Encryption: Encrypted communication
  • Zero Trust: No implicit trust

9. Observability

Q21: What is observability?

Observability is the ability to understand a system's internal state from its external outputs. It consists of three pillars: logs, metrics, and traces.

Three Pillars:

  • Logs: Discrete events (errors, warnings, info)
  • Metrics: Numerical measurements (CPU, memory, latency)
  • Traces: Request journey across services (distributed tracing)
Q22: What is distributed tracing?

Distributed Tracing tracks a request as it travels through multiple services, providing visibility into the entire request lifecycle.

Key Concepts:

  • Trace ID: Unique identifier for the entire request
  • Span ID: Identifier for each operation
  • Parent Span: Relationship between spans

Tools:

  • Jaeger, Zipkin, AWS X-Ray, Azure Application Insights

10. Deployment Strategies

Q23: What are the deployment strategies for microservices?
  • Blue-Green Deployment: Two identical environments. Switch traffic to new version.
  • Canary Deployment: Gradually roll out to subset of users.
  • Rolling Deployment: Gradually replace instances.
  • A/B Testing: Deploy different versions to test features.

11. Microservices Patterns

Q25: What is the Strangler Fig pattern?

Strangler Fig Pattern is a strategy for migrating a monolithic application to microservices by gradually replacing functionality with new services.

Process:

  1. Identify a piece of functionality
  2. Create a new microservice for it
  3. Route traffic to the new service
  4. Remove old functionality from monolith
  5. Repeat until monolith is replaced
Q26: What is the Sidecar pattern?

Sidecar Pattern deploys a helper process (sidecar) alongside the main service to provide cross-cutting concerns like logging, monitoring, or networking.

Benefits:

  • Separation of Concerns: Main service focuses on business logic
  • Language Agnostic: Sidecar can be in any language
  • Shared Infrastructure: Same lifecycle as main service
Q27: What is Service Mesh?

Service Mesh is a dedicated infrastructure layer for handling service-to-service communication. It provides features like load balancing, service discovery, and security.

Popular Service Meshes:

  • Istio: Feature-rich, Kubernetes-native
  • Linkerd: Lightweight, focused on performance
  • Consul Connect: Part of Consul ecosystem

12. Popular Frameworks

Q28: What is Spring Cloud?

Spring Cloud is a framework for building distributed systems and microservices in Java. It provides tools for service discovery, configuration management, circuit breakers, and more.

Key Components:

  • Spring Cloud Netflix: Eureka, Hystrix, Zuul
  • Spring Cloud Config: Centralized configuration
  • Spring Cloud Gateway: API gateway
  • Spring Cloud Sleuth: Distributed tracing
Q29: What is .NET Core Microservices?

.NET Core is a cross-platform framework well-suited for building microservices. It provides built-in support for dependency injection, configuration, and hosting.

Key Technologies:

  • ASP.NET Core: Web API framework
  • MassTransit: Message bus for distributed systems
  • Steeltoe: .NET tools for cloud-native apps
  • Docker: Containerization
  • Kubernetes: Orchestration

Q41: What are the best practices for microservices?
  • Single Responsibility: Each service does one thing well
  • Database per Service: Each service has its own database
  • API Gateway: Use API gateway for external access
  • Service Discovery: Implement service discovery
  • Observability: Implement logging, metrics, tracing
  • Resilience: Implement circuit breakers, retries
  • Security: Implement authentication, authorization
  • CI/CD: Automate deployment
  • Containerization: Use Docker for packaging
  • Orchestration: Use Kubernetes for orchestration
Q42: What are the best practices for API design in microservices?
  • RESTful Design: Follow REST principles
  • Versioning: Version your APIs
  • Documentation: Use Swagger/OpenAPI
  • Error Handling: Use standard error codes
  • Rate Limiting: Implement rate limiting
  • Validation: Validate input data
  • Security: Implement authentication, authorization
  • Testing: Test APIs thoroughly
Q43: What are the best practices for data consistency in microservices?
  • Eventual Consistency: Accept eventual consistency
  • Saga Pattern: Use saga for distributed transactions
  • Idempotency: Design idempotent operations
  • Event Sourcing: Consider event sourcing
  • CQRS: Separate read and write models
  • Compensating Transactions: Implement rollback logic
Q44: What are the best practices for testing microservices?
  • Unit Testing: Test individual services
  • Integration Testing: Test service interactions
  • Contract Testing: Test API contracts
  • End-to-End Testing: Test entire system
  • Consumer-Driven Contracts: Use Pact for contract testing
  • Test Doubles: Use mocks, stubs for dependencies
  • Test Automation: Automate tests in CI/CD
Q45: What are the common challenges in microservices?
  • Complexity: Increased system complexity
  • Network Latency: Inter-service communication overhead
  • Data Consistency: Distributed transactions are complex
  • Testing: End-to-end testing is difficult
  • Deployment: Complex deployment infrastructure
  • Monitoring: Need distributed monitoring
  • Debugging: Debugging across services is difficult
  • Team Coordination: Requires good team coordination
  • Cultural Change: Requires organizational change
  • Skill Requirements: Requires diverse skills
