1. Microservices Fundamentals

Q1: What is Microservices Architecture?

Microservices Architecture is an architectural style that structures an application as a collection of small, autonomous services modeled around a business domain.

Key Characteristics:

  • Small Services: Each service is small and focused
  • Autonomous: Services are independently deployable
  • Decentralized: No central database or governance
  • Technology Agnostic: Different services can use different technologies
  • Resilient: Failure of one service doesn't bring down the system
Q2: What are the benefits of Microservices?
  • Scalability: Scale individual services independently
  • Flexibility: Use different technologies for different services
  • Faster Development: Small teams can work independently
  • Better Fault Isolation: Failure in one service doesn't affect others
  • Easier Maintenance: Smaller codebases are easier to understand
  • Continuous Deployment: Deploy services independently
Q3: What are the challenges of Microservices?
  • Complexity: More complex to design and manage
  • Network Latency: Inter-service communication adds latency
  • Data Consistency: Distributed transactions are complex
  • Testing: End-to-end testing is more complex
  • Deployment: More complex deployment infrastructure
  • Monitoring: Need distributed tracing and monitoring

3. Service Communication

Q6: What are the types of service communication?
  • Synchronous: Request-response pattern. Services wait for response. Examples: REST, gRPC.
  • Asynchronous: Event-driven pattern. Services don't wait for response. Examples: Message queues, event streams.

Q8: What is message queuing?

Message Queuing is a method of asynchronous communication between services using a message broker. Services send messages to a queue, and other services consume them.

Benefits:

  • Decoupling: Services are decoupled
  • Asynchronous: Non-blocking communication
  • Reliability: Messages are persisted
  • Scalability: Consumers can scale independently

Popular Message Brokers:

  • RabbitMQ, Apache Kafka, AWS SQS, Azure Service Bus

4. API Gateway

Q9: What is an API Gateway?

API Gateway is a server that acts as an API front-end, receiving API requests, enforcing throttling and security policies, passing requests to the appropriate service, and returning the response.

Key Functions:

  • Request Routing: Route requests to appropriate services
  • Load Balancing: Distribute load across services
  • Authentication: Centralized authentication
  • Rate Limiting: Protect services from overload
  • Caching: Cache responses to improve performance
  • Request/Response Transformation: Transform requests and responses
Q10: What are the patterns for API Gateway?
  • Single Gateway: One gateway for all services. Simple but potential bottleneck.
  • Per-Service Gateway: Gateway for each service. More complex but no single point of failure.
  • Backend for Frontend (BFF): Gateway per client type. Optimized for specific clients.

5. Service Discovery

Q11: What is service discovery?

Service Discovery is the process of automatically detecting devices and services on a computer network. In microservices, it allows services to find each other without hardcoding addresses.

Types:

  • Client-Side Discovery: Client queries service registry. Example: Netflix Eureka.
  • Server-Side Discovery: Client calls load balancer. Load balancer queries registry. Example: AWS ALB.
Q12: What is a service registry?

Service Registry is a database of available service instances. Services register themselves on startup and deregister on shutdown.

Popular Service Registries:

  • Netflix Eureka: Java-based, REST-based
  • Consul: Go-based, DNS-based
  • Zookeeper: Java-based, consensus-based
  • Kubernetes: Built-in service discovery

6. Data Management

Q13: What is database per service pattern?

Database per Service Pattern is where each microservice has its own database. This ensures loose coupling and independent scaling.

Benefits:

  • Loose Coupling: Services are independent
  • Data Isolation: Each service owns its data
  • Technology Freedom: Use appropriate database for each service
  • Independent Scaling: Scale databases independently
Q14: How do you handle distributed transactions?

Distributed Transactions involve multiple services and databases. Traditional ACID transactions don't work well in distributed systems.

Patterns:

  • 2-Phase Commit (2PC): Coordinated commit. Strong consistency but blocking.
  • Saga Pattern: Sequence of local transactions. Eventual consistency.
  • TCC (Try-Confirm-Cancel): Business-level compensation.
Q15: What is the Saga pattern?

Saga Pattern is a sequence of local transactions where each transaction updates data within a single service. If a transaction fails, compensating transactions undo the changes.

Types:

  • Choreography: Services communicate via events. Decentralized coordination.
  • Orchestration: Central coordinator manages saga. Centralized coordination.

7. Resilience Patterns

Q16: What is the Circuit Breaker pattern?

Circuit Breaker Pattern prevents cascading failures by stopping requests to a failing service after a threshold of failures is reached.

States:

  • Closed: Normal operation, requests pass through
  • Open: Failing state, requests are blocked
  • Half-Open: Testing if service has recovered
Q17: What is the Retry pattern?

Retry Pattern automatically retries failed requests with exponential backoff to handle transient failures.

Best Practices:

  • Use exponential backoff
  • Limit number of retries
  • Retry only idempotent operations
  • Handle non-transient failures appropriately
Q18: What is the Bulkhead pattern?

Bulkhead Pattern isolates resources so that a failure in one part of the system doesn't affect other parts. It's inspired by ship bulkheads that prevent flooding from spreading.

Implementation:

  • Thread pool isolation
  • Semaphore isolation
  • Connection pool isolation

8. Security in Microservices

Q19: How do you secure microservices?
  • Authentication: Use OAuth 2.0, OpenID Connect
  • Authorization: Implement RBAC, ABAC
  • Service-to-Service: Use mTLS, service tokens
  • API Gateway: Centralized security at gateway
  • Network Security: Service mesh, network policies
  • Secrets Management: Use vault, Kubernetes secrets
Q20: What is mTLS?

mTLS (Mutual TLS) is an extension of TLS where both client and server authenticate each other using certificates. It's used for secure service-to-service communication.

Benefits:

  • Authentication: Both parties authenticate
  • Encryption: Encrypted communication
  • Zero Trust: No implicit trust

9. Observability

Q21: What is observability?

Observability is the ability to understand a system's internal state from its external outputs. It consists of three pillars: logs, metrics, and traces.

Three Pillars:

  • Logs: Discrete events (errors, warnings, info)
  • Metrics: Numerical measurements (CPU, memory, latency)
  • Traces: Request journey across services (distributed tracing)
Q22: What is distributed tracing?

Distributed Tracing tracks a request as it travels through multiple services, providing visibility into the entire request lifecycle.

Key Concepts:

  • Trace ID: Unique identifier for the entire request
  • Span ID: Identifier for each operation
  • Parent Span: Relationship between spans

Tools:

  • Jaeger, Zipkin, AWS X-Ray, Azure Application Insights

11. Microservices Patterns

Q25: What is the Strangler Fig pattern?

Strangler Fig Pattern is a strategy for migrating a monolithic application to microservices by gradually replacing functionality with new services.

Process:

  1. Identify a piece of functionality
  2. Create a new microservice for it
  3. Route traffic to the new service
  4. Remove old functionality from monolith
  5. Repeat until monolith is replaced
Q26: What is the Sidecar pattern?

Sidecar Pattern deploys a helper process (sidecar) alongside the main service to provide cross-cutting concerns like logging, monitoring, or networking.

Benefits:

  • Separation of Concerns: Main service focuses on business logic
  • Language Agnostic: Sidecar can be in any language
  • Shared Infrastructure: Same lifecycle as main service
Q27: What is Service Mesh?

Service Mesh is a dedicated infrastructure layer for handling service-to-service communication. It provides features like load balancing, service discovery, and security.

Popular Service Meshes:

  • Istio: Feature-rich, Kubernetes-native
  • Linkerd: Lightweight, focused on performance
  • Consul Connect: Part of Consul ecosystem

Event-driven architecture enables loose coupling through asynchronous messaging.

{
    BootstrapServers = "kafka:9092",

};

{
    Value = JsonSerializer.Serialize(orderCreatedEvent)
});

{
    BootstrapServers = "kafka:9092",
    GroupId = "order-processor",
    AutoOffsetReset = AutoOffsetReset.Earliest,
    EnableAutoCommit = false
};

using var consumer = new ConsumerBuilder<Ignore, string>(consumerConfig)
    .SetKeyDeserializer(Deserializers.Ignore)
    .SetValueDeserializer(Deserializers.Utf8)
    .Build();

consumer.Subscribe(new[] { "orders" });

// Consume events
while (true)
{
    var consumeResult = consumer.Consume(cancellationToken);
    
    try
    {
        var orderEvent = JsonSerializer.Deserialize<OrderCreatedEvent>(consumeResult.Message.Value);
        await ProcessOrderAsync(orderEvent);
        
        consumer.Commit(consumeResult);
    }
    catch (Exception ex)
    {
        _logger.LogError(ex, "Error processing order");
    }
}

using (var transaction = producer.BeginTransaction())
{
    await producer.ProduceAsync("orders", new Message<Null, string>
    {

}

public async Task GradualTrafficShiftAsync(string serviceName, int targetPercentage)
{
    var currentPercentage = await GetCurrentPercentageAsync(serviceName);
    
    while (currentPercentage < targetPercentage)
    {
        currentPercentage = Math.Min(currentPercentage + 5, targetPercentage);
        await UpdateVirtualServiceAsync(serviceName, currentPercentage);
        
        await Task.Delay(TimeSpan.FromMinutes(5));
        
        var metrics = await GetMetricsAsync(serviceName);
        if (metrics.ErrorRate > 0.01)
        {
            await RollbackAsync(serviceName);
            throw new Exception("Canary deployment failed, rolled back");
        }
    }
}

public class ChaosTestingService
{
    public async Task RunChaosExperimentAsync(string experimentName)
    {
        // Create chaos experiment
        await CreateChaosExperimentAsync(experimentName);
        
        // Monitor system health
        var health = await MonitorSystemHealthAsync(TimeSpan.FromMinutes(5));
        
        if (!health.IsHealthy)
        {
            // Stop experiment
            await StopChaosExperimentAsync(experimentName);
            
            // Generate report
            await GenerateFailureReportAsync(experimentName, health);
        }
    }
    
    private async Task<SystemHealth> MonitorSystemHealthAsync(TimeSpan duration)
    {
        var startTime = DateTime.UtcNow;
        var metrics = new List<SystemMetrics>();
        
        while (DateTime.UtcNow - startTime < duration)
        {
            var metric = await CollectMetricsAsync();
            metrics.Add(metric);
            
            if (metric.ErrorRate > 0.05 || metric.Latency > TimeSpan.FromSeconds(5))
            {
                return new SystemHealth { IsHealthy = false, Metrics = metrics };
            }
            
            await Task.Delay(TimeSpan.FromSeconds(10));
        }
        
        return new SystemHealth { IsHealthy = true, Metrics = metrics };
    }
}

// Cross-region service communication
public class RegionalServiceClient
{
    private readonly IServiceDiscovery _discovery;
    
    public async Task<T> CallServiceAsync<T>(string serviceName, string region)
    {
        var endpoints = await _discovery.DiscoverAsync(serviceName, region);
        
        // Prefer local region, fallback to others
        var localEndpoints = endpoints.Where(e => e.Region == region);
        if (localEndpoints.Any())
        {
            return await CallEndpointAsync<T>(localEndpoints.First());
        }
        
        return await CallEndpointAsync<T>(endpoints.OrderBy(e => e.Latency).First());
    }
}
Q52: How do you implement API composition patterns in microservices?

API composition combines data from multiple services into a single response.

// API Gateway composition pattern
public class OrderCompositionService
{
    private readonly IOrderService _orderService;
    private readonly ICustomerService _customerService;
    private readonly IProductService _productService;
    private readonly IInventoryService _inventoryService;
    
    public async Task<OrderDetailDto> GetOrderDetailAsync(int orderId)
    {
        // Parallel calls to independent services
        var orderTask = _orderService.GetOrderAsync(orderId);
        var customerTask = _customerService.GetCustomerAsync(orderId);
        
        await Task.WhenAll(orderTask, customerTask);
        
        var order = await orderTask;
        var customer = await customerTask;
        
        // Compose product details
        var productTasks = order.Items.Select(item => 
            _productService.GetProductAsync(item.ProductId));
        
        var products = await Task.WhenAll(productTasks);
        
        // Compose inventory information
        var inventoryTasks = order.Items.Select(item =>
            _inventoryService.GetInventoryAsync(item.ProductId));
        
        var inventories = await Task.WhenAll(inventoryTasks);
        
        // Compose final response
        return new OrderDetailDto
        {
            Order = order,
            Customer = customer,
            Items = order.Items.Select((item, index) => new OrderItemDetailDto
            {
                Item = item,
                Product = products[index],
                Inventory = inventories[index]
            }).ToList()
        };
    }
}

Q54: How do you implement rate limiting and throttling in microservices?

Rate limiting protects services from being overwhelmed by excessive requests.

// Token bucket rate limiter
public class TokenBucketRateLimiter
{
    private readonly int _capacity;
    private readonly int _refillRate;
    private int _tokens;
    private DateTime _lastRefill;
    
    public TokenBucketRateLimiter(int capacity, int refillRatePerSecond)
    {
        _capacity = capacity;
        _refillRate = refillRatePerSecond;
        _tokens = capacity;
        _lastRefill = DateTime.UtcNow;
    }
    
    public bool TryAcquire(int tokens = 1)
    {
        RefillTokens();
        
        if (_tokens >= tokens)
        {
            _tokens -= tokens;
            return true;
        }
        
        return false;
    }
    
    private void RefillTokens()
    {
        var now = DateTime.UtcNow;
        var elapsed = now - _lastRefill;
        var tokensToAdd = (int)(elapsed.TotalSeconds * _refillRate);
        
        _tokens = Math.Min(_capacity, _tokens + tokensToAdd);
        _lastRefill = now;
    }
}

// Redis-based distributed rate limiting
public class RedisRateLimiter
{
    private readonly IDatabase _redis;
    
    public async Task<bool> IsAllowedAsync(string key, int limit, TimeSpan window)
    {
        var now = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        var windowStart = now - (long)window.TotalMilliseconds;
        
        var transaction = _redis.CreateTransaction();
        
        // Remove old entries
        transaction.AddCondition(Condition.KeyNotExists(key));
        transaction.KeyDeleteAsync(key);
        
        // Add current request
        transaction.SortedSetAddAsync(key, now, now);
        
        // Remove entries outside window
        transaction.SortedSetRemoveRangeByScoreAsync(key, 0, windowStart);
        
        // Count requests in window
        var countTask = transaction.SortedSetLengthAsync(key);
        
        await transaction.ExecuteAsync();
        var count = await countTask;
        
        return count <= limit;
    }
}

// ASP.NET Core rate limiting middleware
public class RateLimitMiddleware
{
    private readonly RequestDelegate _next;
    private readonly IRateLimiter _rateLimiter;
    
    public async Task InvokeAsync(HttpContext context)
    {
        var clientId = context.Request.Headers["X-Client-ID"].FirstOrDefault();
        
        if (!await _rateLimiter.IsAllowedAsync(clientId, 100, TimeSpan.FromMinutes(1)))
        {
            context.Response.StatusCode = 429;
            await context.Response.WriteAsync("Rate limit exceeded");
            return;
        }
        
        await _next(context);
    }
}

Q57: How do you implement event sourcing in microservices?

Event sourcing stores state changes as a sequence of events.

// Event store interface
public interface IEventStore
{
    Task<IEnumerable<IDomainEvent>> GetEventsAsync(string aggregateId);
    Task SaveEventsAsync(string aggregateId, IEnumerable<IDomainEvent> events, int expectedVersion);
}

// Aggregate root with event sourcing
public abstract class AggregateRoot
{
    private readonly List<IDomainEvent> _events = new();
    public IReadOnlyCollection<IDomainEvent> Events => _events.AsReadOnly();
    
    public int Version { get; private set; }
    
    protected void AddEvent(IDomainEvent @event)
    {
        _events.Add(@event);
    }
    
    public void LoadFromHistory(IEnumerable<IDomainEvent> events)
    {
        foreach (var @event in events)
        {
            ApplyChange(@event);
            Version++;
        }
    }
    
    protected abstract void ApplyChange(IDomainEvent @event);
}

// Order aggregate
public class Order : AggregateRoot
{
    public Guid Id { get; private set; }
    public List<OrderItem> Items { get; private set; }
    public OrderStatus Status { get; private set; }
    
    public void CreateOrder(List<OrderItem> items)
    {
        AddEvent(new OrderCreatedEvent
        {
            OrderId = Guid.NewGuid(),
            Items = items,
            CreatedAt = DateTime.UtcNow
        });
    }
    
    public void ConfirmOrder()
    {
        AddEvent(new OrderConfirmedEvent
        {
            OrderId = Id,
            ConfirmedAt = DateTime.UtcNow
        });
    }
    
    protected override void ApplyChange(IDomainEvent @event)
    {
        switch (@event)
        {
            case OrderCreatedEvent e:
                Id = e.OrderId;
                Items = e.Items;
                Status = OrderStatus.Created;
                break;
            case OrderConfirmedEvent e:
                Status = OrderStatus.Confirmed;
                break;
        }
    }
}

// Event store implementation with Kafka
public class KafkaEventStore : IEventStore
{
    public async Task SaveEventsAsync(string aggregateId, IEnumerable<IDomainEvent> events, int expectedVersion)
    {
        foreach (var @event in events)
        {
            var message = new Message<Null, string>
            {
                Value = JsonSerializer.Serialize(@event),
                Headers = new Headers
                {
                    { "aggregateId", Encoding.UTF8.GetBytes(aggregateId) },
                    { "eventType", Encoding.UTF8.GetBytes(@event.GetType().Name) }
                }
            };
            
            await _producer.ProduceAsync("events", message);
        }
    }
}

public class BlueGreenDeploymentService
{
    public async Task DeployAsync(string serviceName, string newVersion)
    {
        // Deploy to green
        await DeployToEnvironmentAsync(serviceName, "green", newVersion);
        
        // Run smoke tests on green
        var greenHealthy = await RunSmokeTestsAsync(serviceName, "green");
        
        if (!greenHealthy)
        {
            await RollbackAsync(serviceName, "green");
            throw new Exception("Smoke tests failed on green environment");
        }
        
        // Switch traffic to green
        await SwitchTrafficAsync(serviceName, "green");
        
        // Verify production
        var productionHealthy = await VerifyProductionAsync(serviceName);
        
        if (!productionHealthy)
        {
            // Rollback to blue
            await SwitchTrafficAsync(serviceName, "blue");
            throw new Exception("Production verification failed");
        }
        
        // Clean up old blue
        await CleanupOldVersionAsync(serviceName, "blue");
    }
}

public class ApiGatewayAuthenticationMiddleware
{
    public async Task InvokeAsync(HttpContext context)
    {
        var token = context.Request.Headers["Authorization"].FirstOrDefault()?.Split(" ").Last();
        
        if (string.IsNullOrEmpty(token))
        {

            return;
        }

        if (!isValid)

        context.Items["User"] = claims;
        
        await _next(context);
    }
}
