Chapter 1: Introduction to SOLID

1.1 What are SOLID Principles?

SOLID is an acronym for five design principles intended to make software designs more understandable, flexible, and maintainable. These principles help developers create robust, scalable, and easy-to-maintain software systems.

Key Insight: SOLID principles are guidelines, not strict rules. Apply them judiciously based on your specific context and requirements.

1.2 The Five Principles

  • S - Single Responsibility Principle
  • O - Open/Closed Principle
  • L - Liskov Substitution Principle
  • I - Interface Segregation Principle
  • D - Dependency Inversion Principle

1.3 Benefits of SOLID

  • Improved code maintainability
  • Enhanced testability
  • Better code reusability
  • Easier to understand and modify
  • Reduced coupling between components

Chapter 2: Single Responsibility Principle (SRP)

2.1 What is SRP?

A class should have only one reason to change, meaning it should have only one job or responsibility.

// Violation of SRP
class User {
    public string Name { get; set; }
    public string Email { get; set; }
    
    public void Save() { /* Save to database */ }
    public void SendEmail() { /* Send email */ }
    public void Validate() { /* Validate user */ }
}

// Following SRP
class User {
    public string Name { get; set; }
    public string Email { get; set; }
}

class UserRepository {
    public void Save(User user) { /* Save to database */ }
}

class EmailService {
    public void SendEmail(User user, string message) { /* Send email */ }
}

class UserValidator {
    public bool Validate(User user) { /* Validate user */ }
}

2.2 Benefits of SRP

  • Easier to understand and maintain
  • Reduced risk of breaking other functionality
  • Better testability
  • Clearer separation of concerns

Chapter 3: Open/Closed Principle (OCP)

3.1 What is OCP?

Software entities should be open for extension but closed for modification. You should be able to extend behavior without modifying existing code.

// Violation of OCP
class AreaCalculator {
    public double CalculateArea(object shape) {
        if (shape is Rectangle) {
            var rect = (Rectangle)shape;
            return rect.Width * rect.Height;
        }
        else if (shape is Circle) {
            var circle = (Circle)shape;
            return 3.14 * circle.Radius * circle.Radius;
        }
        return 0;
    }
}

// Following OCP
interface IShape {
    double CalculateArea();
}

class Rectangle : IShape {
    public double Width { get; set; }
    public double Height { get; set; }
    public double CalculateArea() => Width * Height;
}

class Circle : IShape {
    public double Radius { get; set; }
    public double CalculateArea() => 3.14 * Radius * Radius;
}

class AreaCalculator {
    public double CalculateArea(IShape shape) => shape.CalculateArea();
}

2.2 Benefits of OCP

  • Reduces risk of introducing bugs
  • Makes code more extensible
  • Follows the "don't repeat yourself" principle
  • Easier to add new features

Chapter 4: Liskov Substitution Principle (LSP)

4.1 What is LSP?

Objects of a superclass should be replaceable with objects of a subclass without affecting the correctness of the program.

// Violation of LSP
class Bird {
    public virtual void Fly() { Console.WriteLine("Flying"); }
}

class Penguin : Bird {
    public override void Fly() { 
        throw new NotImplementedException("Penguins can't fly"); 
    }
}

// Following LSP
class Bird { }

class FlyingBird : Bird {
    public virtual void Fly() { Console.WriteLine("Flying"); }
}

class Sparrow : FlyingBird {
    public override void Fly() { Console.WriteLine("Sparrow flying"); }
}

class Penguin : Bird {
    public void Swim() { Console.WriteLine("Swimming"); }
}

4.2 Key Guidelines

  • Subclasses must not remove base class behavior
  • Subclasses must not violate base class invariants
  • Subclasses must not strengthen preconditions
  • Subclasses must not weaken postconditions

Chapter 5: Interface Segregation Principle (ISP)

5.1 What is ISP?

Clients should not be forced to depend on interfaces they don't use. Prefer smaller, more specific interfaces over large, general-purpose ones.

// Violation of ISP
interface IWorker {
    void Work();
    void Eat();
    void Sleep();
}

class Robot : IWorker {
    public void Work() { /* Work */ }
    public void Eat() { throw new NotImplementedException(); }
    public void Sleep() { throw new NotImplementedException(); }
}

// Following ISP
interface IWorkable {
    void Work();
}

interface IEatable {
    void Eat();
}

interface ISleepable {
    void Sleep();
}

class Human : IWorkable, IEatable, ISleepable {
    public void Work() { /* Work */ }
    public void Eat() { /* Eat */ }
    public void Sleep() { /* Sleep */ }
}

class Robot : IWorkable {
    public void Work() { /* Work */ }
}

5.2 Benefits of ISP

  • Reduces coupling between components
  • Makes interfaces more focused
  • Improves code maintainability
  • Prevents forcing implementations of unused methods

Chapter 6: Dependency Inversion Principle (DIP)

6.1 What is DIP?

Depend on abstractions, not concretions. High-level modules should not depend on low-level modules; both should depend on abstractions.

// Violation of DIP
class LightBulb {
    public void TurnOn() { Console.WriteLine("Light on"); }
    public void TurnOff() { Console.WriteLine("Light off"); }
}

class Switch {
    private LightBulb _bulb;
    
    public Switch() {
        _bulb = new LightBulb();
    }
    
    public void Toggle() {
        // Toggle bulb
    }
}

// Following DIP
interface ISwitchable {
    void TurnOn();
    void TurnOff();
}

class LightBulb : ISwitchable {
    public void TurnOn() { Console.WriteLine("Light on"); }
    public void TurnOff() { Console.WriteLine("Light off"); }
}

class Fan : ISwitchable {
    public void TurnOn() { Console.WriteLine("Fan on"); }
    public void TurnOff() { Console.WriteLine("Fan off"); }
}

class Switch {
    private ISwitchable _device;
    
    public Switch(ISwitchable device) {
        _device = device;
    }
    
    public void Toggle() {
        // Toggle device
    }
}

6.2 Benefits of DIP

  • Reduces coupling between components
  • Makes code more testable
  • Enables easier substitution of implementations
  • Follows the Hollywood Principle ("Don't call us, we'll call you")

Chapter 7: Applying SOLID Together

7.1 Real-World Example

Let's see how all SOLID principles work together in a practical example.

// A complete example applying all SOLID principles
interface ILogger {
    void Log(string message);
}

interface IEmailSender {
    void SendEmail(string to, string subject, string body);
}

interface IOrderProcessor {
    void ProcessOrder(Order order);
}

class Order {
    public int Id { get; set; }
    public string CustomerEmail { get; set; }
    public decimal Amount { get; set; }
}

class FileLogger : ILogger {
    public void Log(string message) {
        File.WriteAllText("log.txt", message);
    }
}

class SmtpEmailSender : IEmailSender {
    public void SendEmail(string to, string subject, string body) {
        // Send email via SMTP
    }
}

class OrderProcessor : IOrderProcessor {
    private readonly ILogger _logger;
    private readonly IEmailSender _emailSender;
    
    public OrderProcessor(ILogger logger, IEmailSender emailSender) {
        _logger = logger;
        _emailSender = emailSender;
    }
    
    public void ProcessOrder(Order order) {
        _logger.Log($"Processing order {order.Id}");
        // Process order logic
        _emailSender.SendEmail(order.CustomerEmail, "Order Confirmed", "Your order is confirmed");
    }
}

Chapter 8: Best Practices

8.1 When to Apply SOLID

  • Apply SOLID principles during design and refactoring
  • Don't over-engineer simple solutions
  • Consider the context and team size
  • Balance principles with practical constraints
  • Refactor towards SOLID as code evolves

8.2 Common Pitfalls

  • Over-abstraction leading to complexity
  • Applying principles blindly without context
  • Creating too many small interfaces
  • Ignoring performance implications
  • Focusing on principles over business value
Tip: SOLID principles are tools to guide design decisions, not strict rules to follow blindly. Use them to improve code quality when appropriate.
Important: The goal of SOLID is to create maintainable, flexible software. If applying a principle makes the code harder to understand or maintain, reconsider your approach.
Final Advice: Master SOLID principles through practice. Start with simple examples and gradually apply them to more complex scenarios. The best code is often the simplest code that solves the problem effectively.
