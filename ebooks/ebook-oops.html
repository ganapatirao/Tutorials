Chapter 1: Introduction to OOP

1.1 What is Object-Oriented Programming?

Object-Oriented Programming (OOP) is a programming paradigm based on the concept of "objects", which can contain data and code. The data is in the form of fields (often known as attributes or properties), and the code is in the form of procedures (often known as methods).

1.2 History of OOP

OOP concepts originated with Simula 67, a programming language designed for discrete event simulation. The concepts were further developed in Smalltalk, which introduced the term "object-oriented programming". Today, OOP is used in many popular languages including Java, C++, C#, Python, and more.

1.3 Why Use OOP?

  • Modularity: Break down complex problems into smaller, manageable pieces
  • Reusability: Write code once and reuse it in different parts of the application
  • Maintainability: Easier to understand, debug, and maintain
  • Flexibility: Easy to modify and extend existing code
  • Security: Data hiding protects internal state from external interference

1.4 Key OOP Concepts

  • Class: Blueprint or template for creating objects
  • Object: Instance of a class with state and behavior
  • Encapsulation: Bundling data and methods together
  • Inheritance: Creating new classes from existing classes
  • Polymorphism: Objects taking many forms
  • Abstraction: Hiding implementation details

Chapter 2: Classes and Objects

2.1 What is a Class?

A class is a blueprint or template that defines the properties and behaviors of objects. It specifies what data an object will hold and what operations can be performed on that data.

// Class definition in C#
public class Car
{
    // Properties (state)
    public string Brand { get; set; }
    public string Model { get; set; }
    public int Year { get; set; }
    
    // Constructor
    public Car(string brand, string model, int year)
    {
        Brand = brand;
        Model = model;
        Year = year;
    }
    
    // Methods (behavior)
    public void Start()
    {
        Console.WriteLine("Car is starting");
    }
    
    public void Accelerate()
    {
        Console.WriteLine("Car is accelerating");
    }
    
    public void Brake()
    {
        Console.WriteLine("Car is braking");
    }
}

2.2 What is an Object?

An object is an instance of a class. It has state (values of properties) and behavior (methods). Multiple objects can be created from a single class.

// Creating objects
Car car1 = new Car("Toyota", "Camry", 2022);
Car car2 = new Car("Honda", "Civic", 2023);

// Accessing object properties
Console.WriteLine(car1.Brand); // Toyota
Console.WriteLine(car2.Model); // Civic

// Calling object methods
car1.Start();
car2.Accelerate();
Real-world Example: Think of a class as a blueprint for a house, and objects as the actual houses built from that blueprint. The blueprint defines the structure, but each house can have different paint colors, furniture, etc.

Chapter 3: Encapsulation

3.1 What is Encapsulation?

Encapsulation is the bundling of data and methods that operate on that data within a single unit (class). It also involves restricting access to the internal state of an object, only allowing access through public methods.

3.2 Access Modifiers

Access modifiers control the visibility and accessibility of class members:

Modifier Same Class Same Package Subclass World
public ✓ ✓ ✓ ✓
protected ✓ ✓ ✓ ✗
default ✓ ✓ ✗ ✗
private ✓ ✗ ✗ ✗

3.3 Encapsulation Example

public class BankAccount
{
    // Private fields (encapsulated data)
    private decimal _balance;
    private string _accountNumber;
    
    // Constructor
    public BankAccount(string accountNumber, decimal initialBalance)
    {
        _accountNumber = accountNumber;
        _balance = initialBalance;
    }
    
    // Public methods to access/modify private data
    public decimal GetBalance()
    {
        return _balance;
    }
    
    public void Deposit(decimal amount)
    {
        if (amount > 0)
        {
            _balance += amount;
            Console.WriteLine($"Deposited: {amount:C}");
        }
        else
        {
            Console.WriteLine("Invalid deposit amount");
        }
    }
    
    public bool Withdraw(decimal amount)
    {
        if (amount > 0 && _balance >= amount)
        {
            _balance -= amount;
            Console.WriteLine($"Withdrawn: {amount:C}");
            return true;
        }
        Console.WriteLine("Insufficient funds or invalid amount");
        return false;
    }
}
Tip: Use properties (getters/setters) instead of public fields to implement encapsulation. This allows you to add validation logic later without breaking existing code.

Chapter 4: Inheritance

4.1 What is Inheritance?

Inheritance is a mechanism where a new class derives properties and behaviors from an existing class. It promotes code reusability and establishes a parent-child relationship between classes.

4.2 Types of Inheritance

  • Single Inheritance: A class inherits from one parent class
  • Multiple Inheritance: A class inherits from multiple parent classes (not supported in C#, supported in C++)
  • Multilevel Inheritance: A class inherits from a derived class (chain inheritance)
  • Hierarchical Inheritance: Multiple classes inherit from a single parent class
  • Hybrid Inheritance: Combination of multiple inheritance types

4.3 Inheritance Example

// Base class (Parent)
public class Animal
{
    public string Name { get; set; }
    public int Age { get; set; }
    
    public Animal(string name, int age)
    {
        Name = name;
        Age = age;
    }
    
    public void Eat()
    {
        Console.WriteLine($"{Name} is eating");
    }
    
    public void Sleep()
    {
        Console.WriteLine($"{Name} is sleeping");
    }
    
    public virtual void MakeSound()
    {
        Console.WriteLine("Animal makes a sound");
    }
}

// Derived class (Child)
public class Dog : Animal
{
    public string Breed { get; set; }
    
    public Dog(string name, int age, string breed) : base(name, age)
    {
        Breed = breed;
    }
    
    public void Fetch()
    {
        Console.WriteLine($"{Name} is fetching the ball");
    }
    
    public override void MakeSound()
    {
        Console.WriteLine("Woof!");
    }
}

// Another derived class
public class Cat : Animal
{
    public Cat(string name, int age) : base(name, age) { }
    
    public void Scratch()
    {
        Console.WriteLine($"{Name} is scratching");
    }
    
    public override void MakeSound()
    {
        Console.WriteLine("Meow!");
    }
}

4.4 Method Overriding

Method overriding allows a derived class to provide a specific implementation of a method that is already defined in its base class.

Animal myDog = new Dog("Buddy", 5, "Golden Retriever");
myDog.MakeSound(); // Outputs: Woof! (Dog's implementation)

Animal myCat = new Cat("Whiskers", 3);
myCat.MakeSound(); // Outputs: Meow! (Cat's implementation)

Chapter 5: Polymorphism

5.1 What is Polymorphism?

Polymorphism means "many forms". It allows objects of different types to be treated as objects of a common type. The same method can behave differently based on the object it's called on.

5.2 Types of Polymorphism

Compile-time Polymorphism (Static Binding): Method overloading and operator overloading.

public class Calculator
{
    // Method overloading
    public int Add(int a, int b)
    {
        return a + b;
    }
    
    public double Add(double a, double b)
    {
        return a + b;
    }
    
    public int Add(int a, int b, int c)
    {
        return a + b + c;
    }
    
    public string Add(string a, string b)
    {
        return a + b;
    }
}

Runtime Polymorphism (Dynamic Binding): Method overriding and virtual methods.

public class Shape
{
    public virtual void Draw()
    {
        Console.WriteLine("Drawing shape");
    }
}

public class Circle : Shape
{
    public override void Draw()
    {
        Console.WriteLine("Drawing circle");
    }
}

public class Rectangle : Shape
{
    public override void Draw()
    {
        Console.WriteLine("Drawing rectangle");
    }
}

// Polymorphism in action
List shapes = new List
{
    new Circle(),
    new Rectangle(),
    new Circle()
};

foreach (Shape shape in shapes)
{
    shape.Draw(); // Calls appropriate method based on object type
}
Important: Polymorphism enables you to write flexible and maintainable code. You can add new types without modifying existing code that uses the base type.

Chapter 6: Abstraction

6.1 What is Abstraction?

Abstraction is the concept of hiding complex implementation details and showing only the essential features of an object. It focuses on what an object does rather than how it does it.

6.2 Abstract Classes

An abstract class cannot be instantiated and can contain both abstract methods (without implementation) and concrete methods (with implementation).

public abstract class Vehicle
{
    public string Brand { get; set; }
    public string Model { get; set; }
    
    public Vehicle(string brand, string model)
    {
        Brand = brand;
        Model = model;
    }
    
    // Abstract method - must be implemented by derived classes
    public abstract void Start();
    public abstract void Stop();
    
    // Concrete method - can be used as-is or overridden
    public void Honk()
    {
        Console.WriteLine("Beep beep!");
    }
}

public class Car : Vehicle
{
    public Car(string brand, string model) : base(brand, model) { }
    
    public override void Start()
    {
        Console.WriteLine("Car engine starting");
    }
    
    public override void Stop()
    {
        Console.WriteLine("Car engine stopping");
    }
}

6.3 Interfaces

An interface defines a contract that classes must implement. It contains only method signatures (before C# 8) and no implementation.

public interface IShape
{
    double GetArea();
    double GetPerimeter();
}

public interface IResizable
{
    void Resize(double factor);
}

public class Circle : IShape, IResizable
{
    public double Radius { get; set; }
    
    public Circle(double radius)
    {
        Radius = radius;
    }
    
    public double GetArea()
    {
        return Math.PI * Radius * Radius;
    }
    
    public double GetPerimeter()
    {
        return 2 * Math.PI * Radius;
    }
    
    public void Resize(double factor)
    {
        Radius *= factor;
    }
}
Tip: Use abstract classes when you want to share code among related classes. Use interfaces when you want to define a contract for unrelated classes or when you need multiple inheritance.

Chapter 7: SOLID Principles

7.1 Single Responsibility Principle (SRP)

A class should have only one reason to change. It should have a single responsibility.

// Bad - Multiple responsibilities
public class User
{
    public string Name { get; set; }
    public string Email { get; set; }
    
    public void SaveToDatabase() { }
    public void SendEmail() { }
    public void Validate() { }
}

// Good - Single responsibility
public class User
{
    public string Name { get; set; }
    public string Email { get; set; }
}

public class UserRepository
{
    public void Save(User user) { }
}

public class EmailService
{
    public void SendEmail(User user) { }
}

public class UserValidator
{
    public bool Validate(User user) { }
}

7.2 Open/Closed Principle (OCP)

Software entities should be open for extension but closed for modification.

// Bad - Needs modification for new shapes
public class AreaCalculator
{
    public double CalculateArea(object shape)
    {
        if (shape is Circle)
            return Math.PI * Math.Pow(((Circle)shape).Radius, 2);
        else if (shape is Rectangle)
            return ((Rectangle)shape).Width * ((Rectangle)shape).Height;
        // Need to add more conditions for new shapes
    }
}

// Good - Open for extension
public interface IShape
{
    double CalculateArea();
}

public class Circle : IShape
{
    public double Radius { get; set; }
    public double CalculateArea()
    {
        return Math.PI * Math.Pow(Radius, 2);
    }
}

public class Rectangle : IShape
{
    public double Width { get; set; }
    public double Height { get; set; }
    public double CalculateArea()
    {
        return Width * Height;
    }
}

7.3 Liskov Substitution Principle (LSP)

Subtypes must be substitutable for their base types without affecting the correctness of the program.

// Bad - Violates LSP
public class Bird
{
    public virtual void Fly()
    {
        Console.WriteLine("Flying");
    }
}

public class Penguin : Bird
{
    public override void Fly()
    {
        throw new Exception("Penguins can't fly");
    }
}

// Good - Follows LSP
public class Bird { }

public class FlyingBird : Bird
{
    public virtual void Fly()
    {
        Console.WriteLine("Flying");
    }
}

public class Sparrow : FlyingBird
{
    public override void Fly()
    {
        Console.WriteLine("Sparrow flying");
    }
}

public class Penguin : Bird
{
    public void Swim()
    {
        Console.WriteLine("Swimming");
    }
}

7.4 Interface Segregation Principle (ISP)

Clients should not be forced to depend on interfaces they don't use.

// Bad - Fat interface
public interface IWorker
{
    void Work();
    void Eat();
    void Sleep();
}

public class Robot : IWorker
{
    public void Work() { }
    public void Eat() { throw new Exception(); }
    public void Sleep() { throw new Exception(); }
}

// Good - Segregated interfaces
public interface IWorker
{
    void Work();
}

public interface IEater
{
    void Eat();
}

public interface ISleeper
{
    void Sleep();
}

7.5 Dependency Inversion Principle (DIP)

Depend on abstractions, not on concretions. High-level modules should not depend on low-level modules.

// Bad - Depends on concrete implementation
public class Switch
{
    private LightBulb bulb;
    
    public Switch()
    {
        bulb = new LightBulb();
    }
    
    public void Toggle()
    {
        bulb.TurnOn();
    }
}

// Good - Depends on abstraction
public interface ISwitchable
{
    void TurnOn();
    void TurnOff();
}

public class LightBulb : ISwitchable
{
    public void TurnOn() { Console.WriteLine("Light on"); }
    public void TurnOff() { Console.WriteLine("Light off"); }
}

public class Switch
{
    private ISwitchable device;
    
    public Switch(ISwitchable device)
    {
        this.device = device;
    }
    
    public void Toggle()
    {
        device.TurnOn();
    }
}

Chapter 8: Design Patterns

8.1 Creational Patterns

Singleton Pattern: Ensures a class has only one instance.

public sealed class Singleton
{
    private static Singleton instance = null;
    private static readonly object padlock = new object();
    
    private Singleton() { }
    
    public static Singleton Instance
    {
        get
        {
            if (instance == null)
            {
                lock (padlock)
                {
                    if (instance == null)
                    {
                        instance = new Singleton();
                    }
                }
            }
            return instance;
        }
    }
}

Factory Method Pattern: Creates objects without specifying the exact class.

public interface IShape
{
    void Draw();
}

public class Circle : IShape
{
    public void Draw() => Console.WriteLine("Drawing Circle");
}

public class Rectangle : IShape
{
    public void Draw() => Console.WriteLine("Drawing Rectangle");
}

public class ShapeFactory
{
    public IShape GetShape(string shapeType)
    {
        return shapeType.ToLower() switch
        {
            "circle" => new Circle(),
            "rectangle" => new Rectangle(),
            _ => throw new ArgumentException("Invalid shape type")
        };
    }
}

8.2 Structural Patterns

Adapter Pattern: Allows incompatible interfaces to work together.

public interface ITarget
{
    string GetRequest();
}

public class Adaptee
{
    public string GetSpecificRequest()
    {
        return "Specific request";
    }
}

public class Adapter : ITarget
{
    private readonly Adaptee _adaptee;
    
    public Adapter(Adaptee adaptee)
    {
        _adaptee = adaptee;
    }
    
    public string GetRequest()
    {
        return _adaptee.GetSpecificRequest();
    }
}

8.3 Behavioral Patterns

Observer Pattern: Defines a subscription mechanism to notify multiple objects about events.

public interface IObserver
{
    void Update(string message);
}

public interface ISubject
{
    void Attach(IObserver observer);
    void Detach(IObserver observer);
    void Notify(string message);
}

public class NewsPublisher : ISubject
{
    private List observers = new List();
    
    public void Attach(IObserver observer) => observers.Add(observer);
    public void Detach(IObserver observer) => observers.Remove(observer);
    
    public void Notify(string message)
    {
        foreach (var observer in observers)
        {
            observer.Update(message);
        }
    }
}

Chapter 9: OOP vs Procedural Programming

9.1 Key Differences

Aspect OOP Procedural Programming
Focus Objects and their interactions Functions and procedures
Data Data and methods bundled together Data separate from functions
Access Controlled via access modifiers Global access to data
Reuse Inheritance and composition Function libraries
Complexity Handles complex systems better Simpler for small programs
Example C#, Java, Python C, Pascal, Fortran

9.2 When to Use Each

Use OOP when:
  • Building large, complex applications
  • Need for code reusability and maintainability
  • Modeling real-world entities
  • Multiple developers working on the project
Use Procedural Programming when:
  • Building small, simple programs
  • Performance is critical
  • Learning programming basics
  • System-level programming

Chapter 10: Best Practices

10.1 General OOP Best Practices

  • Follow SOLID principles
  • Use composition over inheritance
  • Keep classes small and focused
  • Use meaningful names for classes and methods
  • Favor interfaces over abstract classes when possible
  • Implement proper encapsulation
  • Write unit tests for your classes
  • Document your code with XML comments

10.2 Common Mistakes to Avoid

  • Creating god classes (classes that do too much)
  • Overusing inheritance
  • Breaking encapsulation with public fields
  • Tight coupling between classes
  • Ignoring error handling
  • Not following naming conventions
Important: Remember that OOP is a tool, not a solution. Use it when it makes sense for your problem, and don't force OOP patterns where simpler solutions would work better.
Tip: The best way to learn OOP is through practice. Start with small projects and gradually apply OOP principles. Review your code regularly and refactor it to follow best practices.
