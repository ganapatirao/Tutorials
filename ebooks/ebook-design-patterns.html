Chapter 1: Introduction to Design Patterns

1.1 What are Design Patterns?

Design patterns are reusable solutions to common problems in software design. They represent best practices and provide a standard approach to solve recurring architectural challenges. Patterns are not code, but templates for solving problems that can be applied in many different situations.

Key Insight: Design patterns are not code, but templates for solving problems that can be applied in many different situations.

1.2 Why Use Design Patterns?

  • Reusability: Proven solutions that can be applied across projects
  • Communication: Common vocabulary for developers
  • Maintainability: Well-structured, organized code
  • Best Practices: Industry-standard approaches
  • Efficiency: Faster development with proven patterns

1.3 Pattern Categories

  • Creational: Object creation mechanisms
  • Structural: Class and object composition
  • Behavioral: Communication between objects

Chapter 2: Creational Patterns

2.1 Singleton Pattern

Ensures a class has only one instance and provides a global point of access to it.

class Singleton {
    private static instance: Singleton;
    
    private constructor() {}
    
    public static getInstance(): Singleton {
        if (!Singleton.instance) {
            Singleton.instance = new Singleton();
        }
        return Singleton.instance;
    }
}

// Usage
const instance1 = Singleton.getInstance();
const instance2 = Singleton.getInstance();
console.log(instance1 === instance2); // true

2.2 Factory Method Pattern

Defines an interface for creating objects but lets subclasses decide which class to instantiate.

interface Product {
    operation(): string;
}

class ConcreteProductA implements Product {
    operation(): string {
        return "Product A";
    }
}

class ConcreteProductB implements Product {
    operation(): string {
        return "Product B";
    }
}

interface Creator {
    factoryMethod(): Product;
}

class ConcreteCreatorA implements Creator {
    factoryMethod(): Product {
        return new ConcreteProductA();
    }
}

2.3 Builder Pattern

Separates the construction of complex objects from their representation.

class Builder {
    private product: Product = new Product();
    
    buildPartA(): this {
        this.product.addPart("Part A");
        return this;
    }
    
    buildPartB(): this {
        this.product.addPart("Part B");
        return this;
    }
    
    getResult(): Product {
        return this.product;
    }
}

// Usage
const product = new Builder()
    .buildPartA()
    .buildPartB()
    .getResult();

2.4 Prototype Pattern

Creates new objects by cloning existing ones.

class Prototype implements Cloneable {
    constructor(private data: any) {}
    
    clone(): Prototype {
        return new Prototype(this.data);
    }
    
    getData(): any {
        return this.data;
    }
}

Chapter 3: Structural Patterns

3.1 Adapter Pattern

Allows incompatible interfaces to work together.

interface Target {
    request(): string;
}

class Adaptee {
    specificRequest(): string {
        return "Adaptee's specific request";
    }
}

class Adapter implements Target {
    constructor(private adaptee: Adaptee) {}
    
    request(): string {
        return this.adaptee.specificRequest();
    }
}

3.2 Decorator Pattern

Adds behavior to objects dynamically without affecting other objects.

interface Component {
    operation(): string;
}

class ConcreteComponent implements Component {
    operation(): string {
        return "Concrete Component";
    }
}

class Decorator implements Component {
    constructor(protected component: Component) {}
    
    operation(): string {
        return this.component.operation();
    }
}

class ConcreteDecorator extends Decorator {
    operation(): string {
        return Decorated();
    }
}

3.3 Facade Pattern

Provides a simplified interface to a complex subsystem.

class SubsystemA {
    operationA(): string {
        return "Subsystem A operation";
    }
}

class SubsystemB {
    operationB(): string {
        return "Subsystem B operation";
    }
}

class Facade {
    constructor(
        private subsystemA: SubsystemA,
        private subsystemB: SubsystemB
    ) {}
    
    operation(): string {
        return ${this.subsystemA.operationA()} + ;
    }
}

3.4 Proxy Pattern

Provides a surrogate or placeholder to control access to an object.

interface Subject {
    request(): string;
}

class RealSubject implements Subject {
    request(): string {
        return "Real Subject request";
    }
}

class Proxy implements Subject {
    private realSubject: RealSubject;
    
    request(): string {
        if (!this.realSubject) {
            this.realSubject = new RealSubject();
        }
        return this.realSubject.request();
    }
}

Chapter 4: Behavioral Patterns

4.1 Observer Pattern

Defines a subscription mechanism to notify multiple objects about events.

interface Observer {
    update(data: any): void;
}

class Subject {
    private observers: Observer[] = [];
    
    attach(observer: Observer): void {
        this.observers.push(observer);
    }
    
    detach(observer: Observer): void {
        this.observers = this.observers.filter(o => o !== observer);
    }
    
    notify(data: any): void {
        this.observers.forEach(observer => observer.update(data));
    }
}

4.2 Strategy Pattern

Defines a family of algorithms and makes them interchangeable.

interface Strategy {
    execute(data: any): any;
}

class ConcreteStrategyA implements Strategy {
    execute(data: any): any {
        return data * 2;
    }
}

class ConcreteStrategyB implements Strategy {
    execute(data: any): any {
        return data + 10;
    }
}

class Context {
    constructor(private strategy: Strategy) {}
    
    setStrategy(strategy: Strategy): void {
        this.strategy = strategy;
    }
    
    executeStrategy(data: any): any {
        return this.strategy.execute(data);
    }
}

4.3 Command Pattern

Encapsulates a request as an object.

interface Command {
    execute(): void;
}

class Receiver {
    action(): void {
        console.log("Receiver action");
    }
}

class ConcreteCommand implements Command {
    constructor(private receiver: Receiver) {}
    
    execute(): void {
        this.receiver.action();
    }
}

class Invoker {
    private command: Command;
    
    setCommand(command: Command): void {
        this.command = command;
    }
    
    executeCommand(): void {
        this.command.execute();
    }
}

4.4 State Pattern

Allows an object to alter its behavior when its internal state changes.

interface State {
    handle(context: Context): void;
}

class ConcreteStateA implements State {
    handle(context: Context): void {
        console.log("State A");
        context.setState(new ConcreteStateB());
    }
}

class ConcreteStateB implements State {
    handle(context: Context): void {
        console.log("State B");
        context.setState(new ConcreteStateA());
    }
}

class Context {
    private state: State;
    
    setState(state: State): void {
        this.state = state;
    }
    
    request(): void {
        this.state.handle(this);
    }
}

Chapter 5: Pattern Selection

5.1 When to Use Patterns

  • Singleton: When you need exactly one instance
  • Factory: When you want to defer object creation
  • Builder: For complex object construction
  • Adapter: To make incompatible interfaces work
  • Decorator: To add behavior dynamically
  • Observer: For publish-subscribe scenarios
  • Strategy: For interchangeable algorithms

5.2 Pattern Combinations

Patterns often work together. Common combinations include:

  • Factory + Singleton
  • Observer + Mediator
  • Decorator + Composite
  • Strategy + State

Chapter 6: Anti-Patterns

6.1 Common Anti-Patterns

  • God Object: Class that does too much
  • Spaghetti Code: Unstructured, tangled code
  • Golden Hammer: Using the same solution everywhere
  • Dead Code: Unused code that remains in the codebase
  • Magic Numbers: Hard-coded values without explanation
Tip: Recognize anti-patterns early and refactor them using appropriate design patterns.

Chapter 7: Best Practices

7.1 Pattern Guidelines

  • Don't force patterns where they don't fit
  • Understand the problem before choosing a pattern
  • Keep patterns simple and focused
  • Document pattern usage in your code
  • Review and refactor pattern implementations

7.2 Code Quality

  • Use meaningful names for pattern classes
  • Follow SOLID principles alongside patterns
  • Write tests for pattern implementations
  • Maintain separation of concerns
  • Keep pattern implementations maintainable
Important: Design patterns are tools, not rules. Use them judiciously to solve real problems, not to show off knowledge.
Final Advice: Master the patterns, but remember that the best code is often the simplest code that solves the problem effectively.
