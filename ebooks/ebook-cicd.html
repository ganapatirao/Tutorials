Chapter 1: Introduction to CI/CD

1.1 What is CI/CD?

CI/CD is a method to frequently deliver apps to customers by introducing automation into the stages of app development. The main concepts are Continuous Integration (CI) and Continuous Deployment (CD).

1.2 Benefits of CI/CD

  • Faster time to market
  • Early bug detection
  • Automated testing
  • Consistent deployments
  • Reduced manual errors
  • Improved collaboration

Chapter 2: Git and Version Control

2.1 Git Fundamentals

  • Repository: Storage for project files and history
  • Commit: Snapshot of changes
  • Branch: Independent line of development
  • Merge: Combine changes from branches
  • Pull Request: Request to merge changes

2.2 Git Workflow

# Clone repository
git clone https://github.com/user/repo.git

# Create branch
git checkout -b feature/new-feature

# Stage and commit
git add .
git commit -m "Add new feature"

# Push branch
git push origin feature/new-feature

# Merge via pull request

Chapter 3: Continuous Integration

3.1 CI Pipeline Stages

  • Build: Compile code and resolve dependencies
  • Test: Run unit tests, integration tests
  • Code Quality: Static analysis, linting
  • Package: Create deployable artifacts

3.2 CI Best Practices

  • Run CI on every commit
  • Keep builds fast
  • Use caching for dependencies
  • Parallelize test execution
  • Fail fast on errors

Chapter 4: Continuous Deployment

4.1 CD Pipeline Stages

  • Deploy to Staging: Test environment deployment
  • Integration Testing: End-to-end tests
  • Deploy to Production: Production environment
  • Health Checks: Verify deployment success

4.2 Deployment Strategies

  • Blue-Green: Two identical production environments
  • Canary: Gradual rollout to subset of users
  • Rolling: Incrementally replace instances
  • Feature Flags: Control feature availability

Chapter 5: GitHub Actions

5.1 Workflow Configuration

name: CI/CD Pipeline

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest
    
    steps:
    - uses: actions/checkout@v3
    
    - name: Setup .NET
      uses: actions/setup-dotnet@v3
      with:
        dotnet-version: '7.0'
    
    - name: Restore dependencies
      run: dotnet restore
    
    - name: Build
      run: dotnet build --configuration Release
    
    - name: Test
      run: dotnet test --no-build --configuration Release

Chapter 6: Azure DevOps Pipelines

6.1 Pipeline Types

  • Classic Pipelines: UI-based pipeline configuration
  • YAML Pipelines: Code-based pipeline configuration
  • Multi-stage Pipelines: Separate stages for CI and CD

6.2 YAML Pipeline Example

trigger:
- main

pool:
  vmImage: 'ubuntu-latest'

stages:
- stage: Build
  jobs:
  - job: Build
    steps:
    - task: DotNetCoreCLI@2
      inputs:
        command: 'build'
        projects: '**/*.csproj'

- stage: Deploy
  dependsOn: Build
  jobs:
  - deployment: DeployWeb
    environment: 'production'

Chapter 7: Docker in CI/CD

7.1 Building Docker Images

# Dockerfile
FROM mcr.microsoft.com/dotnet/sdk:7.0 AS build
WORKDIR /src
COPY ["MyApp.csproj", "./"]
RUN dotnet restore
COPY . .
RUN dotnet build -c Release

FROM mcr.microsoft.com/dotnet/aspnet:7.0 AS runtime
WORKDIR /app
COPY --from=build /src/publish .
ENTRYPOINT ["dotnet", "MyApp.dll"]

7.2 CI/CD with Docker

  • Build Docker image in CI
  • Push to container registry
  • Deploy containers to production
  • Use Docker Compose for local testing

Chapter 8: Kubernetes Deployment

8.1 Deployment Manifest

apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: myregistry.azurecr.io/myapp:v1
        ports:
        - containerPort: 80

8.2 CI/CD to Kubernetes

  • Build and push Docker image
  • Update Kubernetes manifests
  • Apply changes to cluster
  • Use Helm for package management
