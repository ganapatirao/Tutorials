1. CI/CD Fundamentals

Q1: What is CI/CD and why is it important?

CI/CD (Continuous Integration/Continuous Deployment) is a practice to automate the integration and deployment of code changes.

Continuous Integration (CI):

  • Developers frequently merge code changes
  • Automated builds and tests run on every commit
  • Early detection of integration issues
  • Reduced integration problems

Continuous Deployment (CD):

  • Automated deployment of code changes
  • Code changes automatically deployed to production
  • Faster time to market
  • Reduced manual errors

Benefits:

  • Faster development cycles
  • Higher code quality
  • Reduced deployment risks
  • Improved collaboration
  • Faster time to market
Q2: What is the difference between CI, CD, and CD?
  • CI (Continuous Integration): Frequent code integration with automated builds and tests. Focus on code quality and early bug detection.
  • CD (Continuous Delivery): Code changes are automatically built, tested, and prepared for release to production. Manual approval required for deployment.
  • CD (Continuous Deployment): Code changes are automatically deployed to production without manual intervention. Fully automated pipeline.

2. CI Pipeline Concepts

Q3: What are the key stages in a CI pipeline?
  1. Source: Code commit to version control
  2. Build: Compile code, resolve dependencies
  3. Unit Tests: Run automated unit tests
  4. Code Analysis: Static code analysis, security scanning
  5. Package: Create deployable artifacts
  6. Publish: Store artifacts for deployment
// Example CI pipeline (GitHub Actions)
name: CI Pipeline

on:
  push:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest
    
    steps:
    - uses: actions/checkout@v2
    
    - name: Setup .NET
      uses: actions/setup-dotnet@v1
      with:
        dotnet-version: '6.0.x'
    
    - name: Restore dependencies
      run: dotnet restore
    
    - name: Build
      run: dotnet build --configuration Release --no-restore
    
    - name: Test
      run: dotnet test --no-build --configuration Release
    
    - name: Publish
      run: dotnet publish --configuration Release --no-build -o ./publish
Q4: What is a build artifact?

A build artifact is the output of the build process that can be deployed. Examples include:

  • Compiled binaries (DLL, JAR, WAR)
  • Docker images
  • Package files (npm, NuGet, pip)
  • Static assets (HTML, CSS, JS)
  • Infrastructure templates (CloudFormation, Terraform)
Important: Artifacts should be versioned and stored in a repository for traceability and rollback capability.

4. Infrastructure as Code

Q7: What is Infrastructure as Code (IaC)?

Infrastructure as Code (IaC) is the practice of managing and provisioning infrastructure through code rather than manual processes.

Benefits:

  • Consistency: Same infrastructure every time
  • Version control: Infrastructure in version control
  • Reusability: Reuse infrastructure templates
  • Speed: Quick provisioning
  • Documentation: Code serves as documentation

IaC Tools:

  • Terraform: Multi-cloud IaC tool
  • CloudFormation: AWS native IaC
  • ARM Templates: Azure native IaC
  • Kubernetes YAML: Container orchestration
  • Ansible: Configuration management
Q8: What is the difference between imperative and declarative IaC?
  • Imperative: Define how to achieve desired state (step-by-step). Example: Ansible, Chef, Puppet.
  • Declarative: Define desired state, tool figures out how to achieve it. Example: Terraform, CloudFormation, ARM Templates.
// Declarative (Terraform)
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  tags = {
    Name = "WebServer"
  }
}

// Imperative (Ansible)
- name: Launch EC2 instance
  ec2:
    image: ami-0c55b159cbfafe1f0
    instance_type: t2.micro
    tags:
      Name: WebServer

11. Common Interview Questions

Q20: What is the difference between Git Merge and Git Rebase?
  • Git Merge: Creates a merge commit. Preserves history. Non-linear history. Safe for shared branches.
  • Git Rebase: Rewrites history. Linear history. Can cause conflicts. Not recommended for shared branches.
Tip: Use merge for shared branches (main, develop). Use rebase for local feature branches before merging.
Q21: What is a pull request?

A pull request (PR) is a mechanism to propose changes to a codebase. It:

  • Allows code review before merging
  • Runs CI checks automatically
  • Enables discussion about changes
  • Provides traceability
  • Can include approvals from team members
Q22: What is the difference between branch protection rules and branch policies?
  • Branch Protection Rules: GitHub/GitLab. Protect branches from force pushes, deletions, and require reviews, status checks, and approvals.
  • Branch Policies: Azure DevOps. Similar functionality with additional options like build validation and work item linking.
Q23: What is a pipeline as code?

Pipeline as Code is the practice of defining CI/CD pipelines in code files stored in version control alongside application code.

Benefits:

  • Version control for pipeline definitions
  • Code review for pipeline changes
  • Reusability and modularity
  • Documentation through code
  • Consistency across environments
// Examples
// GitHub Actions: .github/workflows/ci.yml
// GitLab CI: .gitlab-ci.yml
// Azure Pipelines: azure-pipelines.yml
// Jenkins: Jenkinsfile
// CircleCI: .circleci/config.yml
Q24: What is the difference between blue/green and canary deployment?
  • Blue/Green: Two identical environments. Switch traffic instantly. Full cutover. Good for low-risk deployments.
  • Canary: Gradual rollout to subset of users. Monitor metrics before full rollout. Good for high-risk changes.
Q25: What is feature flagging?

Feature flagging is a technique that enables teams to modify system behavior without changing code. It allows:

  • Gradual feature rollout
  • A/B testing
  • Instant rollback
  • Testing in production
  • Targeted user segments
// Feature flag example
if (featureFlags.isEnabled('new-feature')) {
  // New feature code
} else {
  // Old feature code
}
Q26: What is the difference between synchronous and asynchronous deployment?
  • Synchronous: Pipeline waits for deployment to complete before proceeding. Slower but sequential. Good for critical deployments.
  • Asynchronous: Pipeline continues without waiting for deployment. Faster but harder to track. Good for non-critical deployments.
Q27: What is a rollback strategy?

Rollback strategies ensure you can revert to a previous version if deployment fails:

  • Versioned artifacts: Keep previous versions available
  • Database migrations: Reversible migrations
  • Blue/Green: Switch back to previous environment
  • Canary: Stop canary and revert
  • Feature flags: Disable feature
  • Automated rollback: Rollback on health check failure
Q28: What is the difference between monorepo and polyrepo?
Aspect Monorepo Polyrepo
Structure Single repository for all projects Separate repository per project
CI/CD Complex (need to trigger only affected) Simple (one pipeline per repo)
Sharing Easy code sharing Requires packages
Build Time Can be slower Faster per project
Q29: What is the difference between Docker and Kubernetes?
  • Docker: Container runtime. Packages applications into containers. Runs on single host. Good for development and simple deployments.
  • Kubernetes: Container orchestration. Manages containers across multiple hosts. Provides scaling, load balancing, self-healing. Good for production deployments.
Q30: What is the difference between stateful and stateless applications in CI/CD?
  • Stateless: No local state. Easy to deploy and scale. Each instance is identical. Ideal for CI/CD.
  • Stateful: Maintains state locally. Harder to deploy and scale. Requires careful state management. Requires special deployment strategies.
Important: Design applications to be stateless for easier CI/CD. Store state in external services (databases, caches).

11. Advanced CI/CD Patterns (Expert Level)

Q31: What is GitOps?

GitOps is a methodology for operating and managing infrastructure and applications using Git as the single source of truth.

Key Principles:

  • Declarative: Infrastructure defined in Git
  • Versioned: All changes tracked in Git
  • Pulled: Agents pull changes from Git
  • Continuous reconciliation: Desired state vs actual state

Tools:

  • ArgoCD
  • Flux
  • Jenkins X
Q32: What is Trunk-Based Development?

Trunk-Based Development is a branching model where developers work on a single branch called "trunk".

Benefits:

  • No long-lived branches
  • Continuous integration
  • Faster feedback
  • Reduced merge conflicts

Practices:

  • Feature flags for incomplete features
  • Small, frequent commits
  • Automated testing on every commit
  • Continuous deployment to production
Q33: What is Progressive Delivery?

Progressive Delivery is the practice of releasing new features to a subset of users gradually.

Techniques:

  • Canary deployments: Roll out to small percentage of users
  • A/B testing: Compare different versions
  • Feature flags: Enable/disable features without deployment
  • Dark launching: Deploy without user visibility
Q34: What is the difference between CI/CD and DevOps?

Components CI, CD, testing, deployment CI/CD, monitoring, culture, collaboration

12. CI/CD Monitoring and Observability (Expert Level)

Q35: How do you monitor CI/CD pipelines?
  • Pipeline metrics: Build duration, success rate, failure rate
  • Deployment metrics: Deployment frequency, lead time, change failure rate
  • Alerting: Notify on pipeline failures

  • Logs: Collect and analyze pipeline logs

Q36: What are DORA metrics?

DORA (DevOps Research and Assessment) metrics measure software delivery performance:

  • Deployment Frequency: How often you deploy
  • Lead Time for Changes: Time from commit to deployment
  • Time to Restore Service: Time to recover from failure
  • Change Failure Rate: Percentage of deployments that fail
Q37: How do you implement observability in CI/CD?
  • Structured logging: Consistent log format with metadata
  • Distributed tracing: Track requests across services
  • Metrics: Collect performance and business metrics
  • Alerting: Proactive notification of issues
  • APM tools: Application Performance Monitoring
  • Log aggregation: Centralized log collection
Q38: What is the difference between logs, metrics, and traces?
Type Purpose Example

Metrics Numerical measurements over time Request rate, error rate
Traces Request path across services Request flow through microservices

Q53: How do you implement CI/CD security with supply chain protection?

Supply chain security ensures the integrity of artifacts throughout the CI/CD pipeline.

# GitHub Actions with Sigstore signing
name: Build and Sign

on:
  push:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v3

      run: docker build -t myapp:${{ github.sha }} .
    
    - name: Sign image with Cosign
      uses: sigstore/cosign-installer@v2
      with:

      run: |
        cosign sign --key env://COSIGN_PRIVATE_KEY \
          myapp:${{ github.sha }}
      env:
        COSIGN_PRIVATE_KEY: ${{ secrets.COSIGN_PRIVATE_KEY }}

# SBOM generation with Syft

Q54: How do you implement feature flags in CI/CD?

Feature flags enable controlled rollout of features without deployment.

# LaunchDarkly integration
using Launchdarkly.Client;

public class FeatureFlagService
{
    private readonly ILdClient _ldClient;
    
    public FeatureFlagService(ILdClient ldClient)
    {
        _ldClient = ldClient;
    }
    
    public async Task<bool> IsFeatureEnabledAsync(string featureKey, string userId)
    {
        var user = User.WithKey(userId);
        return await _ldClient.BoolVariationAsync(featureKey, user, false);
    }
}

# Feature flag in code

{
    return View("NewUI");
}
else
{
    return View("LegacyUI");
}
