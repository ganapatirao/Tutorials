1. SQL Database Fundamentals

Q1: What are the ACID properties in databases?

ACID stands for Atomicity, Consistency, Isolation, Durability - the key properties that guarantee database transactions are processed reliably:

  • Atomicity: All operations in a transaction succeed or fail together
  • Consistency: Database moves from one valid state to another valid state
  • Isolation: Concurrent transactions don't interfere with each other
  • Durability: Committed transactions persist even after system failure
Q2: What is the difference between DELETE and TRUNCATE?

WHERE clause Supported Not supported

Identity reset No Yes
Triggers Fires Doesn't fire
Rollback Can be rolled back Cannot be rolled back
Q3: What are the different types of joins in SQL?
Join Type Description
INNER JOIN Returns rows when there is a match in both tables
LEFT JOIN Returns all rows from left table, matching rows from right
RIGHT JOIN Returns all rows from right table, matching rows from left
FULL OUTER JOIN Returns all rows when there is a match in either table
CROSS JOIN Returns Cartesian product of tables 

// Inner Join
SELECT p.Name, c.CategoryName
FROM Products p
INNER JOIN Categories c ON p.CategoryId = c.Id;

// Left Join
SELECT p.Name, c.CategoryName
FROM Products p
LEFT JOIN Categories c ON p.CategoryId = c.Id;
Q4: What is the difference between WHERE and HAVING?
  • WHERE: Filters rows before aggregation. Cannot use aggregate functions.
  • HAVING: Filters groups after aggregation. Can use aggregate functions.

// WHERE - filters before grouping
SELECT CategoryId, COUNT(*) as Count
FROM Products
WHERE Price > 100
GROUP BY CategoryId;

// HAVING - filters after grouping
SELECT CategoryId, COUNT(*) as Count
FROM Products
GROUP BY CategoryId
HAVING COUNT(*) > 10;

2. MongoDB Fundamentals

Q5: What is MongoDB and what are its key features?

MongoDB is a document-oriented NoSQL database that stores data in flexible, JSON-like documents.

Key Features:

  • Document-oriented: Stores data in BSON (Binary JSON) format
  • Schema-less: Flexible schema design
  • Horizontal scaling: Sharding for large datasets
  • High availability: Replica sets for automatic failover
  • Rich queries: Powerful aggregation framework
  • Indexing: Secondary indexes for performance
Q6: What is the difference between SQL and MongoDB?

Scaling Vertical (mostly) Horizontal (sharding)

Transactions ACID compliant Multi-document ACID (since 4.0)
Query Language SQL MQL (MongoDB Query Language)

Q8: What is the difference between embedding and referencing in MongoDB?
  • Embedding: Store related data within the same document. Good for one-to-few relationships.
  • Referencing: Store references to other documents. Good for one-to-many or many-to-many relationships.

// Embedding - one-to-few
{
  "_id": ObjectId("..."),
  "name": "User",
  "addresses": [
    { "street": "123 Main St", "city": "NYC" },
    { "street": "456 Oak Ave", "city": "LA" }
  ]
}

// Referencing - one-to-many
{
  "_id": ObjectId("..."),
  "name": "Author",
  "bookIds": [ObjectId("..."), ObjectId("...")]
}

3. Kafka Fundamentals

Q9: What is Apache Kafka and what are its key components?

Apache Kafka is a distributed event streaming platform for high-throughput, fault-tolerant, and scalable real-time data streaming.

Key Components:

  • Topics: Categories or feeds of messages
  • Producers: Applications that publish messages to topics
  • Consumers: Applications that subscribe to topics and process messages
  • Brokers: Kafka servers that store and serve messages
  • Partitions: Divisions of topics for parallelism
  • Consumer Groups: Groups of consumers that share topic consumption
  • Zookeeper: Manages cluster metadata (deprecated in newer versions)
Q10: How do you produce and consume messages in Kafka using .NET? 

// Install Confluent.Kafka NuGet package
using Confluent.Kafka;

// Producer
var config = new ProducerConfig
{
    BootstrapServers = "localhost:9092",
    Acks = Acks.All
};

using var producer = new ProducerBuilder<Null, string>(config).Build();
await producer.ProduceAsync("my-topic", 
    new Message<Null, string> { Value = "Hello Kafka" });

// Consumer
var consumerConfig = new ConsumerConfig
{
    BootstrapServers = "localhost:9092",
    GroupId = "my-consumer-group",
    AutoOffsetReset = AutoOffsetReset.Earliest
};

using var consumer = new ConsumerBuilder<Ignore, string>(consumerConfig).Build();
consumer.Subscribe("my-topic");

while (true)
{
    var consumeResult = consumer.Consume();
    Console.WriteLine($"Message: {consumeResult.Message.Value}");
    consumer.Commit(consumeResult);
}
Q11: What is the difference between partitions and consumer groups?
  • Partitions: Divide a topic into multiple logs for parallelism. Each partition is ordered, but the topic as a whole is not.
  • Consumer Groups: Allow multiple consumers to share the load of consuming from a topic. Each partition is consumed by exactly one consumer in a group.
Important: The number of consumers in a group should not exceed the number of partitions. Extra consumers will remain idle.

4. Redis Fundamentals

Q12: What is Redis and what are its key features?

Redis (Remote Dictionary Server) is an in-memory data structure store used as a database, cache, and message broker.

Key Features:

  • In-memory storage: Extremely fast read/write operations
  • Data structures: Strings, Hashes, Lists, Sets, Sorted Sets
  • Persistence: Optional disk persistence (RDB, AOF)
  • Replication: Master-slave replication for high availability
  • Clustering: Horizontal scaling with Redis Cluster
  • Pub/Sub: Publish/subscribe messaging
  • Transactions: Multi-command transactions
Q13: How do you use Redis with .NET? 

// Install StackExchange.Redis NuGet package

db.SortedSetAdd("leaderboard", "user1", 100);
var rank = db.SortedSetRank("leaderboard", "user1");

// Expiration
db.KeyExpire("key", TimeSpan.FromMinutes(5));

Q21: How do you integrate Redis with ASP.NET Core? 

// Add Redis distributed cache
builder.Services.AddStackExchangeRedisCache(options =>
{
    options.Configuration = "localhost:6379";
    options.InstanceName = "MyApp_";
});

// Use IDistributedCache
public class ProductService
{
    private readonly IDistributedCache _cache;
    private readonly IProductRepository _repository;
    
    public ProductService(IDistributedCache cache, IProductRepository repository)
    {
        _cache = cache;
        _repository = repository;
    }
    
    public async Task<Product> GetProductAsync(int id)
    {
        var cacheKey = $"product_{id}";
        var cachedProduct = await _cache.GetStringAsync(cacheKey);
        
        if (cachedProduct != null)
        {
            return JsonSerializer.Deserialize<Product>(cachedProduct);
        }
        
        var product = await _repository.GetByIdAsync(id);
        await _cache.SetStringAsync(cacheKey, 
            JsonSerializer.Serialize(product),
            new DistributedCacheEntryOptions
            {
                AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(10)
            });
        
        return product;
    }
}

Q23: What is database indexing and when should you use it?

Database indexing creates data structures to improve query performance. Use indexes when:

  • Frequently queried columns
  • Columns used in WHERE, JOIN, ORDER BY clauses
  • Columns with high cardinality (many unique values)
  • Large tables where full table scans are slow

// Create index
CREATE INDEX idx_product_name ON Products(Name);

// Composite index
CREATE INDEX idx_product_category_price ON Products(CategoryId, Price);

// Unique index
CREATE UNIQUE INDEX idx_product_code ON Products(ProductCode);
Important: Indexes improve read performance but slow down write operations. Don't over-index.
Q24: What is database normalization?

Normalization is the process of organizing data to minimize redundancy and dependency. It involves dividing large tables into smaller, related tables and defining relationships between them.

Normal Forms:

  • 1NF: Atomic values, no repeating groups
  • 2NF: 1NF + no partial dependencies
  • 3NF: 2NF + no transitive dependencies
  • BCNF: Stronger version of 3NF

Q26: What is database sharding?

Sharding is a database scaling technique that splits large datasets across multiple database instances (shards) based on a shard key.

Benefits:

  • Horizontal scaling
  • Improved performance
  • Better resource utilization

Challenges:

  • Complex queries across shards
  • Data rebalancing
  • Transaction management
Q27: What is database replication?

Replication is the process of copying data from one database server to another to ensure data availability and redundancy.

Replication Types:

  • Master-Slave: One master, multiple read-only slaves
  • Master-Master: Multiple masters, all can read/write
  • Multi-Master: All nodes can read/write

Q30: What is the difference between synchronous and asynchronous database operations?
  • Synchronous: Blocks the thread until the operation completes. Can lead to thread pool exhaustion under load.
  • Asynchronous: Returns immediately, operation completes in background. Better scalability and resource utilization.

// Synchronous (avoid in web apps)

var products = await context.Products.ToListAsync();

9. Database Performance & Optimization (Expert Level)

Q31: How do you optimize SQL query performance?
  • Indexing: Create appropriate indexes on frequently queried columns
  • Query optimization: Use EXPLAIN/EXPLAIN ANALYZE to analyze query execution plans
  • SELECT only needed columns: Avoid SELECT *
  • Use WHERE instead of HAVING: Filter rows before aggregation
  • Avoid functions on indexed columns: Prevents index usage
  • Use proper JOIN types: INNER JOIN for required relationships, LEFT JOIN for optional
  • Limit result sets: Use LIMIT/OFFSET or pagination
  • Use stored procedures: For complex, frequently executed logic

-- Bad: Function on indexed column
SELECT * FROM Products WHERE YEAR(CreatedAt) = 2024;

-- Good: Range query on indexed column
SELECT * FROM Products WHERE CreatedAt >= '2024-01-01' AND CreatedAt < '2025-01-01';

-- Bad: SELECT *
SELECT * FROM Users WHERE Id = 1;

-- Good: Select specific columns
SELECT Id, Name, Email FROM Users WHERE Id = 1;
Q32: What are database indexing strategies?
  • B-Tree Index: Default for most databases. Good for equality and range queries.
  • Hash Index: Only for equality comparisons. Faster than B-Tree for exact matches.
  • GIN Index: For array values and full-text search (PostgreSQL).
  • GiST Index: For geometric data and full-text search.
  • Composite Index: Index on multiple columns. Order matters.
  • Partial Index: Index on subset of rows matching a condition.
  • Covering Index: Index includes all columns needed by query.
  • Clustered Index: Data stored in index order (one per table).
  • Non-Clustered Index: Separate index structure with pointers to data.

-- Create composite index
CREATE INDEX idx_product_category_price ON Products(CategoryId, Price);

-- Create partial index
CREATE INDEX idx_active_products ON Products(Name) WHERE IsActive = true;

-- Create covering index (INCLUDE clause in SQL Server)
CREATE INDEX idx_product_covering ON Products(CategoryId) 
INCLUDE (Name, Price, Stock);

Q34: What is database partitioning and when should you use it?

Partitioning divides large tables into smaller, more manageable pieces while maintaining a single logical table.

  • Range Partitioning: Based on value ranges (e.g., date ranges)
  • List Partitioning: Based on discrete value lists
  • Hash Partitioning: Based on hash function for even distribution
  • Composite Partitioning: Combination of above methods

-- Range partitioning by date (PostgreSQL)
CREATE TABLE orders (
    id SERIAL,
    order_date DATE,
    customer_id INT,
    amount DECIMAL
) PARTITION BY RANGE (order_date);

-- Create partitions
CREATE TABLE orders_2024 PARTITION OF orders
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

CREATE TABLE orders_2025 PARTITION OF orders
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');

-- Benefits:
-- - Improved query performance (partition pruning)
-- - Easier maintenance (drop old partitions)
-- - Parallel query execution
-- - Better index efficiency

10. Database Security & Compliance (Expert Level)

Q36: How do you implement database security best practices?
  • Principle of Least Privilege: Grant minimum required permissions
  • Encryption at Rest: Use TDE (Transparent Data Encryption) or file system encryption
  • Encryption in Transit: Use SSL/TLS for database connections
  • Row-Level Security: Implement RLS policies for data access control
  • Column-Level Encryption: Encrypt sensitive columns (SSN, credit cards)
  • Auditing: Enable audit logging for sensitive operations
  • Regular Backups: Automated, encrypted backups with retention policies
  • Network Security: Database in private subnet, firewall rules
  • Authentication: Strong passwords, MFA, integrated authentication
  • Regular Updates: Keep database software patched
Q37: What is SQL injection and how do you prevent it?

SQL Injection is a code injection technique where malicious SQL is inserted into query strings.


-- Vulnerable code (DON'T DO THIS)
string query = "SELECT * FROM Users WHERE Name = '" + userName + "'";
// If userName = "' OR '1'='1", returns all users

-- Prevention methods:

// 1. Parameterized queries (BEST)
string query = "SELECT * FROM Users WHERE Name = @Name";
SqlCommand command = new SqlCommand(query, connection);
command.Parameters.AddWithValue("@Name", userName);

// 2. ORM (Entity Framework)
var user = await context.Users.FirstOrDefaultAsync(u => u.Name == userName);

// 3. Stored procedures
CREATE PROCEDURE GetUserByName
    @Name NVARCHAR(100)
AS
BEGIN
    SELECT * FROM Users WHERE Name = @Name
END

// 4. Input validation
if (!Regex.IsMatch(userName, @"^[a-zA-Z0-9 ]*$"))
{
    throw new ArgumentException("Invalid input");
}
Q38: How do you implement data masking and anonymization? 

-- Dynamic Data Masking (SQL Server)
CREATE TABLE Customers (
    Id INT PRIMARY KEY,
    Name NVARCHAR(100),
    Email NVARCHAR(100) MASKED WITH (FUNCTION = 'email()'),
    CreditCard NVARCHAR(20) MASKED WITH (FUNCTION = 'partial(0,"XXXX-XXXX-XXXX-",4)'),

-- Grant unmask permission
GRANT UNMASK TO UserWithAccess;

-- PostgreSQL using pgcrypto
CREATE EXTENSION pgcrypto;

-- Encrypt data
INSERT INTO sensitive_data (encrypted_field)
VALUES (pgp_sym_encrypt('secret data', 'encryption_key'));

-- Decrypt data
SELECT pgp_sym_decrypt(encrypted_field::bytea, 'encryption_key') 
FROM sensitive_data;

-- Application-level masking
public static string MaskEmail(string email)
{
    if (string.IsNullOrEmpty(email)) return email;
    var parts = email.Split('@');
    return parts[0].Substring(0, 2) + "****@" + parts[1];
}
Q39: How do you implement GDPR compliance in databases?
  • Data Minimization: Store only necessary data
  • Right to be Forgotten: Implement data deletion procedures
  • Data Portability: Export data in standard formats
  • Consent Management: Track and manage user consent
  • Data Breach Notification: Implement monitoring and alerting
  • Pseudonymization: Replace identifying information with pseudonyms
  • Access Control: Strict role-based access control
  • Audit Logging: Log all data access and modifications

Right to be forgotten implementation
CREATE PROCEDURE DeleteUserData
    @UserId INT
AS
BEGIN
    BEGIN TRANSACTION;
    
    -- Delete from child tables first
    DELETE FROM UserOrders WHERE UserId = @UserId;
    DELETE FROM UserSessions WHERE UserId = @UserId;

        Address = NULL,
        DeletedAt = GETUTCDATE()
    WHERE Id = @UserId;
    
    COMMIT TRANSACTION;
END
Q40: How do you implement database auditing? 

SQL Server Audit
CREATE SERVER AUDIT MyServerAudit
TO FILE (FILEPATH = 'C:\Audit\');

ALTER SERVER AUDIT MyServerAudit WITH (STATE = ON);

CREATE DATABASE AUDIT SPECIFICATION MyDatabaseAudit
FOR SERVER AUDIT MyServerAudit
ADD (SELECT, INSERT, UPDATE, DELETE ON SCHEMA::dbo BY PUBLIC);

ALTER DATABASE AUDIT SPECIFICATION MyDatabaseAudit WITH (STATE = ON);

-- PostgreSQL using triggers
CREATE TABLE audit_log (
    id SERIAL PRIMARY KEY,
    table_name VARCHAR(100),
    operation VARCHAR(10),
    old_data JSONB,
    new_data JSONB,
    changed_by VARCHAR(100),
    changed_at TIMESTAMP DEFAULT NOW()
);

CREATE OR REPLACE FUNCTION audit_trigger()
RETURNS TRIGGER AS $$
BEGIN
    IF TG_OP = 'DELETE' THEN
        INSERT INTO audit_log (table_name, operation, old_data, changed_by)
        VALUES (TG_TABLE_NAME, TG_OP, row_to_json(OLD), current_user);
        RETURN OLD;
    ELSIF TG_OP = 'UPDATE' THEN
        INSERT INTO audit_log (table_name, operation, old_data, new_data, changed_by)
        VALUES (TG_TABLE_NAME, TG_OP, row_to_json(OLD), row_to_json(NEW), current_user);
        RETURN NEW;
    ELSIF TG_OP = 'INSERT' THEN
        INSERT INTO audit_log (table_name, operation, new_data, changed_by)
        VALUES (TG_TABLE_NAME, TG_OP, row_to_json(NEW), current_user);
        RETURN NEW;
    END IF;
END;
$$ LANGUAGE plpgsql;

Q42: How do you implement database replication?
  • Master-Slave Replication: One master writes, multiple slaves read. Good for read-heavy workloads.
  • Master-Master Replication: Multiple nodes can write. Needs conflict resolution.
  • Multi-Master Replication: All nodes can read and write. Complex but highly available.
  • Synchronous Replication: Data copied before commit. Strong consistency, slower.
  • Asynchronous Replication: Data copied after commit. Faster, potential data loss.
  • Semi-Synchronous Replication: At least one slave confirms before commit. Balance of consistency and performance.

PostgreSQL streaming replication
-- Primary server (postgresql.conf)
wal_level = replica
max_wal_senders = 5
wal_keep_size = 1GB

-- Create replication user
CREATE USER replicator WITH REPLICATION ENCRYPTED PASSWORD 'password';

-- pg_hba.conf on primary
host replication replicator 192.168.1.0/24 md5

-- Standby server (recovery.conf)
standby_mode = 'on'
primary_conninfo = 'host=primary_host port=5432 user=replicator password=password'
restore_command = 'cp /var/lib/postgresql/archive/%f %p'
Q43: What is database sharding and how do you implement it?

Sharding partitions data across multiple database instances (horizontal partitioning at the database level).

  • Hash-based Sharding: Consistent hash of shard key determines shard
  • Range-based Sharding: Data ranges assigned to specific shards
  • Directory-based Sharding: Lookup service maps keys to shards
  • Geographic Sharding: Data located based on user location

// Consistent hashing for sharding
public class ConsistentHash
{
    private readonly SortedDictionary<int, string> _ring = new SortedDictionary<int, string>();
    private readonly int _virtualNodes = 100;

    public void AddNode(string node)
    {
        for (int i = 0; i < _virtualNodes; i++)
        {
            int hash = GetHashCode($"{node}:{i}");
            _ring[hash] = node;
        }
    }

    public string GetNode(string key)
    {
        int hash = GetHashCode(key);
        var node = _ring.FirstOrDefault(x => x.Key >= hash);
        return node.Value ?? _ring.First().Value;
    }
}

Q48: How do you implement distributed transactions with Kafka? 

Using Kafka Transactions (exactly-once semantics)
public class OrderProcessor
{
    private readonly IProducer<string, string> _producer;
    private readonly AppDbContext _dbContext;

    public async Task ProcessOrderAsync(Order order)
    {
        // Begin transaction
        using (var transaction = await _dbContext.Database.BeginTransactionAsync())
        {
            try
            {
                // Update database
                _dbContext.Orders.Add(order);
                await _dbContext.SaveChangesAsync();

                // Produce to Kafka within transaction
                _producer.InitTransactions(TimeSpan.FromSeconds(30));
                _producer.BeginTransaction();

                var message = new Message<string, string>
                {
                    Key = order.Id.ToString(),
                    Value = JsonSerializer.Serialize(order)
                };

                _producer.Produce("orders-topic", message);

                // Commit both transactions
                _producer.CommitTransaction();
                await transaction.CommitAsync();
            }
            catch (Exception)
            {
                _producer.AbortTransaction();
                await transaction.RollbackAsync();
                throw;
            }
        }
    }
}

// Using Saga pattern for distributed transactions
public class OrderSaga
{
    private readonly List<ISagaStep> _steps;

    public async Task ExecuteAsync(Order order)
    {
        var executedSteps = new List<ISagaStep>();

        foreach (var step in _steps)
        {
            try
            {
                await step.ExecuteAsync(order);
                executedSteps.Add(step);
            }
            catch (Exception)
            {
                // Compensating actions
                foreach (var executedStep in executedSteps.AsEnumerable().Reverse())
                {
                    await executedStep.CompensateAsync(order);
                }
                throw;
            }
        }
    }
}

public class ChangeStreamProcessor
{
    private readonly IMongoCollection<Order> _ordersCollection;

    public async Task ProcessChangesAsync()
    {
        var pipeline = new EmptyPipelineDefinition<ChangeStreamDocument<Order>>()
            .Match(change => change.OperationType == ChangeOperationType.Insert ||
                           change.OperationType == ChangeOperationType.Update ||
                           change.OperationType == ChangeOperationType.Delete);

        var options = new ChangeStreamOptions
        {
            FullDocument = ChangeStreamFullDocumentOption.UpdateLookup
        };

        using var cursor = await _ordersCollection.WatchAsync(pipeline, options);
        
        await cursor.ForEachAsync(async change =>
        {
            switch (change.OperationType)
            {
                case ChangeOperationType.Insert:
                    await HandleInsertAsync(change.FullDocument);
                    break;
                case ChangeOperationType.Update:
                    await HandleUpdateAsync(change.FullDocument);
                    break;
                case ChangeOperationType.Delete:
                    await HandleDeleteAsync(change.DocumentKey["_id"].AsGuid);
                    break;
            }
        });
    }

    private async Task HandleInsertAsync(Order order)
    {
        // Update search index
        await _searchIndex.IndexDocumentAsync(order);
        
        // Invalidate cache
        await _cache.RemoveAsync($"order_{order.Id}");
        
        // Send notification
        await _notificationService.SendAsync($"New order {order.Id}");
    }
}

Q51: What is the CAP theorem and how does it apply to databases?

CAP Theorem states that a distributed system can only provide two of the following three guarantees:

  • Consistency: All nodes see the same data at the same time
  • Availability: Every request receives a response (success or failure)
  • Partition Tolerance: System continues operating despite network partitions
System Type Properties Examples
CA Consistency + Availability (no partition tolerance) Single-node RDBMS
CP Consistency + Partition Tolerance MongoDB, HBase, Redis
AP Availability + Partition Tolerance Cassandra, DynamoDB, CouchDB

Aspect OLTP (Online Transaction Processing) OLAP (Online Analytical Processing)
Purpose Day-to-day operations Analysis and reporting
Queries Simple, fast, frequent Complex, long-running, batch
Data Volume Current data Historical data
Users Many concurrent users Few analysts
Normalization Highly normalized Denormalized (star schema)
Examples Order processing, banking Data warehousing, BI
Q53: What is eventual consistency?

Eventual Consistency is a consistency model where, given no new updates, all accesses to a data item will eventually return the last updated value.

  • Strong Consistency: Reads always return the latest write (CP systems)
  • Eventual Consistency: Reads may return stale data, but converge over time (AP systems)
  • Causal Consistency: Causally related operations are seen by all nodes in order
  • Read Your Writes: Client always sees its own writes

Q57: How do you handle database schema drift?

Schema Drift occurs when the actual database schema differs from the expected schema due to manual changes or failed migrations.


Detect schema drift
public class SchemaDriftDetector
{
    public async Task<List<string>> DetectDriftAsync()
    {
        var drift = new List<string>();
        
        using var context = new AppDbContext();
        var model = context.Model;
        
        // Get actual database schema
        var connection = context.Database.GetDbConnection();
        await connection.OpenAsync();
        
        using var command = connection.CreateCommand();
        command.CommandText = @"
            SELECT TABLE_NAME, COLUMN_NAME, DATA_TYPE 
            FROM INFORMATION_SCHEMA.COLUMNS
            WHERE TABLE_SCHEMA = 'dbo'";
        
        using var reader = await command.ExecuteReaderAsync();
        var actualColumns = new List<(string Table, string Column, string Type)>();
        
        while (await reader.ReadAsync())
        {
            actualColumns.Add((
                reader.GetString(0),
                reader.GetString(1),
                reader.GetString(2)
            ));
        }

        // Compare with expected schema
        foreach (var entity in model.GetEntityTypes())
        {
            var tableName = entity.GetTableName();
            foreach (var property in entity.GetProperties())
            {
                var columnName = property.GetColumnName();
                var expectedType = GetColumnType(property);
                
                var actual = actualColumns.FirstOrDefault(c => 
                    c.Table == tableName && c.Column == columnName);
                
                if (actual == default)
                {
                    drift.Add($"Missing column: {tableName}.{columnName}");
                }
                else if (actual.Type != expectedType)
                {
                    drift.Add($"Type mismatch: {tableName}.{columnName} (expected {expectedType}, actual {actual.Type})");
                }
            }
        }

        return drift;
    }
}

public class DatabaseResiliencePolicy
{
    public static IAsyncPolicy<HttpResponseMessage> CreateRetryPolicy()
    {
        return Policy
            .Handle<SqlException>(ex => IsTransientError(ex))
            .Or<TimeoutException>()
            .WaitAndRetryAsync(
                retryCount: 3,
                sleepDurationProvider: retryAttempt => TimeSpan.FromSeconds(Math.Pow(2, retryAttempt)),
                onRetry: (exception, timeSpan, retryCount, context) =>
                {
                    Console.WriteLine($"Retry {retryCount} after {timeSpan.TotalSeconds}s due to: {exception.Message}");
                });
    }

    private static bool IsTransientError(SqlException ex)
    {
        // Error codes for transient errors
        int[] transientErrorNumbers = { 4060, 40197, 40501, 40613, 49918, 49919, 49920, 11001 };
        return transientErrorNumbers.Contains(ex.Number);
    }
}
