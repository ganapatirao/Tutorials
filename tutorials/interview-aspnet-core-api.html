1. Web API Fundamentals

Q1: What is ASP.NET Core Web API?

ASP.NET Core Web API is a framework for building HTTP services that can be consumed by various clients including browsers, mobile devices, and other applications. It's built on top of ASP.NET Core and follows RESTful architecture principles.

Key Features:

  • RESTful Support: Built-in support for REST architectural patterns
  • Content Negotiation: Automatic response format selection (JSON, XML, etc.)
  • Model Binding: Automatic mapping of HTTP requests to action parameters
  • Validation: Built-in model validation with data annotations
  • Dependency Injection: Full DI support for service management
Controllers/ProductController.cs
using Microsoft.AspNetCore.Mvc; namespace MyApi.Controllers; [ApiController] [Route("api/[controller]")] public class ProductController : ControllerBase { [HttpGet] public IEnumerable<string> Get() { return new string[] { "Product1", "Product2" }; } }
Q2: What are the main HTTP methods used in REST APIs?

REST APIs use standard HTTP methods, each with specific semantics and idempotency characteristics:

HTTP Methods:

  • GET: Retrieve resources. Idempotent and safe.
  • POST: Create new resources. Not idempotent.
  • PUT: Update/replace entire resources. Idempotent.
  • PATCH: Partial update of resources. Not idempotent.
  • DELETE: Remove resources. Idempotent.
Controllers/UserController.cs
[HttpGet("{id}")] // GET api/user/5 public User Get(int id) { } [HttpPost] // POST api/user public IActionResult Create([FromBody] User user) { } [HttpPut("{id}")] // PUT api/user/5 public IActionResult Update(int id, [FromBody] User user) { } [HttpPatch("{id}")] // PATCH api/user/5 public IActionResult Patch(int id, [FromBody] JsonPatchDocument<User> patch) { } [HttpDelete("{id}")] // DELETE api/user/5 public IActionResult Delete(int id) { }
Q3: What is the difference between ControllerBase and Controller?

ControllerBase is the base class for API controllers without view support. Controller inherits from ControllerBase and adds view support for MVC applications.

Key Differences:

  • ControllerBase: Used for Web APIs. No view-related methods. Lightweight.
  • Controller: Used for MVC with views. Includes View(), PartialView(), etc.
  • Recommendation: Use ControllerBase for APIs to avoid unnecessary dependencies.
Controllers/ApiController.cs
// Use ControllerBase for Web API [ApiController] public class MyApiController : ControllerBase { // Has Ok(), NotFound(), BadRequest(), etc. // No view support } // Use Controller for MVC with views public class MyMvcController : Controller { // Has View(), PartialView() plus all ControllerBase methods }
Q4: What does the [ApiController] attribute do?

The [ApiController] attribute enables automatic API-specific behaviors that simplify Web API development.

Automatic Behaviors:

  • Automatic 400 Response: Returns HTTP 400 on validation errors
  • Model Binding Inference: Automatically binds from request body for complex types
  • Problem Details: Standardized error responses (RFC 7807)
  • Route Inference: Infers routes from action names
Controllers/OrderController.cs
[ApiController] [Route("api/[controller]")] public class OrderController : ControllerBase { [HttpPost] public IActionResult Create(OrderDto order) { // Automatic 400 response if validation fails // No need for if (!ModelState.IsValid) check return Ok(order); } }
Q5: What is attribute routing in Web API?

Attribute routing uses attributes to define routes directly on controllers and actions, providing more control over URL patterns. It's the default routing method in ASP.NET Core Web API.

Benefits:

  • Flexibility: Define custom URL patterns
  • Clarity: Routes are visible on the controller/action
  • Versioning: Easy to implement API versioning
  • Constraints: Apply route constraints at attribute level
Controllers/CustomerController.cs
[Route("api/customers")] public class CustomerController : ControllerBase { [HttpGet] // GET api/customers [Route("active")] // GET api/customers/active public IActionResult GetActive() { } [HttpGet("{id:int}")] // GET api/customers/5 public IActionResult GetById(int id) { } [HttpGet("{id}/orders")] // GET api/customers/5/orders public IActionResult GetOrders(int id) { } }

2. Model Binding and Validation

Q6: What are the different model binding sources?

Model binding sources determine where data comes from in the HTTP request:

Binding Sources:

  • [FromBody]: Request body (JSON, XML)
  • [FromQuery]: Query string parameters
  • [FromRoute]: URL route data
  • [FromForm]: Form data
  • [FromHeader]: HTTP headers
  • [FromServices]: Dependency injection
Controllers/SearchController.cs
[HttpPost("search")] public IActionResult Search( [FromBody] SearchCriteria criteria, // Request body JSON [FromQuery] int page = 1, // ?page=1 [FromRoute] string category, // Route parameter [FromHeader] string authorization, // HTTP header [FromServices] ILogger<SearchController> logger) // DI { return Ok(); }
Q7: How do you validate models in Web API?

Use data annotations attributes on model properties for validation. With [ApiController], validation errors automatically return HTTP 400 Bad Request.

Validation Attributes:

  • [Required]: Property must have a value
  • [StringLength]: String length constraints
  • [Range]: Numeric range validation
  • [EmailAddress]: Email format validation
  • [RegularExpression]: Pattern matching
Models/ProductDto.cs
public class ProductDto { public int Id { get; set; } [Required] [StringLength(100)] public string Name { get; set; } [Range(0.01, 10000)] public decimal Price { get; set; } [EmailAddress] public string Email { get; set; } }

3. Dependency Injection

Q8: What is Dependency Injection in ASP.NET Core?

Dependency Injection (DI) is a design pattern where dependencies are injected into classes rather than created internally. ASP.NET Core has built-in DI support with three service lifetimes.

Service Lifetimes:

  • Transient: New instance each time it's requested
  • Scoped: One instance per HTTP request
  • Singleton: One instance for the entire application lifetime
Program.cs
var builder = WebApplication.CreateBuilder(args); // Register services builder.Services.AddTransient<IMyTransientService, MyTransientService>(); builder.Services.AddScoped<IMyScopedService, MyScopedService>(); builder.Services.AddSingleton<IMySingletonService, MySingletonService>(); var app = builder.Build();
Q9: What is the difference between AddTransient, AddScoped, and AddSingleton?

The three service lifetimes determine how instances are created and managed:

Lifetime Comparison:

  • AddTransient: Creates a new instance every time it's requested. Use for lightweight, stateless services.
  • AddScoped: Creates one instance per HTTP request. Use for services that need to maintain state within a request.
  • AddSingleton: Creates one instance for the entire application lifetime. Use for stateless services or caches.
Program.cs
// Transient: New instance every time builder.Services.AddTransient<ILogger, ConsoleLogger>(); // Scoped: One instance per HTTP request builder.Services.AddScoped<IRepository, Repository>(); // Singleton: One instance for app lifetime builder.Services.AddSingleton<ICacheService, MemoryCacheService>();

4. Middleware

Q10: What is Middleware in ASP.NET Core?

Middleware is software that handles requests and responses in a pipeline. Each middleware component can process requests, pass them to the next component, and modify responses.

Middleware Pipeline:

  • Sequential Processing: Middleware executes in the order registered
  • Request/Response: Can modify both incoming requests and outgoing responses
  • Short-circuiting: Can terminate the pipeline early
  • Order Matters: Registration order is critical
Program.cs
var app = builder.Build(); // Middleware pipeline order app.UseExceptionHandler("/error"); app.UseHttpsRedirection(); app.UseStaticFiles(); app.UseRouting(); app.UseAuthentication(); app.UseAuthorization(); app.MapControllers(); app.Run();
Q11: How do you create custom middleware?

Custom middleware can be created as a class with an Invoke or InvokeAsync method that accepts HttpContext and a RequestDelegate.

Steps to Create Custom Middleware:

  • Create a class with a constructor accepting RequestDelegate
  • Implement InvokeAsync method with HttpContext parameter
  • Call _next(context) to pass to the next middleware
  • Register with app.UseMiddleware<T>() in Program.cs
Middleware/LoggingMiddleware.cs
public class LoggingMiddleware { private readonly RequestDelegate _next; public LoggingMiddleware(RequestDelegate next) { _next = next; } public async Task InvokeAsync(HttpContext context) { Console.WriteLine($"Request: {context.Request.Path}"); await _next(context); Console.WriteLine($"Response: {context.Response.StatusCode}"); } } // Registration in Program.cs app.UseMiddleware<LoggingMiddleware>();

5. Authentication and Authorization

Q12: How is authentication implemented in ASP.NET Core?

Authentication is implemented using ASP.NET Core Identity, JWT Bearer tokens, or other authentication schemes. Configure authentication in Program.cs with AddAuthentication and UseAuthentication middleware.

Authentication Options:

  • JWT Bearer: Token-based authentication for APIs
  • Cookie: Session-based authentication for web apps
  • OpenID Connect: OAuth 2.0 / OpenID Connect providers
  • Custom: Implement custom authentication handlers
Program.cs
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme) .AddJwtBearer(options => { options.TokenValidationParameters = new TokenValidationParameters { ValidateIssuer = true, ValidateAudience = true, ValidateLifetime = true, ValidIssuer = "https://myapi.com", ValidAudience = "https://myapp.com", IssuerSigningKey = new SymmetricSecurityKey( Encoding.UTF8.GetBytes(builder.Configuration["Jwt:Key"])) }; }); app.UseAuthentication();
Q13: What is the difference between Authentication and Authorization?

Authentication verifies who the user is (identity). Authorization determines what the user can do (permissions). Authentication must happen before authorization.

Key Differences:

  • Authentication: "Who are you?" - Identity verification
  • Authorization: "What can you do?" - Permission control
  • Order: Authentication first, then authorization
  • Use Case: Use authentication to identify users, authorization to control access
Program.cs
// Authentication: Who are you? app.UseAuthentication(); // Authorization: What can you do? app.UseAuthorization();

6. Entity Framework Core

Q14: What is Entity Framework Core?

Entity Framework Core (EF Core) is an ORM (Object-Relational Mapper) that enables .NET developers to work with databases using .NET objects. It eliminates most data access code and supports LINQ queries, change tracking, and migrations.

Key Features:

  • ORM: Maps database tables to .NET classes
  • LINQ: Query databases using LINQ syntax
  • Change Tracking: Automatically tracks entity changes
  • Migrations: Database schema versioning
  • Multiple Providers: SQL Server, PostgreSQL, MySQL, SQLite, etc.

The Repository pattern mediates between domain and data mapping layers, providing collection-like access to domain objects. It abstracts data access logic and makes code more testable and maintainable.

Benefits:

  • Abstraction: Hides data access implementation details
  • Testability: Easy to mock for unit testing
  • Separation of Concerns: Separates business logic from data access
  • Centralized Logic: Data access logic in one place

7. API Versioning and Documentation

Q16: How do you implement API versioning?

API versioning allows multiple versions of an API to coexist, enabling backward compatibility. Common approaches include URL path versioning, query string versioning, header versioning, or media type versioning.

Versioning Strategies:

  • URL Path: /api/v1/products, /api/v2/products
  • Query String: /api/products?version=1
  • Header: api-version: 1.0
  • Media Type: application/vnd.myapi.v1+json
Program.cs
builder.Services.AddApiVersioning(options => { options.DefaultApiVersion = new ApiVersion(1, 0); options.AssumeDefaultVersionWhenUnspecified = true; options.ReportApiVersions = true; }); builder.Services.AddVersionedApiExplorer(options => { options.GroupNameFormat = "'v'VVV"; options.SubstituteApiVersionInUrl = true; }); [ApiVersion("1.0")] [Route("api/v{version:apiVersion}/[controller]")] public class ProductsController : ControllerBase { }
Q17: What is Swagger/OpenAPI?

OpenAPI (formerly Swagger) is a specification for describing REST APIs. It provides interactive documentation, client SDK generation, and API testing. ASP.NET Core includes Swashbuckle for Swagger integration.

Swagger Benefits:

  • Interactive Documentation: UI for exploring API endpoints
  • Client Generation: Auto-generate client SDKs
  • Testing: Test API directly from the UI
  • Standardization: Industry-standard API description format
Program.cs
builder.Services.AddEndpointsApiExplorer(); builder.Services.AddSwaggerGen(c => { c.SwaggerDoc("v1", new OpenApiInfo { Title = "My API", Version = "v1", Description = "API Documentation" }); }); app.UseSwagger(); app.UseSwaggerUI(c => { c.SwaggerEndpoint("/swagger/v1/swagger.json", "My API V1"); });

8. CORS and Security

Q18: What is CORS and why is it needed?

CORS (Cross-Origin Resource Sharing) is a security feature that restricts web pages from making requests to a different domain than the one that served the page. It's needed when your API is called from a frontend on a different origin.

CORS Headers:

  • Access-Control-Allow-Origin: Specifies allowed origins
  • Access-Control-Allow-Methods: Specifies allowed HTTP methods
  • Access-Control-Allow-Headers: Specifies allowed headers
  • Access-Control-Allow-Credentials: Allows cookies
Program.cs
builder.Services.AddCors(options => { options.AddPolicy("AllowSpecificOrigin", builder => builder .WithOrigins("https://myapp.com") .AllowAnyMethod() .AllowAnyHeader() .AllowCredentials()); }); app.UseCors("AllowSpecificOrigin");
Q19: How do you handle exceptions in Web API?

Use exception handling middleware or exception filters to handle errors globally and return appropriate HTTP responses.

Exception Handling Approaches:

  • UseExceptionHandler: Global exception handling middleware
  • Exception Filters: Controller-level exception handling
  • Problem Details: RFC 7807 standardized error format
  • Custom Error Responses: Return specific error DTOs
Program.cs
app.UseExceptionHandler("/error"); app.Map("/error", (app) => { app.Run(async context => { context.Response.StatusCode = 500; await context.Response.WriteAsJsonAsync(new { error = "An error occurred processing your request." }); }); });

9. Caching and Performance

Q20: What is response caching in Web API?

Response caching stores HTTP responses to reduce server load and improve performance. Use the [ResponseCache] attribute to control caching behavior.

Caching Options:

  • Duration: How long to cache the response
  • Location: Where to cache (client, CDN, server)
  • NoStore: Disable caching
  • VaryByHeader: Cache based on headers
Controllers/ProductController.cs
[HttpGet] [ResponseCache(Duration = 60, Location = ResponseCacheLocation.Any)] public IActionResult Get() { return Ok(_repository.GetAll()); }
Q21: What is in-memory caching vs distributed caching?

In-memory caching stores data in the application's memory, while distributed caching stores data in a shared cache like Redis, accessible across multiple server instances.

Comparison:

  • In-Memory: Fast, single-server, lost on restart
  • Distributed: Slower, multi-server, persists across restarts
  • Use In-Memory: Single instance, simple apps
  • Use Distributed: Multi-instance, production apps
Program.cs
// In-Memory Cache builder.Services.AddMemoryCache(); // Distributed Cache (Redis) builder.Services.AddStackExchangeRedisCache(options => { options.Configuration = "localhost:6379"; });

10. File Uploads and Streaming

Q22: How do you handle file uploads in Web API?

Use IFormFile to handle file uploads. Configure maximum file size limits and validate file types for security.

File Upload Best Practices:

  • Size Limits: Configure max file size in Program.cs
  • Validation: Check file type and content
  • Storage: Save to disk, cloud storage, or database
  • Security: Validate and sanitize file names
Controllers/FileController.cs
[HttpPost("upload")] [RequestSizeLimit(10485760)] public async Task<IActionResult> Upload(IFormFile file) { if (file == null || file.Length == 0) return BadRequest(); var path = Path.Combine(Directory.GetCurrentDirectory(), "Uploads", file.FileName); using (var stream = new FileStream(path, FileMode.Create)) { await file.CopyToAsync(stream); } return Ok(new { path }); }

14. Rate Limiting

Q26: How do you implement rate limiting in Web API?

Use rate limiting middleware or third-party libraries like AspNetCoreRateLimit to control the number of requests a client can make within a time period.

Rate Limiting Strategies:

  • IP-based: Limit by client IP address
  • API Key: Limit by API key
  • Endpoint-specific: Different limits per endpoint
  • Sliding Window: Time-based rolling limits
Program.cs
builder.Services.AddRateLimiter(options => { options.AddPolicy("FixedWindow", context => RateLimitPartition.GetSlidingWindowLimiter( context.Connection.RemoteIpAddress?.ToString() ?? "anonymous", _ => new SlidingWindowRateLimiterOptions { PermitLimit = 100, Window = TimeSpan.FromMinutes(1), SegmentsPerWindow = 2 })); }); app.UseRateLimiter();

16. Minimal APIs

Q28: What are Minimal APIs?

Minimal APIs are a simplified way to build HTTP APIs with minimal code. They eliminate the need for controllers and use lambda-based endpoint definitions directly in Program.cs.

Minimal API Benefits:

  • Simplicity: Less boilerplate code
  • Performance: Lower overhead
  • Modern: Uses modern C# features
  • Flexibility: Mix with controller-based APIs
Program.cs
var builder = WebApplication.CreateBuilder(args); var app = builder.Build(); app.MapGet("/products", () => new[] { "Product1", "Product2" }); app.MapPost("/products", (Product product) => Results.Created("/products/1", product)); app.MapPut("/products/{id}", (int id, Product product) => Results.NoContent()); app.MapDelete("/products/{id}", (int id) => Results.NoContent()); app.Run();

gRPC is a high-performance RPC framework using Protocol Buffers. It's ideal for microservices communication, offering better performance than REST for internal service-to-service calls.

gRPC Benefits:

19. Entity Framework Core Advanced

Q31: What are EF Core migrations?

Migrations allow you to evolve your database schema over time. Each migration represents a change to the database schema, applied incrementally.

Migration Commands:

  • Add-Migration: Create a new migration
  • Update-Database: Apply pending migrations
  • Remove-Migration: Remove the last migration
  • Script-Migration: Generate SQL script
Package Manager Console
Add-Migration InitialCreate Update-Database Script-Migration
Q32: What is the difference between eager, lazy, and explicit loading?

These are different strategies for loading related data in EF Core.

Loading Strategies:

  • Eager Loading: Load related data with Include() in single query
  • Lazy Loading: Load related data automatically when accessed
  • Explicit Loading: Load related data manually with Load()
Controllers/ProductController.cs
// Eager Loading var products = _context.Products.Include(p => p.Category).ToList(); // Lazy Loading (requires virtual navigation properties) var product = _context.Products.First(); var category = product.Category; // Loaded automatically // Explicit Loading var product = _context.Products.First(); _context.Entry(product).Reference(p => p.Category).Load();
Q33: What is the difference between Add, Attach, and Update?

These methods determine how EF Core tracks entity state when adding or modifying entities.

Entity States:

  • Add: Marks entity as Added (insert on SaveChanges)
  • Attach: Marks entity as Unchanged (no changes)
  • Update: Marks entity as Modified (update on SaveChanges)
Data/ProductRepository.cs
// Add - Insert new entity _context.Products.Add(product); // Attach - Entity exists, no changes _context.Products.Attach(product); // Update - Entity exists, has changes _context.Products.Update(product);
Q34: What is the difference between HasData and Seed Data?

HasData is used in OnModelCreating to seed data that's part of the model, while custom seed data can be added in OnModelCreating or using migration SQL.

Seeding Approaches:

  • HasData: Seed data in model configuration
  • Migration SQL: Custom SQL in migrations
  • OnModelCreating: Programmatic seeding
Data/AppDbContext.cs
protected override void OnModelCreating(ModelBuilder modelBuilder) { modelBuilder.Entity<Category>().HasData( new Category { Id = 1, Name = "Electronics" }, new Category { Id = 2, Name = "Books" } ); }

HTTPS Benefits:

Secret Management:

  • Never commit secrets to source control

builder.Configuration.AddUserSecrets<Program>(); builder.Configuration.AddAzureKeyVault(vaultUri, clientId, clientSecret);

ASP.NET Core supports multiple authentication schemes including JWT Bearer, Cookie, OpenID Connect, and custom schemes.

Authentication Schemes:

  • JWT Bearer: Token-based for APIs

  • OpenID Connect: OAuth 2.0 / OIDC providers
  • Custom: Implement IAuthenticationHandler

21. Performance Optimization

Q38: What is async/await in Web API?

Async/await allows non-blocking I/O operations, improving scalability by freeing threads while waiting for I/O operations to complete.

Async Benefits:

  • Scalability: Handle more concurrent requests
  • Responsiveness: Better user experience
  • Resource Efficiency: Fewer blocked threads
  • Best Practice: Use async for I/O operations
Controllers/ProductController.cs
[HttpGet] public async Task<IActionResult> Get() { var products = await _repository.GetAllAsync(); return Ok(products); }
Q39: What is the difference between synchronous and asynchronous operations?

Synchronous operations block the thread until complete, while asynchronous operations return immediately and continue when the operation completes.

Comparison:

  • Sync: Simpler code, blocks thread, less scalable
  • Async: More complex code, non-blocking, more scalable
  • Use Sync: CPU-bound operations
  • Use Async: I/O-bound operations (database, HTTP)
Q40: What is response compression?

Response compression reduces the size of HTTP responses using Gzip or Brotli compression, improving bandwidth usage and load times.

Program.cs
builder.Services.AddResponseCompression(options => { options.EnableForHttps = true; options.Providers.Add<BrotliCompressionProvider>(); options.Providers.Add<GzipCompressionProvider>(); }); app.UseResponseCompression();

22. Deployment

Q41: What is the difference between development and production environments?

Development environment has detailed errors, debug symbols, and hot reload. Production environment is optimized, secure, and has minimal error details.

Environment Differences:

  • Development: Detailed errors, debug mode, hot reload
  • Production: Optimized, secure, generic errors
  • Staging: Production-like for testing
  • Use ASPNETCORE_ENVIRONMENT variable

Q43: What is the difference between IIS and Kestrel?

Kestrel is the built-in cross-platform web server. IIS is a Windows web server that can act as a reverse proxy for Kestrel.

Server Roles:

23. API Design Patterns

Q44: What is the difference between REST and GraphQL?

REST uses multiple endpoints for different resources. GraphQL uses a single endpoint with a query language to fetch exactly the data needed.

Comparison:

  • REST: Multiple endpoints, over/under-fetching, caching built-in
  • GraphQL: Single endpoint, exact data, no built-in caching
  • Use REST: Standard APIs, caching important
  • Use GraphQL: Complex data requirements, mobile apps
Q45: What is the difference between GET and POST for data retrieval?

GET is idempotent and safe, used for retrieving data. POST is not idempotent, typically used for creating data but can retrieve with complex parameters.

When to Use:

  • GET: Simple queries, cacheable, bookmarkable
  • POST: Complex queries, large payloads, non-cacheable
  • GET Limit: URL length limits
  • POST: Body has no size limit
Q46: What is HATEOAS in REST APIs?

HATEOAS (Hypermedia as the Engine of Application State) is a REST constraint where responses contain links to related resources, allowing clients to discover actions dynamically.

Unit tests test individual components in isolation. Integration tests test how components work together, including database and external services.

Middleware runs globally for all requests in the pipeline. Filters run at specific stages (before/after action) and can be scoped to controllers or actions.

MapControllers routes to controller-based actions. MapGet/MapPost are for minimal APIs with inline lambda handlers.

Routing Approaches:

  • MapControllers: Traditional controller-based APIs
  • MapGet/MapPost: Minimal APIs with inline handlers

34. Advanced Security

Q72: What is the difference between authentication and authorization policies?

Authentication verifies identity. Authorization policies define complex authorization rules based on claims, roles, or custom requirements.

Program.cs
builder.Services.AddAuthorization(options => { options.AddPolicy("AdminOnly", policy => policy.RequireRole("Admin")); options.AddPolicy("Over18", policy => policy.RequireClaim("Age", "18")); });
Q73: What is the difference between roles and claims?

Roles are simple groupings of users (Admin, User). Claims are key-value pairs representing user attributes (Age, Email, Permissions).

When to Use:

  • Roles: Simple authorization scenarios
  • Claims: Fine-grained authorization, custom attributes
Q74: What is the difference between RequireClaim and RequireRole?

RequireClaim checks for specific claim values. RequireRole checks if the user is in a specific role. Roles are often implemented as claims internally.

AsNoTracking disables change tracking for read-only queries, improving performance. Tracking (default) tracks changes for updates.

FromSqlRaw executes raw SQL with parameterization. FromSqlInterpolated uses string interpolation with automatic parameterization.

ToListAsync returns a List. ToArrayAsync returns an array. Both execute the query asynchronously. Use List for mutable collections, array for fixed-size.

46. Advanced API Design

Q96: What is the difference between PUT and PATCH?

PUT replaces the entire resource. PATCH applies partial updates to a resource.

Q97: What is idempotency in REST APIs?

Connection pooling reuses database connections instead of creating new ones, improving performance by reducing connection overhead.
