1. Microservices Fundamentals

Q1: What are microservices and how do they differ from monolithic architecture?

Microservices is an architectural style that structures an application as a collection of small, autonomous services modeled around a business domain.

Aspect Monolithic Microservices
Structure Single codebase
Aspect Monolithic Microservices
Structure Single codebase Multiple independent services
Deployment Deploy entire application Deploy individual services
Scaling Scale entire application Scale specific services
Technology Single technology stack Different technologies per service
Q2: What are the benefits of microservices architecture?

Key Benefits:

  • Scalability: Individual services can be scaled independently
  • Flexibility: Different services can use different technologies
  • Faster Development: Teams can work on different services in parallel
  • Better Fault Isolation: Failure in one service doesn't bring down the entire system
  • Independent Deployment: Services can be deployed without affecting others
Q3: What are the challenges of microservices architecture?

Key Challenges:

  • Increased Complexity: Managing distributed systems is more complex
  • Network Latency: Inter-service communication adds latency
  • Data Consistency: Maintaining data consistency across services is difficult
  • Testing: Integration testing becomes more complex
  • Monitoring: Need to monitor multiple services and their interactions
Q4: When should you use microservices vs monolithic architecture?

Use Microservices When:

  • Application is large and complex
  • Different teams need to work on different features independently
  • Services have different scaling requirements
  • You need to use different technologies for different parts

Use Monolithic When:

  • Application is small and simple
  • Team size is small
  • You want to minimize operational complexity

2. Architecture Patterns

Q5: What is the API Gateway pattern?

API Gateway is a server that acts as an API front-end, receiving API requests, enforcing throttling and security policies, and passing requests to appropriate services.

Responsibilities:

  • Request routing to appropriate microservices
  • Authentication and authorization
  • Rate limiting and throttling
  • Load balancing
  • Response aggregation
  • Caching
api_gateway.cs
// API Gateway using Ocelot in .NET public class Startup { public void ConfigureServices(IServiceCollection services) { services.AddOcelot(); } } // ocelot.json { "Routes": [ { "DownstreamPathTemplate": "/api/products", "DownstreamScheme": "http", "DownstreamHostAndPorts": [{ "Host": "localhost", "Port": 5001 }], "UpstreamPathTemplate": "/products" } ] }
Q6: What is the Service Mesh pattern?

Service Mesh is a dedicated infrastructure layer for handling service-to-service communication. It provides features like load balancing, service discovery, encryption, and observability.

Key Features:

  • Service discovery and load balancing
  • TLS management (mTLS)
  • Circuit breaking and fault tolerance
  • Metrics, logs, and tracing
  • Traffic management
Q7: What is the Saga pattern?

Saga Pattern is a design pattern for managing distributed transactions. It breaks a transaction into a sequence of local transactions, each updating data within a single service.

saga_pattern.cs
// Orchestration-based Saga in C# public class OrderSaga { public async Task ProcessOrder(Order order) { try { await _orderService.CreateOrder(order); await _paymentService.ProcessPayment(order); await _inventoryService.ReserveItems(order.Items); } catch (Exception ex) { // Compensating transactions await _inventoryService.ReleaseItems(order.Items); await _paymentService.RefundPayment(order); await _orderService.CancelOrder(order.Id); throw; } } }
Q8: What is the CQRS pattern?

CQRS (Command Query Responsibility Segregation) separates read and write operations into different models. This can improve performance, scalability, and maintainability.

cqrs_pattern.cs
// CQRS pattern in C# public interface ICommandHandler<TCommand> { Task Handle(TCommand command); } public interface IQueryHandler<TQuery, TResult> { Task<TResult> Handle(TQuery query); } // Command side public class CreateOrderCommandHandler : ICommandHandler<CreateOrderCommand> { public async Task Handle(CreateOrderCommand command) { await _orderRepository.Create(command); } } // Query side public class GetOrderQueryHandler : IQueryHandler<GetOrderQuery, OrderDto> { public async Task<OrderDto> Handle(GetOrderQuery query) { return await _orderReadRepository.GetById(query.OrderId); } }

3. Service Communication

Q9: What is the difference between synchronous and asynchronous communication?
Aspect Synchronous Asynchronous
Blocking Blocks until response received Non-blocking, continues processing
Coupling Tight coupling between services Loose coupling through events/messages
Reliability Requires both services to be available More resilient to failures
Q10: What is gRPC and how does it differ from REST?

gRPC is a high-performance RPC framework that uses Protocol Buffers for serialization. It's designed for efficient communication between microservices.

Aspect REST gRPC
Protocol HTTP/1.1 or HTTP/2 HTTP/2
Data Format JSON/XML Protocol Buffers (binary)
Performance Slower due to text-based Faster due to binary format
Q11: What is message queuing and when should you use it?

Message Queuing enables asynchronous communication between services through a message broker. Services produce messages to a queue, and consumers consume them at their own pace.

4. Data Management

Q12: What is the Database per Service pattern?

Database per Service is a pattern where each microservice has its own database. This ensures loose coupling and allows services to choose the best database for their needs.

Benefits:

  • Loose coupling between services
  • Independent scaling of databases
  • Technology flexibility (SQL, NoSQL, etc.)
  • Data isolation and security
Q13: What is the Event Sourcing pattern?

Event Sourcing is a pattern where state changes are stored as a sequence of events. Instead of storing the current state, you store the history of all changes.

event_sourcing.cs
// Event Sourcing in C# public class Order { private readonly List<IDomainEvent> _events = new List<IDomainEvent>(); public void CreateOrder(List<OrderItem> items) { _events.Add(new OrderCreatedEvent { OrderId = Guid.NewGuid(), Items = items, CreatedAt = DateTime.UtcNow }); } }

5. API Gateway

Q14: What are the key features of an API Gateway?

Key Features:

  • Request Routing: Route requests to appropriate microservices
  • Load Balancing: Distribute traffic across service instances
  • Authentication & Authorization: Centralized security enforcement
  • Rate Limiting: Protect services from overload
  • Caching: Cache responses to reduce load

6. Service Discovery

Q15: What is service discovery and why is it needed?

Service Discovery is the process of automatically detecting devices and services on a computer network. In microservices, it allows services to find and communicate with each other without hardcoding hostnames and ports.

service_discovery.cs
// Service discovery using Consul in C# public class ConsulServiceDiscovery { public async Task RegisterService(string serviceName, string serviceUrl, int port) { var registration = new AgentServiceRegistration { ID = $"{serviceName}-{Guid.NewGuid()}", Name = serviceName, Address = serviceUrl, Port = port }; await _consulClient.Agent.ServiceRegister(registration); } }

Q17: What is externalized configuration?

Externalized Configuration is the practice of storing configuration outside the application code. This allows configuration to be changed without redeploying the application.

12-Factor App is a methodology for building software-as-a-service apps that includes: Codebase, Dependencies, Config, Backing Services, Build/Release/Run, Processes, Port Binding, Concurrency, Disposability, Dev/Prod Parity, Logs, Admin Processes.

8. Resilience Patterns

Q19: What is the Circuit Breaker pattern?

Circuit Breaker prevents cascading failures by stopping requests to a failing service after a threshold is reached, allowing the service time to recover.

circuit_breaker.cs
// Circuit Breaker using Polly in C# public class OrderService { private readonly IAsyncPolicy<HttpResponseMessage> _circuitBreakerPolicy; public OrderService() { _circuitBreakerPolicy = Policy .Handle<HttpRequestException>() .CircuitBreakerAsync( exceptionsAllowedBeforeBreaking: 3, durationOfBreak: TimeSpan.FromSeconds(30) ); } }
Q20: What is the Retry pattern?

Retry Pattern automatically retries failed operations with configurable delays and retry counts to handle transient failures.

retry_pattern.cs
// Retry pattern using Polly in C# public class DatabaseService { private readonly IAsyncPolicy _retryPolicy; public DatabaseService() { _retryPolicy = Policy .Handle<SqlException>() .WaitAndRetryAsync( retryCount: 3, sleepDurationProvider: retryAttempt => TimeSpan.FromSeconds(Math.Pow(2, retryAttempt)) ); } }
Q21: What is the Bulkhead pattern?

Bulkhead Pattern limits the number of concurrent calls to a resource to prevent resource exhaustion and cascading failures.

bulkhead_pattern.cs
// Bulkhead pattern using Polly in C# public class ExternalApiService { private readonly IAsyncPolicy _bulkheadPolicy; public ExternalApiService() { _bulkheadPolicy = Policy.BulkheadAsync( maxParallelization: 10, maxQueuedActions: 5 ); } }
Q22: What is the Timeout pattern?

Timeout Pattern prevents operations from blocking indefinitely by setting a maximum time for an operation to complete.

timeout_pattern.cs
// Timeout pattern in C# public class ApiService { private readonly HttpClient _httpClient; public ApiService() { _httpClient = new HttpClient { Timeout = TimeSpan.FromSeconds(30) }; } }

9. Security

Q23: How do you secure microservices?

Security Best Practices:

  • Authentication: OAuth 2.0, OpenID Connect, JWT tokens
  • Authorization: Role-based access control (RBAC)
  • Service-to-Service: mTLS, API keys, service accounts
  • API Gateway: Centralized authentication and rate limiting
  • Secrets Management: HashiCorp Vault, Azure Key Vault

Q24: What is OAuth 2.0 and how is it used in microservices?

OAuth 2.0 is an authorization framework that enables applications to obtain limited access to user accounts. In microservices, it's commonly used for service-to-service authentication.

10. Monitoring & Observability

Q25: What is observability and why is it important?

Observability is the ability to understand a system's internal state by examining its external outputs. It goes beyond monitoring to provide insights into system behavior.

Three Pillars of Observability:

  • Logs: Discrete events that happened in the system
  • Metrics: Numerical measurements over time
  • Traces: Request paths through distributed systems
Q26: What is distributed tracing?

Distributed Tracing tracks a request as it travels through multiple microservices, providing visibility into the entire request lifecycle and identifying performance bottlenecks.

opentelemetry.cs
// Distributed tracing using OpenTelemetry services.AddOpenTelemetry() .ConfigureResource(resource => resource.AddService("order-service")) .WithTracing(builder => builder .AddAspNetCoreInstrumentation() .AddHttpClientInstrumentation() .AddJaegerExporter(o => { o.AgentHost = "localhost"; o.AgentPort = 6831; }));

11. Deployment & Orchestration

Q28: What is containerization and why is it important for microservices?

Containerization packages applications with their dependencies into lightweight, portable containers. It's essential for microservices because it provides consistent deployment environments.

Dockerfile
// Dockerfile for .NET microservice FROM mcr.microsoft.com/dotnet/aspnet:7.0 AS base WORKDIR /app EXPOSE 80 FROM mcr.microsoft.com/dotnet/sdk:7.0 AS build WORKDIR /src COPY ["OrderService.csproj", "./"] RUN dotnet restore "OrderService.csproj" COPY . . RUN dotnet build "OrderService.csproj" -c Release -o /app/build FROM build AS publish RUN dotnet publish "OrderService.csproj" -c Release -o /app/publish FROM base AS final WORKDIR /app COPY --from=publish /app/publish . ENTRYPOINT ["dotnet", "OrderService.dll"]
Q29: What is Kubernetes and how does it help with microservices?

Kubernetes is an open-source container orchestration platform that automates deployment, scaling, and management of containerized applications.

Key Features:

  • Service discovery and load balancing
  • Auto-scaling with Horizontal Pod Autoscaler
  • Self-healing (restarts failed containers)
  • Rolling updates for zero-downtime deployments
  • Secrets management
  • ConfigMaps for configuration

12. Testing Strategies

Q31: How do you test microservices?

Testing Levels:

  • Unit Testing: Test individual components in isolation
  • Integration Testing: Test interactions between services
  • Contract Testing: Verify API contracts between services
  • End-to-End Testing: Test complete user flows
  • Performance Testing: Test system under load
  • Chaos Engineering: Test system resilience
unit_test.cs
// Unit test using xUnit in C# public class OrderServiceTests { [Fact] public async Task CreateOrder_WithValidItems_ReturnsOrderId() { var mockRepository = new Mock<IOrderRepository>(); var service = new OrderService(mockRepository.Object); var order = new Order { Items = new List<OrderItem>() }; var result = await service.CreateOrderAsync(order); Assert.NotNull(result); Assert.NotEqual(Guid.Empty, result.Id); } }
Q32: What is contract testing?

Contract Testing ensures that the contract between a service and its consumers is maintained. It catches breaking changes early in the development cycle.

Q33: What is chaos engineering?

Chaos Engineering is the practice of intentionally introducing failures into a system to test its resilience and identify weaknesses before they cause real problems.

Chaos Engineering Practices:

  • Failure injection
  • Network latency
  • Resource exhaustion
  • Pod kill
  • Zone failure
Q34: What is the Strangler Fig pattern?

Strangler Fig Pattern is a strategy for gradually migrating a monolithic application to microservices by creating new services around the old system and gradually replacing it.

Q35: What is the Sidecar pattern?

Sidecar Pattern deploys helper components alongside the main application container in the same pod. These sidecars handle cross-cutting concerns like logging, monitoring, and configuration.

Q36: What is the Ambassador pattern?

Ambassador Pattern uses a helper service (ambassador) to send requests on behalf of a service, handling cross-cutting concerns like retry logic, circuit breaking, and security.

Q37: What is the Anti-Corruption Layer?

Anti-Corruption Layer (ACL) is a pattern that isolates your domain model from external systems or legacy code, preventing their influence from corrupting your clean architecture.

Q38: What is the Backend for Frontend pattern?

Backend for Frontend (BFF) is a pattern where a dedicated backend service is created for each frontend application. This simplifies the frontend by aggregating calls to multiple microservices.

Q39: What is the Aggregator pattern?

Aggregator Pattern is a pattern where a service collects data from multiple microservices and combines it into a single response for the client. This reduces the number of calls the client needs to make.

Q40: What is the Composite UI pattern?

Composite UI Pattern is a pattern where the UI is composed by assembling fragments from multiple microservices. Each service provides its own UI component, and the client composes them into a complete page.

Q41: What is the Shared Database pattern?

Shared Database is a pattern where multiple microservices share the same database. This is generally not recommended as it creates tight coupling, but can be used in certain scenarios.

Challenges:

  • Tight coupling between services
  • Data integrity issues
  • Performance bottlenecks
  • Difficult to scale individual services
Q42: What is the Shared Kernel pattern?

Shared Kernel is a pattern where common functionality is extracted into a shared library that can be used by multiple microservices. This reduces code duplication but must be managed carefully to avoid tight coupling.

Q43: What is the Blue-Green Deployment pattern?

Blue-Green Deployment is a technique for reducing downtime by having two identical production environments (blue and green). One environment handles live traffic while the other is updated, then traffic is switched.

Q44: What is the Canary Deployment pattern?

Canary Deployment is a technique where a new version is rolled out to a small subset of users first, then gradually to the entire user base. This allows for early detection of issues.

Q45: What is the Rolling Update pattern?

Rolling Update is a deployment strategy where instances of the new version are gradually rolled out while replacing instances of the old version, ensuring zero downtime.

Q46: What is the Feature Toggle pattern?

Feature Toggle is a technique that allows you to enable or disable features without deploying new code. This is useful for A/B testing, canary releases, and rapid rollback.

feature_toggle.cs
// Feature toggle using LaunchDarkly in C# public class OrderService { private readonly IFeatureFlag _featureFlag; public async Task<Order> GetOrderAsync(Guid orderId) { if (await _featureFlag.IsEnabledAsync("new-order-api", User.Current)) { return await _newOrderRepository.GetByIdAsync(orderId); } else { return await _legacyOrderRepository.GetByIdAsync(orderId); } } }
Q47: What is the Idempotency pattern?

Idempotency is the property of an operation where applying it multiple times has the same effect as applying it once. This is crucial in distributed systems to handle duplicate requests.

idempotency.cs
// Idempotency in C# public class OrderService { public async Task<Order> CreateOrderAsync(Order order) { // Check if order already exists var existingOrder = await _orderRepository.GetByIdAsync(order.Id); if (existingOrder != null) { return existingOrder; // Return existing order (idempotent) } return await _orderRepository.CreateAsync(order); } }
Q48: What is the Distributed Transaction pattern?

Distributed Transaction is a transaction that spans multiple microservices. Implementing distributed transactions is challenging and typically uses patterns like Saga or 2PC (Two-Phase Commit).

Q49: What is the Event-Driven Architecture pattern?

Event-Driven Architecture is a pattern where services communicate through events. This provides loose coupling and enables asynchronous processing.

event_driven.cs
// Event-driven architecture using MassTransit in C# public class OrderCreatedEventConsumer : IConsumer<OrderCreatedEvent> { public async Task Consume(ConsumeContext<OrderCreatedEvent> context) { var @event = context.Message; // Process order await _inventoryService.ReserveItems(@event.Items); await _notificationService.SendOrderConfirmation(@event.OrderId); } }
Q50: What is the CQRS with Event Sourcing pattern?

CQRS with Event Sourcing combines both patterns to create a powerful architecture where commands generate events, and queries use read models built from those events. This provides excellent scalability and auditability.

cqrs_event_sourcing.cs
// CQRS with Event Sourcing public class OrderCommandHandler : ICommandHandler<CreateOrderCommand> { private readonly IEventStore _eventStore; private readonly IEventBus _eventBus; public async Task Handle(CreateOrderCommand command) { var order = new Order(); order.CreateOrder(command.Items); await _eventStore.SaveEvents(order.Id, order.GetUncommittedEvents()); foreach (var @event in order.GetUncommittedEvents()) { await _eventBus.Publish(@event); } } }
