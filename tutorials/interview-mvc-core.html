1. MVC Pattern Fundamentals

Q1: What is the MVC pattern and how does it work in ASP.NET Core?

MVC (Model-View-Controller) is an architectural pattern that separates an application into three main components:

  • Model: Represents the data and business logic. It contains the application's data and business rules.
  • View: Represents the user interface. It displays the data provided by the Model.
  • Controller: Handles user input and interactions. It processes requests, manipulates the Model, and selects the View to display.

Benefits of MVC:

  • Separation of concerns
  • Easier maintenance and testing
  • Parallel development
  • Multiple views for the same model

2. Controllers and Actions

Q3: What is a Controller in ASP.NET Core MVC?

A Controller is a class that handles HTTP requests and generates responses. It:

  • Inherits from the Controller base class
  • Contains public methods called "actions"
  • Receives user input and processes it
  • Interacts with the Model
  • Selects and returns a View or data
public class ProductsController : Controller
{
    private readonly IProductService _service;
    
    public ProductsController(IProductService service)
    {
        _service = service;
    }
    
    // GET: Products
    public IActionResult Index()
    {
        var products = _service.GetAll();
        return View(products);
    }
    
    // GET: Products/Details/5
    public IActionResult Details(int id)
    {
        var product = _service.GetById(id);
        if (product == null) return NotFound();
        return View(product);
    }
    
    // POST: Products/Create
    [HttpPost]
    [ValidateAntiForgeryToken]
    public IActionResult Create(ProductViewModel model)
    {
        if (!ModelState.IsValid) return View(model);
        _service.Create(model);
        return RedirectToAction(nameof(Index));
    }
}

3. Views and Razor Syntax

Q5: What is Razor and how does it work?

Razor is a templating engine that combines HTML with C# code. It uses the @ symbol to transition between HTML and C#.

Razor Syntax Basics:

  • @variable - Outputs variable value
  • @{ code } - Code block
  • @(expression) - Explicit expression
  • @if (condition) { } - Conditional
  • @for, @foreach, @while - Loops
  • @model - Specifies the model type
  • @inject - Injects services
@model IEnumerable<Product>
@{
    ViewData["Title"] = "Products";
}

<h1>@ViewData["Title"]</h1>

@if (Model.Any())
{
    @foreach (var product in Model)
    {
        <div class="product">
            <h3>@product.Name</h3>
            <p>Price: @product.Price.ToString("C")</p>
            <a asp-action="Details" asp-route-id="@product.Id">View Details</a>
        </div>
    }
}
else
{
    <p>No products available.</p>
}

4. Models and Model Binding

Q7: What is Model Binding in ASP.NET Core MVC?

Model Binding is the process of mapping HTTP request data to action method parameters. It automatically converts:

  • Form data
  • Route values
  • Query string parameters
  • Request body (JSON, XML)

to strongly-typed objects.

// Model binding example
public class ProductViewModel
{
    public int Id { get; set; }
    [Required]
    public string Name { get; set; }
    [Range(0.01, 10000)]
    public decimal Price { get; set; }
}

[HttpPost]
public IActionResult Create(ProductViewModel model)
{
    if (!ModelState.IsValid)
    {
        return View(model);
    }
    // Process the model
    return RedirectToAction(nameof(Index));
}
Q8: What are the different sources of model binding?

Model binding sources in order of priority:

  1. Form values: HTTP POST form data
  2. Route values: URL route parameters
  3. Query string: URL query parameters
  4. Body: Request body (JSON, XML)
  5. Header: HTTP headers
// Explicit binding sources
public IActionResult Edit(
    [FromRoute] int id,
    [FromQuery] string returnUrl,
    [FromHeader] string authorization,
    [FromBody] ProductViewModel model)
{
    // ...
}

5. Routing in MVC

Q9: What are the different types of routing in ASP.NET Core MVC?

1. Convention-based Routing:

app.MapControllerRoute(
    name: "default",
    pattern: "{controller=Home}/{action=Index}/{id?}");

2. Attribute Routing:

[Route("products")]
public class ProductsController : Controller
{
    [Route("")]
    public IActionResult Index() { }
    
    [Route("{id:int}")]
    public IActionResult Details(int id) { }
    
    [Route("create")]
    [HttpPost]
    public IActionResult Create(ProductViewModel model) { }
}

3. Mixed Routing:

Both convention-based and attribute routing can be used together.

Q10: What is the difference between [Route] and [HttpGet] attributes?
  • [Route]: Defines the URL pattern for the action. It can be used for any HTTP method.
  • [HttpGet], [HttpPost], etc.: Combines route definition with HTTP method constraint. More specific and preferred for API actions.
// [Route] - Any HTTP method
[Route("products/{id}")]
public IActionResult Get(int id) { }

// [HttpGet] - Only GET requests
[HttpGet("products/{id}")]
public IActionResult Get(int id) { }

6. View Models and DTOs

Q11: What is a View Model and why is it important?

A View Model is a class designed specifically for a view, containing only the data needed for that view.

Benefits of View Models:

  • Separation of concerns - domain models separate from presentation
  • Security - prevents over-posting attacks
  • Flexibility - can shape data for specific views
  • Validation - view-specific validation rules
  • Composition - can combine data from multiple sources
// Domain Model
public class Product
{
    public int Id { get; set; }
    public string Name { get; set; }
    public decimal Price { get; set; }
    public int CategoryId { get; set; }
    public Category Category { get; set; }
}

// View Model
public class ProductViewModel
{
    public int Id { get; set; }
    public string Name { get; set; }
    public string FormattedPrice { get; set; }
    public string CategoryName { get; set; }
    public bool IsInStock { get; set; }
}
Q12: What is the over-posting vulnerability?

Over-posting occurs when a user submits form data that includes fields that should not be modifiable, such as ID or role fields. This can be exploited to:

  • Modify sensitive data
  • Elevate privileges
  • Bypass business rules
Important: Always use View Models to prevent over-posting. Never bind directly to domain entities in POST actions. 
// Vulnerable - binds to all properties
public IActionResult Create(Product product) { }

// Secure - uses view model
public IActionResult Create(ProductViewModel model) { }

A Partial View is a reusable view fragment that can be rendered within other views. It's useful for:

8. Tag Helpers

Q15: What are Tag Helpers and why are they preferred over HTML Helpers?

Tag Helpers enable server-side code to participate in creating and rendering HTML elements in Razor files. They look like standard HTML attributes.

Advantages over HTML Helpers:

  • More readable - looks like HTML
  • Better IntelliSense support
  • Easier to work with前端工具
  • More natural syntax
// HTML Helper (old way)
@Html.TextBoxFor(m => m.Name)
@Html.DropDownListFor(m => m.CategoryId, Model.Categories)

// Tag Helper (new way)
<input asp-for="Name" />
<select asp-for="CategoryId" asp-items="Model.Categories"></select>
Q16: What are some built-in Tag Helpers?
  • asp-controller: Specifies the controller
  • asp-action: Specifies the action
  • asp-route-{value}: Route parameter
  • asp-for: Model binding
  • asp-validation-for: Validation message
  • asp-antiforgery: Anti-forgery token
  • asp-area: Area specification
  • asp-page: Razor Page reference

9. Layouts and Sections

Q17: What is a Layout page?

A Layout page defines the common structure for multiple views, such as header, footer, and navigation. It uses @RenderBody() to specify where the view content will be inserted.

// _Layout.cshtml
<!DOCTYPE html>
<html>
<head>
    <title>@ViewData["Title"]</title>
    @RenderSection("head", required: false)
</head>
<body>
    <header>@RenderSection("Header", required: false)</header>
    <main>@RenderBody()</main>
    <footer>@RenderSection("Footer", required: false)</footer>
</body>
</html>
Q18: What are Sections in Razor views?

Sections allow views to inject content into specific areas of the layout page.

// In view
@{
    Layout = "_Layout";
}

@section Header {
    <nav>Custom navigation</nav>
}

@section Footer {
    <p>Custom footer</p>
}

<h1>Main content</h1>

10. Areas in MVC

Q19: What are Areas in ASP.NET Core MVC?

Areas allow you to organize large MVC applications into functional groups, each with its own controllers, views, and models. Each area is essentially a mini-MVC application.

Benefits:

  • Better organization for large applications
  • Separation of functional modules
  • Easier team collaboration
  • Independent routing for each area
// Area controller
[Area("Admin")]
[Route("admin/[controller]/[action]")]
public class DashboardController : Controller
{
    public IActionResult Index()
    {
        return View();
    }
}

// Area routing
app.MapControllerRoute(
    name: "areas",
    pattern: "{area:exists}/{controller=Home}/{action=Index}/{id?}");

11. Filters in MVC

Q20: What are the different types of filters in MVC?
  • Authorization filters: Run first to determine if user is authorized
  • Resource filters: Run after authorization, can short-circuit pipeline
  • Action filters: Run before and after action execution
  • Exception filters: Handle exceptions globally
  • Result filters: Run before and after result execution

    {
        Console.WriteLine($"Executed: {context.ActionDescriptor.DisplayName}");
    }
}

12. Validation

Q21: How does validation work in ASP.NET Core MVC?

ASP.NET Core MVC uses Data Annotations for validation. The framework automatically validates models based on these attributes.

public class ProductViewModel
{
    public int Id { get; set; }
    
    [Required(ErrorMessage = "Name is required")]
    [StringLength(100, MinimumLength = 2)]
    public string Name { get; set; }
    
    [Range(0.01, 10000)]
    public decimal Price { get; set; }
    
    [EmailAddress]
    public string Email { get; set; }
    
    [RegularExpression(@"^[A-Z]{2}\d{4}$")]
    public string ProductCode { get; set; }
}

// In controller
[HttpPost]
public IActionResult Create(ProductViewModel model)
{
    if (!ModelState.IsValid)
    {
        return View(model);
    }
    // Process valid model
    return RedirectToAction(nameof(Index));
}
Q22: What is custom validation?

You can create custom validation attributes by inheriting from ValidationAttribute.

public class CustomDateAttribute : ValidationAttribute
{
    protected override ValidationResult IsValid(object value, ValidationContext context)
    {
        if (value is DateTime date)
        {
            if (date < DateTime.Now)
            {
                return new ValidationResult("Date must be in the future");
            }
        }
        return ValidationResult.Success;
    }
}

// Usage
[CustomDate]
public DateTime ExpiryDate { get; set; }

14. Advanced MVC Patterns (All Levels Level)

Q31: What is the Repository Pattern in MVC?

Repository Pattern mediates between the domain and data mapping layers, acting like an in-memory domain object collection.

 Repository interface
public interface IRepository<T> where T : class
{
    Task<IEnumerable<T>> GetAllAsync();
    Task<T> GetByIdAsync(int id);
    Task<T> AddAsync(T entity);
    Task UpdateAsync(T entity);
    Task DeleteAsync(T entity);
}

// Repository implementation
public class Repository<T> : IRepository<T> where T : class
{
    private readonly AppDbContext _context;
    private readonly DbSet<T> _dbSet;

    public Repository(AppDbContext context)
    {
        _context = context;
        _dbSet = context.Set<T>();
    }

    public async Task<IEnumerable<T>> GetAllAsync()
    {
        return await _dbSet.ToListAsync();
    }

    public async Task<T> GetByIdAsync(int id)
    {
        return await _dbSet.FindAsync(id);
    }

    public async Task<T> AddAsync(T entity)
    {
        await _dbSet.AddAsync(entity);
        await _context.SaveChangesAsync();
        return entity;
    }

    public async Task UpdateAsync(T entity)
    {
        _dbSet.Update(entity);
        await _context.SaveChangesAsync();
    }

    public async Task DeleteAsync(T entity)
    {
        _dbSet.Remove(entity);
        await _context.SaveChangesAsync();
    }
}
Q32: What is the Unit of Work Pattern in MVC?

Unit of Work Pattern maintains a list of objects affected by a business transaction and coordinates the writing out of changes.

 Unit of Work interface
public interface IUnitOfWork : IDisposable
{
    IRepository<Product> Products { get; }
    IRepository<Category> Categories { get; }
    Task SaveAsync();
}

// Unit of Work implementation
public class UnitOfWork : IUnitOfWork
{
    private readonly AppDbContext _context;
    public IRepository<Product> Products { get; }
    public IRepository<Category> Categories { get; }

    public UnitOfWork(AppDbContext context)
    {
        _context = context;
        Products = new Repository<Product>(context);
        Categories = new Repository<Category>(context);
    }

    public async Task SaveAsync()
    {
        await _context.SaveChangesAsync();
    }

    public void Dispose()
    {
        _context.Dispose();
    }
}
Q33: What is the Service Layer Pattern in MVC?

Service Layer Pattern defines an application's boundary and its set of available operations from the perspective of interfacing client layers.

 Service interface
public interface IProductService
{
    Task<ProductViewModel> GetProductAsync(int id);
    Task<IEnumerable<ProductViewModel> GetAllProductsAsync();
    Task<ProductViewModel> CreateProductAsync(CreateProductViewModel model);
    Task UpdateProductAsync(ProductViewModel model);
    Task DeleteProductAsync(int id);
}

// Service implementation
public class ProductService : IProductService
{
    private readonly IUnitOfWork _unitOfWork;
    private readonly IMapper _mapper;

    public ProductService(IUnitOfWork unitOfWork, IMapper mapper)
    {
        _unitOfWork = unitOfWork;
        _mapper = mapper;
    }

    public async Task<ProductViewModel> GetProductAsync(int id)
    {
        var product = await _unitOfWork.Products.GetByIdAsync(id);
        return _mapper.Map<ProductViewModel>(product);
    }

    public async Task<ProductViewModel> CreateProductAsync(CreateProductViewModel model)
    {
        var product = _mapper.Map<Product>(model);
        await _unitOfWork.Products.AddAsync(product);
        await _unitOfWork.SaveAsync();
        return _mapper.Map<ProductViewModel>(product);
    }
}
Q34: What is the CQRS Pattern in MVC?

CQRS (Command Query Responsibility Segregation) separates read and write operations into different models.

 Query
public class GetProductQuery
{
    public int Id { get; set; }
}

public class GetProductQueryHandler
{
    private readonly AppDbContext _context;

    public GetProductQueryHandler(AppDbContext context)
    {
        _context = context;
    }

    public async Task<ProductViewModel> Handle(GetProductQuery query)
    {
        var product = await _context.Products.FindAsync(query.Id);
        return new ProductViewModel { Id = product.Id, Name = product.Name };
    }
}

// Command
public class CreateProductCommand
{
    public string Name { get; set; }
    public decimal Price { get; set; }
}

public class CreateProductCommandHandler
{
    private readonly AppDbContext _context;

    public CreateProductCommandHandler(AppDbContext context)
    {
        _context = context;
    }

    public async Task<int> Handle(CreateProductCommand command)
    {
        var product = new Product { Name = command.Name, Price = command.Price };
        _context.Products.Add(product);
        await _context.SaveChangesAsync();
        return product.Id;
    }
}

Q36: How do you implement policy-based authorization in MVC? 
 Define policy
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("AtLeast21", policy =>
        policy.Requirements.Add(new MinimumAgeRequirement(21)));
});

// Requirement handler
public class MinimumAgeRequirement : IAuthorizationRequirement
{
    public int MinimumAge { get; }

    public MinimumAgeRequirement(int minimumAge)
    {
        MinimumAge = minimumAge;
    }
}

public class MinimumAgeHandler : AuthorizationHandler<MinimumAgeRequirement>
{
    protected override Task HandleRequirementAsync(
        AuthorizationHandlerContext context,
        MinimumAgeRequirement requirement)
    {
        var ageClaim = context.User.FindFirst(c => c.Type == "Age");
        if (ageClaim != null && int.Parse(ageClaim.Value) >= requirement.MinimumAge)
        {
            context.Succeed(requirement);
        }
        return Task.CompletedTask;
    }
}

// Use policy
[Authorize(Policy = "AtLeast21")]
public IActionResult AdultContent()
{
    return View();
}
Q37: How do you implement resource-based authorization in MVC? 
 Requirement
public class SameAuthorRequirement : IAuthorizationRequirement
{
}

// Handler
public class SameAuthorHandler : AuthorizationHandler<SameAuthorRequirement, Document>
{
    protected override Task HandleRequirementAsync(
        AuthorizationHandlerContext context,
        SameAuthorRequirement requirement,
        Document resource)
    {
        if (resource.AuthorId == context.User.FindFirstValue(ClaimTypes.NameIdentifier))
        {
            context.Succeed(requirement);
        }
        return Task.CompletedTask;
    }
}

// Controller
public class DocumentController : Controller
{
    private readonly IAuthorizationService _authorizationService;

    public async Task<IActionResult> Edit(int id)
    {
        var document = await _documentRepository.GetByIdAsync(id);
        var result = await _authorizationService.AuthorizeAsync(
            User, document, "SameAuthor");
        
        if (!result.Succeeded)
        {
            return Forbid();
        }
        
        return View(document);
    }
}

Q39: What is the difference between TempData and Session?
  • TempData: Stores data for the current and next request only. Used for passing data between actions. Automatically cleared after second request.
  • Session: Stores data for the entire user session. Persists across multiple requests. Requires explicit clearing.
Q40: What is the difference between @Html.Action and @Html.RenderAction?
  • @Html.Action: Returns HTML string. Can be stored in variable. Less efficient.
  • @Html.RenderAction: Renders directly to response. More efficient. Cannot be stored in variable.
Q41: What is the difference between @Html.Partial and @Html.RenderPartial?
  • @Html.Partial: Returns HTML string. Can be stored in variable. Less efficient.
  • @Html.RenderPartial: Renders directly to response. More efficient. Cannot be stored in variable.
Q42: What is the difference between @Html.DisplayFor and @Html.EditorFor?
  • @Html.DisplayFor: Displays data in read-only mode. Uses display templates. Good for showing data.
  • @Html.EditorFor: Displays data in editable mode. Uses editor templates. Good for forms.
Q43: What is the difference between @Html.LabelFor and @Html.DisplayNameFor?
  • @Html.LabelFor: Renders HTML label element. Can be used with form inputs. Includes for attribute.
  • @Html.DisplayNameFor: Returns display name as string. No HTML rendering. Used for headers, tables.
Q44: What is the difference between [Bind] and [BindNever]?
  • [Bind]: Specifies which properties should be included in model binding. Whitelist approach.
  • [BindNever]: Specifies which properties should never be included in model binding. Blacklist approach.
Q45: What is the difference between [HttpGet] and [HttpPost]?
  • [HttpGet]: Restricts action to HTTP GET requests. Used for retrieving data. Idempotent.
  • [HttpPost]: Restricts action to HTTP POST requests. Used for creating/updating data. Not idempotent.
Q46: What is the difference between [Route] and [HttpGet] with route template?
  • [Route]: Applies to all HTTP methods. Defines route for controller or action. More general.
  • [HttpGet]: Applies only to GET requests. Combines HTTP method constraint with route. More specific.
Q47: What is the difference between RedirectToAction and RedirectToRoute?
  • RedirectToAction: Redirects to action using action name and controller name. Simpler syntax.
  • RedirectToRoute: Redirects using route name and route values. More flexible for complex routing.
Q48: What is the difference between PartialView and ViewComponent?
  • PartialView: Simple view fragment. No logic. Receives data from parent view. Good for static content.
  • ViewComponent: Has its own logic and model. Can have dependencies. Good for dynamic components with business logic.
