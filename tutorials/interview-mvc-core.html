1. MVC Fundamentals

Q1: What is MVC?

MVC (Model-View-Controller) is an architectural pattern that separates an application into three main components: Model, View, and Controller. This separation of concerns promotes better code organization, maintainability, and testability.

MVC Components:

  • Model: Represents data and business logic. Contains application data, validation rules, and business operations. Independent of UI.
  • View: Represents the UI and displays data. Renders the model data to the user. Handles user presentation logic.
  • Controller: Handles user input and coordinates Model and View. Processes requests, interacts with the model, and selects the appropriate view.

Benefits of MVC:

  • Separation of Concerns: Each component has a specific responsibility
  • Testability: Components can be tested independently
  • Maintainability: Easier to modify and extend
  • Parallel Development: Different developers can work on different components simultaneously
MVC.cs
// Model public class Product { public int Id { get; set; } public string Name { get; set; } public decimal Price { get; set; } } // Controller public class ProductController : Controller { public IActionResult Index() { var products = new[] { new Product { Id = 1, Name = "Laptop", Price = 999.99m } }; return View(products); } } // View (Index.cshtml) @model IEnumerable<Product> @foreach (var product in Model) { <div>@product.Name - @product.Price</div> }
Q2: What is the difference between MVC and Web API?

MVC for web applications with views. Web API for HTTP services without views. MVC is designed for building traditional web applications that return HTML views, while Web API is designed for building RESTful HTTP services that return data (JSON, XML) to be consumed by various clients including browsers, mobile apps, and other services.

Key Differences:

  • Purpose: MVC builds UI-focused web apps; Web API builds data services
  • Return Type: MVC returns HTML views; Web API returns JSON/XML data
  • Content Negotiation: MVC limited; Web API supports multiple formats
  • Base Class: MVC uses Controller; Web API uses ControllerBase
  • Use Case: MVC for server-rendered pages; Web API for SPA/mobile/backend
MVCvsWebAPI.cs
// MVC Controller - Returns Views public class HomeController : Controller { public IActionResult Index() { return View(); // Returns HTML view } } // Web API Controller - Returns Data [ApiController] [Route("api/[controller]")] public class ProductsController : ControllerBase { [HttpGet] public IActionResult Get() { return Ok(new[] { "Product1", "Product2" }); // Returns JSON } }
Q3: What is the Model in MVC?

Model represents the data and business logic of the application. It is the core component that encapsulates the application's data structure, validation rules, and business operations. The model is independent of the user interface and can be reused across different views and controllers.

Model Responsibilities:

  • Data Representation: Defines the structure and properties of application data
  • Business Logic: Contains validation rules and business operations
  • Data Access: May include database interaction logic
  • State Management: Maintains application state independent of UI
Model.cs
public class Product { public int Id { get; set; } public string Name { get; set; } public decimal Price { get; set; } public decimal CalculateDiscount() { return Price * 0.9m; } }
Q4: What is the View in MVC?

View represents the UI and displays data to the user. Views are responsible for rendering the model data into a format that can be presented to the user, typically HTML for web applications. Views should contain minimal logic and focus primarily on presentation concerns.

View Responsibilities:

  • Rendering: Converts model data into user-readable format (HTML)
  • User Interaction: Handles user input through forms and links
  • Presentation Logic: Contains UI-specific logic like formatting and display rules
  • Separation: Should not contain business logic or data access code
View.cshtml
@model Product @{ ViewData["Title"] = "Product Details"; } <h1>@Model.Name</h1> <p>Price: $@Model.Price</p> <p>Discounted: $@Model.CalculateDiscount()</p>
Q5: What is the Controller in MVC?

Controller handles user input, interacts with Model, and selects View. Controllers act as the intermediary between the user (through the browser) and the application's data and business logic. They process HTTP requests, invoke appropriate model operations, and return the appropriate view or data response.

Controller Responsibilities:

  • Request Handling: Receives and processes HTTP requests
  • Model Interaction: Calls business logic and data operations
  • View Selection: Chooses the appropriate view to render
  • Response Generation: Returns HTML views, JSON data, or other response types
  • Input Validation: Validates user input before processing
Controller.cs
public class ProductController : Controller { private readonly IProductService _service; public ProductController(IProductService service) { _service = service; } public IActionResult Details(int id) { var product = _service.GetById(id); return View(product); } }
Q6: What is the difference between MVC and MVVM?

MVC for web. MVVM for desktop/mobile with data binding. MVC is primarily used for web applications where the server generates HTML, while MVVM is designed for client-side applications (WPF, Xamarin, MAUI, Blazor) that use data binding to automatically synchronize the UI with the underlying data.

Key Differences:

  • Platform: MVC for web server-side; MVVM for desktop/mobile client-side
  • Data Binding: MVC manual data passing; MVVM automatic two-way binding
  • ViewModel: MVVM has a dedicated ViewModel that implements INotifyPropertyChanged
  • View Updates: MVC controller updates view; MVVM data binding updates view automatically
  • Testing: Both support testing, but MVVM ViewModel is easier to unit test
MVCvsMVVM.cs
// MVC - Model-View-Controller (Web) public class ProductController : Controller { public IActionResult Index() { var model = new ProductViewModel(); return View(model); } } // MVVM - Model-View-ViewModel (Desktop/Mobile with data binding) // Used in WPF, Xamarin, MAUI, Blazor public class ProductViewModel : INotifyPropertyChanged { private string _name; public string Name { get => _name; set { _name = value; OnPropertyChanged(nameof(Name)); } } public event PropertyChangedEventHandler PropertyChanged; }
Q7: What is the difference between MVC and MVP?

MVC View updates Model. MVP Presenter updates View. In MVC, the View can directly interact with the Model, while in MVP, the Presenter acts as a complete intermediary, handling all communication between the View and the Model. This makes MVP more testable and provides better separation of concerns.

Key Differences:

  • View-Model Interaction: MVC view can access model; MVP view cannot access model directly
  • Mediator: MVC controller mediates loosely; MVP presenter mediates strictly
  • View Updates: MVC view updates itself; MVP presenter updates view through interface
  • Testability: MVP is more testable as view can be mocked completely
  • Complexity: MVP adds more layers but provides better separation
MVCvsMVP.cs
// MVC - View can directly access Model public class ProductController : Controller { public IActionResult Index() { var product = _repository.GetById(1); return View(product); // View has direct access to Model } } // MVP - Presenter mediates between View and Model public class ProductPresenter { private readonly IProductView _view; private readonly IProductRepository _repository; public ProductPresenter(IProductView view, IProductRepository repository) { _view = view; _repository = repository; } public void LoadProduct(int id) { var product = _repository.GetById(id); _view.DisplayProduct(product); // Presenter updates View } }
Q8: What is the difference between ASP.NET MVC and ASP.NET Core MVC?

ASP.NET MVC runs on .NET Framework. ASP.NET Core MVC runs on .NET Core. ASP.NET Core MVC is a complete rewrite of ASP.NET MVC, designed to be cross-platform, high-performance, and modular. It represents the modern evolution of the MVC framework with significant improvements.

Key Differences:

  • Platform: ASP.NET MVC is Windows-only (.NET Framework); ASP.NET Core MVC is cross-platform (.NET Core/.NET 5+)
  • Performance: Core MVC is significantly faster and more lightweight
  • Dependency Injection: Core MVC has built-in DI; ASP.NET MVC requires external containers
  • Configuration: Core MVC uses modern configuration system; ASP.NET MVC uses web.config
  • Unified Framework: Core MVC unifies MVC and Web API; ASP.NET MVC has separate frameworks
  • Deployment: Core MVC supports self-contained deployment; ASP.NET MVC requires IIS
MVCvsCore.cs
// ASP.NET MVC (Legacy - .NET Framework) public class HomeController : Controller { public ActionResult Index() { return View(); } } // ASP.NET Core MVC (Modern - .NET Core/.NET 5+) public class HomeController : Controller { public IActionResult Index() { return View(); } } // Key differences: // - Core: Cross-platform, faster, lightweight // - Core: Built-in DI, better configuration // - Core: Unified MVC + Web API
Q9: What is the difference between Razor Pages and MVC?

Razor Pages page-focused. MVC controller-focused. Razor Pages is a newer, page-based programming model that makes building dynamic web applications easier and more productive. It organizes code around pages, with each page having its own view and code-behind file, while MVC separates concerns across controllers and views.

Key Differences:

  • Organization: Razor Pages organizes by page; MVC organizes by controller
  • File Structure: Razor Pages has .cshtml and .cshtml.cs together; MVC separates Controllers and Views folders
  • Handler Methods: Razor Pages uses OnGet, OnPost handlers; MVC uses action methods
  • Simplicity: Razor Pages is simpler for page-focused scenarios; MVC is better for complex APIs
  • Use Case: Razor Pages for traditional web apps; MVC for API-heavy applications
RazorPagesvsMVC.cs
// Razor Pages (Page-focused) // Pages/Index.cshtml.cs public class IndexModel : PageModel { public void OnGet() { } public void OnPost() { } } // Pages/Index.cshtml @page @model IndexModel <form method="post"><button type="submit">Submit</button></form> // MVC (Controller-focused) // Controllers/HomeController.cs public class HomeController : Controller { public IActionResult Index() => View(); [HttpPost] public IActionResult Index() => RedirectToAction("Index"); }
Q10: What is the difference between Blazor and MVC?

Blazor runs C# in browser. MVC runs C# on server. Blazor is a modern web framework that allows you to build interactive web UIs using C# instead of JavaScript. It runs C# code directly in the browser via WebAssembly (Blazor WebAssembly) or on the server (Blazor Server), while MVC runs all C# code on the server and sends HTML to the browser.

Key Differences:

  • Execution: Blazor runs C# in browser (WebAssembly) or server; MVC runs C# only on server
  • Interactivity: Blazor provides rich client-side interactivity; MVC requires JavaScript for interactivity
  • State Management: Blazor maintains client-side state; MVC is stateless
  • JavaScript: Blazor reduces need for JavaScript; MVC often requires JavaScript libraries
  • Performance: Blazor WebAssembly has initial load but fast interactions; MVC has faster initial load but slower interactions
  • Use Case: Blazor for SPA-like experiences; MVC for traditional server-rendered apps
BlazorvsMVC.cs
// Blazor (C# in browser via WebAssembly) // Counter.razor @page "/counter" <button @onclick="IncrementCount">Click me</button> <p>Current count: @currentCount</p> @code { private int currentCount = 0; private void IncrementCount() => currentCount++; } // MVC (C# on server, HTML to browser) // Views/Home/Index.cshtml @model HomeViewModel <button onclick="location.href='@Url.Action("Increment")'">Click me</button> <p>Current count: @Model.Count</p>

2. Controllers

Q11: What is a controller?

Controller handles HTTP requests and returns responses. It inherits from ControllerBase. Controllers are classes that define a set of action methods which handle incoming HTTP requests. They are the entry point for your application's logic and are responsible for processing user input, coordinating with models, and selecting appropriate views or returning data.

Controller Characteristics:

  • Inheritance: Inherits from Controller (for MVC) or ControllerBase (for Web API)
  • Public Methods: Public methods are treated as action methods by default
  • Routing: Mapped to URL patterns through routing configuration
  • Dependency Injection: Supports constructor injection of services
  • Filters: Can apply filters for cross-cutting concerns
Controller.cs
public class HomeController : Controller { // Action method public IActionResult Index() { return View(); } [HttpGet] public IActionResult Details(int id) { return Json(new { Id = id }); } [HttpPost] public IActionResult Create(Product product) { if (ModelState.IsValid) { return RedirectToAction("Index"); } return View(product); } }
Q12: What is an action method?

Action Method is a public method in a controller that handles requests. Action methods are responsible for executing the business logic for a specific request and returning an appropriate response. They are mapped to URLs through the routing system and can accept parameters from the request.

Action Method Characteristics:

  • Public: Must be public to be accessible by the routing system
  • Return Type: Typically returns IActionResult or a specific result type
  • Parameters: Can accept parameters from route data, query strings, or form data
  • Attributes: Can be decorated with HTTP verb attributes ([HttpGet], [HttpPost], etc.)
  • NonAction: Methods marked with [NonAction] are not treated as actions
ActionMethod.cs
public class ProductController : Controller { // Action method - public, returns IActionResult public IActionResult Index() { return View(); } // Not an action method - private private void HelperMethod() { } // Not an action method - has NonAction attribute [NonAction] public void UtilityMethod() { } }
Q13: What is the difference between IActionResult and ActionResult?

IActionResult is interface. ActionResult is base class. IActionResult is the recommended return type for action methods as it provides flexibility to return different types of results. ActionResult is a generic version that allows returning a specific type or an ActionResult.

Key Differences:

  • Type: IActionResult is an interface; ActionResult is an abstract base class
  • Flexibility: IActionResult allows returning any result type; ActionResult allows specific type or result
  • Usage: IActionResult is recommended for most actions; ActionResult for API actions returning data
  • Type Safety: ActionResult provides compile-time type checking for the return type
ActionResult.cs
// IActionResult - Interface (recommended) public IActionResult Get() { return Ok("data"); } // ActionResult - Generic with specific type public ActionResult<Product> GetProduct(int id) { var product = _service.GetById(id); if (product == null) return NotFound(); return product; }
Q14: What is the difference between ViewResult and PartialViewResult?

ViewResult full page. PartialViewResult partial view. ViewResult renders a complete HTML page with layout, while PartialViewResult renders a portion of HTML without layout, typically used for reusable components or AJAX updates.

Key Differences:

  • Layout: ViewResult includes _Layout page; PartialViewResult does not include layout
  • Usage: ViewResult for full pages; PartialViewResult for reusable components
  • AJAX: PartialViewResult commonly used for AJAX partial updates
  • Rendering: ViewResult renders complete HTML document; PartialViewResult renders HTML fragment
ViewResults.cs
// ViewResult - Full page with layout public IActionResult Index() { return View(); // Renders Views/Home/Index.cshtml with _Layout } // PartialViewResult - Partial without layout public IActionResult ProductList() { return PartialView("_ProductList"); // Renders partial only } // Usage in view @Html.PartialAsync("_ProductList")
Q15: What is the difference between ContentResult and JsonResult?

ContentResult plain text. JsonResult JSON data. ContentResult returns raw string content with a specified content type, while JsonResult serializes an object to JSON format. ContentResult is useful for returning plain text, XML, or other custom formats, while JsonResult is ideal for API responses.

Key Differences:

  • Content: ContentResult returns raw string; JsonResult serializes object to JSON
  • Content Type: ContentResult requires explicit content type; JsonResult sets application/json automatically
  • Usage: ContentResult for custom formats; JsonResult for API responses
  • Modern Equivalent: Ok() method is preferred over JsonResult in modern ASP.NET Core
ContentResults.cs
// ContentResult - Plain text/string public IActionResult GetText() { return Content("Hello World", "text/plain"); } // JsonResult - JSON data public IActionResult GetData() { return Json(new { Name = "Product", Price = 99.99 }); } // Modern equivalent public IActionResult GetData() { return Ok(new { Name = "Product", Price = 99.99 }); }
Q16: What is the difference between FileResult and RedirectResult?

FileResult returns file. RedirectResult redirects to URL. FileResult is used to return binary file content to the browser, such as PDFs, images, or documents. RedirectResult sends an HTTP redirect response to the browser, instructing it to navigate to a different URL.

Key Differences:

  • Purpose: FileResult returns file content; RedirectResult redirects to another URL
  • HTTP Status: FileResult returns 200 OK with file; RedirectResult returns 302/301 redirect
  • Browser Behavior: FileResult prompts download or displays file; RedirectResult navigates to new URL
  • Use Case: FileResult for file downloads; RedirectResult for navigation flow
FileRedirect.cs
// FileResult - Returns file public IActionResult Download() { byte[] fileBytes = System.IO.File.ReadAllBytes("document.pdf"); return File(fileBytes, "application/pdf", "download.pdf"); } // RedirectResult - Redirects to URL public IActionResult External() { return Redirect("https://example.com"); }
Q17: What is the difference between RedirectToAction and RedirectToRoute?

RedirectToAction to action. RedirectToRoute to route. RedirectToAction generates a URL based on the controller and action names, while RedirectToRoute generates a URL based on a named route defined in the routing configuration. RedirectToRoute is more flexible when you have complex routing scenarios.

Key Differences:

  • Target: RedirectToAction targets controller/action; RedirectToRoute targets named route
  • Flexibility: RedirectToRoute supports custom route parameters more flexibly
  • Usage: RedirectToAction for simple redirects; RedirectToRoute for complex routing
  • Route Configuration: RedirectToRoute requires named routes to be defined
RedirectTypes.cs
// RedirectToAction - Redirect to action public IActionResult Create() { return RedirectToAction("Index", "Home"); // Or with parameters return RedirectToAction("Details", new { id = 5 }); } // RedirectToRoute - Redirect to named route public IActionResult GoToProduct() { return RedirectToRoute("productDetails", new { id = 5 }); }
Q18: What is the difference between RedirectToAction and Redirect?

RedirectToAction to action. Redirect to URL. RedirectToAction generates a URL based on the routing system using controller and action names, making it more maintainable if routes change. Redirect uses a literal URL string, which is less flexible but necessary for external URLs or when you need exact control over the URL.

Key Differences:

  • URL Generation: RedirectToAction uses routing; Redirect uses literal string
  • Maintainability: RedirectToAction is more maintainable; Redirect is brittle
  • External URLs: Redirect supports external URLs; RedirectToAction only internal
  • Route Changes: RedirectToAction adapts to route changes; Redirect does not
RedirectComparison.cs
// RedirectToAction - Generates URL from action return RedirectToAction("Index", "Home"); // Generates: /Home/Index // Redirect - Direct URL return Redirect("/Home/Index"); // Or external return Redirect("https://example.com");
Q19: What is the difference between Permanent and Temporary redirect?

Permanent 301. Temporary 302. Permanent redirects (301) indicate that the resource has permanently moved to a new location, causing browsers and search engines to update their indexes. Temporary redirects (302) indicate a temporary move, and browsers/search engines continue to use the original URL.

Key Differences:

  • HTTP Status: Permanent returns 301; Temporary returns 302
  • SEO Impact: 301 transfers SEO value to new URL; 302 does not
  • Browser Caching: Browsers cache 301 redirects; 302 redirects are not cached
  • Use Case: 301 for permanent URL changes; 302 for temporary redirects
  • Methods: RedirectPermanent() vs Redirect() in ASP.NET Core
RedirectCodes.cs
// Permanent redirect (301) - SEO friendly public IActionResult OldPage() { return RedirectPermanent("/new-page"); // Or return RedirectToActionPermanent("NewPage"); } // Temporary redirect (302) - Default public IActionResult TempRedirect() { return Redirect("/temp-page"); }
Q20: What is the difference between async and sync actions?

Async improves scalability. Sync blocks thread. Async actions use async/await to perform I/O operations without blocking the thread pool, allowing the server to handle more concurrent requests. Sync actions block the thread while waiting for I/O operations, which can limit scalability under high load.

Key Differences:

  • Thread Usage: Async releases thread during I/O; Sync blocks thread
  • Scalability: Async handles more concurrent requests; Sync limited by thread pool
  • Performance: Async better for I/O-bound operations; Sync simpler for CPU-bound
  • Complexity: Async requires async/await pattern; Sync is simpler
  • Use Case: Async for database/API calls; Sync for simple CPU operations
Async.cs
// Synchronous - blocks thread public IActionResult GetProducts() { var products = _repository.GetAll(); // Blocks thread return View(products); } // Asynchronous - non-blocking public async Task<IActionResult> GetProductsAsync() { var products = await _repository.GetAllAsync(); // Doesn't block return View(products); }

3. Views

Q21: What is a view?

View is a template that generates HTML response. Uses Razor syntax. Views are .cshtml files that contain HTML markup mixed with server-side C# code using Razor syntax. They are responsible for presenting data to the user and handling user interactions through forms and links.

View Characteristics:

  • File Extension: .cshtml files using Razor syntax
  • Location: Stored in Views folder organized by controller
  • Model Binding: Can be strongly-typed with @model directive
  • Layout: Can use shared layout pages for consistent structure
  • Rendering: Server-side rendering to HTML sent to browser
View.cshtml
@* View file: Views/Home/Index.cshtml *@ @model IEnumerable<Product> @{ ViewData["Title"] = "Product List"; } <h1>@ViewData["Title"]</h1> <table class="table"> <thead> <tr> <th>Name</th> <th>Price</th> </tr> </thead> <tbody> @foreach (var product in Model) { <tr> <td>@product.Name</td> <td>@product.Price.ToString("C")</td> </tr> } </tbody> </table>
Q22: What is Razor?

Razor is a markup syntax for embedding server code in HTML. Razor provides a clean, lightweight syntax for combining server-side C# code with HTML markup. It uses the @ symbol to transition between HTML and C# code, making it intuitive and easy to read.

Razor Features:

  • Syntax: Uses @ symbol for code transitions (@if, @foreach, @model)
  • Code Blocks: @{ } for multi-line code blocks
  • Implicit Expressions: @VariableName automatically renders
  • HTML Encoding: Automatically encodes output to prevent XSS
  • Layouts: Supports layout pages for consistent structure
Razor.cshtml
@* Razor syntax examples *@ @model ProductViewModel @{ var title = "Product Details"; var price = Model.Price; } <h1>@title</h1> @if (Model.IsAvailable) { <p>In Stock</p> } else { <p>Out of Stock</p> } @foreach (var item in Model.Features) { <li>@item</li> } @Html.ActionLink("Edit", "Edit", new { id = Model.Id })
Q23: What is the difference between Razor and ASPX?

Razor modern, cleaner. ASPX legacy, verbose. Razor is the modern view engine for ASP.NET Core, providing a clean and concise syntax. ASPX (Web Forms) is the legacy view engine from classic ASP.NET, using verbose syntax with <% %> tags. Razor is more readable, easier to maintain, and better suited for modern web development.

Key Differences:

  • Syntax: Razor uses @ symbol; ASPX uses <% %> tags
  • Readability: Razor is cleaner and more readable; ASPX is verbose
  • Modern Support: Razor is supported in ASP.NET Core; ASPX is legacy only
  • Code Blocks: Razor uses @{ }; ASPX uses <% %>
  • Output: Razor uses @Variable; ASPX uses <%= %>
RazorvsASPX.cshtml
@* Razor (Modern) *@ @model Product @if (Model.Price > 100) { <span class="expensive">@Model.Name</span> } <%-- ASPX (Legacy) --%> <%@ Page Inherits="System.Web.Mvc.ViewPage<Product>" %> <% if (Model.Price > 100) { %> <span class="expensive"><%= Model.Name %></span> <% } %>
Q24: What is the difference between View and PartialView?

View complete page. PartialView reusable component. A View is a complete page that includes the layout and renders a full HTML document. A PartialView is a reusable component that renders a portion of HTML without the layout, typically used for shared UI elements like headers, footers, or list items.

Key Differences:

  • Layout: View includes _Layout; PartialView does not include layout
  • Structure: View is a complete page; PartialView is a fragment
  • Rendering: View renders full HTML; PartialView renders HTML fragment
  • Usage: View for main pages; PartialView for reusable components
  • AJAX: PartialView commonly used for AJAX updates
ViewvsPartial.cshtml
@* View - Complete page with layout *@ @model Product @{ ViewData["Title"] = "Product"; } <h1>@Model.Name</h1> @* PartialView - Reusable component *@ @model Product <div class="product-card"> <h3>@Model.Name</h3> <p>@Model.Price</p> </div> @* Usage *@ @Html.PartialAsync("_ProductCard", product)
Q25: What is the difference between PartialView and ViewComponent?

PartialView simple. ViewComponent with logic. PartialView is a simple reusable UI fragment without any logic. ViewComponent is a more powerful component that includes its own C# logic, can perform data retrieval, and supports dependency injection. ViewComponents are like mini controllers that render reusable UI with business logic.

Key Differences:

  • Logic: PartialView has no logic; ViewComponent has C# logic
  • Data Access: PartialView receives data from parent; ViewComponent fetches its own data
  • Dependency Injection: PartialView doesn't support DI; ViewComponent supports DI
  • Complexity: PartialView is simpler; ViewComponent is more powerful
  • Use Case: PartialView for simple UI; ViewComponent for complex reusable components
ViewComponent.cs
// ViewComponent - Has logic public class CartSummaryViewComponent : ViewComponent { private readonly ICartService _cart; public CartSummaryViewComponent(ICartService cart) { _cart = cart; } public async Task<IViewComponentResult> InvokeAsync() { var items = await _cart.GetItemsAsync(); return View(items); } } // Usage in view @await Component.InvokeAsync("CartSummary")
Q26: What is the difference between _ViewStart and _ViewImports?

_ViewStart common layout. _ViewImports common using statements. _ViewStart.cshtml is executed before each view and is used to specify a common layout file. _ViewImports.cshtml is used to include common using statements, tag helpers, and other directives that should be available in all views.

Key Differences:

  • Purpose: _ViewStart sets common layout; _ViewImports sets common directives
  • Execution: _ViewStart runs before each view; _ViewImports is included in each view
  • Content: _ViewStart contains Layout assignment; _ViewImports contains using statements
  • Scope: Both apply to views in the same folder and subfolders
  • Inheritance: Both support hierarchical folder structure
ViewFiles.cshtml
@* _ViewStart.cshtml - Common layout *@ @{ Layout = "_Layout"; } @* _ViewImports.cshtml - Common imports *@ @using MyApp.Models @using MyApp.Services @addTagHelper *, Microsoft.AspNetCore.Mvc.TagHelpers
Q27: What is the difference between _Layout and View?

_Layout master template. View content template. _Layout.cshtml is a master template that defines the common structure of your site (header, footer, navigation). Views are content templates that provide the specific page content. Views specify which layout they use, and the layout renders the view content at the RenderBody() location.

Key Differences:

  • Purpose: _Layout defines common structure; View defines page-specific content
  • Reusability: _Layout is shared across multiple views; View is unique per page
  • Content: _Layout contains header/footer; View contains page content
  • Rendering: _Layout wraps View; View is rendered inside _Layout
  • Sections: _Layout defines sections; View fills sections
Layout.cshtml
@* _Layout.cshtml - Master template *@ <!DOCTYPE html> <html> <head> @RenderSection("head", required: false) </head> <body> @RenderBody() @RenderSection("scripts", required: false) </body> </html> @* View.cshtml - Content template *@ @{ Layout = "_Layout"; } <h1>Page Content</h1>
Q28: What is the difference between RenderBody and RenderSection?

RenderBody required content. RenderSection optional content. RenderBody() is a required placeholder in the layout where the view's main content is rendered. RenderSection() is an optional placeholder that allows views to define specific sections (like scripts, styles) that the layout can render at designated locations.

Key Differences:

  • Requirement: RenderBody is required in layout; RenderSection is optional
  • Content: RenderBody renders main view content; RenderSection renders named sections
  • Multiple: Only one RenderBody per layout; Multiple RenderSections allowed
  • Optional: RenderBody cannot be optional; RenderSection can be marked optional
  • Usage: RenderBody for page content; RenderSection for scripts/styles
RenderMethods.cshtml
@* _Layout.cshtml *@ <body> @RenderBody() // Required - renders view content @RenderSection("scripts", required: false) // Optional </body> @* View.cshtml *@ @section scripts { <script src="custom.js"></script> }
Q29: What is the difference between @RenderPage and @RenderPartial?

@RenderPage renders page. @RenderPartial renders partial. @RenderPage is an older method that renders a complete page file, while @Html.Partial (or @Html.PartialAsync) is the modern method for rendering partial views. @RenderPage is rarely used in modern ASP.NET Core applications.

Key Differences:

  • Target: @RenderPage renders any page file; @Html.Partial renders partial views
  • Modern Usage: @RenderPage is legacy; @Html.PartialAsync is modern
  • Context: @RenderPage doesn't inherit parent context; @Html.Partial inherits
  • Recommendation: Use @Html.PartialAsync in modern ASP.NET Core
RenderMethods2.cshtml
@* @RenderPage - Renders a page file *@ @RenderPage("~/Shared/Header.cshtml") @* @Html.Partial - Renders a partial view *@ @Html.Partial("_Header") // Modern equivalent @await Html.PartialAsync("_Header")
Q30: What is the difference between @Html.Partial and @Html.RenderPartial?

@Html.Partial returns string. @Html.RenderPartial writes directly. @Html.Partial renders the partial view to a string and returns it, which can be stored or manipulated. @Html.RenderPartial renders the partial view directly to the response stream, which is more efficient for large partials. In ASP.NET Core, use @Html.PartialAsync or @await Html.PartialAsync instead.

Key Differences:

  • Return Type: @Html.Partial returns IHtmlContent (string-like); @Html.RenderPartial returns void
  • Output: @Html.Partial can be stored; @Html.RenderPartial writes directly to response
  • Performance: @Html.RenderPartial is more efficient for large content
  • Modern ASP.NET Core: Use @Html.PartialAsync instead of both
  • Syntax: @Html.Partial("_Partial") vs @{ Html.RenderPartial("_Partial"); }
PartialMethods.cshtml
@* @Html.Partial - Returns IHtmlContent (can be stored) *@ @{ var partialHtml = @Html.Partial("_Product"); } @partialHtml @* @Html.RenderPartial - Writes directly to response (faster) *@ @{ Html.RenderPartial("_Product"); } // Modern equivalent @await Html.PartialAsync("_Product")

4. Models

Q31: What is a model?

Model represents data and business logic. Can be domain model or view model. Models are classes that represent the data and business logic of your application. They encapsulate the application's state, validation rules, and operations. Models are independent of the user interface and can be used across different views and controllers.

Model Types:

  • Domain Model: Represents business entities and database structure
  • View Model: Contains data specifically formatted for views
  • DTO: Data Transfer Object for API communication
  • Input Model: Used for receiving user input in forms
Model.cs
// Domain Model - Business entity public class Product { public int Id { get; set; } public string Name { get; set; } public decimal Price { get; set; } } // View Model - Presentation data public class ProductViewModel { public int Id { get; set; } public string Name { get; set; } public string FormattedPrice { get; set; } public bool IsInStock { get; set; } }
Q32: What is the difference between domain model and view model?

Domain model business entity. View model presentation data. Domain models represent the core business entities and match your database structure. View models are tailored specifically for views, containing only the data needed for display and may include computed properties, formatted strings, and validation attributes.

Key Differences:

  • Purpose: Domain model for business logic; View model for presentation
  • Structure: Domain model matches database; View model matches UI requirements
  • Properties: Domain model has all fields; View model has only needed fields
  • Security: View model excludes sensitive data (passwords, internal IDs)
  • Validation: View model includes UI-specific validation attributes
DomainvsViewModel.cs
// Domain Model - Matches database/business public class User { public int Id { get; set; } public string FirstName { get; set; } public string LastName { get; set; } public string PasswordHash { get; set; } } // View Model - What the view needs public class UserViewModel { public string FullName { get; set; } // Combined public string AvatarUrl { get; set; } // Computed // No password hash exposed }
Q33: What is the difference between view model and DTO?

View model for views. DTO for data transfer. View models are designed for MVC views and may include validation attributes, display formatting, and UI-specific properties. DTOs (Data Transfer Objects) are used for API communication and service boundaries, focusing on data transfer without UI concerns.

Key Differences:

  • Purpose: View model for UI rendering; DTO for API/service communication
  • Validation: View model has validation attributes; DTO typically has minimal validation
  • Formatting: View model has formatted properties; DTO has raw data
  • Dependencies: View model may reference view-specific types; DTO is framework-agnostic
  • Usage: View model passed to Razor views; DTO serialized to JSON/XML
ViewModelvsDTO.cs
// DTO - Data Transfer Object (API) public class ProductDto { public int Id { get; set; } public string Name { get; set; } public decimal Price { get; set; } } // ViewModel - View-specific (MVC) public class ProductViewModel { public int Id { get; set; } public string Name { get; set; } public string FormattedPrice { get; set; } [Required] public string Quantity { get; set; } // For form input }
Q34: What is the difference between model and entity?

Model application layer. Entity database layer. Models represent application business logic and are independent of the database. Entities are database-specific representations used by ORM frameworks like Entity Framework Core, often decorated with database-specific attributes and containing navigation properties.

Key Differences:

  • Layer: Model belongs to application layer; Entity belongs to data layer
  • Dependencies: Model is framework-agnostic; Entity has ORM dependencies
  • Attributes: Model has business logic; Entity has database attributes
  • Navigation: Model focuses on behavior; Entity includes navigation properties
  • Usage: Model used throughout application; Entity used for database operations
ModelvsEntity.cs
// Entity - Database representation (EF Core) [Table("Products")] public class ProductEntity { [Key] public int Id { get; set; } public string Name { get; set; } public decimal Price { get; set; } } // Model - Application logic public class Product { public int Id { get; set; } public string Name { get; set; } public decimal Price { get; set; } public decimal CalculateTax() { return Price * 0.08m; } }
Q35: What is the difference between POCO and entity?

POCO plain class. Entity with EF tracking. POCO (Plain Old CLR Object) is a simple class with no dependencies on any framework, making it lightweight and testable. Entities in Entity Framework are classes that are tracked by the context and may include navigation properties and change tracking capabilities.

Key Differences:

  • Dependencies: POCO has no framework dependencies; Entity has EF Core dependencies
  • Tracking: POCO is not tracked; Entity is tracked by DbContext
  • Navigation: POCO has simple properties; Entity has navigation properties
  • Inheritance: POCO is standalone; Entity may inherit from EntityBase
  • Testability: POCO is easier to test; Entity requires context setup
POCOvsEntity.cs
// POCO - Plain Old CLR Object (no EF dependencies) public class Product { public int Id { get; set; } public string Name { get; set; } } // Entity - EF Core tracked class public class Product : EntityBase { public int Id { get; set; } public string Name { get; set; } public virtual ICollection<Order> Orders { get; set; } // Navigation property }
Q36: What is the difference between ViewModel and TempData?

ViewModel data structure. TempData state storage. ViewModel is a class that contains data to be displayed in a view, passed from controller to view. TempData is a dictionary object used to store data that persists across HTTP redirects, typically used for passing messages or temporary state between requests.

Key Differences:

  • Purpose: ViewModel structures data for display; TempData stores temporary state
  • Lifetime: ViewModel exists for single request; TempData persists across redirects
  • Type Safety: ViewModel is strongly-typed; TempData is weakly-typed (requires casting)
  • Usage: ViewModel passed to View(); TempData accessed via TempData["key"]
  • Scope: ViewModel is view-specific; TempData is request-scoped
ViewModelvsTempData.cs
// ViewModel - Passed to view public class CreateProductViewModel { public string Name { get; set; } public decimal Price { get; set; } } public IActionResult Create() { return View(new CreateProductViewModel()); } // TempData - Persists across redirects [HttpPost] public IActionResult Create(CreateProductViewModel model) { TempData["Message"] = "Product created!"; return RedirectToAction("Index"); }
Q37: What is the difference between ViewBag and ViewData?

ViewBag dynamic. ViewData dictionary. ViewBag is a dynamic wrapper around ViewData, providing a cleaner syntax using dot notation. ViewData is a dictionary object that uses string keys to store values. Both are used to pass data from controller to view, but ViewBag is more convenient while ViewData provides slightly better performance.

Key Differences:

  • Type: ViewBag is dynamic; ViewData is a dictionary
  • Syntax: ViewBag uses dot notation; ViewData uses string keys
  • Type Safety: Neither is type-safe; both require casting in views
  • Compile-time: ViewBag has no compile-time checking; ViewData has key checking
  • Performance: ViewData is slightly faster; ViewBag has dynamic overhead
ViewBagvsViewData.cs
// ViewBag - Dynamic (no compile-time checking) public IActionResult Index() { ViewBag.Title = "Home"; ViewBag.UserCount = 42; return View(); } // ViewData - Dictionary (requires casting) public IActionResult Index() { ViewData["Title"] = "Home"; ViewData["UserCount"] = 42; return View(); } // In view @ViewBag.Title @((int)ViewData["UserCount"])
Q38: What is the difference between ViewData and TempData?

ViewData current request. TempData current and next request. ViewData is used to pass data from controller to view within the same HTTP request. TempData is used to pass data that persists across HTTP redirects, making it available in the next request. TempData is ideal for passing success/error messages after a POST-Redirect-GET pattern.

Key Differences:

  • Lifetime: ViewData exists only for current request; TempData persists to next request
  • Redirects: ViewData lost on redirect; TempData survives redirects
  • Storage: ViewData stored in controller context; TempData uses session/cookies
  • Use Case: ViewData for view data; TempData for redirect messages
  • Clearing: ViewData cleared after view renders; TempData cleared after being read
ViewDatavsTempData.cs
// ViewData - Current request only public IActionResult Index() { ViewData["Message"] = "Hello"; return View(); } // TempData - Persists to next request (redirect) [HttpPost] public IActionResult Create() { TempData["Success"] = "Created!"; return RedirectToAction("Index"); } // In Index view after redirect @TempData["Success"]
Q39: What is the difference between TempData and Session?

TempData short-lived. Session user session. TempData is designed for short-lived data that persists only until the next request, automatically cleared after being read. Session stores data for the entire user session and persists across multiple requests until the session expires or is explicitly abandoned. Session is used for user-specific data like authentication or shopping cart.

Key Differences:

  • Lifetime: TempData lasts until next request; Session lasts for user session
  • Clearing: TempData auto-clears after read; Session requires explicit removal
  • Scope: TempData for redirect scenarios; Session for user-specific data
  • Storage: TempData uses session/cookies with short TTL; Session uses configured storage
  • Use Case: TempData for messages; Session for user state
TempDatavsSession.cs
// TempData - Available until read (short-lived) TempData["Message"] = "Success"; // Cleared after being read in next request // Session - Persists for user session (longer-lived) HttpContext.Session.SetString("UserId", "123"); var userId = HttpContext.Session.GetString("UserId"); // Persists until session expires or abandoned
Q40: What is the difference between Session and Cache?

Session per user. Cache shared. Session stores user-specific data that is isolated to each user and persists for that user's session. Cache stores application-wide data that is shared across all users and can be used to improve performance by storing frequently accessed data. Session is for user state; Cache is for application data.

Key Differences:

  • Scope: Session is per-user; Cache is application-wide (shared)
  • Isolation: Session data is private to user; Cache data is shared
  • Expiry: Session expires on timeout/abandon; Cache expires on configured policy
  • Use Case: Session for user preferences/auth; Cache for database queries/config
  • Storage: Session uses distributed storage; Cache uses memory/Redis
SessionvsCache.cs
// Session - Per user (isolated) HttpContext.Session.SetString("CartId", Guid.NewGuid().ToString()); // Each user has their own session // Cache - Shared across all users _cache.Set("Products", products, TimeSpan.FromMinutes(30)); // All users share the same cached data

5. Routing

Q41: What is routing?

Routing maps URL patterns to controller actions. Routing is the process of matching incoming HTTP requests to controller action methods. It enables clean, SEO-friendly URLs by mapping URL patterns to specific controllers and actions. ASP.NET Core supports both conventional routing (defined centrally) and attribute routing (defined on controllers/actions).

Routing Features:

  • URL Mapping: Maps URLs to controller actions
  • Pattern Matching: Supports route parameters and constraints
  • Types: Conventional routing and attribute routing
  • Middleware: Implemented as UseRouting() middleware
  • Flexibility: Supports custom routes and route templates
Routing.cs
// Program.cs - Conventional routing builder.Services.AddControllersWithViews(); var app = builder.Build(); app.UseRouting(); app.UseEndpoints(endpoints => { endpoints.MapControllerRoute( name: "default", pattern: "{controller=Home}/{action=Index}/{id?}"); }); // Attribute routing [Route("api/[controller]")] public class ProductsController : Controller { [HttpGet("{id:int}")] public IActionResult Get(int id) { return Json(new { Id = id }); } }
Q42: What is the difference between attribute and conventional routing?

Attribute on controllers. Conventional in Startup. Attribute routing is defined directly on controllers and actions using attributes like [Route] and [HttpGet], providing fine-grained control over individual routes. Conventional routing is defined centrally in Program.cs using MapControllerRoute, following a consistent pattern across the application.

Key Differences:

  • Location: Attribute routing on controllers/actions; Conventional routing in Program.cs
  • Granularity: Attribute routing is per-action; Conventional routing is global
  • Flexibility: Attribute routing more flexible; Conventional routing more consistent
  • Use Case: Attribute routing for APIs; Conventional routing for MVC apps
  • Order: Attribute routes evaluated before conventional routes
RoutingTypes.cs
// Conventional routing (Program.cs) app.UseEndpoints(endpoints => { endpoints.MapControllerRoute( name: "default", pattern: "{controller=Home}/{action=Index}/{id?}"); }); // Attribute routing (on controller) [Route("api/[controller]")] public class ProductsController : Controller { [HttpGet("{id:int}")] public IActionResult Get(int id) => Ok(id); }
Q43: What is the difference between [Route] and [HttpGet]?

[Route] defines route. [HttpGet] defines HTTP method. [Route] attribute defines the URL pattern for an action but doesn't restrict the HTTP method. [HttpGet] attribute restricts the action to respond only to HTTP GET requests and can also define the route pattern. [HttpGet] is more specific as it combines both HTTP method restriction and route definition.

Key Differences:

  • Purpose: [Route] defines URL pattern; [HttpGet] defines HTTP method + route
  • HTTP Method: [Route] accepts any method; [HttpGet] only accepts GET
  • Specificity: [Route] is generic; [HttpGet] is method-specific
  • Combination: [Route] can be combined with [HttpGet]; [HttpGet] includes route
  • Use Case: [Route] for custom routes; [HttpGet] for RESTful APIs
RouteAttributes.cs
// [Route] - Defines URL pattern only [Route("products/{id}")] public IActionResult GetProduct(int id) { return Ok(id); } // [HttpGet] - Defines HTTP method + route [HttpGet("products/{id}")] public IActionResult GetProduct(int id) { return Ok(id); } // Combined [Route("api/[controller]")] public class ProductsController { [HttpGet("{id}")] // Responds to GET /api/products/{id} public IActionResult Get(int id) => Ok(id); }
Q44: What is the difference between route parameters and query strings?

Route parameters in URL path. Query strings after ?. Route parameters are embedded in the URL path and are used for required or optional resource identifiers. Query strings appear after the ? in the URL and are used for optional parameters, filters, or search criteria. Route parameters are more SEO-friendly and hierarchical, while query strings are flexible for optional data.

Key Differences:

  • Location: Route parameters in path; Query strings after ?
  • Purpose: Route parameters for resource identification; Query strings for options/filters
  • SEO: Route parameters are SEO-friendly; Query strings less so
  • Binding: Route parameters use [FromRoute]; Query strings use [FromQuery]
  • Example: /products/123 vs /products?id=123
RouteParams.cs
// Route parameter - in URL path [HttpGet("products/{id}")] public IActionResult GetProduct(int id) { // URL: /products/123 return Ok(id); } // Query string - after ? [HttpGet("products")] public IActionResult Search([FromQuery] string query) { // URL: /products?query=laptop return Ok(query); } // Combined [HttpGet("products/{id}")] public IActionResult GetProduct(int id, [FromQuery] bool details = false) { // URL: /products/123?details=true return Ok(new { Id = id, Details = details }); }
Q45: What is the difference between optional and required parameters?

Optional has default. Required must be provided. Optional parameters have a default value and can be omitted from the URL, allowing the route to match with or without the parameter. Required parameters must be present in the URL for the route to match; if missing, the route returns 404. Optional parameters are marked with ? in the route template or have a default value.

Key Differences:

  • Requirement: Optional can be omitted; Required must be present
  • Default Value: Optional has default value; Required has no default
  • Route Template: Optional uses {param?}; Required uses {param}
  • Matching: Optional matches multiple patterns; Required matches one pattern
  • Use Case: Optional for filters/pagination; Required for resource IDs
OptionalParams.cs
// Optional parameter - has default value [HttpGet("products/{id?}")] public IActionResult GetProduct(int? id = null) { // Matches: /products or /products/123 return Ok(id); } // Required parameter - must be provided [HttpGet("products/{id:int}")] public IActionResult GetProduct(int id) { // Matches: /products/123 only // /products returns 404 return Ok(id); }
Q46: What is the difference between route constraints and validation?

Route constraints URL matching. Validation data validation. Route constraints restrict which URLs match a route based on parameter types (e.g., int, guid, datetime). They operate during routing before the action is selected. Validation checks the data after model binding to ensure it meets business rules. Route constraints prevent invalid URLs from reaching the action; validation ensures data integrity.

Key Differences:

  • Timing: Route constraints during routing; Validation after model binding
  • Purpose: Route constraints for URL matching; Validation for data integrity
  • Scope: Route constraints affect route selection; Validation affects ModelState
  • Examples: Route constraints: {id:int}, {guid:guid}; Validation: [Required], [Range]
  • Failure: Route constraint failure = 404; Validation failure = 400 with errors
Constraints.cs
// Route constraints - URL matching only [HttpGet("products/{id:int}")] public IActionResult GetProduct(int id) { // /products/123 matches // /products/abc returns 404 (not matched) return Ok(id); } // Validation - Data validation after binding public class ProductViewModel { [Range(1, 1000)] public int Id { get; set; } } [HttpPost] public IActionResult Create(ProductViewModel model) { if (!ModelState.IsValid) { // Id > 1000 fails validation return BadRequest(ModelState); } return Ok(); }
Q47: What is the difference between [Area] and [Controller]?

[Area] organizes controllers. [Controller] specifies controller. [Area] attribute is used to organize controllers into logical groups or modules (e.g., Admin, Blog, Store), allowing for better organization and routing separation. [Controller] is a route token that gets replaced with the controller name at runtime, used in route templates to create dynamic routes based on the controller class name.

Key Differences:

  • Purpose: [Area] organizes controllers into modules; [Controller] is a route token
  • Usage: [Area] is an attribute on controllers; [Controller] is used in route templates
  • Routing: [Area] adds area prefix to routes; [Controller] substitutes controller name
  • Scope: [Area] affects multiple controllers; [Controller] affects route pattern
  • Example: [Area("Admin")] vs [Route("api/[controller]")]
AreaController.cs
// [Area] - Organizes controllers into areas [Area("Admin")] [Route("admin/[controller]/[action]")] public class ProductsController : Controller { // URL: /admin/products/index public IActionResult Index() => View(); } // [Controller] - Token replacement for controller name [Route("api/[controller]")] public class ProductsController : Controller { // [controller] replaced with "products" // URL: /api/products public IActionResult Get() => Ok(); }
Q48: What is the difference between UseRouting and UseEndpoints?

UseRouting matches routes. UseEndpoints executes. UseRouting middleware determines which endpoint (controller action, Razor page, etc.) matches the incoming request URL but doesn't execute it. UseEndpoints middleware executes the matched endpoint. This separation allows middleware like authentication and authorization to run between route matching and endpoint execution.

Key Differences:

  • Function: UseRouting matches URL to endpoint; UseEndpoints executes the endpoint
  • Timing: UseRouting runs first; UseEndpoints runs after other middleware
  • Middleware: UseRouting enables endpoint routing; UseEndpoints defines endpoints
  • Separation: Allows auth/authorization middleware between them
  • Configuration: UseRouting has no parameters; UseEndpoints takes endpoint configuration
RoutingPipeline.cs
var app = builder.Build(); // UseRouting - Matches URL to route (doesn't execute) app.UseRouting(); // Middleware between routing and endpoints app.UseAuthentication(); app.UseAuthorization(); // UseEndpoints - Executes the matched route app.UseEndpoints(endpoints => { endpoints.MapControllerRoute("default", "{controller}/{action}"); endpoints.MapRazorPages(); });
Q49: What is the difference between MapControllerRoute and MapAreaControllerRoute?

MapControllerRoute standard. MapAreaControllerRoute for areas. MapControllerRoute is used for standard routing without area support, mapping URL patterns to controller actions. MapAreaControllerRoute is specifically designed for area-based routing, automatically adding the area name to the route pattern and associating controllers with specific areas.

Key Differences:

  • Purpose: MapControllerRoute for standard routing; MapAreaControllerRoute for area routing
  • Area Support: MapControllerRoute no area support; MapAreaControllerRoute includes area prefix
  • Pattern: MapControllerRoute uses {controller}/{action}; MapAreaControllerRoute uses {area}/{controller}/{action}
  • Controller Association: MapControllerRoute matches any controller; MapAreaControllerRoute matches area controllers
  • Use Case: MapControllerRoute for simple apps; MapAreaControllerRoute for modular apps
RouteMapping.cs
// MapControllerRoute - Standard routing endpoints.MapControllerRoute( name: "default", pattern: "{controller=Home}/{action=Index}/{id?}"); // MapAreaControllerRoute - Area routing endpoints.MapAreaControllerRoute( name: "admin", areaName: "Admin", pattern: "Admin/{controller}/{action}/{id?}"); // Combined with area route endpoints.MapControllerRoute( name: "areas", pattern: "{area:exists}/{controller}/{action}/{id?}");
Q50: What is the difference between fallback routing and conventional routing?

Fallback catch-all. Conventional specific patterns. Fallback routing is a catch-all mechanism that handles requests that don't match any other route, commonly used for Single Page Applications (SPAs) to return index.html for client-side routing. Conventional routing defines specific URL patterns that must match exactly for the route to be selected.

Key Differences:

  • Matching: Fallback matches unmatched routes; Conventional matches specific patterns
  • Priority: Fallback has lowest priority; Conventional routes evaluated first
  • Use Case: Fallback for SPAs/404 handling; Conventional for standard MVC
  • Methods: Fallback uses MapFallbackToFile/MapFallbackToController; Conventional uses MapControllerRoute
  • Flexibility: Fallback is broad; Conventional is precise
FallbackRouting.cs
// Conventional routing - specific patterns endpoints.MapControllerRoute( name: "products", pattern: "products/{action}/{id?}"); // Fallback routing - catch-all for SPA endpoints.MapFallbackToFile("index.html"); // Or for controller endpoints.MapFallbackToController("Index", "Home"); // Matches any unmatched route

6. Model Binding

Q51: What is model binding?

Model binding maps HTTP request data to action parameters. Model binding is the process of converting HTTP request data (form fields, query strings, route parameters, JSON body) into .NET objects that can be used as action method parameters. It eliminates manual data extraction and type conversion, making controllers cleaner and more maintainable.

Model Binding Features:

  • Automatic: Automatically maps request data to parameters
  • Type Conversion: Converts string data to appropriate types
  • Sources: Supports form, query, route, body, header
  • Validation: Integrates with model validation
  • Customizable: Supports custom binders and attributes
ModelBinding.cs
public class Product { public int Id { get; set; } public string Name { get; set; } public decimal Price { get; set; } } // Model binding from form data [HttpPost] public IActionResult Create(Product product) { // product is automatically bound from form data return Ok(product); } // Model binding with attributes [HttpGet("search")] public IActionResult Search( [FromQuery] string query, [FromRoute] int page = 1) { return Ok(new { query, page }); }
Q52: What are binding sources?

Binding sources: Form, Query, Route, Body, Header. Binding sources specify where model binding should look for data in the HTTP request. Each source attribute tells the model binder to extract data from a specific location: [FromForm] for form data, [FromQuery] for query strings, [FromRoute] for route parameters, [FromBody] for request body (JSON), and [FromHeader] for HTTP headers.

Binding Sources:

  • [FromForm]: Binds from form-urlencoded data
  • [FromQuery]: Binds from query string parameters
  • [FromRoute]: Binds from route parameters
  • [FromBody]: Binds from request body (JSON/XML)
  • [FromHeader]: Binds from HTTP headers
  • [FromServices]: Injects services from DI container
BindingSources.cs
// FromForm - Form data [HttpPost] public IActionResult Create([FromForm] Product product) { // Binds from form-urlencoded data return Ok(product); } // FromQuery - Query string [HttpGet] public IActionResult Search([FromQuery] string q) { // Binds from ?q=value return Ok(q); } // FromRoute - Route parameter [HttpGet("{id}")] public IActionResult Get([FromRoute] int id) { // Binds from /123 return Ok(id); } // FromBody - Request body (JSON) [HttpPost] public IActionResult Create([FromBody] Product product) { // Binds from JSON body return Ok(product); } // FromHeader - HTTP header [HttpGet] public IActionResult Get([FromHeader] string authorization) { // Binds from Authorization header return Ok(authorization); }
Q53: What is the difference between FromBody and FromForm?

FromBody JSON/XML. FromForm form data. [FromBody] binds data from the HTTP request body, typically used for JSON or XML payloads in API scenarios. [FromForm] binds data from form-urlencoded data, typically used for HTML form submissions. FromBody uses formatters to deserialize the body; FromForm uses name-value pair matching.

Key Differences:

  • Content Type: FromBody for application/json; FromForm for application/x-www-form-urlencoded
  • Format: FromBody uses JSON/XML structure; FromForm uses key-value pairs
  • Use Case: FromBody for APIs; FromForm for HTML forms
  • Complexity: FromBody supports complex nested objects; FromForm for flat structures
  • Formatter: FromBody uses input formatters; FromForm uses form value provider
BodyvsForm.cs
// FromBody - JSON/XML in request body [HttpPost] public IActionResult Create([FromBody] Product product) { // Content-Type: application/json // Body: { "name": "Laptop", "price": 999 } return Ok(product); } // FromForm - Form data (x-www-form-urlencoded) [HttpPost] public IActionResult Create([FromForm] Product product) { // Content-Type: application/x-www-form-urlencoded // Body: name=Laptop&price=999 return Ok(product); }
Q54: What is the difference between FromQuery and FromRoute?

FromQuery query string. FromRoute route parameters. [FromQuery] binds data from the query string portion of the URL (after the ?), used for optional parameters like filters, search terms, or pagination. [FromRoute] binds data from the route parameters embedded in the URL path, used for required resource identifiers. FromQuery is optional; FromRoute is typically required.

Key Differences:

  • Location: FromQuery from query string (?key=value); FromRoute from URL path (/value)
  • Optionality: FromQuery parameters are optional; FromRoute parameters are typically required
  • SEO: FromQuery less SEO-friendly; FromRoute more SEO-friendly
  • Use Case: FromQuery for filters/options; FromRoute for resource IDs
  • Example: /products?page=2 vs /products/2
QueryvsRoute.cs
// FromQuery - Query string parameter [HttpGet("products")] public IActionResult Get([FromQuery] int page = 1) { // URL: /products?page=2 return Ok(page); } // FromRoute - Route parameter [HttpGet("products/{page}")] public IActionResult Get([FromRoute] int page) { // URL: /products/2 return Ok(page); }
Q55: What is the difference between FromHeader and FromBody?

FromHeader HTTP headers. FromBody request body. [FromHeader] binds data from HTTP request headers, used for metadata like authorization tokens, content types, or custom headers. [FromBody] binds data from the request body payload, used for the main data being sent (JSON/XML). FromHeader is for metadata; FromBody is for content.

Key Differences:

  • Location: FromHeader from HTTP headers; FromBody from request body
  • Purpose: FromHeader for metadata; FromBody for content data
  • Size: FromHeader for small values; FromBody for large payloads
  • Use Case: FromHeader for auth/tokens; FromBody for business data
  • Multiple: FromHeader can bind multiple headers; FromBody typically one parameter
HeadervsBody.cs
// FromHeader - HTTP header [HttpGet] public IActionResult Get([FromHeader("Authorization")] string auth) { // Header: Authorization: Bearer token123 return Ok(auth); } // FromBody - Request body [HttpPost] public IActionResult Create([FromBody] Product product) { // Body: { "name": "Laptop" } return Ok(product); }
Q56: What is model validation?

Model validation validates model properties using data annotations. Model validation is the process of ensuring that data submitted to the application meets specified business rules and constraints. It uses data annotation attributes to define validation rules on model properties, automatically validating data during model binding. Validation errors are stored in ModelState and can be displayed to users.

Model Validation Features:

  • Declarative: Uses attributes to define validation rules
  • Automatic: Runs automatically during model binding
  • ModelState: Validation errors stored in ModelState dictionary
  • Client-side: Supports client-side validation with jQuery
  • Customizable: Supports custom validation attributes
Validation.cs
public class ProductViewModel { [Required] [StringLength(100)] public string Name { get; set; } [Range(0.01, 10000)] public decimal Price { get; set; } [EmailAddress] public string Email { get; set; } } [HttpPost] public IActionResult Create(ProductViewModel model) { if (!ModelState.IsValid) { return View(model); } // Process valid data return RedirectToAction("Index"); }
Q57: What are data annotations?

Data annotations: [Required], [StringLength], [Range], etc. Data annotations are attributes that can be applied to class properties to define validation rules and metadata. They provide a declarative way to specify validation requirements, display names, and formatting rules. Common annotations include [Required], [StringLength], [Range], [EmailAddress], [RegularExpression], and [Compare].

Common Data Annotations:

  • [Required]: Property must have a value
  • [StringLength]: Restricts string length
  • [Range]: Restricts numeric value range
  • [EmailAddress]: Validates email format
  • [RegularExpression]: Validates against regex pattern

7. Filters

Q61: What are filters?

Filters add logic before/after action execution. Filters are custom classes that implement cross-cutting concerns like authentication, authorization, logging, caching, and error handling. They allow you to run code before or after specific stages in the request processing pipeline. ASP.NET Core provides several filter types: Authorization, Resource, Action, Exception, and Result filters.

Filter Types:

Q62: What are the types of filters?

Filter types: Authorization, Resource, Action, Exception, Result. ASP.NET Core provides five types of filters that run at different stages of the request pipeline. Authorization filters run first to determine if the request is authorized. Resource filters run after authorization and can short-circuit the pipeline. Action filters run before and after action execution. Exception filters handle unhandled exceptions. Result filters run before and after action result execution.

Filter Execution Order:

  • 1. Authorization: Runs first, checks if user is authorized
  • 2. Resource: Runs after authorization, before model binding
  • 3. Action: Runs before and after action method execution
  • 4. Exception: Runs when an exception is thrown
  • 5. Result: Runs before and after action result execution

Q63: What is the difference between Authorization and Action filter?

Authorization before action. Action before/after action. Authorization filters run first in the pipeline, before model binding, to determine if the user is authorized to access the resource. They can short-circuit the request. Action filters run after authorization and model binding, before and after the action method executes. Authorization is for access control; Action is for action-specific logic.

Key Differences:

  • Timing: Authorization runs first; Action runs after authorization
  • Model Binding: Authorization before model binding; Action after model binding
  • Execution: Authorization runs once; Action runs before/after action
  • Use Case: Authorization for access control; Action for logging/validation
  • Short-circuit: Both can short-circuit, but Authorization is primary

Q64: What is the difference between Resource and Result filter?

Resource before/after action. Result before/after result. Resource filters run after authorization but before model binding and action execution, often used for caching or short-circuiting the pipeline. Result filters run after the action has executed but before and after the action result is executed, used to modify the result. Resource filters wrap the entire action pipeline; Result filters wrap only the result execution.

Key Differences:

  • Timing: Resource runs before model binding; Result runs after action
  • Scope: Resource wraps entire pipeline; Result wraps result execution
  • Model Binding: Resource before model binding; Result after model binding
  • Use Case: Resource for caching/short-circuit; Result for response modification
  • Access: Resource can prevent action execution; Result can modify response

8. Dependency Injection

Q71: What is dependency injection?

DI is a design pattern for achieving Inversion of Control. Dependency Injection is a design pattern where dependencies are provided to a class rather than the class creating them itself. This promotes loose coupling, easier testing, and better maintainability. ASP.NET Core has built-in DI support that manages service lifetimes and automatically injects dependencies into constructors.

DI Benefits:

  • Loose Coupling: Reduces dependencies between components
  • Testability: Easier to unit test with mock dependencies
  • Maintainability: Centralized service registration
  • Flexibility: Easy to swap implementations
  • Lifetime Management: Built-in support for service lifetimes
DI.cs
// Register services in Program.cs var builder = WebApplication.CreateBuilder(args); builder.Services.AddTransient<IProductService, ProductService>(); builder.Services.AddScoped<IRepository, Repository>(); builder.Services.AddSingleton<ICacheService, CacheService>(); // Inject into controller public class ProductController : Controller { private readonly IProductService _service; public ProductController(IProductService service) { _service = service; } public IActionResult Index() { var products = _service.GetAll(); return View(products); } }
Q72: What are DI lifetimes?

Lifetimes: Transient, Scoped, Singleton. DI lifetimes determine how long a service instance lives and when a new instance is created. Transient creates a new instance every time it's requested. Scoped creates a new instance once per HTTP request. Singleton creates a single instance for the entire application lifetime. Choosing the right lifetime is crucial for correct behavior and performance.

Service Lifetimes:

  • Transient: New instance each time it's requested
  • Scoped: New instance per HTTP request
  • Singleton: Single instance for app lifetime
  • Transient Use: Lightweight, stateless services
  • Scoped Use: Services with request-specific state
  • Singleton Use: Caching, configuration, stateless services
ServiceLifetimes.cs
// Transient - New instance each time requested builder.Services.AddTransient<IProductService, ProductService>(); // Scoped - New instance per HTTP request builder.Services.AddScoped<IUserService, UserService>(); // Singleton - Single instance for app lifetime builder.Services.AddSingleton<ICacheService, CacheService>();
Q73: What is the difference between Transient and Scoped?

Transient new each time. Scoped same per request. Transient services create a new instance every time they are injected, even within the same HTTP request. Scoped services create a new instance once per HTTP request and reuse that instance throughout the request. Transient is for lightweight, stateless services; Scoped is for services that maintain state within a request context.

Key Differences:

  • Instance Creation: Transient creates new each injection; Scoped creates once per request
  • State Sharing: Transient no state sharing; Scoped shares state within request
  • Performance: Transient more instances; Scoped fewer instances
  • Thread Safety: Transient thread-safe by design; Scoped needs thread safety
  • Use Case: Transient for stateless; Scoped for request-specific state
TransientvsScoped.cs
// Transient - Different instance each injection builder.Services.AddTransient<GuidService>(); // Scoped - Same instance within HTTP request builder.Services.AddScoped<GuidService>(); public class HomeController : Controller { // Transient: t1.Id != t2.Id // Scoped: s1.Id == s2.Id }
Q74: What is the difference between Scoped and Singleton?

Scoped per request. Singleton per application. Scoped services create a new instance for each HTTP request and are disposed at the end of the request. Singleton services create a single instance when first requested and live for the entire application lifetime. Scoped is for request-specific data; Singleton is for application-wide shared resources. Never inject Scoped into Singleton (captive dependency problem).

Key Differences:

  • Lifetime: Scoped per HTTP request; Singleton per application
  • Disposal: Scoped disposed after request; Singleton never disposed
  • State: Scoped isolated per request; Singleton shared globally
  • Thread Safety: Scoped request-scoped; Singleton requires thread safety
  • Use Case: Scoped for DbContext; Singleton for caching/configuration
ScopedvsSingleton.cs
// Scoped - New per HTTP request builder.Services.AddScoped<DbContext, AppDbContext>(); // Singleton - Single instance for entire app builder.Services.AddSingleton<ICacheService, MemoryCacheService>(); // WARNING: Don't inject Scoped into Singleton! // This causes captive dependency problem
Q75: How do you inject services into controllers?

Use constructor injection with service parameters. Services are injected into controllers through constructor injection, where dependencies are declared as parameters in the controller's constructor. The DI container automatically provides the registered services. This is the recommended approach as it makes dependencies explicit and ensures the controller has all required dependencies to function.

Injection Best Practices:

  • Explicit Dependencies: Make all dependencies visible in constructor
  • Interface Segregation: Depend on abstractions, not concretions

Q91: What are areas?

Areas organize large MVC applications into functional groups. Areas are a way to partition a large MVC application into smaller functional groupings, each with its own set of controllers, views, and models. Each area functions as a mini-MVC application within the main application. Areas help organize code for large projects by separating distinct functional modules like Admin, Sales, or Blog into their own folders.

Area Benefits:

  • Organization: Separates large apps into functional modules
  • Structure: Each area has its own Controllers and Views
  • Maintenance: Easier to maintain and navigate large codebases
  • Routing: Area-specific URL patterns (e.g., /Admin/Home/Index)
  • Team Development: Multiple teams can work on different areas
