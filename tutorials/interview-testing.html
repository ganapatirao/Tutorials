1. Testing Fundamentals

Q1: What is software testing and why is it important?

Software Testing is the process of evaluating a software application to find defects and ensure it meets the specified requirements. It involves executing the software with the intent of finding bugs.

Importance of Testing:

  • Quality Assurance: Ensures software meets quality standards
  • Cost Reduction: Finds bugs early, reducing fix costs
  • User Satisfaction: Delivers reliable and bug-free software
  • Security: Identifies security vulnerabilities
  • Performance: Ensures software performs well under load
  • Compliance: Meets regulatory and industry standards
Q2: What are the different levels of testing?

Testing Levels:

  1. Unit Testing: Testing individual components or functions in isolation
  2. Integration Testing: Testing how different components work together
  3. System Testing: Testing the complete system as a whole
  4. Acceptance Testing: Testing to ensure the system meets user requirements
Q3: What is the difference between verification and validation?
Aspect Verification Validation
Definition Are we building the product right? Are we building the right product?
Focus Process and specifications End product and user needs
Timing During development After development
Activities Reviews, inspections, walkthroughs Testing, user acceptance
Q4: What is the testing pyramid?

Testing Pyramid is a strategy that suggests having a large number of low-level unit tests, fewer integration tests, and even fewer end-to-end tests.

Pyramid Levels:

  • Base (70%): Unit tests - fast, isolated, cover business logic
  • Middle (20%): Integration tests - test component interactions
  • Top (10%): E2E tests - test complete user flows

Benefits:

  • Faster feedback due to more unit tests
  • Lower maintenance cost
  • Better test coverage
  • Reliable and stable tests

2. Manual Testing

Q5: What is manual testing and when should you use it?

Manual Testing is the process of manually testing software for defects without using automated tools. Testers execute test cases manually and compare actual results with expected results.

When to Use Manual Testing:

  • Exploratory testing to discover unknown issues
  • Usability testing to evaluate user experience
  • Ad-hoc testing for quick checks
  • When automation is not cost-effective
  • Testing complex visual elements
  • One-time testing scenarios
Q6: What are the different types of manual testing?

Types of Manual Testing:

  • Black Box Testing: Testing without knowledge of internal code
  • White Box Testing: Testing with knowledge of internal code
  • Gray Box Testing: Testing with partial knowledge of code
  • Exploratory Testing: Learning while testing
  • Usability Testing: Testing user experience
  • Ad-hoc Testing: Informal testing without planning
Q7: What is a test case and what are its components?

Test Case is a set of conditions or variables under which a tester will determine whether a system under test satisfies requirements.

Test Case Components:

  • Test Case ID: Unique identifier
  • Title: Brief description of the test
  • Description: Detailed explanation
  • Preconditions: Conditions before test execution
  • Test Steps: Step-by-step instructions
  • Test Data: Data required for testing
  • Expected Result: Anticipated outcome
  • Actual Result: Actual outcome after execution
  • Status: Pass/Fail

3. Automation Testing

Q8: What is automation testing and its benefits?

Automation Testing uses software tools to execute test cases automatically, comparing actual results with expected results. It involves writing scripts to test the application.

Benefits:

  • Speed: Faster execution of test cases
  • Reusability: Test scripts can be reused
  • Reliability: Eliminates human error
  • Coverage: Can test more scenarios
  • Cost-effective: Reduces long-term testing costs
  • CI/CD Integration: Enables continuous testing
Q9: When should you automate tests?

When to Automate:

  • Repetitive test cases that run frequently
  • Regression testing suites
  • Smoke and sanity tests
  • Performance and load testing
  • Tests with predictable results
  • Tests that require multiple data sets

When NOT to Automate:

  • Exploratory testing
  • Usability testing
  • One-time test cases
  • Frequently changing UI
  • Tests that require human judgment

Reliability Prone to human error More reliable
Coverage Limited Extensive

4. Unit Testing

Q11: What is unit testing?

Unit Testing is the process of testing individual components or functions of a software in isolation. It verifies that each unit of code performs as expected.

Characteristics:

  • Tests individual methods or functions
  • Fast execution
  • Isolated from dependencies (using mocks)
  • Written by developers
  • Run frequently during development
CalculatorTests.cs
// Unit test using xUnit in C# public class CalculatorTests { [Fact] public void Add_WhenCalled_ReturnsSum() { // Arrange var calculator = new Calculator(); // Act var result = calculator.Add(5, 3); // Assert Assert.Equal(8, result); } [Theory] [InlineData(5, 3, 8)] [InlineData(-5, 3, -2)] [InlineData(0, 0, 0)] public void Add_WithVariousInputs_ReturnsCorrectSum(int a, int b, int expected) { var calculator = new Calculator(); var result = calculator.Add(a, b); Assert.Equal(expected, result); } }
Q12: What are the AAA principles in unit testing?

AAA (Arrange-Act-Assert) is a pattern for writing unit tests that organizes the test into three clear sections.

AAA Pattern:

  • Arrange: Set up the test data, objects, and dependencies
  • Act: Execute the method being tested
  • Assert: Verify the result matches expectations
DiscountTests.cs
// AAA pattern example [Fact] public void CalculateDiscount_WithValidCustomer_ReturnsDiscount() { // Arrange var customer = new Customer { IsPremium = true }; var calculator = new DiscountCalculator(); // Act var discount = calculator.CalculateDiscount(customer, 100); // Assert Assert.Equal(20, discount); }
Q13: What is mocking and why is it used?

Mocking is creating fake objects that simulate the behavior of real objects. It's used to isolate the unit under test from its dependencies.

Benefits of Mocking:

  • Isolates the unit being tested
  • Eliminates external dependencies (database, API)
  • Makes tests faster
  • Allows testing edge cases
  • Verifies interaction between components
OrderServiceTests.cs
// Mocking using Moq in C# public class OrderServiceTests { [Fact] public async Task CreateOrder_WithValidData_ReturnsOrderId() { // Arrange var mockRepository = new Mock<IOrderRepository>(); var mockEmailService = new Mock<IEmailService>(); mockRepository .Setup(r => r.Add(It.IsAny<Order>())) .ReturnsAsync(Guid.NewGuid()); var service = new OrderService(mockRepository.Object, mockEmailService.Object); // Act var result = await service.CreateOrderAsync(new Order()); // Assert Assert.NotEqual(Guid.Empty, result); mockRepository.Verify(r => r.Add(It.IsAny<Order>()), Times.Once); } }

5. Integration Testing

Q15: What is integration testing?

Integration Testing tests how different modules or services work together. It verifies that the interfaces between components are correct.

Types of Integration Testing:

  • Big Bang: Test all components together at once
  • Top-Down: Test from top-level modules down
  • Bottom-Up: Test from low-level modules up
  • Sandwich/Hybrid: Combination of top-down and bottom-up

6. End-to-End Testing

Q17: What is end-to-end (E2E) testing?

End-to-End Testing tests the complete application flow from start to finish, simulating real user scenarios. It validates that all components work together correctly.

E2E Testing Characteristics:

  • Tests complete user workflows
  • Uses real browsers and environments
  • Simulates real user interactions
  • Tests across multiple systems
  • Slower but comprehensive
OrderE2ETests.cs
// E2E test using Playwright in C# public class OrderE2ETests : IClassFixture<PlaywrightFixture> { [Fact] public async Task CompleteOrderFlow_ShouldSucceed() { // Arrange var browser = await Playwright.Chromium.LaunchAsync(); var page = await browser.NewPageAsync(); // Act await page.GotoAsync("https://example.com"); await page.ClickAsync("text=Products"); await page.ClickAsync("text=Add to Cart"); await page.ClickAsync("text=Checkout"); await page.FillAsync("#email", "test@example.com"); await page.ClickAsync("text=Place Order"); // Assert var successMessage = await page.TextContentAsync(".success-message"); Assert.Contains("Order placed", successMessage); await browser.CloseAsync(); } }

Q19: What is Page Object Model (POM)?

Page Object Model is a design pattern that creates an object repository for web elements. It improves test maintenance by separating page logic from test logic.

LoginPage.cs
// Page Object Model in C# with Selenium public class LoginPage { private readonly IWebDriver _driver; private readonly By _usernameField = By.Id("username"); private readonly By _passwordField = By.Id("password"); private readonly By _loginButton = By.Id("login"); public LoginPage(IWebDriver driver) { _driver = driver; } public LoginPage EnterUsername(string username) { _driver.FindElement(_usernameField).SendKeys(username); return this; } public LoginPage EnterPassword(string password) { _driver.FindElement(_passwordField).SendKeys(password); return this; } public DashboardPage ClickLogin() { _driver.FindElement(_loginButton).Click(); return new DashboardPage(_driver); } } // Test using POM [Fact] public void Login_WithValidCredentials_ShouldSucceed() { var driver = new ChromeDriver(); var loginPage = new LoginPage(driver); loginPage.EnterUsername("testuser") .EnterPassword("password") .ClickLogin(); Assert.Contains("Welcome", driver.Title); }

7. API Testing

Q20: What is API testing?

API Testing tests the application programming interface (API) to ensure it meets functionality, reliability, performance, and security expectations.

API Testing Aspects:

  • Functionality: Correct response for valid requests
  • Reliability: Consistent behavior under load
  • Performance: Response time and throughput
  • Security: Authentication, authorization, data protection
  • Data Validation: Correct data format and values
ApiTests.cs
// API testing using RestSharp in C# public class ApiTests { [Fact] public async Task GetOrder_WithValidId_ReturnsOrder() { var client = new RestClient("https://api.example.com"); var request = new RestRequest($"/orders/{orderId}", Method.Get); var response = await client.ExecuteAsync<Order>(request); Assert.Equal(HttpStatusCode.OK, response.StatusCode); Assert.NotNull(response.Data); Assert.Equal(orderId, response.Data.Id); } [Fact] public async Task CreateOrder_WithValidData_ReturnsCreated() { var client = new RestClient("https://api.example.com"); var request = new RestRequest("/orders", Method.Post); request.AddJsonBody(new Order { Items = new List<OrderItem>() }); var response = await client.ExecuteAsync<Order>(request); Assert.Equal(HttpStatusCode.Created, response.StatusCode); } }
Q21: What are the common HTTP methods in API testing?

HTTP Methods:

  • GET: Retrieve data from the server
  • POST: Create new resource on the server
  • PUT: Update entire resource on the server
  • PATCH: Partially update resource on the server
  • DELETE: Remove resource from the server
Q22: What is contract testing?

Contract Testing ensures that the contract between a service and its consumers is maintained. It catches breaking changes early in the development cycle.

OrderApiConsumerTests.cs
// Contract testing using Pact in C# public class OrderApiConsumerTests { [Fact] public async Task GetOrder_WithValidId_ReturnsOrder() { var pact = new PactBuilder(); pact .ServiceConsumer("OrderConsumer") .HasPactWith("OrderService") .UponReceiving("a request to get order") .WithRequest(new { Method = HttpVerb.Get, Path = "/api/orders/123" }) .WillRespondWith(new { Status = 200, Headers = new Dictionary<string, string> { { "Content-Type", "application/json" } }, Body = new { id = 123, status = "Created" } }); await pact.VerifyAsync(); } }

8. Performance Testing

Q23: What is performance testing?

Performance Testing evaluates how a system performs in terms of responsiveness and stability under a particular workload. It ensures the application meets performance requirements.

Types of Performance Testing:

  • Load Testing: Tests system under expected load
  • Stress Testing: Tests system beyond expected load
  • Endurance Testing: Tests system under load for extended period
  • Spike Testing: Tests system with sudden load increase
  • Volume Testing: Tests system with large data volume
Q24: What is load testing?

Load Testing tests the system's behavior under expected user load. It helps identify performance bottlenecks and ensures the system can handle expected traffic.

LoadTest.cs
// Load testing using NBomber in C# public class LoadTest { [Fact] public async Task OrderApi_LoadTest_ShouldHandleExpectedLoad() { var scenario = Scenario.Create("order_api_load", async context => { var client = new HttpClient(); var response = await client.GetAsync("https://api.example.com/orders"); return response.IsSuccessStatusCode ? Q25: What are the key performance metrics?

Key Performance Metrics:

  • Response Time: Time taken to respond to a request
  • Throughput: Number of requests processed per second
  • Latency: Time delay in data transfer
  • Error Rate: Percentage of failed requests
  • Concurrent Users: Number of simultaneous users
  • CPU/Memory Usage: Resource utilization
  • Network I/O: Network bandwidth usage

9. Security Testing

Q26: What is security testing?

Security Testing identifies vulnerabilities and threats in a software system. It ensures the application is protected from unauthorized access and data breaches.

Security Testing Types:

  • Vulnerability Scanning: Automated scanning for known vulnerabilities
  • Penetration Testing: Simulated attacks to find weaknesses
  • Security Auditing: Review of security controls
  • Ethical Hacking: Authorized hacking to find vulnerabilities
  • Risk Assessment: Evaluation of security risks
Q27: What are common security vulnerabilities?

OWASP Top 10:

  1. Broken Access Control: Improper implementation of restrictions
  2. Cryptographic Failures: Failures related to cryptography
  3. Injection: SQL, NoSQL, OS, LDAP injection
  4. Insecure Design: Flaws in design and architecture
  5. Security Misconfiguration: Improper security settings
  6. Vulnerable Components: Using components with known vulnerabilities
  7. Authentication Failures: Weak authentication mechanisms
  8. Software Integrity Failures: Code and infrastructure integrity
  9. Logging Failures: Insufficient logging and monitoring
  10. SSRF: Server-Side Request Forgery
Q28: What is SQL injection and how to prevent it?

SQL Injection is a vulnerability where malicious SQL code is inserted into query strings, allowing attackers to manipulate databases.

// Prevention using parameterized queries in C# public class UserRepository { public User GetUser(string username) { // Vulnerable // string query = $"SELECT * FROM Users WHERE Username = '{username}'"; // Safe - Parameterized query string query = "SELECT * FROM Users WHERE Username = @Username"; using (var command = new SqlCommand(query, connection)) { command.Parameters.AddWithValue("@Username", username); var reader = command.ExecuteReader(); // ... } } }

12. CI/CD for Testing

Q34: What is continuous testing in CI/CD?

Continuous Testing is the process of executing automated tests as part of the CI/CD pipeline. It provides immediate feedback on code quality and prevents defects from reaching production.

Benefits:

  • Early defect detection
  • Faster feedback to developers
  • Reduced manual testing effort
  • Higher code quality
  • Faster release cycles

// GitHub Actions workflow for testing name: Run Tests on: [push, pull_request] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v2 - name: Setup .NET uses: actions/setup-dotnet@v2 with: dotnet-version: '7.0' - name: Restore dependencies run: dotnet restore - name: Build run: dotnet build --no-restore - name: Run unit tests run: dotnet test --no-build --verbosity normal --filter "Category=Unit" - name: Run integration tests run: dotnet test --no-build --verbosity normal --filter "Category=Integration" - name: Run E2E tests run: dotnet test --no-build --verbosity normal --filter "Category=E2E"
Q35: What is test-driven development (TDD)?

Test-Driven Development (TDD) is a development process where tests are written before the code. It follows a red-green-refactor cycle.

TDD Cycle:

  1. Red: Write a failing test
  2. Green: Write minimum code to pass the test
  3. Refactor: Improve the code while keeping tests green

Benefits:

  • Better code design
  • Fewer bugs
  • Comprehensive test coverage
  • Documentation through tests
  • Confidence in refactoring
Q36: What is behavior-driven development (BDD)?

Behavior-Driven Development (BDD) is an extension of TDD that focuses on the behavior of the system from the user's perspective. It uses natural language to describe test scenarios.

BDD Benefits:

  • Improved communication between developers and stakeholders
  • Tests written in business language
  • Living documentation
  • Better alignment with business requirements

13. Test Strategies

Q37: What is a test strategy?

Test Strategy is a high-level document that defines the testing approach, scope, resources, and schedule for a project. It guides the testing team throughout the project lifecycle.

Test Strategy Components:

  • Scope: What will be tested
  • Approach: How testing will be done
  • Tools: Testing tools and frameworks
  • Resources: Team and infrastructure
  • Schedule: Testing timeline
  • Risk Analysis: Potential risks and mitigation
Q38: What is a test plan?

Test Plan is a detailed document that outlines the test objectives, scope, approach, resources, and schedule for a specific project or feature.

Test Plan Components:

  • Test objectives and scope
  • Test strategy and approach
  • Test environment and resources
  • Test schedule and milestones
  • Risk and contingency planning
  • Deliverables and reporting
Q39: What is test coverage?

Test Coverage is a metric that measures how much of the code is executed by tests. It helps identify untested code and improve test quality.

Types of Coverage:

  • Statement Coverage: Percentage of statements executed
  • Branch Coverage: Percentage of branches executed
  • Function Coverage: Percentage of functions called
  • Line Coverage: Percentage of lines executed

Q40: What is smoke testing?

Smoke Testing is a type of testing that checks the basic functionality of an application to ensure it is stable enough for further testing. It's also known as build verification testing.

Smoke Testing Characteristics:

  • Tests critical functionality
  • Quick to execute
  • Performed after each build
  • Blocks further testing if fails
  • Identifies major defects early
Q41: What is sanity testing?

Sanity Testing is a type of testing that verifies that a specific functionality or bug fix is working as expected. It's a subset of regression testing.

Sanity vs Smoke Testing:

  • Smoke: Broad, checks entire system stability
  • Sanity: Focused, checks specific functionality
Q42: What is regression testing?

Regression Testing is testing that verifies that recent changes haven't broken existing functionality. It ensures the software still works after modifications.

Regression Testing Approaches:

  • Retest All: Re-run all tests
  • Regression Test Selection: Select relevant tests
  • Prioritization: Prioritize high-risk areas
  • Automated Regression: Use automated test suites
Q43: What is alpha testing?

Alpha Testing is performed by the development team or internal testers before the software is released to external users. It's conducted at the developer's site.

Alpha Testing Characteristics:

  • Performed by internal team
  • Conducted at developer site
  • Identifies major defects
  • Before beta testing
  • Simulates real user environment
Q44: What is beta testing?

Beta Testing is performed by actual users in a real environment before the final release. It helps identify issues that internal testing might miss.

Beta Testing Characteristics:

  • Performed by real users
  • Conducted at user site
  • Identifies usability issues
  • After alpha testing
  • Real-world usage scenarios
Q45: What is A/B testing?

A/B Testing is a method of comparing two versions of a webpage or app to determine which performs better. It's used for optimization and decision-making.

A/B Testing Process:

  1. Identify what to test
  2. Create variations (A and B)
  3. Split traffic between versions
  4. Collect data and metrics
  5. Analyze results and implement winner
Q46: What is usability testing?

Usability Testing evaluates how easy and user-friendly a product is for its intended users. It focuses on the user experience and interface design.

Usability Testing Methods:

  • Think Aloud: Users speak their thoughts while testing
  • Observation: Watch users interact with the product
  • Surveys: Collect user feedback
  • Heuristic Evaluation: Expert review of interface
Q47: What is accessibility testing?

Accessibility Testing ensures that the application is usable by people with disabilities. It verifies compliance with accessibility standards like WCAG.

Accessibility Testing Aspects:

  • Screen reader compatibility
  • Keyboard navigation
  • Color contrast
  • Font size and readability
  • Alternative text for images
  • Video captions
Q48: What is compatibility testing?

Compatibility Testing ensures the application works correctly across different browsers, operating systems, devices, and hardware configurations.

Compatibility Testing Types:

  • Browser Compatibility: Different browsers and versions
  • OS Compatibility: Different operating systems
  • Device Compatibility: Different devices and screen sizes
  • Network Compatibility: Different network conditions
  • Hardware Compatibility: Different hardware configurations
Q49: What is exploratory testing?

Exploratory Testing is an informal testing approach where testers learn about the system while testing it. It relies on tester's intuition and creativity to find defects.

Exploratory Testing Benefits:

  • Finds unexpected defects
  • Requires minimal documentation
  • Adapts to findings during testing
  • Leverages tester's expertise
  • Complements scripted testing
Q50: What is mutation testing?

Mutation Testing is a technique that introduces small changes (mutations) to the code to check if the tests can detect them. It measures the effectiveness of the test suite.

Mutation Testing Process:

  1. Run original tests - should pass
  2. Introduce mutations in code
  3. Run tests with mutated code
  4. If tests still pass, mutation survived (bad)
  5. If tests fail, mutation killed (good)
  6. Calculate mutation score

ATDD Process:

  3. Implement code to pass acceptance tests
  4. Refactor and improve

Q55: What is a test double?

Test Double is a generic term for any object that stands in for a real dependency during testing. It includes mocks, stubs, fakes, and spies.

Types of Test Doubles:

  • Dummy: Objects passed around but never used
  • Stub: Provides canned answers to calls
  • Fake: Working but simplified implementation
  • Mock: Objects with pre-programmed expectations
  • Spy: Records information about calls

Property-Based Testing generates random test cases based on properties that should always hold true. It finds edge cases that example-based tests might miss.

Q64: What is a flaky test?

Flaky Test is a test that passes and fails intermittently without any code changes. It undermines confidence in the test suite.

Causes of Flaky Tests:

  • Timing issues (race conditions)
  • External dependencies (APIs, databases)
  • Non-deterministic test data
  • Order dependency between tests
  • Resource leaks

Solutions:

  • Use proper waits and synchronization
  • Mock external dependencies
  • Isolate tests from each other
  • Use deterministic test data
  • Retry flaky tests with caution

  • Container-level: Run tests in containers
  • Sharding: Split tests across machines
