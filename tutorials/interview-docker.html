1. Docker Fundamentals

Q1: What is Docker?

Docker is a platform for developing, shipping, and running applications in containers. It enables developers to package applications with all dependencies into standardized units called containers. Docker uses containerization technology, which packages code and its dependencies together so the application runs quickly and reliably in different computing environments.

Key Benefits:

  • Consistency: Same environment across development, testing, and production
  • Isolation: Applications run in isolated environments without interference
  • Portability: Run anywhere Docker is installed, eliminating "it works on my machine" issues
  • Efficiency: Lightweight compared to VMs, sharing the host OS kernel
  • Speed: Fast startup times (seconds vs minutes for VMs)
  • Scalability: Easy to scale horizontally with container orchestration
Q2: What is a container?

Container is a lightweight, standalone executable package that includes everything needed to run an application: code, runtime, system tools, libraries, and settings. Containers share the host OS kernel but are isolated from each other and from the host system.

Container Characteristics:

  • Lightweight: Share host OS kernel, no full OS needed
  • Portable: Run consistently across different environments
  • Isolated: Process and resource isolation
  • Ephemeral: Can be created and destroyed quickly
  • Immutable: Built from images, don't change at runtime

Container vs VM:

  • VM: Full OS with hypervisor, heavier, isolated at hardware level
  • Container: Shared host OS kernel, lighter, isolated at process level
  • Startup: VMs take minutes, containers take seconds
Q3: What is the difference between Docker and VM?

Docker Containers and Virtual Machines both provide isolation but at different levels. VMs virtualize the entire hardware stack, while containers virtualize the OS kernel.

Aspect Docker Container Virtual Machine
OS Shared host kernel Separate guest OS
Size MBs (lightweight) GBs (heavyweight)
Startup Seconds Minutes
Performance Near native Overhead due to virtualization
Isolation Process-level Hardware-level
Portability High (same OS kernel) Lower (full OS included)
Q4: What is a Docker image?

Docker Image is a read-only template used to create containers. It contains the application code, libraries, dependencies, and runtime needed to run the application. Images are built from Dockerfiles and stored in registries like Docker Hub.

Image Characteristics:

  • Read-Only: Images are immutable once built
  • Layered: Built in layers, enabling efficient caching
  • Versioned: Tagged with versions (e.g., myapp:1.0)
  • Reusable: Can be used to create multiple containers
docker-commands.sh
# Build a Docker image docker build -t myapp:1.0 . # List images docker images # Remove an image docker rmi myapp:1.0
Q5: What is a Dockerfile?

Dockerfile is a text file that contains instructions to build a Docker image. It defines the base image, dependencies, and commands to run the application. Dockerfiles use a simple DSL (Domain Specific Language) with instructions like FROM, RUN, COPY, and CMD.

Common Dockerfile Instructions:

  • FROM: Base image to build from
  • RUN: Execute commands during build
  • COPY/ADD: Copy files to container
  • WORKDIR: Set working directory
  • ENV: Set environment variables
  • EXPOSE: Document exposed ports
  • CMD/ENTRYPOINT: Default command to run
Dockerfile
# Simple Dockerfile FROM python:3.9-slim WORKDIR /app COPY requirements.txt . RUN pip install -r requirements.txt COPY . . CMD ["python", "app.py"]

2. Docker Architecture

Q6: What are the main components of Docker architecture?

Docker Architecture consists of several key components that work together to build, ship, and run containers. Understanding these components is essential for working effectively with Docker.

Docker Components:

  • Docker Daemon (dockerd): Background service that listens for Docker API requests and manages Docker objects
  • Docker Client: CLI tool that users interact with to send commands to the daemon
  • Docker Registry: Stores and distributes Docker images (Docker Hub, private registries)
  • Docker Objects: Images, containers, networks, volumes, and other Docker entities
  • Docker Engine: The core technology that builds and runs containers
Q7: What is the Docker daemon?

Docker Daemon (dockerd) is the background service that listens for Docker API requests and manages Docker objects like images, containers, networks, and volumes. It's the core component that actually performs the work of building, running, and managing containers.

Docker Daemon Responsibilities:

  • Image Management: Build, pull, and manage images
  • Container Lifecycle: Create, start, stop, and remove containers
  • Network Management: Create and manage container networks
  • Volume Management: Create and manage storage volumes
  • API Server: Expose REST API for Docker client communication
Q8: What is Docker Hub?

Docker Hub is a cloud-based registry service for storing and sharing Docker images. It's the default registry for Docker and provides both public and private repositories for images.

Docker Hub Features:

  • Public Repositories: Free public image hosting
  • Private Repositories: Private image storage (paid)
  • Official Images: Verified official images for popular software
  • Automated Builds: Build images from GitHub/Bitbucket
  • Webhooks: Trigger actions on image push
docker-hub.sh
# Pull from Docker Hub docker pull nginx:latest # Push to Docker Hub docker push myusername/myapp:1.0 # Login to Docker Hub docker login
Q9: What is the difference between ENTRYPOINT and CMD?

ENTRYPOINT and CMD are both Dockerfile instructions that specify what command to run when a container starts, but they serve different purposes and behave differently.

Aspect ENTRYPOINT CMD
Purpose Executable that runs on container start Default arguments for ENTRYPOINT
Override Requires --entrypoint flag Easily overridden with docker run arguments
Use Case Main application executable Default parameters or fallback command
Combination Can be combined with CMD for arguments Provides arguments to ENTRYPOINT
Q10: What is the difference between COPY and ADD?

COPY and ADD are both Dockerfile instructions for copying files from the host to the container, but ADD has additional functionality that COPY doesn't have.

Aspect COPY ADD
Function Copy files from host to container Copy + can extract URLs/tar files
URL Support No Yes, can download from URLs
Tar Extraction No Yes, automatically extracts tar files
Best Practice Preferred for simple copying Use only for URLs/tar extraction

3. Dockerfile

Q11: What are common Dockerfile instructions?

Dockerfile Instructions are commands that define how to build a Docker image. Each instruction creates a layer in the image, and instructions are executed in order from top to bottom.

Common Instructions:

  • FROM: Base image to build from (required first instruction)
  • RUN: Execute commands during build (install dependencies)
  • COPY/ADD: Copy files from host to container
  • WORKDIR: Set working directory for subsequent instructions
  • ENV: Set environment variables
  • EXPOSE: Document which ports the container listens on
  • CMD/ENTRYPOINT: Default command to run when container starts
  • VOLUME: Create mount point for volumes
  • USER: Set user for subsequent instructions
  • ARG: Build-time variables
Q12: What is multi-stage build?

Multi-stage Build uses multiple FROM statements to create intermediate images, copying only needed artifacts to the final image. This significantly reduces the final image size by excluding build tools and intermediate files.

Multi-stage Build Benefits:

  • Smaller Images: Final image contains only runtime dependencies
  • Better Security: Fewer packages means fewer vulnerabilities
  • Separation of Concerns: Build and runtime environments separated
  • Faster Deployment: Smaller images pull and deploy faster
Dockerfile
# Multi-stage build FROM node:16 AS builder WORKDIR /app COPY package*.json ./ RUN npm install COPY . . RUN npm run build FROM nginx:alpine COPY --from=builder /app/dist /usr/share/nginx/html
Q13: What is .dockerignore?

.dockerignore file specifies files and directories to exclude from the build context, similar to .gitignore. This reduces build context size, speeds up builds, and prevents sensitive files from being included in images.

Common .dockerignore Entries:

  • node_modules: Don't copy node dependencies
  • .git: Exclude git history
  • .env: Exclude environment files (sensitive)
  • *.log: Exclude log files
  • dist: Exclude build artifacts (rebuild in container)
  • Dockerfile: Don't include Dockerfile itself
.dockerignore
# .dockerignore node_modules .git .env *.log dist Dockerfile .dockerignore
Q14: How do you optimize Docker image size?

Docker Image Optimization reduces image size, which improves build times, deployment speed, and security. Smaller images pull faster and have fewer potential vulnerabilities.

Optimization Techniques:

  • Use Minimal Base Images: alpine, slim, distroless images
  • Multi-stage Builds: Separate build and runtime environments
  • Combine RUN Commands: Reduce layers by combining commands
  • Clean Up: Remove cache and temp files in same layer
  • .dockerignore: Exclude unnecessary files from build context
  • Layer Caching: Order instructions to maximize cache hits
  • Use .dockerignore: Exclude unnecessary files
Dockerfile (Optimized)
# Optimized Dockerfile FROM node:18-alpine AS builder WORKDIR /app COPY package*.json ./ RUN npm ci --only=production && \ npm cache clean --force COPY . . RUN npm run build && \ rm -rf node_modules src FROM node:18-alpine WORKDIR /app COPY --from=builder /app/dist ./dist COPY --from=builder /app/package*.json ./ RUN npm ci --only=production --no-optional CMD ["node", "dist/index.js"]
Dockerfile (Python)
# Optimized Python Dockerfile FROM python:3.11-slim AS builder WORKDIR /app COPY requirements.txt . RUN pip install --user --no-cache-dir -r requirements.txt FROM python:3.11-slim WORKDIR /app COPY --from=builder /root/.local /root/.local COPY . . ENV PATH=/root/.local/bin:$PATH CMD ["python", "app.py"]
Q15: What is the difference between RUN, CMD, and ENTRYPOINT?

RUN, CMD, and ENTRYPOINT are Dockerfile instructions that execute commands, but at different times and for different purposes.

Instruction When Executed Purpose
RUN Build time (image creation) Install dependencies, setup, configuration
CMD Runtime (container start) Default command/arguments for container
ENTRYPOINT Runtime (container start) Main executable, not easily overridden

4. Docker Compose

Q16: What is Docker Compose?

Docker Compose is a tool for defining and running multi-container Docker applications. It uses YAML files to configure services, networks, and volumes, making it easy to coordinate multi-container applications.

Docker Compose Benefits:

  • Multi-Container Orchestration: Define and run multiple containers together
  • YAML Configuration: Simple, declarative configuration
  • One-Command Start: Start entire application stack with one command
  • Development Focus: Ideal for development and testing environments
  • Service Discovery: Automatic DNS resolution between services
docker-compose.yml
version: '3.8' services: web: build: . ports: - "5000:5000" environment: - DATABASE_URL=postgres://user:pass@db:5432/mydb depends_on: - db db: image: postgres:13 environment: POSTGRES_PASSWORD: example volumes: - db_data:/var/lib/postgresql/data volumes: db_data:
docker-compose.yml (Complex)
version: '3.8' services: frontend: build: ./frontend ports: - "3000:3000" backend: build: ./backend ports: - "8000:8000" environment: - REDIS_HOST=redis redis: image: redis:alpine nginx: image: nginx:alpine ports: - "80:80" volumes: - ./nginx.conf:/etc/nginx/nginx.conf
Q17: What are common Docker Compose commands?

Docker Compose Commands are used to manage multi-container applications defined in docker-compose.yml files. These commands simplify the process of starting, stopping, and managing multiple containers.

docker-compose-commands.sh
# Start services docker-compose up # Start in detached mode docker-compose up -d # Stop services docker-compose down # View logs docker-compose logs # Rebuild and start docker-compose up --build
Q18: What is the difference between docker-compose up and docker-compose start?

docker-compose up creates and starts containers, while docker-compose start only starts existing containers that were previously created but stopped.

Command Action
up Create and start containers (builds if needed)
start Start existing, stopped containers
Use Case up for first run/start, start for restart
Q19: How do you scale services in Docker Compose?

Scaling Services in Docker Compose allows you to run multiple instances of a service to handle increased load. Scaling can be done dynamically with command-line flags or defined in the compose file.

docker-compose-scale.sh
# Scale a service dynamically docker-compose up --scale web=3 # Define replicas in compose file services: web: deploy: replicas: 3
Q20: What are Docker Compose networks?

Docker Compose Networks automatically create a network for services to communicate. Services can reach each other by service name, and custom networks can be defined for more complex topologies.

Network Types:

  • Default Network: Automatically created for all services
  • Custom Networks: User-defined networks for isolation
  • External Networks: Connect to existing Docker networks
docker-compose-networks.yml
# Custom network networks: frontend: backend: services: web: networks: - frontend db: networks: - backend

5. Docker Networking

Q21: What are Docker network drivers?

Docker Network Drivers determine how containers communicate with each other and with the outside world. Each driver provides different networking capabilities suited for different use cases.

Network Drivers:

  • Bridge: Default driver for single-host communication, containers on same bridge can communicate
  • Host: Removes network isolation, container uses host's networking directly
  • Overlay: Multi-host communication for Docker Swarm, enables cross-host container communication
  • Macvlan: Assigns MAC addresses to containers, making them appear as physical devices
  • None: Disables networking for the container
Q22: What is the default Docker network?

Bridge Network is the default network driver in Docker. When you create a container without specifying a network, it's connected to the default bridge network. Containers on the same bridge network can communicate by IP address or container name.

Bridge Network Characteristics:

  • Default Network: Automatically created when Docker is installed
  • IP Allocation: Each container gets its own IP address
  • Communication: Containers can communicate by IP or name
  • Isolation: Provides network isolation from host
docker-network.sh
# Create custom bridge network docker network create my-network # Connect container to network docker network connect my-network mycontainer # List networks docker network ls # Inspect network docker network inspect my-network
docker-compose-network.yml
version: '3.8' services: app: build: . networks: - my-network db: image: postgres:13 networks: - my-network networks: my-network: driver: bridge
Q23: How do containers communicate?

Container Communication happens through Docker networking. Containers can communicate with each other and with the outside world using various methods depending on the network configuration.

Communication Methods:

  • Same Network: Containers on the same network can communicate by container name or IP
  • Exposed Ports: Port mapping allows external access to container services
  • Links: Legacy method for container communication (deprecated, use networks instead)
  • Host Networking: Container uses host's network stack directly
Q24: What is an overlay network?

Overlay Network enables multi-host communication for Docker Swarm. It creates a distributed network across multiple Docker hosts, allowing containers on different hosts to communicate as if they were on the same network.

Overlay Network Features:

  • Multi-Host: Spans across multiple Docker hosts
  • Encrypted: Optional encryption for secure communication
  • Service Discovery: Automatic DNS resolution across hosts
  • Load Balancing: Built-in load balancing for services
Q25: What is the difference between bridge and host network?

Bridge Network provides network isolation with slight overhead, while Host Network removes isolation for maximum performance by using the host's network stack directly.

Aspect Bridge Host
Isolation Network isolation from host No isolation, shares host network
Performance Slight overhead due to NAT No overhead, native performance
Use Case Most applications, default choice High performance needs, networking tools
Port Binding Requires port mapping No port mapping needed

6. Docker Storage

Q26: What is a Docker volume?

Docker Volume is a persistent storage mechanism that exists independently of containers. Data in volumes persists even after containers are removed, making them ideal for storing data that needs to survive container lifecycle.

Volume Benefits:

  • Persistence: Data survives container removal and recreation
  • Managed: Managed by Docker, independent of host filesystem
  • Portable: Can be moved between hosts
  • Performance: Better performance than bind mounts in many cases
  • Security: Better isolation and permissions management
docker-volume.sh
# Create volume docker volume create myvolume # Use volume in container docker run -v myvolume:/data myapp # List volumes docker volume ls # Remove volume docker volume rm myvolume
docker-compose-volume.yml
version: '3.8' services: db: image: postgres:13 volumes: - db_data:/var/lib/postgresql/data app: build: . volumes: - ./data:/app/data - logs:/var/log volumes: db_data: logs:
Q27: What is the difference between volume and bind mount?

Docker Volumes are managed by Docker and stored in a specific location on the host, while Bind Mounts map a host directory or file directly into the container.

Aspect Volume Bind Mount
Location Managed by Docker in /var/lib/docker/volumes Host file system, any location
Portability High, managed by Docker Host-specific, less portable
Performance Better, optimized by Docker Variable, depends on host filesystem
Use Case Database data, persistent storage Development, configuration files
Q28: What is a bind mount?

Bind Mount mounts a file or directory from the host filesystem directly into a container. It's useful for development where you want to access host files from within the container.

Bind Mount Use Cases:

  • Development: Live code editing from host
  • Configuration: Share config files between host and container
  • Debugging: Access host logs from container
  • Temporary Storage: Quick file sharing
docker-bind-mount.sh
# Bind mount docker run -v /host/path:/container/path myapp
Q29: What is a tmpfs mount?

tmpfs Mount stores data in the host's memory (RAM) instead of disk. It's non-persistent and ideal for temporary, sensitive data that shouldn't persist after container stops.

tmpfs Use Cases:

  • Temporary Files: Cache, temporary processing data
  • Sensitive Data: Data that shouldn't persist on disk
  • Performance: Fast I/O for temporary operations
  • Security: Data is lost when container stops
Q30: How do you backup Docker volumes?

Docker Volume Backup involves creating a container that mounts the volume and backs up the data to a tar file or another location. This ensures data persistence and disaster recovery.

docker-volume-backup.sh
# Backup volume to tar file docker run --rm -v myvolume:/data -v $(pwd):/backup \ alpine tar czf /backup/backup.tar.gz /data # Restore from tar file docker run --rm -v myvolume:/data -v $(pwd):/backup \ alpine tar xzf /backup/backup.tar.gz -C /

7. Docker Security

Q31: What are Docker security best practices?

Docker Security Best Practices help protect containers and the host system from security threats. Following these practices reduces the attack surface and improves overall security posture.

Security Practices:

  • Use Official Images: Pull from trusted sources like Docker Hub official images
  • Minimal Base Images: Use alpine or distroless to reduce attack surface
  • Run as Non-Root: Use USER instruction to run as non-privileged user
  • Scan Images: Regular vulnerability scanning with Trivy, Snyk, or Docker Scout
  • Limit Capabilities: Drop unnecessary Linux capabilities
  • Network Isolation: Use separate networks for different services
  • Read-Only Filesystem: Mount filesystem as read-only when possible
Dockerfile (Secure)
# Secure Dockerfile FROM node:18-alpine RUN addgroup -g 1001 -S nodejs && \ adduser -S nodejs -u 1001 WORKDIR /app COPY --chown=nodejs:nodejs package*.json ./ RUN npm ci --only=production && \ npm cache clean --force COPY --chown=nodejs:nodejs . . USER nodejs EXPOSE 3000 CMD ["node", "app.js"]
docker-security.sh
# Scan image for vulnerabilities docker scan myapp:latest # Run with limited capabilities docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE myapp # Run with read-only filesystem docker run --read-only --tmpfs /tmp myapp # Run as non-root user docker run --user 1001:1001 myapp
Q32: What is Docker Content Trust?

Docker Content Trust (DCT) enables digital signing of Docker images. It ensures images are pulled from trusted sources and haven't been tampered with during transit or storage.

DCT Benefits:

  • Image Verification: Verify image authenticity and integrity
  • Chain of Trust: Establish trust from publisher to consumer
  • Tamper Detection: Detect if images have been modified
  • Publisher Identity: Verify the identity of image publishers
docker-dct.sh
# Enable DCT export DOCKER_CONTENT_TRUST=1 # Sign an image docker trust sign myapp:1.0
Q33: What is a non-root container?

Non-root Container runs applications as a non-privileged user instead of root. This reduces the impact of container compromise, as the attacker gains only limited privileges.

Non-Root Benefits:

  • Reduced Attack Surface: Limited privileges if container is compromised
  • Compliance: Meets security compliance requirements
  • Defense in Depth: Additional security layer
  • Principle of Least Privilege: Follows security best practices
Dockerfile
# Run as non-root FROM alpine RUN adduser -D appuser USER appuser CMD ["./app"]
Q34: What are Docker secrets?

Docker Secrets provide a secure way to manage sensitive data like passwords, API keys, and certificates in Docker Swarm mode. Secrets are encrypted at rest and only accessible to authorized services.

Secrets Benefits:

  • Secure Storage: Encrypted at rest and in transit
  • Access Control: Only accessible to authorized services
  • Swarm Integration: Native integration with Docker Swarm
  • Runtime Injection: Injected into containers at runtime
docker-secrets.sh
# Create secret echo "mysecret" | docker secret create my_secret - # Use secret in service docker service create --secret my_secret myapp # List secrets docker secret ls # Remove secret docker secret rm my_secret
docker-compose-secrets.yml
version: '3.8' services: app: image: myapp secrets: - db_password - api_key secrets: db_password: file: ./secrets/db_password.txt api_key: external: true
Q35: How do you scan Docker images for vulnerabilities?

Docker Image Scanning identifies known vulnerabilities in images and their dependencies. Regular scanning helps catch security issues before deployment.

Scanning Tools:

  • Trivy: Open-source vulnerability scanner
  • Docker Scout: Built-in Docker scanning
  • Snyk: Commercial vulnerability scanning
docker-scan.sh
# Scan with Docker Scout docker scout quickview myapp:latest # Scan with Trivy trivy image myapp:latest # Scan with Snyk snyk container test myapp:latest
.github/workflows/scan.yml
# Scan in CI/CD pipeline - name: Scan image uses: aquasecurity/trivy-action@master with: image-ref: myapp:${{ github.sha }} format: 'sarif' output: 'trivy-results.sarif'

8. Docker Best Practices

Q36: What are Docker best practices?

Docker Best Practices ensure efficient, secure, and maintainable containerized applications. Following these practices improves performance, security, and developer experience.

Best Practices:

  • Use .dockerignore: Exclude unnecessary files from build context
  • Multi-stage Builds: Separate build and runtime for smaller images
  • Layer Caching: Order instructions to maximize cache hits
  • Minimal Base Images: Use alpine or distroless images
  • Don't Run as Root: Use non-root user for security
  • Scan Images: Regular vulnerability scanning
  • Use Specific Tags: Avoid latest tag, use specific versions
  • Combine RUN Commands: Reduce layers by combining commands
Q37: What is the difference between CMD and ENTRYPOINT in practice?

CMD provides default arguments that can be easily overridden, while ENTRYPOINT defines the main executable that's harder to override. They can be combined for flexible container configuration.

Practical Usage:

  • CMD Only: For containers that can run different commands
  • ENTRYPOINT Only: For containers with a single purpose
  • Combined: ENTRYPOINT for executable, CMD for default arguments
Q38: How do you debug Docker containers?

Docker Container Debugging involves inspecting container state, logs, and processes to identify and fix issues. Various tools and techniques help debug containerized applications.

Debugging Techniques:

  • Container Logs: View logs with docker logs
  • Interactive Shell: Access container with docker exec
  • Inspect: View container details with docker inspect
  • Stats: Monitor resource usage with docker stats
  • Network Debugging: Check network connectivity
docker-debug.sh
# View logs docker logs mycontainer # Interactive shell docker exec -it mycontainer /bin/bash # Inspect container docker inspect mycontainer
Q39: What is Docker layer caching?

Docker Layer Caching caches build layers to speed up subsequent builds. When a Dockerfile instruction hasn't changed, Docker uses the cached layer instead of rebuilding it.

Caching Best Practices:

  • Order Instructions: Put frequently changing instructions last
  • Copy Dependencies First: Copy package files before source code
  • Combine RUN Commands: Reduce number of layers
  • Use BuildKit: Enable BuildKit for better caching
Dockerfile (Optimized for Caching)
# Optimized for layer caching FROM node:18-alpine WORKDIR /app # Copy dependencies first (changes less often) COPY package*.json ./ RUN npm ci --only=production # Copy source code (changes more often) COPY . . # Build (changes with source code) RUN npm run build CMD ["node", "dist/index.js"]
docker-build-cache.sh
# Build with BuildKit enabled DOCKER_BUILDKIT=1 docker build -t myapp . # Build with cache from registry docker build --cache-from myapp:latest -t myapp:new . # Build without cache docker build --no-cache -t myapp .
Q40: What is Docker BuildKit?

Docker BuildKit is the next-generation build engine for Docker. It provides improved performance, better caching, and advanced build features compared to the legacy builder.

BuildKit Features:

  • Better Caching: More efficient layer caching
  • Parallel Builds: Build stages in parallel
  • Secret Mounting: Secure secret handling during build
  • Build Cache Export: Export and import build cache

9. Advanced Docker Concepts

Q41: What is Docker Swarm?

Docker Swarm is Docker's native clustering and orchestration tool. It turns multiple Docker hosts into a single virtual host, enabling container orchestration with service discovery, load balancing, and scaling.

Swarm Features:

  • Clustering: Multiple hosts as single virtual host
  • Service Discovery: Automatic DNS resolution
  • Load Balancing: Built-in load balancing
  • Scaling: Easy scaling of services
  • Rolling Updates: Zero-downtime deployments
docker-swarm.sh
# Initialize swarm docker swarm init # Join worker node docker swarm join --token $TOKEN $MANAGER_IP:2377 # Deploy service docker service create --name web --replicas 3 -p 80:80 nginx # Scale service docker service scale web=5 # Update service docker service update --image nginx:latest web
docker-compose-swarm.yml
version: '3.8' services: web: image: nginx deploy: replicas: 3 update_config: parallelism: 1 delay: 10s ports: - "80:80"
Q42: What is the difference between Docker and Docker Swarm?

Docker is the container platform for building and running containers on a single host, while Docker Swarm is the orchestration tool for managing containers across multiple hosts.

Aspect Docker Docker Swarm
Scope Single host Multiple hosts (cluster)
Orchestration Manual Automated
Scaling Manual Automated
Use Case Development, single host Production, multi-host
Q43: What is a Docker service?

Docker Service in Swarm mode is a definition of how to run containers, including the image, replicas, ports, and networks. Services enable scaling and load balancing across the swarm.

Service Characteristics:

  • Desired State: Define desired number of replicas
  • Scaling: Scale up or down easily
  • Load Balancing: Automatic load balancing
  • Self-Healing: Automatically restart failed containers
Q44: What is the difference between Docker service and container?

Container is a single instance of an image, while Service manages multiple containers (replicas) in Swarm mode with scaling and load balancing.

Aspect Container Service
Scope Single instance Multiple replicas
Scaling Manual Automated
Load Balancing None Built-in
Use Case Standalone Swarm orchestration
Q45: What is Docker stack?

Docker Stack is a group of interrelated services that share dependencies and can be orchestrated together. Stacks are defined using docker-compose files and deployed to Swarm.

Stack Benefits:

  • Multi-Service: Deploy multiple services together
  • Dependencies: Define service dependencies
  • Single Command: Deploy entire stack with one command
  • Rolling Updates: Update entire stack atomically
docker-stack.sh
# Deploy stack docker stack deploy -c docker-compose.yml mystack # List stacks docker stack ls # List services in stack docker stack services mystack # Remove stack docker stack rm mystack
docker-compose-stack.yml
version: '3.8' services: web: image: nginx deploy: replicas: 3 ports: - "80:80" db: image: postgres:13 deploy: replicas: 1 environment: POSTGRES_PASSWORD: example networks: default: driver: overlay
Q46: What is a Docker node?

Docker Node is a physical or virtual machine participating in a Docker Swarm cluster. Nodes can be managers or workers, each with different responsibilities.

Node Types:

  • Manager Node: Manages cluster, accepts commands, orchestrates services
  • Worker Node: Executes containers, reports to managers
Q47: What is the difference between manager and worker node?

Manager Nodes handle cluster management and orchestration, while Worker Nodes execute containers and report status to managers.

Aspect Manager Node Worker Node
Role Cluster management Container execution
API Access Yes, accepts commands No, only manager access
Quorum Participates in Raft quorum Does not participate
Q48: What is Docker health check?

Docker Health Check is a command that runs inside a container to check if the application is healthy. Docker can automatically restart unhealthy containers based on health check results.

Dockerfile
HEALTHCHECK --interval=30s --timeout=3s \ CMD curl -f http://localhost/ || exit 1
Q49: What is Docker resource limits?

Docker Resource Limits control the amount of CPU and memory a container can use. This prevents a single container from consuming all host resources.

docker-run.sh
# Set resource limits docker run --cpus="0.5" --memory="512m" myapp
Q50: What is Docker restart policy?

Docker Restart Policy defines what happens when a container stops. It can automatically restart containers based on the exit status or always restart them.

Restart Policies:

  • no: Don't restart automatically
  • on-failure: Restart only on non-zero exit
  • always: Always restart
  • unless-stopped: Always restart unless manually stopped

10. Docker Orchestration

Q51: What is the difference between Docker Swarm and Kubernetes?

Docker Swarm is Docker's native orchestration tool, simpler to set up but with fewer features. Kubernetes is a more powerful, feature-rich orchestration platform with a steeper learning curve.

Aspect Docker Swarm Kubernetes
Complexity Simple, easy to learn Complex, steep learning curve
Features Basic orchestration Advanced features (auto-scaling, RBAC, etc.)

Q52: What is Docker registry?

Docker Registry is a storage and distribution system for Docker images. It can be public (Docker Hub) or private (self-hosted or cloud-based).

Registry Types:

  • Docker Hub: Default public registry
  • Private Registry: Self-hosted with Docker Registry
  • Cloud Registries: AWS ECR, Google GCR, Azure ACR

Docker Private Registry is a self-hosted registry for storing and distributing Docker images within an organization. It provides control over image storage and access.

Private Registry Benefits:

  • Privacy: Images stay within organization
  • Control: Full control over storage and access
  • Compliance: Meet regulatory requirements

Q55: What is Docker plugin?

Docker Plugin extends Docker's functionality. Plugins can add new capabilities like volume drivers, network drivers, or logging drivers.

Plugin Types:

  • Network Plugins: Custom network drivers
  • Logging Plugins: Custom log aggregation
  • Authorization Plugins: Custom authentication
Q56: What is Docker API?

Docker API is a REST API that allows programmatic control of Docker. It's used by Docker CLI, Docker Compose, and other tools to interact with the Docker daemon.

API Use Cases:

  • Automation: Integrate Docker into automation pipelines
  • Monitoring: Monitor Docker daemon state
  • Integration: Integrate with other systems
Q57: What is Docker machine?

Docker Machine is a tool that lets you install Docker Engine on virtual hosts and manage them with docker-machine commands. It's useful for setting up Docker on multiple platforms.

Docker Machine Use Cases:

  • Local Development: Run Docker on Mac/Windows (legacy)
  • Cloud Provisioning: Provision Docker hosts on cloud providers
  • Testing: Test on multiple platforms
Q58: What is Docker Desktop?

Docker Desktop is an application for Mac and Windows that provides Docker Engine, Docker CLI, Docker Compose, and other tools in a single package. It's the recommended way to run Docker on desktop platforms.

Docker Desktop Features:

  • Kubernetes: Built-in single-node Kubernetes

Q59: What is Docker namespace?

Docker Namespace is a Linux kernel feature that provides isolation for containers. Namespaces isolate process IDs, network, mount points, and other system resources.

Namespace Types:

  • PID Namespace: Isolates process IDs
  • Network Namespace: Isolates network interfaces
  • Mount Namespace: Isolates filesystem mounts
  • UTS Namespace: Isolates hostname and domain name
  • IPC Namespace: Isolates inter-process communication
  • User Namespace: Isolates user and group IDs
Q60: What is Docker cgroups?

Docker cgroups (control groups) is a Linux kernel feature that limits and accounts resource usage for processes. Docker uses cgroups to limit CPU, memory, and other resources for containers.

cgroups Capabilities:

  • Resource Limiting: Limit CPU, memory, disk I/O
  • Accounting: Track resource usage
  • Control: Freeze or thaw processes
  • Priority: Set process priorities
Q61: What is the difference between namespace and cgroups?

Namespaces provide isolation by virtualizing system resources, while cgroups provide resource limiting and accounting. Together they enable containerization.

Aspect Namespaces cgroups
Purpose Isolation Resource limiting
What it Does Virtualizes resources Limits and accounts resources
Example Separate process trees Limit CPU usage
Q62: What is Docker copy-on-write?

Docker Copy-on-Write (CoW) is a storage strategy where layers share data, and modifications create copies only of changed data. This makes images efficient and fast to build.

CoW Benefits:

  • Efficiency: Share unchanged data between layers
  • Speed: Fast builds by reusing layers
  • Storage: Reduced storage requirements
Q63: What is Docker union filesystem?

Docker Union Filesystem (UnionFS) is a layered filesystem that combines multiple directories into a single directory. Docker uses UnionFS to build images layer by layer.

UnionFS Drivers:

  • Overlay2: Default, recommended driver
  • AUFS: Older driver, deprecated
  • Btrfs: Btrfs filesystem driver
  • ZFS: ZFS filesystem driver
Q64: What is Docker overlay network?

Docker Overlay Network is a multi-host network driver for Swarm. It creates a distributed network across multiple hosts, enabling containers to communicate across hosts.

Overlay Network Features:

  • Multi-Host: Spans across multiple Docker hosts
  • Encrypted: Optional encryption (IPsec)
  • Service Discovery: Automatic DNS resolution
  • Load Balancing: Built-in load balancing
Q65: What is Docker ingress network?

Docker Ingress Network is a special overlay network in Swarm that handles external traffic routing to services. It provides load balancing for services exposed to the outside world.

Ingress Network Features:

  • External Access: Routes external traffic to services
  • Load Balancing: Load balances across service replicas
  • Automatic: Automatically created in Swarm
  • Routing Mesh: All nodes can receive external traffic
Q66: What is Docker routing mesh?

Docker Routing Mesh in Swarm routes incoming traffic to service replicas. All nodes in the swarm can accept traffic for the service, even if they don't run the service.

Routing Mesh Benefits:

  • High Availability: Any node can accept traffic
  • Load Balancing: Automatic load balancing
  • Simplicity: No external load balancer needed
Q67: What is Docker service update?

Docker Service Update updates a service in Swarm with zero downtime. It performs rolling updates, replacing old containers with new ones gradually.

Update Strategies:

  • Rolling Update: Gradual replacement of containers
  • Parallelism: Control how many containers update at once
  • Rollback: Automatic rollback on failure
  • Delay: Delay between updates
Q68: What is Docker service rollback?

Docker Service Rollback reverts a service to a previous version. It's useful when an update causes issues and you need to quickly revert to a known good state.

docker-rollback.sh
# Rollback service docker service rollback myservice
Q69: What is Docker config?

Docker Config in Swarm stores non-sensitive configuration data. Unlike secrets, configs are not encrypted and are suitable for configuration files.

Config vs Secret:

  • Config: Non-sensitive data, not encrypted
  • Secret: Sensitive data, encrypted at rest
Q70: What is Docker global mode?

Docker Global Mode in Swarm deploys one service task per swarm node. It's useful for monitoring agents, logging agents, or other services that need to run on every node.

Global Mode Use Cases:

  • Monitoring: Run monitoring on every node
  • Logging: Run log collection on every node
  • Networking: Run network plugins on every node
Q71: What is Docker replicated mode?

Docker Replicated Mode in Swarm deploys a specified number of service tasks across the swarm. It's the default mode and provides load balancing and scaling.

Replicated Mode Features:

  • Specified Replicas: Define number of replicas
  • Load Balancing: Automatic load balancing
  • Scaling: Easy scaling up or down
  • Placement: Swarm decides where to place tasks

Docker Placement Constraints control where service tasks are placed in the swarm. They can be based on node labels, engine labels, or other node attributes.

Docker Service Logs aggregate logs from all service tasks in Swarm. This provides centralized logging for distributed services.

# View service logs docker service logs myservice

Docker Service Scale changes the number of replicas for a service. Scaling can be done dynamically to handle increased load or reduce resource usage.

Docker Service PS lists the tasks of a service, showing which nodes they're running on and their current state. This helps monitor service distribution and health.

docker-ps.sh
# List service tasks docker service ps myservice
