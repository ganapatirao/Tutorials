1. ASP.NET Core Fundamentals

Q1: What is ASP.NET Core?

ASP.NET Core is a cross-platform, high-performance framework for building modern cloud-based web applications.

Key Features:

  • Cross-platform: Runs on Windows, Linux, and macOS
  • High-performance: Optimized for speed and scalability
  • Modular: Lightweight and modular architecture
  • Open-source: Part of the .NET platform
  • Unified: Combines MVC, Web API, and Web Pages
  • Cloud-ready: Built for cloud deployment
  • Dependency Injection: Built-in DI container
Q2: What is the difference between ASP.NET Core and ASP.NET?

ASP.NET Core is cross-platform, modular, and high-performance. ASP.NET is Windows-only and monolithic.

Aspect ASP.NET Core ASP.NET
Platform Cross-platform (Windows, Linux, macOS) Windows-only
Architecture Modular, lightweight Monolithic, heavy
Performance High performance, optimized Slower, legacy
Deployment Self-contained, Docker support IIS required
Configuration JSON-based, flexible XML-based web.config
Q3: What is the difference between .NET Core and .NET Framework?

.NET Core is cross-platform and open-source. .NET Framework is Windows-only and proprietary.

Aspect .NET Core .NET Framework
Platform Cross-platform Windows-only

Installation Side-by-side, self-contained System-wide, single version
Support Active development Maintenance mode

2. Middleware

Q11: What is middleware?

Middleware is software that handles requests and responses in a pipeline. Each middleware can process the request, pass it to the next middleware, and process the response on the way back.

Middleware Pipeline:

  • Request processing: Middleware executes in the order registered
  • Response processing: Middleware executes in reverse order
  • Short-circuiting: Middleware can terminate the pipeline
  • Branching: Middleware can branch based on conditions
Pipeline.cs
// Request flows down app.UseExceptionHandler(); app.UseHttpsRedirection(); app.UseStaticFiles(); app.UseRouting(); app.UseAuthentication(); app.UseAuthorization(); app.UseEndpoints(); // Response flows up (reverse order)
Q12: How do you create custom middleware?

Create a class with Invoke method or use app.Use() in Configure.

CustomMiddleware.cs
public class CustomMiddleware { private readonly RequestDelegate _next; public CustomMiddleware(RequestDelegate next) { _next = next; } public async Task Invoke(HttpContext context) { // Custom logic before await _next(context); // Custom logic after } }
Q13: What is the difference between Use, Map, and Run?

Use passes to next. Map branches. Run terminates.

Middleware.cs
app.Use(middleware); // Passes to next app.Map("/api", app); // Branches app.Run(context => context.Response.WriteAsync("End")); // Terminates
Q14: What is the order of middleware?

Middleware runs in the order registered for requests, and in reverse order for responses.

Order Matters:

  • Exception handling: Should be first (catches errors from all middleware)
  • HTTPS redirection: Should be early (redirects before processing)
  • Static files: Should be before routing (serves files directly)
  • Authentication: Must be before authorization
  • Authorization: Must be after authentication
  • Endpoints: Should be last (executes the actual handler)
MiddlewareOrder.cs
var app = builder.Build(); // 1. Exception handling (FIRST) if (env.IsDevelopment()) { app.UseDeveloperExceptionPage(); } else { app.UseExceptionHandler("/Error"); } // 2. HTTPS redirection app.UseHttpsRedirection(); // 3. Static files app.UseStaticFiles(); // 4. Routing app.UseRouting(); // 5. Authentication (before authorization) app.UseAuthentication(); // 6. Authorization (after authentication) app.UseAuthorization(); // 7. CORS (before endpoints) app.UseCors(); // 8. Endpoints (LAST) app.UseEndpoints(endpoints => { endpoints.MapControllers(); endpoints.MapRazorPages(); });
Tip: Always register exception handling middleware first to catch errors from all subsequent middleware. Register endpoints last to ensure all other middleware has a chance to process the request.
Q15: What is the difference between UseRouting and UseEndpoints?

UseRouting matches routes to endpoints. UseEndpoints executes the matched endpoint.

Aspect UseRouting UseEndpoints
Purpose Route matching Endpoint execution
Timing Matches routes early in pipeline Executes at end of pipeline
Function Determines which endpoint to use Runs the endpoint handler
Order Must come before UseEndpoints Must come after UseRouting
Q16: What is the difference between UseAuthentication and UseAuthorization?

UseAuthentication authenticates the user (who they are). UseAuthorization authorizes the user (what they can do).

Aspect UseAuthentication UseAuthorization
Purpose Verifies user identity Checks user permissions
Timing Must come before UseAuthorization Must come after UseAuthentication
What it does Creates user principal from token/cookie Enforces policies, roles, requirements
Order importance Critical - must be first Critical - must be after auth
Q17: What is the difference between UseCors and UseExceptionHandler?

UseCors handles Cross-Origin Resource Sharing (CORS) policies. UseExceptionHandler handles unhandled exceptions globally.

Aspect UseCors UseExceptionHandler
Purpose Enable cross-origin requests Handle unhandled exceptions
When to use When API accessed from other domains For global error handling
Configuration Define allowed origins, methods, headers Define error response handler
Order Should be before UseEndpoints Should be first in pipeline
Q18: What is the difference between UseStaticFiles and UseDefaultFiles?

UseStaticFiles serves static files from the web root. UseDefaultFiles serves default files (index.html, default.html) when a directory is requested.

Aspect UseStaticFiles UseDefaultFiles
Purpose Serve any static file Serve default file for directory
Default files None (must specify file) index.html, default.html, etc.
Order UseDefaultFiles should come first Should come before UseStaticFiles
Use case Serving CSS, JS, images Serving SPA entry points
Q19: What is the difference between UseHttpsRedirection and UseHsts?

UseHttpsRedirection redirects HTTP requests to HTTPS. UseHsts adds the HTTP Strict Transport Security (HSTS) header to enforce HTTPS.

Aspect UseHttpsRedirection UseHsts
Purpose Redirect HTTP to HTTPS Enforce HTTPS via HSTS header
Behavior 301/307 redirect Adds Strict-Transport-Security header
Environment All environments Production only
Use case Force HTTPS redirect Prevent downgrade attacks
HttpsMiddleware.cs
app.UseHttpsRedirection(); // Redirect HTTP to HTTPS app.UseHsts(); // Add HSTS header (production only)
Q20: What is the difference between UseDeveloperExceptionPage and UseExceptionHandler?

UseDeveloperExceptionPage shows detailed error information for development. UseExceptionHandler provides custom error handling for production.

Aspect UseDeveloperExceptionPage UseExceptionHandler
Environment Development only Production
Error details Detailed stack trace, request info Custom error page/message
Security Reveals sensitive info Hides sensitive info
Use case Debugging in development User-friendly errors in production
ExceptionMiddleware.cs
if (env.IsDevelopment()) { app.UseDeveloperExceptionPage(); } else { app.UseExceptionHandler("/Error"); }

3. Dependency Injection

Q21: What is dependency injection?

Dependency Injection (DI) is a design pattern for achieving Inversion of Control (IoC). It allows a class to receive its dependencies from external sources rather than creating them internally.

Benefits of DI:

  • Loose coupling: Components depend on abstractions, not concrete implementations
  • Testability: Easy to mock dependencies for unit testing
  • Maintainability: Easier to change implementations
  • Reusability: Components can be reused in different contexts

DI in ASP.NET Core:

  • Built-in DI container (IServiceProvider)
  • Constructor injection (preferred)
  • Property injection (optional)
  • Method injection (rare)
DIExample.cs
// 1. Define interface public interface IMyService { string GetData(); } // 2. Implement service public class MyService : IMyService { public string GetData() => "Hello from DI!"; } // 3. Register in Program.cs var builder = WebApplication.CreateBuilder(args); builder.Services.AddSingleton<IMyService, MyService>(); // 4. Inject via constructor public class MyController : ControllerBase { private readonly IMyService _service; public MyController(IMyService service) { _service = service; } [HttpGet] public IActionResult Get() { return Ok(_service.GetData()); } }
Q22: What are DI lifetimes?

DI lifetimes determine how long a service instance lives and when a new instance is created.

Lifetime Description When Created Use Case
Transient New instance each time Every injection Lightweight, stateless services
Scoped Same instance per request Once per HTTP request Request-specific data, DbContext
Singleton Same instance per app First injection only Caching, configuration, stateful services
Q23: What is the difference between Transient and Scoped?

Transient creates a new instance every time it's injected. Scoped creates one instance per HTTP request.

Aspect Transient Scoped
Instance creation Every injection Once per request
State sharing No state shared State shared within request
Thread safety Thread-safe by default Thread-safe per request
Memory usage Higher (more instances) Lower (fewer instances)
Best for Stateless services DbContext, request data
Q24: What is the difference between Scoped and Singleton?

Scoped creates one instance per HTTP request. Singleton creates one instance for the entire application lifetime.

Aspect Scoped Singleton
Instance creation Once per HTTP request Once per application
State sharing Within request only Across all requests
Thread safety Thread-safe per request Must be thread-safe
Memory usage Medium Low (single instance)
Best for DbContext, request-scoped data Caching, configuration, services
Warning: Never inject a Scoped service into a Singleton service. This can cause the Scoped service to behave like a Singleton, leading to incorrect behavior and potential memory leaks.
Q25: How do you inject services?

Use constructor injection with service parameters.

ServiceInjection.cs
public class MyController : ControllerBase { private readonly IMyService _service; public MyController(IMyService service) { _service = service; } }
Q26: What is the difference between constructor and property injection?

Constructor injection is the preferred method in ASP.NET Core. Property injection is optional and less common.

Aspect Constructor Injection Property Injection
When to use Required dependencies Optional dependencies
Immutability Dependencies are immutable Properties can be modified
Testing Easy to test Harder to test
Support Built-in DI supports it Requires IServiceProvider
Best practice Preferred approach Use sparingly
Q27: What is the difference between IServiceProvider and service locator?

IServiceProvider is the built-in DI container in ASP.NET Core. Service locator is an anti-pattern where components explicitly request dependencies.

Aspect IServiceProvider (DI) Service Locator
Approach Dependencies injected Dependencies requested
Control Container controls dependencies Component controls dependencies
Testing Easy to mock Harder to test
Coupling Loose coupling Tight coupling to container
Pattern Recommended Anti-pattern
Q28: How do you register services?

Use services.AddTransient, services.AddScoped, services.AddSingleton.

ServiceRegistration.cs
builder.Services.AddTransient<IMyService, MyService>(); builder.Services.AddScoped<IRepository, Repository>(); builder.Services.AddSingleton<ICache, Cache>();
Q29: What is the difference between AddTransient and AddSingleton?

AddTransient creates a new instance every time. AddSingleton creates a single instance for the entire application.

Aspect AddTransient AddSingleton
Instance creation Every injection First injection only
Lifetime Shortest Longest
State No state shared State shared globally
Thread safety Thread-safe by default Must be thread-safe
Use case Lightweight, stateless Caching, configuration
Q30: What is the difference between AddScoped and AddTransient?

AddScoped creates one instance per HTTP request. AddTransient creates a new instance every time it's injected.

Aspect AddScoped AddTransient
Instance creation Once per request Every injection
State sharing Within request Never shared
Performance Better (fewer instances) Worse (more instances)
Use case DbContext, request data Stateless services

4. Configuration

Q31: What is configuration?

Configuration in ASP.NET Core provides settings from various sources like JSON files, environment variables, command-line arguments, and more.

Configuration Sources (in order of precedence):

  • appsettings.json - Default configuration file
  • appsettings.{Environment}.json - Environment-specific overrides
  • Environment variables - System environment variables
  • Command-line arguments - Arguments passed at startup
  • Key-per-file configuration - Directory-based configuration
  • Secret Manager - Development secrets
  • Azure Key Vault - Cloud-based secrets
Configuration.cs
// appsettings.json { "Logging": { "LogLevel": { "Default": "Information" } }, "ConnectionStrings": { "DefaultConnection": "Server=localhost;Database=MyDb" } } // Program.cs - Access configuration var builder = WebApplication.CreateBuilder(args); var config = builder.Configuration; // Get configuration value var connectionString = config.GetConnectionString("DefaultConnection"); var logLevel = config["Logging:LogLevel:Default"]; // Inject IConfiguration into services builder.Services.AddSingleton<IConfiguration>(config);
Tip: Later configuration sources override earlier ones. Environment variables and command-line arguments take highest precedence, making them ideal for deployment-specific overrides.
Q32: What are configuration providers?

Configuration providers read configuration data from various sources and make it available to the application.

Common Providers:

  • JSON Configuration Provider - Reads from .json files
  • XML Configuration Provider - Reads from .xml files
  • INI Configuration Provider - Reads from .ini files
  • Environment Variables Provider - Reads system environment variables
  • Command-line Provider - Reads command-line arguments
  • Key-per-file Provider - Reads from directory structure
  • Memory Provider - In-memory configuration (for testing)
  • Secret Manager - Development secrets storage
  • Azure Key Vault Provider - Cloud secrets management
ConfigProviders.cs
// Add custom configuration providers var builder = WebApplication.CreateBuilder(args); // Add JSON provider builder.Configuration .AddJsonFile("appsettings.json", optional: false, reloadOnChange: true) .AddJsonFile($"appsettings.{env.EnvironmentName}.json", optional: true); // Add environment variables provider builder.Configuration.AddEnvironmentVariables(); // Add command-line provider builder.Configuration.AddCommandLine(args); // Add memory provider (for testing) var dict = new Dictionary<string, string?> { ["TestKey"] = "TestValue" }; builder.Configuration.AddInMemoryCollection(dict);
Q33: What is the difference between appsettings.json and appsettings.Development.json?

appsettings.json contains default configuration. appsettings.Development.json contains environment-specific overrides for development.

Aspect appsettings.json appsettings.Development.json
Purpose Default/base configuration Development-specific overrides
Environment All environments Development only
Precedence Lower Higher (overrides base)
Contents Common settings Dev-specific (debugging, local DB)
Git tracking Tracked Often excluded

IOptionsSnapshot is scoped and reloads per request. IOptionsMonitor is a singleton with change notifications and current value access.

Aspect IOptionsSnapshot IOptionsMonitor

Q36: What is the difference between IOptions and IConfiguration?

IOptions provides strongly-typed configuration classes. IConfiguration provides raw key-value access to configuration.

Aspect IOptions IConfiguration
Type safety Strongly-typed String-based
Access pattern Object properties Key-value strings
Compile-time checking Yes No
IntelliSense Yes No

Q37: How do you bind configuration to classes?

Use Get() or Bind() methods with configuration section.

ConfigBinding.cs
var settings = config.GetSection("AppSettings").Get<AppSettings>(); // or var settings = new AppSettings(); config.GetSection("AppSettings").Bind(settings);
Q38: What is the difference between Get and Bind?

Get creates and returns a new instance. Bind populates an existing instance with configuration values.

Aspect Get<T>() Bind()
Instance creation Creates new instance Uses existing instance
Return value Returns the instance Returns void
Use case Simple configuration binding Populating existing objects
Flexibility Less flexible More flexible
Q39: What is the difference between configuration and settings?

Configuration is the ASP.NET Core feature for managing app settings. Settings is a general concept for application configuration values.

Aspect Configuration Settings
Type ASP.NET Core feature General concept
Implementation IConfiguration, Options pattern Any storage mechanism
Sources JSON, env vars, command line, etc. Files, database, registry, etc.

// ASP.NET Core Configuration var config = builder.Configuration; var value = config["MySetting"]; // Options pattern (typed configuration) builder.Services.Configure<MyOptions>(config.GetSection("MyOptions"));
Q40: What is the difference between environment variables and configuration?

Environment variables are one source of configuration. Configuration is the ASP.NET Core system that aggregates multiple sources including environment variables.

Aspect Environment Variables Configuration
Scope Single source (OS-level) Multiple sources (JSON, env vars, CLI, etc.)
Priority High priority (overrides JSON) Merges sources with priority
Structure Flat key-value pairs Hierarchical (supports sections)
Use case Deployment-specific values Complete app configuration
EnvConfig.cs
// Environment variable (OS level) // MYAPP__ConnectionString=Server=localhost;Database=MyDb // Configuration (ASP.NET Core) var builder = WebApplication.CreateBuilder(args); builder.Configuration.AddEnvironmentVariables(); var connString = builder.Configuration.GetConnectionString("MyDb");

5. Logging

Q41: What is logging?

Logging in ASP.NET Core records application events for debugging, monitoring, and auditing purposes.

Logging Features:

  • Built-in logging API: ILogger interface for structured logging
  • Multiple providers: Console, Debug, Event Log, Application Insights, etc.
  • Log levels: Trace, Debug, Information, Warning, Error, Critical
  • Structured logging: Support for named parameters and scopes
  • Filtering: Filter by log level and category
  • Performance: Async logging support
Logging.cs
public class MyService { private readonly ILogger<MyService> _logger; public MyService(ILogger<MyService> logger) { _logger = logger; } public void DoWork() { _logger.LogInformation("Starting work..."); try { // Do some work _logger.LogDebug("Processing item {ItemId}", itemId); } catch (Exception ex) { _logger.LogError(ex, "Error processing item"); } _logger.LogWarning("Work completed with warnings"); } } // Configure logging in Program.cs builder.Logging.ClearProviders(); builder.Logging.AddConsole(); builder.Logging.AddDebug(); builder.Logging.SetMinimumLevel(LogLevel.Information);
Q42: What are log levels?

Log levels indicate the severity of log messages and help filter logs based on importance.

Level Severity Use Case
Trace Lowest Detailed diagnostic information
Debug Low Debugging information
Information Medium General application flow
Warning High Unexpected but recoverable issues
Error Critical Errors that prevent operation
Critical Highest System-failure level events
Q43: What is the difference between ILogger and ILoggerFactory?

ILogger is used to write log messages. ILoggerFactory is used to create ILogger instances.

Aspect ILogger ILoggerFactory
Purpose Write log messages Create logger instances
Usage Injected into classes Used in configuration
Methods LogInformation, LogError, etc. CreateLogger
When to use In services/controllers In Program.cs/Startup
Q44: What is the difference between ILogger and ILogger?

ILogger<T> is the generic version that automatically includes the category name (type name). ILogger is the non-generic version where you must specify the category name manually.

Aspect ILogger<T> ILogger
Category name Automatic (type name) Manual (specified)
Type safety Strongly-typed String-based
Usage ILogger<MyService> loggerFactory.CreateLogger("MyService")
Best practice Preferred For dynamic categories
Q45: What are logging providers?

Logging providers determine where log messages are written - console, files, databases, cloud services, etc.

Built-in Providers:

  • Console: Writes to console output
  • Debug: Writes to debugger output
  • EventSource: Writes to Event Tracing for Windows
  • EventLog: Writes to Windows Event Log
  • Azure App Service: Writes to Azure diagnostics

Third-Party Providers:

  • Serilog: Rich structured logging with sinks
  • NLog: Flexible logging with targets
  • log4net: Apache logging framework
  • Application Insights: Azure monitoring
LoggingProviders.cs
// Built-in providers builder.Logging.ClearProviders(); builder.Logging.AddConsole(); builder.Logging.AddDebug(); builder.Logging.AddEventSourceLogger(); builder.Logging.AddEventLog(); // Serilog integration using Serilog; Log.Logger = new LoggerConfiguration() .WriteTo.Console() .WriteTo.File("logs/log-.txt", rollingInterval: RollingInterval.Day) .WriteTo.ApplicationInsights(telemetryConfig) .CreateLogger(); builder.Host.UseSerilog();

Logging records discrete events in the application. Tracing tracks the flow of a request across multiple services.

OpenTelemetry is an open-source standard for observability. Application Insights is Azure's monitoring service.

6. Hosting

Q51: What is hosting?

Hosting in ASP.NET Core manages the application lifecycle, startup, and shutdown processes.

Hosting Responsibilities:

  • Application startup: Configure services and request pipeline
  • Lifecycle management: Handle start and stop events
  • Dependency injection: Build and provide DI container
  • Configuration: Load and provide configuration
  • Logging: Configure logging providers
  • Hosted services: Manage background services
Hosting.cs
// Program.cs - Minimal hosting model (.NET 6+) var builder = WebApplication.CreateBuilder(args); // Configure services builder.Services.AddControllers(); builder.Services.AddDbContext<AppDbContext>(...); builder.Services.AddScoped<IMyService, MyService>(); // Configure the HTTP request pipeline var app = builder.Build(); app.UseHttpsRedirection(); app.UseAuthorization(); app.MapControllers(); // Run the application app.Run();
Q52: What is the difference between IHost and IWebHost?

IHost is the generic host for any application type. IWebHost is the web-specific host (legacy).

Aspect IHost IWebHost
Type Generic host Web-specific host
Use case Any application (web, worker, console) Web applications only
Status Current approach Legacy (deprecated)
Features Minimal, focused Web-specific features
Recommendation Use for new projects Migrate to IHost
Q53: What is the difference between Host.CreateDefaultBuilder and WebHost.CreateDefaultBuilder?

Host.CreateDefaultBuilder is the generic host builder for any application type. WebHost.CreateDefaultBuilder is the web-specific host builder (legacy).

Aspect Host.CreateDefaultBuilder WebHost.CreateDefaultBuilder
Type Generic host Web-specific host
Use case Any application (web, worker, console) Web applications only
Status Current approach (.NET Core 3.0+) Legacy (deprecated)
Features Minimal, focused Web-specific features (Kestrel, etc.)
Recommendation Use for new projects Migrate to generic host
HostBuilder.cs
// Generic Host (Current approach) var host = Host.CreateDefaultBuilder(args) .ConfigureWebHostDefaults(webBuilder => { webBuilder.UseStartup<Startup>(); }) .Build(); // Web Host (Legacy - deprecated) var host = WebHost.CreateDefaultBuilder(args) .UseStartup<Startup>() .Build();
Q54: What is the difference between IHostedService and BackgroundService?

IHostedService is the interface for background services. BackgroundService is a base class that simplifies implementing IHostedService.

Aspect IHostedService BackgroundService
Type Interface Abstract base class
Methods to implement StartAsync, StopAsync ExecuteAsync only
Cancellation handling Manual Built-in
Complexity More complex Simpler
Recommendation For custom scenarios For most cases
BackgroundService.cs
// Using BackgroundService (recommended) public class TimedHostedService : BackgroundService { private readonly ILogger<TimedHostedService> _logger; public TimedHostedService(ILogger<TimedHostedService> logger) { _logger = logger; } protected override async Task ExecuteAsync(CancellationToken stoppingToken) { while (!stoppingToken.IsCancellationRequested) { _logger.LogInformation("Worker running at: {time}", DateTimeOffset.Now); await Task.Delay(TimeSpan.FromSeconds(5), stoppingToken); } } } // Register in Program.cs builder.Services.AddHostedService<TimedHostedService>();
Q55: What is the difference between hosted service and worker service?

Hosted service runs within a web application. Worker service is a standalone background service template.

Aspect Hosted Service Worker Service

IHostBuilder is the older builder pattern. WebApplicationBuilder is the newer, simplified builder introduced in .NET 6.

Q57: What is the difference between Configure and ConfigureServices?

Q124: What is the difference between gRPC and REST?

gRPC uses binary protocol buffers with streaming support. REST uses JSON with request-response pattern.

Aspect gRPC REST
Data format Protocol Buffers (binary) JSON (text)
Performance Faster, smaller payload Slower, larger payload
Streaming Yes (uni/bidirectional) No (request-response)
Browser support Limited (needs gRPC-web) Native
Use case Microservices, internal APIs Public APIs, web apps
Q125: What is Blazor?

Blazor is a framework for building interactive web UIs with C# instead of JavaScript, enabling full-stack .NET development.

Blazor Features:

  • C# in browser: Write client-side logic in C#
  • Component-based: Reusable UI components
  • Two hosting models: Server and WebAssembly
  • JavaScript interop: Call JS from C# and vice versa
  • Full-stack .NET: Share code between client and server
Blazor.razor
// Counter.razor - Blazor component @page "/counter" @rendermode InteractiveServer <PageTitle>Counter</PageTitle> <h1>Counter</h1> <p role="status">Current count: @currentCount</p> <button class="btn btn-primary" @onclick="IncrementCount">Click me</button> @code { private int currentCount = 0; private void IncrementCount() { currentCount++; } } // Program.cs - Register Blazor builder.Services.AddRazorComponents() .AddInteractiveServerComponents(); app.MapRazorComponents<App>() .AddInteractiveServerRenderMode();
Q126: What is the difference between Blazor Server and Blazor WebAssembly?

Blazor Server runs UI logic on the server with SignalR for updates. Blazor WebAssembly runs UI logic in the browser using WebAssembly.

Aspect Blazor Server Blazor WebAssembly
Execution Server-side Client-side (browser)
Initial load Faster (smaller download) Slower (downloads WASM)
Latency Higher (SignalR roundtrip) Lower (local execution)
Offline support No Yes
SEO Better (server rendering) Worse (client rendering)
Use case Internal apps, fast initial load Public apps, offline support
