1. SOLID Principles Overview

Q1: What are the SOLID principles?

SOLID is an acronym for five design principles intended to make software designs more understandable, flexible, and maintainable.

The Five Principles:

  1. S - Single Responsibility Principle (SRP): A class should have one reason to change
  2. O - Open/Closed Principle (OCP): Open for extension, closed for modification
  3. L - Liskov Substitution Principle (LSP): Subtypes must be substitutable for base types
  4. I - Interface Segregation Principle (ISP): Clients shouldn't depend on unused interfaces
  5. D - Dependency Inversion Principle (DIP): Depend on abstractions, not concretions
Q2: Who introduced the SOLID principles?

The SOLID principles were introduced by Robert C. Martin (Uncle Bob) in the early 2000s. These principles are part of object-oriented design and help developers create more maintainable and scalable software systems.

Q3: Why are SOLID principles important?

Importance of SOLID Principles:

  • Maintainability: Easier to maintain and modify code
  • Scalability: Systems can grow without becoming unmanageable
  • Testability: Easier to write unit tests
  • Flexibility: Code is more flexible to changes
  • Reusability: Components can be reused in different contexts
  • Readability: Code is more organized and easier to understand
Q4: What is the relationship between SOLID and design patterns?

SOLID principles are the foundation for many design patterns. Design patterns are specific solutions to common problems, while SOLID principles are guidelines for creating good object-oriented designs. Many design patterns (like Strategy, Factory, Observer) naturally follow SOLID principles.

2. Single Responsibility Principle (SRP)

Q5: What is the Single Responsibility Principle?

Single Responsibility Principle (SRP) states that a class should have only one reason to change, meaning it should have only one job or responsibility.

SRP.cs
// Violating SRP public class User { public string Name { get; set; } public string Email { get; set; } public void SaveToDatabase() { } public void SendEmail() { } public void Validate() { } } // Following SRP public class User { public string Name { get; set; } public string Email { get; set; } } public class UserRepository { public void Save(User user) { } } public class EmailService { public void SendEmail(User user) { } } public class UserValidator { public bool Validate(User user) { return true; } }
Q6: How do you identify if a class violates SRP?

Signs of SRP Violation:

  • Class has too many methods (more than 10-15)
  • Class handles multiple concerns (UI, database, validation)
  • Class has multiple reasons to change
  • Class name includes "and" or "or" (e.g., UserAndOrderManager)
  • Difficult to test due to multiple responsibilities
Q7: Can you give another example of SRP violation and its fix?
OrderService.cs
// Violating SRP public class OrderService { public void ProcessOrder(Order order) { // Validate order if (order.Items.Count == 0) throw new Exception(); // Calculate total decimal total = order.Items.Sum(i => i.Price); // Save to database _context.Orders.Add(order); _context.SaveChanges(); // Send email _emailService.Send(order.CustomerEmail); // Update inventory foreach (var item in order.Items) { _inventoryService.UpdateStock(item.Id, -1); } } } // Following SRP public class OrderService { private readonly IOrderValidator _validator; private readonly IOrderCalculator _calculator; private readonly IOrderRepository _repository; private readonly IEmailService _emailService; private readonly IInventoryService _inventory; public void ProcessOrder(Order order) { _validator.Validate(order); order.Total = _calculator.CalculateTotal(order); _repository.Save(order); _emailService.SendConfirmation(order); _inventory.UpdateStock(order); } }
Q8: Does SRP apply to methods as well?

Yes, SRP also applies to methods. A method should do one thing and do it well. If a method is doing multiple things, it should be split into smaller methods.

UserService.cs
// Method violating SRP public void ProcessUser(User user) { if (user == null) return; if (string.IsNullOrEmpty(user.Name)) return; user.Name = user.Name.Trim(); user.Email = user.Email.ToLower(); _repository.Save(user); _emailService.SendWelcome(user); } // Following SRP public void ProcessUser(User user) { ValidateUser(user); NormalizeUser(user); SaveUser(user); SendWelcomeEmail(user); }
Q9: What is the difference between SRP and separation of concerns?

SRP is about a class having one reason to change, while Separation of Concerns (SoC) is a broader concept about separating different aspects of a system. SoC is the principle, and SRP is one way to achieve it at the class level.

3. Open/Closed Principle (OCP)

Q10: What is the Open/Closed Principle?

Open/Closed Principle (OCP) states that software entities should be open for extension but closed for modification. You should be able to extend behavior without modifying existing code.

OCP.cs
// Violating OCP public class AreaCalculator { public double CalculateArea(object shape) { if (shape is Rectangle r) return r.Width * r.Height; if (shape is Circle c) return Math.PI * c.Radius * c.Radius; throw new ArgumentException("Unknown shape"); } } // Following OCP public abstract class Shape { public abstract double Area(); } public class Rectangle : Shape { public double Width { get; set; } public double Height { get; set; } public override double Area() => Width * Height; } public class Circle : Shape { public double Radius { get; set; } public override double Area() => Math.PI * Radius * Radius; }
Q11: How can you achieve OCP in practice?

Ways to Achieve OCP:

  • Abstraction: Use abstract classes or interfaces
  • Inheritance: Extend classes instead of modifying them
  • Composition: Use composition over inheritance
  • Strategy Pattern: Encapsulate algorithms in separate classes
  • Dependency Injection: Inject dependencies instead of hardcoding
Q12: Give an example of OCP with discount calculation.
Discount.cs
// Following OCP with Strategy Pattern public interface IDiscount { double Calculate(double price); } public class NoDiscount : IDiscount { public double Calculate(double price) => price; } public class RegularDiscount : IDiscount { public double Calculate(double price) => price * 0.9; } public class PremiumDiscount : IDiscount { public double Calculate(double price) => price * 0.8; } public class DiscountCalculator { public double CalculatePrice(double price, IDiscount discount) { return discount.Calculate(price); } }
Q13: How does OCP relate to the Strategy pattern?

The Strategy pattern is a direct application of OCP. It defines a family of algorithms, encapsulates each one, and makes them interchangeable. This allows you to add new strategies without modifying existing code.

Q14: Can you show OCP with filter example?
Specification.cs
// Following OCP with Specification Pattern public interface ISpecification<T> { bool IsSatisfied(T item); } public class Product { public string Color { get; set; } public string Size { get; set; } } public class ColorSpecification : ISpecification<Product> { private readonly string _color; public ColorSpecification(string color) => _color = color; public bool IsSatisfied(Product p) => p.Color == _color; } public class SizeSpecification : ISpecification<Product> { private readonly string _size; public SizeSpecification(string size) => _size = size; public bool IsSatisfied(Product p) => p.Size == _size; } public class ProductFilter { public IEnumerable<Product> Filter(IEnumerable<Product> products, ISpecification<Product> spec) { return products.Where(p => spec.IsSatisfied(p)); } }

4. Liskov Substitution Principle (LSP)

Q15: What is the Liskov Substitution Principle?

Liskov Substitution Principle (LSP) states that objects of a superclass should be replaceable with objects of a subclass without breaking the application. Subtypes must be substitutable for their base types.

LSP.cs
// Violating LSP public class Rectangle { public virtual int Width { get; set; } public virtual int Height { get; set; } public int Area() => Width * Height; } public class Square : Rectangle { public override int Width { set { base.Width = base.Height = value; } } public override int Height { set { base.Width = base.Height = value; } } } // Problem: Square breaks Rectangle behavior Rectangle r = new Square(); r.Width = 5; r.Height = 10; Console.WriteLine(r.Area()); // Expected 50, but returns 100
Q16: How do you fix the LSP violation in the Rectangle/Square example?
LSP.cs
// Following LSP public abstract class Shape { public abstract int Area(); } public class Rectangle : Shape { public int Width { get; set; } public int Height { get; set; } public override int Area() => Width * Height; } public class Square : Shape { public int Side { get; set; } public override int Area() => Side * Side; }
Q17: What are common signs of LSP violation?

Signs of LSP Violation:

  • Subclass throws exceptions that base class doesn't throw
  • Subclass returns null when base class returns object
  • Subclass modifies base class invariants
  • Subclass has weaker preconditions than base class
  • Subclass has stronger postconditions than base class
Q18: Give an example of LSP with birds and flying.
Bird.cs
// Violating LSP public class Bird { public virtual void Fly() { } } public class Penguin : Bird { public override void Fly() { throw new InvalidOperationException("Penguins can't fly"); } } // Following LSP public abstract class Bird { public abstract void Move(); } public class FlyingBird : Bird { public override void Move() => Fly(); public virtual void Fly() { } } public class Penguin : Bird { public override void Move() => Swim(); public void Swim() { } }
Q19: How does LSP relate to inheritance?

LSP is a principle that guides proper use of inheritance. It ensures that inheritance is used correctly - the "is-a" relationship must hold true. If a subclass cannot be substituted for its parent, then the inheritance hierarchy is flawed.

5. Interface Segregation Principle (ISP)

Q20: What is the Interface Segregation Principle?

Interface Segregation Principle (ISP) states that clients should not be forced to depend on interfaces they don't use. Split large interfaces into smaller, more specific ones.

ISP.cs
// Violating ISP public interface IWorker { void Work(); void Eat(); } public class Robot : IWorker { public void Work() { } public void Eat() { } // Robot doesn't need to eat } // Following ISP public interface IWorkable { void Work(); } public interface IEatable { void Eat(); } public class Human : IWorkable, IEatable { public void Work() { } public void Eat() { } } public class Robot : IWorkable { public void Work() { } }
Q21: Why is ISP important?

Benefits of ISP:

  • Loose Coupling: Clients only depend on what they need
  • Flexibility: Easier to implement interfaces
  • Maintainability: Changes to one interface don't affect others
  • Clarity: Smaller interfaces are more focused and clear
Q22: Give an example of ISP with printer interfaces.
Printer.cs
// Following ISP public interface IPrinter { void Print(Document doc); } public interface IScanner { void Scan(Document doc); } public interface IFax { void Fax(Document doc); } public class MultiFunctionPrinter : IPrinter, IScanner, IFax { public void Print(Document doc) { } public void Scan(Document doc) { } public void Fax(Document doc) { } } public class SimplePrinter : IPrinter { public void Print(Document doc) { } }
Q23: How does ISP relate to SRP?

ISP is similar to SRP but at the interface level. Just as a class should have one responsibility, an interface should represent one cohesive set of behaviors. Both principles aim to reduce coupling and increase cohesion.

Q24: What is the difference between ISP and OCP?

ISP focuses on creating small, focused interfaces that clients can implement without being forced to depend on unused methods. OCP focuses on being able to extend behavior without modifying existing code. ISP helps achieve OCP by making it easier to add new implementations without affecting existing ones.

6. Dependency Inversion Principle (DIP)

Q25: What is the Dependency Inversion Principle?

Dependency Inversion Principle (DIP) states that high-level modules should not depend on low-level modules. Both should depend on abstractions. Abstractions should not depend on details. Details should depend on abstractions.

DIP.cs
// Violating DIP public class LightSwitch { private LightBulb _bulb; public LightSwitch() { _bulb = new LightBulb(); } public void Toggle() { _bulb.Toggle(); } } // Following DIP public interface ISwitchable { void Toggle(); } public class LightBulb : ISwitchable { public void Toggle() { } } public class Fan : ISwitchable { public void Toggle() { } } public class LightSwitch { private readonly ISwitchable _device; public LightSwitch(ISwitchable device) { _device = device; } public void Toggle() { _device.Toggle(); } }
Q26: What is Dependency Injection?

Dependency Injection (DI) is a technique to implement DIP. It involves providing dependencies to a class rather than the class creating them. This can be done through constructor injection, property injection, or method injection.

OrderService.cs
// Constructor Injection public class OrderService { private readonly IOrderRepository _repository; private readonly IEmailService _emailService; public OrderService(IOrderRepository repository, IEmailService emailService) { _repository = repository; _emailService = emailService; } }
Q27: What are the types of Dependency Injection?

Types of Dependency Injection:

  • Constructor Injection: Dependencies provided through constructor
  • Property Injection: Dependencies set through public properties
  • Method Injection: Dependencies passed to methods
  • Service Locator: Dependencies requested from a central registry (anti-pattern)
Q28: Give an example of DIP with logging.
Logger.cs
// Following DIP public interface ILogger { void Log(string message); } public class ConsoleLogger : ILogger { public void Log(string message) => Console.WriteLine(message); } public class FileLogger : ILogger { public void Log(string message) => File.AppendAllText("log.txt", message); } public class UserService { private readonly ILogger _logger; public UserService(ILogger logger) { _logger = logger; } public void CreateUser(User user) { _logger.Log($"Creating user: {user.Name}"); // Create user logic } }
Q29: What is Inversion of Control (IoC)?

Inversion of Control (IoC) is a broader principle where the control of object creation and lifecycle is transferred from the application to a framework or container. DI is a pattern that implements IoC. IoC containers like ASP.NET Core's built-in DI container manage object creation and dependency resolution.

Q30: How does DIP relate to the other SOLID principles?

DIP is closely related to all other SOLID principles. It enables OCP by allowing you to extend behavior through new implementations. It supports ISP by depending on small, focused interfaces. It works with LSP by ensuring you depend on abstractions that can be substituted. It also supports SRP by separating concerns through dependency injection.

7. SOLID in Practice

Q31: How do you apply SOLID principles in a real project?

Applying SOLID in Practice:

  • Start with SRP: Break down large classes into smaller, focused ones
  • Use interfaces: Define contracts for dependencies
  • Apply DI: Use dependency injection throughout the application
  • Refactor regularly: Continuously improve code to follow SOLID
  • Code reviews: Use code reviews to ensure SOLID compliance
  • Testing: Write tests to verify SOLID principles
Q32: Can you show a complete example applying all SOLID principles?
OrderProcessing.cs
// Complete SOLID example: Order Processing // Interfaces (ISP, DIP) public interface IOrderValidator { bool Validate(Order order); } public interface IOrderRepository { void Save(Order order); } public interface INotificationService { void Send(Order order); } public interface IPaymentProcessor { bool Process(Order order); } // Implementations (SRP) public class OrderValidator : IOrderValidator { public bool Validate(Order order) { return order.Items.Count > 0 && order.Customer != null; } } public class OrderRepository : IOrderRepository { public void Save(Order order) { // Database logic } } public class EmailNotificationService : INotificationService { public void Send(Order order) { // Email logic } } public class CreditCardProcessor : IPaymentProcessor { public bool Process(Order order) { // Payment logic return true; } } // Service (OCP, DIP) public class OrderService { private readonly IOrderValidator _validator; private readonly IOrderRepository _repository; private readonly INotificationService _notification; private readonly IPaymentProcessor _payment; public OrderService( IOrderValidator validator, IOrderRepository repository, INotificationService notification, IPaymentProcessor payment) { _validator = validator; _repository = repository; _notification = notification; _payment = payment; } public void ProcessOrder(Order order) { if (!_validator.Validate(order)) throw new ValidationException(); if (!_payment.Process(order)) throw new PaymentException(); _repository.Save(order); _notification.Send(order); } }
Q33: How do SOLID principles help with testing?

SOLID and Testing:

  • SRP: Small classes are easier to unit test
  • OCP: New functionality can be tested independently
  • LSP: Mocks can substitute real implementations
  • ISP: Smaller interfaces are easier to mock
  • DIP: Dependencies can be injected with test doubles
Q34: What are the trade-offs of following SOLID principles?

Trade-offs:

  • Initial Complexity: More classes and interfaces initially
  • Development Time: May take longer to implement initially
  • Over-engineering: Risk of over-applying principles in simple projects
  • Learning Curve: Requires understanding of principles

Benefits Outweigh Trade-offs:

  • Easier maintenance in long run
  • Better testability
  • More flexible to changes
  • Higher code quality
Q35: When should you NOT follow SOLID principles?

When to Avoid SOLID:

  • Simple Scripts: One-off scripts don't need SOLID
  • Prototypes: Quick prototypes may skip SOLID
  • Performance Critical: May need to optimize over design
  • Small Projects: Overhead may not be justified
Q36: How do you refactor code to follow SOLID?

Refactoring Steps:

  1. Identify violations (large classes, tight coupling)
  2. Extract methods and classes (SRP)
  3. Introduce interfaces for dependencies (DIP)
  4. Replace conditionals with polymorphism (OCP)
  5. Split large interfaces (ISP)
  6. Ensure proper inheritance hierarchies (LSP)
  7. Add tests to verify refactoring
Q37: What tools help with SOLID compliance?

Tools for SOLID:

  • Static Analysis: SonarQube, ReSharper, CodeAnalysis
  • Code Metrics: Cyclomatic complexity, coupling metrics
  • Design Pattern Libraries: Help implement patterns correctly
  • DI Containers: Autofac, Unity, ASP.NET Core DI
  • Testing Frameworks: xUnit, NUnit for verifying design
Q38: How does SOLID relate to microservices architecture?

SOLID principles are essential in microservices. Each microservice should follow SRP (single responsibility). Services communicate through interfaces (ISP, DIP). Services can be extended without modification (OCP). Services can be substituted (LSP). This makes microservices more maintainable and scalable.

Q39: Can you show SOLID with a payment system example?
PaymentSystem.cs
// SOLID Payment System public interface IPaymentMethod { bool Process(decimal amount); } public class CreditCardPayment : IPaymentMethod { public bool Process(decimal amount) { // Credit card logic return true; } } public class PayPalPayment : IPaymentMethod { public bool Process(decimal amount) { // PayPal logic return true; } } public class PaymentService { private readonly IPaymentMethod _paymentMethod; public PaymentService(IPaymentMethod paymentMethod) { _paymentMethod = paymentMethod; } public bool MakePayment(decimal amount) { return _paymentMethod.Process(amount); } }
Q40: How do you teach SOLID to junior developers?

Teaching SOLID:

  • Start with simple examples and real-world analogies
  • Show before/after code comparisons
  • Use code reviews to reinforce principles
  • Encourage refactoring existing code
  • Pair programming to demonstrate application
  • Provide reading materials and resources

8. Benefits and Trade-offs

Q41: What are the long-term benefits of SOLID?

Long-term Benefits:

  • Reduced Technical Debt: Code stays maintainable over time
  • Faster Development: Easier to add new features
  • Better Team Collaboration: Clear structure helps teams work together
  • Easier Onboarding: New developers understand code faster
  • Higher Quality: Fewer bugs and issues
Q42: How does SOLID improve code maintainability?

SOLID improves maintainability by ensuring that changes are localized. SRP means changes affect only one class. OCP means new features don't require modifying existing code. LSP ensures inheritance is correct. ISP prevents unnecessary dependencies. DIP reduces coupling between components.

Q43: What is the relationship between SOLID and Clean Code?

SOLID principles are a subset of Clean Code practices. Clean Code encompasses SOLID plus other practices like meaningful names, small functions, proper comments, and good formatting. SOLID provides the structural foundation for writing clean, maintainable code.

Q44: How do you measure SOLID compliance?

Measuring SOLID Compliance:

  • Code Metrics: Class size, method complexity, coupling
  • Static Analysis: Tools that detect violations
  • Code Reviews: Manual review by team members
  • Test Coverage: High coverage often indicates good design
  • Refactoring Frequency: Less refactoring needed over time
Q45: Can SOLID principles be applied to functional programming?

While SOLID is an object-oriented principle, the concepts can be adapted to functional programming. SRP applies to functions (one responsibility). OCP applies through composition and higher-order functions. LSP is less relevant but substitutability still matters. ISP applies to module interfaces. DIP applies through dependency injection of functions.

Q46: How does SOLID relate to Domain-Driven Design (DDD)?

SOLID and DDD complement each other. DDD focuses on domain modeling and bounded contexts, while SOLID provides principles for implementing those models. DDD's aggregates, entities, and value objects naturally follow SRP. DDD's repositories and services follow DIP. Both aim to create maintainable, domain-focused software.

Q47: What are common mistakes when applying SOLID?

Common Mistakes:

  • Over-abstraction: Creating too many interfaces
  • Premature Optimization: Applying SOLID too early
  • Ignoring Context: Applying principles without considering project needs
  • Rigid Adherence: Following principles blindly
  • Naming Confusion: Poor names for classes and interfaces
Q48: How do you balance SOLID with YAGNI (You Aren't Gonna Need It)?

Balance SOLID with YAGNI by applying principles pragmatically. Don't create abstractions for features you don't need yet. Apply SOLID when you see actual complexity or anticipate clear future needs. Refactor to SOLID when code becomes difficult to maintain, not preemptively.

Q49: What is the role of SOLID in Agile development?

SOLID principles support Agile development by enabling iterative development. Agile requires frequent changes, and SOLID makes changes easier. Small, focused classes (SRP) can be modified quickly. Open for extension (OCP) allows adding features without breaking existing code. This aligns with Agile's emphasis on responding to change.

Q50: How do you stay updated on SOLID best practices?

Staying Updated:

  • Read books by Robert C. Martin and other experts
  • Follow blogs and articles on software design
  • Attend conferences and meetups
  • Participate in code reviews and discussions
  • Experiment with different approaches
  • Learn from open-source projects
  • Stay connected with the developer community

6. Advanced SOLID Concepts

Q51: What is the Interface Segregation Principle (ISP) in detail?

Interface Segregation Principle (ISP) states that clients should not be forced to depend on interfaces they don't use. Instead of one large interface, create smaller, more specific interfaces.

ISP Benefits:

  • Reduces coupling between components
  • Makes interfaces more focused
  • Prevents implementing unnecessary methods
  • Improves code maintainability
ISPExample.cs
// Bad: Fat interface public interface IMultiFunctionDevice { void Print(); void Scan(); void Fax(); } // Good: Segregated interfaces public interface IPrinter { void Print(); } public interface IScanner { void Scan(); } public interface IFax { void Fax(); } // Client depends only on what it needs public class SimplePrinter : IPrinter { public void Print() { /* Print implementation */ } }
Q52: What is the Dependency Inversion Principle (DIP) in detail?

Dependency Inversion Principle (DIP) states that high-level modules should not depend on low-level modules. Both should depend on abstractions. Abstractions should not depend on details. Details should depend on abstractions.

DIP Benefits:

  • Reduces coupling between modules
  • Makes code more testable
  • Enables easier changes to implementations
  • Supports dependency injection
DIPExample.cs
// Bad: High-level depends on low-level public class LightSwitch { private LightBulb _bulb; public LightSwitch() { _bulb = new LightBulb(); } } // Good: Both depend on abstraction public interface ISwitchable { void TurnOn(); void TurnOff(); } public class LightSwitch { private ISwitchable _device; public LightSwitch(ISwitchable device) { _device = device; } public void Toggle() { _device.TurnOn(); } }
Q53: What is the difference between abstraction and implementation?

Abstraction defines what something does without specifying how it's done. Implementation provides the actual code that performs the operations defined by the abstraction.

Key Differences:

  • Abstraction: Interface, abstract class, contract
  • Implementation: Concrete class, actual code
  • Abstraction: Stable, changes rarely
  • Implementation: Can change frequently
Q54: What is dependency injection?

Dependency Injection (DI) is a design pattern where dependencies are provided to a class rather than created internally. It implements the Dependency Inversion Principle.

DI Types:

  • Constructor Injection: Dependencies via constructor
  • Property Injection: Dependencies via properties
  • Method Injection: Dependencies via method parameters
DIExample.cs
// Constructor Injection public class UserService { private readonly IUserRepository _repository; public UserService(IUserRepository repository) { _repository = repository; } } // Property Injection public class UserService { public IUserRepository Repository { get; set; } } // Method Injection public class UserService { public void ProcessUser(int userId, IUserRepository repository) { var user = repository.GetById(userId); } }
Q55: What is an Inversion of Control (IoC) container?

Inversion of Control (IoC) Container is a framework that automates dependency injection. It manages object creation and lifecycle, resolving dependencies automatically.

Popular IoC Containers:

  • .NET: ASP.NET Core DI, Autofac, StructureMap
  • Java: Spring, Guice
  • JavaScript: InversifyJS, Awilix
ServiceCollection.cs
// ASP.NET Core DI Container public void ConfigureServices(IServiceCollection services) { // Register services services.AddSingleton<IUserRepository, UserRepository>(); services.AddScoped<IUserService, UserService>(); services.AddTransient<IEmailService, EmailService>(); } // Lifetimes: // Singleton: One instance for app lifetime // Scoped: One instance per HTTP request // Transient: New instance each time requested
Q56: What is the difference between composition and inheritance?

Composition combines objects to achieve complex behavior. Inheritance creates new classes from existing ones. Composition is generally preferred over inheritance (Composition over Inheritance).

Composition vs Inheritance:

  • Composition: Has-a relationship, flexible, runtime
  • Inheritance: Is-a relationship, rigid, compile-time
  • Composition: Easier to change behavior
  • Inheritance: Tight coupling, fragile base class
CompositionExample.cs
// Composition (Preferred) public class Car { private readonly IEngine _engine; private readonly IWheel _wheel; public Car(IEngine engine, IWheel wheel) { _engine = engine; _wheel = wheel; } } // Inheritance (Use sparingly) public class SportsCar : Car { // Inherits all Car behavior // Tight coupling to base class }
Q57: What is the Strategy pattern and how does it relate to OCP?

Strategy Pattern defines a family of algorithms, encapsulates each one, and makes them interchangeable. It implements the Open/Closed Principle by allowing new strategies without modifying existing code.

Strategy Pattern Benefits:

  • Open for extension (new strategies)
  • Closed for modification (existing code unchanged)
  • Runtime algorithm selection
  • Eliminates conditional statements
StrategyPattern.cs
public interface ISortStrategy { void[] Sort(int[] array); } public class BubbleSort : ISortStrategy { public int[] Sort(int[] array) { /* Bubble sort */ } } public class QuickSort : ISortStrategy { public int[] Sort(int[] array) { /* Quick sort */ } } public class Sorter { private ISortStrategy _strategy; public Sorter(ISortStrategy strategy) { _strategy = strategy; } public int[] Sort(int[] array) { return _strategy.Sort(array); } }
Q58: What is the Template Method pattern and how does it relate to OCP?

Template Method Pattern defines the skeleton of an algorithm in a base class, allowing subclasses to override specific steps without changing the algorithm's structure. It supports OCP by enabling extension through inheritance.

Template Method Benefits:

  • Code reuse in base class
  • Consistent algorithm structure
  • Subclasses customize specific steps
  • Follows Hollywood Principle
TemplateMethod.cs
public abstract class DataProcessor { // Template method public void Process() { LoadData(); ValidateData(); TransformData(); SaveData(); } protected abstract void LoadData(); protected abstract void TransformData(); protected virtual void ValidateData() { } protected virtual void SaveData() { } } public class CsvProcessor : DataProcessor { protected override void LoadData() { /* Load CSV */ } protected override void TransformData() { /* Transform CSV */ } }
Q59: What is the Adapter pattern and how does it relate to ISP?

Adapter Pattern allows incompatible interfaces to work together. It relates to ISP by creating focused interfaces that clients can depend on, rather than forcing clients to use large, incompatible interfaces.

Adapter Pattern Benefits:

  • Enables integration of incompatible systems
  • Follows ISP by creating client-specific interfaces
  • Reuses existing functionality
  • Decouples client from implementation
AdapterPattern.cs
// Client interface public interface ITarget { void Request(); } // Incompatible interface public class Adaptee { public void SpecificRequest() { } } // Adapter public class Adapter : ITarget { private readonly Adaptee _adaptee; public Adapter(Adaptee adaptee) { _adaptee = adaptee; } public void Request() { _adaptee.SpecificRequest(); } }
Q60: What is the Decorator pattern and how does it relate to OCP?

Decorator Pattern adds behavior to objects dynamically without modifying their classes. It implements OCP by extending functionality through composition rather than inheritance.

Decorator Pattern Benefits:

  • Adds behavior without modifying classes
  • Open for extension (new decorators)
  • Closed for modification (original class unchanged)
  • Flexible alternative to inheritance
DecoratorPattern.cs
public interface ICoffee { string GetDescription(); decimal GetCost(); } public class SimpleCoffee : ICoffee { public string GetDescription() => "Simple Coffee"; public decimal GetCost() => 2.00m; } public class MilkDecorator : ICoffee { private readonly ICoffee _coffee; public MilkDecorator(ICoffee coffee) { _coffee = coffee; } public string GetDescription() => _coffee.GetDescription() + ", Milk"; public decimal GetCost() => _coffee.GetCost() + 0.50m; }
Q61: What is the Facade pattern and how does it relate to SRP?

Facade Pattern provides a simplified interface to a complex subsystem. It relates to SRP by giving each subsystem a single responsibility while providing a unified interface to clients.

Facade Pattern Benefits:

  • Simplifies complex subsystems
  • Reduces coupling between client and subsystem
  • Each subsystem maintains SRP
  • Provides a clean API for clients
FacadePattern.cs
// Complex subsystem public class OrderSystem { public void CreateOrder() { } } public class PaymentSystem { public void ProcessPayment() { } } public class InventorySystem { public void UpdateInventory() { } } // Facade public class OrderFacade { private readonly OrderSystem _order; private readonly PaymentSystem _payment; private readonly InventorySystem _inventory; public void PlaceOrder() { _order.CreateOrder(); _payment.ProcessPayment(); _inventory.UpdateInventory(); } }
Q62: What is the difference between SRP and separation of concerns?

SRP focuses on a single class having one reason to change. Separation of Concerns (SoC) is a broader principle about separating different aspects of a system. SRP is a specific application of SoC at the class level.

Key Differences:

  • SRP: Class-level, one responsibility per class
  • SoC: System-level, separate different concerns
  • SRP: About change reasons
  • SoC: About architectural boundaries
Q63: What is the Law of Demeter and how does it relate to SOLID?

Law of Demeter (LoD) states that an object should only talk to its immediate friends. It reduces coupling and supports SRP by keeping interactions focused and simple.

Law of Demeter:

  • Don't talk to strangers
  • Only call methods on immediate dependencies
  • Reduces coupling between objects
  • Makes code more maintainable
LawOfDemeter.cs
// Bad: Violates Law of Demeter public class OrderProcessor { public void Process(Order order) { // Too many dots var address = order.Customer.Address.Street; } } // Good: Follows Law of Demeter public class OrderProcessor { public void Process(Order order) { // Ask Order for the address var address = order.GetShippingAddress(); } }
Q64: What is the difference between coupling and cohesion?

Coupling measures how closely connected classes are. Cohesion measures how related the responsibilities within a class are. SOLID aims for low coupling and high cohesion.

Goals:

  • Low Coupling: Classes are independent
  • High Cohesion: Class has focused responsibility
  • SRP: Increases cohesion
  • DIP: Reduces coupling
Q65: What is the difference between association, aggregation, and composition?

Association is a general relationship between objects. Aggregation is a weak "has-a" relationship. Composition is a strong "has-a" relationship where the child cannot exist without the parent.

Relationship Types:

  • Association: Objects know about each other
  • Aggregation: Can exist independently
  • Composition: Dependent lifecycle
Q66: What is the difference between abstract class and interface?

Abstract Class can have implementation and state. Interface defines a contract without implementation. Both support polymorphism but serve different purposes.

Key Differences:

  • Abstract Class: Can have fields, methods, constructors
  • Interface: Only method signatures (mostly)
  • Abstract Class: Single inheritance
  • Interface: Multiple implementation
AbstractVsInterface.cs
// Abstract Class public abstract class Animal { protected string _name; public Animal(string name) { _name = name; } public abstract void MakeSound(); public void Sleep() { } } // Interface public interface IFlyable { void Fly(); } // Class can implement multiple interfaces public class Bird : Animal, IFlyable { public Bird(string name) : base(name) { } public override void MakeSound() { } public void Fly() { } }
Q67: What is the difference between tight coupling and loose coupling?

Tight Coupling means classes are highly dependent on each other. Loose Coupling means classes are independent and interact through abstractions. SOLID promotes loose coupling.

Benefits of Loose Coupling:

  • Easier to test
  • Easier to maintain
  • Easier to replace components
  • More flexible architecture
Q68: What is the difference between high cohesion and low cohesion?

High Cohesion means a class has related responsibilities. Low Cohesion means a class has unrelated responsibilities. SRP promotes high cohesion.

Benefits of High Cohesion:

  • Easier to understand
  • Easier to maintain
  • Easier to reuse
  • Follows SRP
Q69: What is the difference between composition and aggregation?

Composition is a strong relationship where the child cannot exist without the parent. Aggregation is a weak relationship where the child can exist independently.

Key Differences:

  • Composition: Same lifecycle, strong ownership
  • Aggregation: Independent lifecycle, weak ownership
  • Composition: "part-of" relationship
  • Aggregation: "has-a" relationship
Q70: What is the difference between inheritance and polymorphism?

Inheritance creates new classes from existing ones. Polymorphism allows objects of different types to be treated as objects of a common type. Inheritance enables polymorphism.

Key Differences:

  • Inheritance: Code reuse, is-a relationship
  • Polymorphism: Flexibility, runtime behavior
  • Inheritance: Compile-time relationship
  • Polymorphism: Runtime behavior
Q71: What is the difference between overloading and overriding?

Overloading creates multiple methods with the same name but different parameters. Overriding provides a new implementation for a method in a derived class.

Key Differences:

  • Overloading: Same name, different parameters
  • Overriding: Same name, same parameters
  • Overloading: Compile-time polymorphism
  • Overriding: Runtime polymorphism
Q72: What is the difference between static and dynamic binding?

Static Binding happens at compile time. Dynamic Binding happens at runtime. Overloading uses static binding, overriding uses dynamic binding.

Key Differences:

  • Static Binding: Compile-time, faster
  • Dynamic Binding: Runtime, flexible
  • Static Binding: Methods, properties
  • Dynamic Binding: Virtual methods, interfaces
Q73: What is the difference between early binding and late binding?

Early Binding resolves method calls at compile time. Late Binding resolves method calls at runtime. Early binding is faster, late binding is more flexible.

Key Differences:

  • Early Binding: Compile-time, type-safe
  • Late Binding: Runtime, flexible
  • Early Binding: Better performance
  • Late Binding: Dynamic features
Q74: What is the difference between abstraction and encapsulation?

Abstraction hides implementation details and shows only functionality. Encapsulation hides data and provides access through methods. Both are OOP principles that work together.

Key Differences:

  • Abstraction: Hides complexity, shows what
  • Encapsulation: Hides data, protects state
  • Abstraction: Design focus
  • Encapsulation: Implementation focus
Q75: What is the future of SOLID principles?

Future of SOLID involves adapting to new paradigms like functional programming, microservices, and cloud-native development. The principles remain relevant but are applied differently in modern contexts.

Modern Applications:

  • Microservices: SRP at service level
  • Functional Programming: Adapted principles
  • Cloud-Native: DIP with cloud services
  • Event-Driven: Loose coupling through events
