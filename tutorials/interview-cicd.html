1. CI/CD Fundamentals

Q1: What is CI/CD?

CI/CD stands for Continuous Integration and Continuous Delivery/Deployment. It's a methodology to automate the integration, testing, and deployment of code changes, enabling teams to deliver software more frequently and reliably. CI/CD bridges the gap between development and operations, creating a seamless pipeline that transforms code commits into deployable artifacts.

Core Components:

  • Continuous Integration (CI): Developers frequently merge code changes into a shared repository. Each merge is automatically verified by automated builds and tests to detect integration errors early.
  • Continuous Delivery (CD): Code changes are automatically built, tested, and prepared for release to a testing or production environment. It ensures the software is always in a deployable state.
  • Continuous Deployment: An extension of continuous delivery where code changes are automatically deployed to production without manual intervention after passing all tests.

CI/CD Pipeline Flow:

  • Code Commit: Developer pushes code to version control
  • Build: Compile code and create artifacts
  • Test: Run automated tests (unit, integration, security)
  • Package: Create deployable artifacts (Docker images, JARs)
  • Deploy: Deploy to staging or production environment
  • Monitor: Track application health and performance
Q2: What is the difference between CI and CD?

Continuous Integration (CI) focuses on the integration and testing phase, ensuring that code changes are automatically built and tested before merging. Continuous Delivery/Deployment (CD) focuses on the delivery and deployment phase, ensuring that tested code is automatically deployed to production or staging environments.

Aspect CI (Continuous Integration) CD (Continuous Delivery/Deployment)
Primary Focus Code integration and automated testing Application delivery and deployment
Main Goal Detect integration errors early Release software reliably and quickly
Execution Frequency Multiple times daily on every commit On demand or continuous after CI passes
Key Activities Build, unit tests, integration tests Deploy to staging, production, monitoring
Failure Impact Blocks merge, requires fix Blocks deployment, may require rollback

Key Insight:

CI ensures code quality and prevents integration issues, while CD ensures that quality code reaches users quickly and reliably. Together, they form a complete automated software delivery pipeline.

Q3: What is the difference between Continuous Delivery and Continuous Deployment?

Continuous Delivery ensures that code is always in a deployable state, but requires manual approval before deploying to production. Continuous Deployment automates the entire process, deploying code to production automatically after passing all tests without human intervention.

Aspect Continuous Delivery Continuous Deployment
Deployment Trigger Manual approval required Fully automated after tests pass
Human Control Human in the loop for production No human intervention needed
Risk Level Lower risk (human gatekeeper) Higher risk (requires strong testing)
Release Speed Slower (waiting for approval) Faster (instant deployment)
Best For Production with compliance needs High-frequency release environments

When to Use Each:

  • Continuous Delivery: Use when you need manual control over production releases, have regulatory requirements, or want to review changes before they go live.
  • Continuous Deployment: Use when you have comprehensive automated testing, need maximum release velocity, and trust your automated processes completely.
Q4: What are the benefits of CI/CD?

CI/CD provides numerous benefits that transform software development from a manual, error-prone process into an automated, reliable pipeline. These benefits span across development speed, code quality, team collaboration, and business outcomes.

Key Benefits:

  • Faster Time to Market: Automated pipelines reduce manual work, enabling frequent releases. Teams can ship features multiple times per day instead of weekly or monthly.
  • Improved Code Quality: Automated testing catches bugs early in the development cycle, reducing the cost and effort of fixing issues later.
  • Reduced Deployment Risk: Smaller, incremental changes are easier to test and debug. If something breaks, it's easier to identify and fix the specific change.
  • Enhanced Collaboration: Developers integrate code frequently, reducing merge conflicts and improving team communication and visibility.
  • Consistent Environments: Standardized build and deployment processes ensure consistency across development, testing, and production environments.
  • Faster Feedback Loop: Developers receive immediate feedback on their code changes, enabling quick iterations and continuous improvement.
  • Cost Reduction: Automation reduces manual effort, minimizes downtime, and reduces the cost of fixing bugs in production.
  • Scalability: Automated pipelines can handle increased workload and team growth without proportional increases in manual effort.
Q5: What is a pipeline?

Pipeline is a set of automated processes that take code from version control to production. It defines the workflow for building, testing, and deploying applications, typically organized into stages that execute sequentially or in parallel. Pipelines are the backbone of CI/CD, ensuring consistent and repeatable software delivery.

Pipeline Components:

  • Stages: Logical divisions in the pipeline (build, test, deploy)
  • Steps: Individual tasks within a stage (compile, run tests, deploy)
  • Agents: Compute resources that execute pipeline steps
  • Artifacts: Build outputs passed between stages
  • Triggers: Events that start pipeline execution (push, PR, schedule)

Typical Pipeline Stages:

  • Build Stage: Compile code, resolve dependencies, create artifacts
  • Test Stage: Run unit tests, integration tests, security scans
  • Package Stage: Create deployable packages (Docker images, installers)
  • Deploy Stage: Deploy to staging or production environments
  • Post-Deploy Stage: Run smoke tests, health checks, monitoring

2. Continuous Integration

Q6: What is Continuous Integration?

Continuous Integration (CI) is the practice of merging all developers' working copies to a shared mainline multiple times a day. Each merge is verified by automated build and tests to detect integration errors early. CI ensures that the codebase is always in a buildable state and that integration issues are identified and fixed immediately.

CI Core Practices:

  • Frequent Code Commits: Developers commit code multiple times per day to the main branch
  • Automated Builds: Every commit triggers an automated build process
  • Automated Testing: Unit tests, integration tests run automatically on every build
  • Fast Feedback: Developers receive immediate feedback on their changes
  • Maintain a Single Source of Truth: Keep the main branch always deployable

CI Benefits:

  • Reduces integration problems and conflicts
  • Enables early bug detection
  • Improves code quality through continuous testing
  • Accelerates development velocity
Q7: What is a build server?

Build Server is a dedicated server or cloud service that compiles code, runs tests, and produces build artifacts. It's the workhorse of CI, executing pipeline tasks on every code change. Build servers can be self-hosted (Jenkins, TeamCity) or cloud-based (GitHub Actions, CircleCI, GitLab CI).

Build Server Responsibilities:

  • Code Checkout: Fetch latest code from version control
  • Dependency Resolution: Install required libraries and packages
  • Compilation: Build the application from source code
  • Test Execution: Run automated test suites
  • Artifact Creation: Package build outputs (JARs, Docker images)
  • Reporting: Generate test reports and build status

Popular Build Servers:

  • Jenkins: Open-source, highly customizable with plugins
  • GitHub Actions: Integrated with GitHub, YAML-based workflows
  • GitLab CI: Built into GitLab, uses .gitlab-ci.yml
  • CircleCI: Cloud-native, fast with Docker support
  • TeamCity: JetBrains' solution with excellent UI
Q8: What is a commit stage?

Commit Stage is the first and most critical stage in a CI pipeline. It compiles code, runs unit tests, and provides fast feedback to developers. The commit stage should complete quickly (typically under 5-10 minutes) to provide rapid feedback and prevent blocking the main branch.

Commit Stage Characteristics:

  • Fast Execution: Must complete quickly to avoid developer waiting
  • Unit Tests: Runs fast unit tests, not slow integration tests
  • Code Quality Checks: Includes linting, static analysis
  • Compilation: Ensures code compiles without errors
  • Gatekeeper: Blocks merge if any checks fail

Commit Stage Best Practices:

  • Keep tests fast and focused
  • Run in parallel when possible
  • Use caching for dependencies
  • Fail fast on first error
Q9: What is artifact management?

Artifact Management stores and manages build artifacts like JARs, WARs, Docker images, npm packages, and other build outputs. Artifact repositories provide a centralized location for storing, versioning, and distributing build artifacts across environments and teams.

Artifact Repository Benefits:

  • Centralized Storage: Single source of truth for all artifacts
  • Versioning: Track different versions of artifacts
  • Dependency Management: Resolve and manage dependencies
  • Access Control: Secure access to artifacts
  • Integration: Works with CI/CD tools

Popular Artifact Repositories:

  • JFrog Artifactory: Universal repository manager
  • Sonatype Nexus: Open-source and enterprise options
  • Docker Hub: Docker container registry
  • GitHub Packages: Integrated with GitHub
  • AWS ECR: Amazon Elastic Container Registry
Q10: What is branching strategy?

Branching Strategy defines how teams create and manage branches for development, testing, and release. A good branching strategy enables parallel development, facilitates code reviews, and supports release management while maintaining a clean, releasable main branch.

Common Branching Strategies:

  • Trunk-Based Development: Single main branch with short-lived feature branches. Developers commit directly to main or use very short-lived branches (1-2 days). Best for CI/CD and small teams.
  • Git Flow: Uses master, develop, feature, release, and hotfix branches. More structured but complex. Good for projects with scheduled releases.
  • GitHub Flow: Simpler than Git Flow with just main and feature branches. Features are developed in branches and merged via pull requests. Ideal for continuous deployment.
  • GitLab Flow: Combines feature branches with environment branches (production, staging). Each environment has its own branch.

Choosing a Strategy:

  • Team Size: Smaller teams benefit from simpler strategies
  • Release Cadence: Frequent releases favor trunk-based, scheduled releases favor Git Flow
  • CI/CD Maturity: Advanced CI/CD enables trunk-based development

3. Continuous Delivery

Q11: What is Continuous Delivery?

Continuous Delivery (CD) ensures code is always in a deployable state. It extends CI by automating the release process up to production deployment. With CD, every change that passes the automated tests is a potential release candidate, ready to be deployed to production with a single click.

CD Key Principles:

  • Always Deployable: Codebase is always in a state that can be deployed to production
  • Automated Testing: Comprehensive automated tests validate every change
  • Environment Parity: Development, staging, and production environments are identical
  • Manual Approval: Deployment to production requires manual approval
  • Release Automation: Deployment process is fully automated

CD Pipeline Stages:

  • Build: Compile and package the application
  • Test: Run unit, integration, and acceptance tests
  • Deploy to Staging: Deploy to staging environment for validation
  • Manual Approval: Team reviews and approves deployment
  • Deploy to Production: Deploy to production environment
Q12: What is a deployment pipeline?

Deployment Pipeline is a series of stages that code goes through from commit to production. Each stage validates the code before moving to the next, ensuring that only high-quality, tested code reaches production. The pipeline provides visibility into the deployment process and enables automated rollback if issues are detected.

Typical Pipeline Stages:

  • Build Stage: Compile code, resolve dependencies, create artifacts
  • Test Stage: Run unit tests, integration tests, security scans
  • Staging Stage: Deploy to staging environment for validation
  • Production Stage: Deploy to production environment
  • Post-Deploy Stage: Run smoke tests, health checks, monitoring

Pipeline Benefits:

  • Visibility: Clear view of deployment status at each stage
  • Automation: Reduces manual errors and increases speed
  • Quality Gates: Each stage acts as a quality checkpoint
  • Consistency: Standardized deployment process
  • Rollback: Easy rollback if issues are detected
Q13: What is environment promotion?

Environment Promotion is the process of moving code through environments (dev → test → staging → production) with validation at each stage. Each environment serves a specific purpose, and code must pass validation in one environment before being promoted to the next.

Environment Hierarchy:

  • Development Environment: For active development and initial testing
  • Test Environment: For integration testing and QA validation
  • Staging Environment: Production-like environment for final validation
  • Production Environment: Live environment serving end users

Validation at Each Stage:

  • Dev: Unit tests, basic functionality checks
  • Test: Integration tests, API testing, performance tests
  • Staging: End-to-end tests, user acceptance testing, security scans
  • Production: Smoke tests, health checks, monitoring
Q14: What is feature flagging?

Feature Flagging allows enabling/disabling features without deploying new code. It enables canary releases, A/B testing, and instant rollback by controlling feature availability through configuration rather than code deployment.

Feature Flag Benefits:

  • Instant Rollback: Disable problematic features without redeploying
  • Canary Releases: Roll out features to a subset of users
  • A/B Testing: Test different feature variations
  • Trunk-Based Development: Merge incomplete features behind flags
  • Gradual Rollout: Increase feature adoption gradually
feature-flag.js
// Feature flag example in JavaScript if (featureFlags.isEnabled("new-feature")) { // New feature code renderNewFeature(); } else { // Old feature code renderOldFeature(); }
Q15: What is blue-green deployment?

Blue-Green Deployment maintains two identical production environments. Traffic is switched from blue (old) to green (new) after testing. This strategy enables zero-downtime deployments and instant rollback by keeping the old version running until the new version is fully validated.

How It Works:

  • Blue Environment: Current production version serving live traffic
  • Green Environment: New version deployed and tested
  • Switch: Traffic routing switched from blue to green
  • Rollback: If issues occur, switch back to blue instantly

Benefits:

  • Zero Downtime: No service interruption during deployment
  • Instant Rollback: Switch back to previous version in seconds
  • Easy Testing: Test new version in production-like environment
  • Reduced Risk: Old version remains available as fallback
blue-green-deploy.sh (Bash)
# Blue-Green Deployment Script BLUE_ENV="blue.example.com" GREEN_ENV="green.example.com" # Deploy to green environment kubectl apply -f deployment-green.yaml kubectl rollout status deployment/green-app # Run health checks on green curl -f http://$GREEN_ENV/health || exit 1 # Switch traffic to green kubectl patch service my-service -p '{"spec":{"selector":{"version":"green"}}}' # Monitor and rollback if needed if ! curl -f http://$GREEN_ENV/health; then kubectl patch service my-service -p '{"spec":{"selector":{"version":"blue"}}}' echo "Rolled back to blue" fi
blue-green-deploy.py (Python)
import requests import subprocess def blue_green_deploy(): # Deploy to green environment subprocess.run(["kubectl", "apply", "-f", "deployment-green.yaml"]) # Wait for rollout subprocess.run(["kubectl", "rollout", "status", "deployment/green-app"]) # Health check if requests.get("http://green.example.com/health").status_code != 200: raise Exception("Health check failed") # Switch traffic subprocess.run(["kubectl", "patch", "service", "my-service", "-p", '{"spec":{"selector":{"version":"green"}}}']) if __name__ == "__main__": blue_green_deploy()

4. CI/CD Tools

Q16: What are popular CI/CD tools?

CI/CD Tools automate the build, test, and deployment processes. The choice of tool depends on factors like cloud vs. on-premise, integration with existing tools, team size, and specific requirements.

Popular CI/CD Tools:

  • Jenkins: Open-source, highly customizable with plugins, self-hosted, steep learning curve
  • GitHub Actions: Integrated with GitHub, YAML-based workflows, cloud-hosted, free for public repos
  • GitLab CI: Built into GitLab, uses .gitlab-ci.yml, includes container registry
  • CircleCI: Cloud-native, fast with Docker support, excellent documentation
  • Azure DevOps: Microsoft's solution, integrates with Azure, enterprise features
  • Bitbucket Pipelines: Integrated with Bitbucket, simple YAML configuration
  • Travis CI: Simple configuration, good for open-source projects
  • TeamCity: JetBrains' solution, excellent UI, self-hosted
Q17: What is Jenkins?

Jenkins is an open-source automation server that provides hundreds of plugins to support building, deploying, and automating any project. It's highly extensible, supports distributed builds, and can be self-hosted on any platform. Jenkins is one of the most widely used CI/CD tools due to its flexibility and large community.

Jenkins Key Features:

  • Plugin Ecosystem: Over 1,800 plugins for various integrations
  • Distributed Builds: Run builds across multiple agents
  • Pipeline as Code: Jenkinsfile defines pipelines in version control
  • Extensible: Custom plugins and shared libraries
  • Self-Hosted: Full control over infrastructure
Jenkinsfile
pipeline { agent any stages { stage('Build') { steps { sh 'mvn clean package' } } stage('Test') { steps { sh 'mvn test' } } } }
Q18: What is GitHub Actions?

GitHub Actions is GitHub's CI/CD platform that automates build, test, and deployment workflows directly within GitHub repositories. It uses YAML workflow files and integrates seamlessly with GitHub's ecosystem, including pull requests, issues, and releases.

GitHub Actions Key Features:

  • Native Integration: Works seamlessly with GitHub repositories
  • YAML Workflows: Simple, declarative workflow configuration
  • Marketplace: Thousands of pre-built actions and integrations
  • Free for Public Repos: No cost for open-source projects
  • Matrix Builds: Test across multiple OS and language versions
  • Self-Hosted Runners: Use your own infrastructure if needed
.github/workflows/ci.yml
name: CI on: [push] jobs: build: runs-on: ubuntu-latest steps: - uses: actions/checkout@v2 - name: Build run: npm run build - name: Test run: npm test
Q19: What is GitLab CI?

GitLab CI is integrated into GitLab, providing a complete DevOps platform. It uses .gitlab-ci.yml files for pipeline configuration and includes built-in container registry, deployment features, and Kubernetes integration. GitLab CI is part of GitLab's all-in-one solution.

GitLab CI Key Features:

  • Built-in Integration: Native to GitLab, no separate setup needed
  • Container Registry: Built-in Docker image registry
  • Kubernetes Integration: Easy deployment to K8s clusters
  • Auto DevOps: Automated CI/CD configuration
  • Environments: Built-in environment management
  • Self-Hosted: Can be self-hosted on your infrastructure
.gitlab-ci.yml
stages: - build - test - deploy build: stage: build script: - npm install - npm run build test: stage: test script: - npm test
Q20: What is the difference between Jenkins and GitHub Actions?

Jenkins and GitHub Actions are both popular CI/CD tools but have different approaches. Jenkins is a self-hosted, highly customizable solution with a plugin ecosystem, while GitHub Actions is a cloud-native solution integrated directly into GitHub.

Aspect Jenkins GitHub Actions
Hosting Self-hosted (requires infrastructure) Cloud-hosted (managed service)
Maintenance High (server maintenance, updates) Low (managed by GitHub)
Integration Plugin-based (requires setup) Native GitHub integration
Configuration Jenkinsfile (Groovy or declarative) YAML workflows
Cost Infrastructure costs only Free for public repos, paid for private
Extensibility 1,800+ plugins, highly extensible Marketplace actions, limited by GitHub

5. Pipeline Configuration

Q21: What is a Jenkinsfile?

Jenkinsfile is a text file that contains the definition of a Jenkins Pipeline. It's checked into source control alongside the application code, enabling pipeline as code. The Jenkinsfile can be written in declarative or scripted syntax and defines the entire build, test, and deployment process.

Jenkinsfile Benefits:

  • Pipeline as Code: Pipeline definition in version control
  • Reviewable: Pipeline changes go through code review
  • Versioned: Track pipeline changes over time
  • Single Source of Truth: Pipeline definition with the code
Jenkinsfile
pipeline { agent any stages { stage('Build') { steps { sh './gradlew build' } } stage('Deploy') { when { branch 'main' } steps { sh './deploy.sh' } } } }
Q22: What is the difference between declarative and scripted pipelines?

Declarative Pipelines use a structured, predefined syntax that's easier to read and write. Scripted Pipelines use Groovy scripting language, offering more flexibility but with a steeper learning curve. Declarative is recommended for most use cases, while scripted is for complex scenarios.

Aspect Declarative Pipeline Scripted Pipeline
Syntax Structured, predefined blocks Groovy scripting language
Complexity Simpler, easier to learn More flexible, steeper learning curve
Use Case Standard pipelines, most scenarios Complex logic, advanced scenarios
Error Handling Built-in error handling Manual error handling required
Readability High, self-documenting Lower, requires Groovy knowledge
Q23: What are pipeline stages?

Stages are logical divisions in a pipeline that group related steps together. Each stage represents a phase in the software delivery process (build, test, deploy). Stages can be executed sequentially or in parallel, and they provide clear visualization of the pipeline progress.

Stage Characteristics:

  • Logical Grouping: Groups related steps (e.g., all test steps)
  • Sequential Execution: Stages run in order by default
  • Parallel Execution: Independent stages can run in parallel
  • Visualization: Clear view of pipeline progress
  • Failure Handling: Pipeline stops if a stage fails
Q24: What are pipeline steps?

Steps are individual tasks within a stage that perform specific actions. Steps can be shell commands, script calls, plugin invocations, or other operations. Each step executes sequentially within a stage, and if a step fails, the stage fails.

Common Step Types:

  • Shell Commands: Execute bash, PowerShell, or shell scripts
  • Script Calls: Invoke build scripts (npm, maven, gradle)
  • Plugin Actions: Use CI/CD tool plugins
  • Checkout: Fetch code from version control
  • Upload/Download: Handle artifacts between stages
Q25: What is a pipeline trigger?

Trigger starts a pipeline execution automatically or manually. Triggers define when the pipeline should run based on events like code push, pull request, schedule, or manual initiation. Proper trigger configuration ensures pipelines run at the right time.

Common Trigger Types:

  • Push Trigger: Runs on code push to specific branches
  • Pull Request Trigger: Runs on PR creation or update
  • Scheduled Trigger: Runs at specific times (cron)
  • Manual Trigger: Requires manual initiation
  • Webhook Trigger: Triggered by external webhooks
.github/workflows/ci.yml
on: push: branches: [ main ] pull_request: branches: [ main ]

6. Testing in CI/CD

Q26: What types of tests should run in CI?

CI Test Types include various automated tests that validate code quality and functionality. The goal is to catch issues early and provide fast feedback to developers. Tests should be organized by speed and complexity, with faster tests running first.

CI Test Types:

  • Unit Tests: Fast tests that test individual components in isolation (seconds)
  • Integration Tests: Test component interactions and APIs (minutes)
  • Static Analysis: Code quality checks, linting, style checks (seconds)
  • Security Scans: Dependency vulnerability detection (minutes)
  • Contract Tests: Verify API contracts between services (minutes)

Test Pyramid:

  • Base: Many unit tests (fast, cheap)
  • Middle: Fewer integration tests (slower, more expensive)
  • Top: Fewest end-to-end tests (slowest, most expensive)
test.spec.js (JavaScript/Jest)
// Unit test example describe('Calculator', () => { test('adds two numbers', () => { const result = add(2, 3); expect(result).toBe(5); }); }); // Integration test example describe('API Integration', () => { test('fetches user data', async () => { const response = await fetch('/api/users/1'); const data = await response.json(); expect(data.id).toBe(1); }); });
test_calculator.py (Python/pytest)
import pytest # Unit test example def test_addition(): result = add(2, 3) assert result == 5 # Integration test example def test_api_user(client): response = client.get('/api/users/1') assert response.status_code == 200 assert response.json()['id'] == 1
Q27: What is test parallelization?

Test Parallelization runs tests in parallel across multiple agents or containers to reduce overall execution time. Instead of running tests sequentially, parallelization splits the test suite and runs subsets simultaneously, significantly reducing feedback time.

Parallelization Strategies:

  • File-Based: Split test files across agents
  • Test-Based: Split individual tests across agents
  • Matrix Builds: Run tests across multiple OS/language versions
  • Sharding: Divide tests into shards based on execution time

Benefits:

  • Faster Feedback: Reduce test execution time significantly
  • Better Resource Utilization: Use multiple agents efficiently
  • Scalability: Handle growing test suites
Q28: What is code coverage?

Code Coverage measures how much code is executed by tests, typically expressed as a percentage. It helps identify untested code and ensures adequate test coverage. Coverage metrics include line coverage, branch coverage, function coverage, and statement coverage.

Coverage Metrics:

  • Line Coverage: Percentage of executable lines executed
  • Branch Coverage: Percentage of decision branches executed
  • Function Coverage: Percentage of functions called
  • Statement Coverage: Percentage of statements executed
package.json
# Run tests with coverage npm test -- --coverage # Set coverage threshold in package.json "jest": { "coverageThreshold": { "global": { "branches": 80, "functions": 80, "lines": 80, "statements": 80 } } }
Q29: What is SonarQube?

SonarQube is an automatic code review tool that detects bugs, code smells, and security vulnerabilities. It provides continuous inspection of code quality and integrates seamlessly with CI pipelines. SonarQube tracks technical debt and provides actionable insights for improving code quality.

SonarQube Features:

  • Bug Detection: Identifies potential bugs in code
  • Code Smells: Detects maintainability issues
  • Security Vulnerabilities: Finds security hotspots
  • Code Coverage: Tracks test coverage metrics
  • Duplication Detection: Identifies duplicated code
  • Quality Gates: Enforces quality standards
Q30: What is test reporting?

Test Reporting generates and displays test results from CI pipelines. Most CI tools provide built-in test reporting with pass/fail status, execution time, and historical trends. Good test reporting helps teams understand test health and identify flaky tests.

Test Report Components:

  • Pass/Fail Status: Overall test suite result
  • Test Details: Individual test results and failure messages
  • Execution Time: Time taken for each test and suite
  • Historical Trends: Test performance over time
  • Flaky Test Detection: Identify inconsistent tests

7. Deployment Strategies

Q31: What is canary deployment?

Canary Deployment rolls out new version to a small subset of users before full rollout. It reduces risk by catching issues early with minimal impact. The deployment gradually increases the percentage of users receiving the new version based on metrics and success criteria.

Canary Deployment Process:

  • Initial Rollout: Deploy to 1-5% of users
  • Monitor Metrics: Track error rates, latency, user feedback
  • Gradual Increase: Increase traffic if metrics are healthy
  • Rollback: Roll back if issues are detected
  • Full Rollout: Complete rollout after validation

Benefits:

  • Risk Reduction: Issues affect only small user subset
  • Real-World Testing: Test with actual users and traffic
  • Gradual Rollout: Control deployment pace
  • Quick Rollback: Easy to revert if problems occur
canary-deploy.sh (Bash)
# Canary Deployment Script CANARY_PERCENTAGE=5 # Deploy canary version kubectl apply -f deployment-canary.yaml # Update service to route 5% traffic to canary kubectl patch service my-service -p '{"spec":{"selector":{"version":"canary"}}}' kubectl patch service my-service -p '{"spec":{"selector":{"canary-weight":"5"}}}' # Monitor for 5 minutes for i in {1..30}; do ERROR_RATE=$(curl -s http://canary.example.com/metrics | jq '.error_rate') if (( $(echo "$ERROR_RATE > 0.01" | bc) )); then kubectl patch service my-service -p '{"spec":{"selector":{"version":"stable"}}}' echo "Canary failed, rolled back" exit 1 fi sleep 10 done # Gradually increase traffic for pct in 10 25 50 100; do kubectl patch service my-service -p "{'spec':{'selector':{'canary-weight':'$pct'}}}" sleep 60 done
canary-deploy.js (Node.js)
const { exec } = require('child_process'); const axios = require('axios'); async function canaryDeploy() { // Deploy canary version await exec('kubectl apply -f deployment-canary.yaml'); // Route 5% traffic to canary await exec('kubectl patch service my-service -p \'{"spec":{"selector":{"version":"canary"}}}\''); // Monitor metrics for (let i = 0; i < 30; i++) { const response = await axios.get('http://canary.example.com/metrics'); if (response.data.error_rate > 0.01) { await exec('kubectl patch service my-service -p \'{"spec":{"selector":{"version":"stable"}}}\''); throw new Error('Canary failed, rolled back'); } await new Promise(resolve => setTimeout(resolve, 10000)); } // Gradual rollout const percentages = [10, 25, 50, 100]; for (const pct of percentages) { await exec(`kubectl patch service my-service -p '{\"spec\":{\"selector\":{\"canary-weight\":\"${pct}\"}}}'`); await new Promise(resolve => setTimeout(resolve, 60000)); } } canaryDeploy().catch(console.error);
Q32: What is rolling deployment?

Rolling Deployment gradually replaces old instances with new ones, maintaining availability during deployment. It updates instances one by one or in batches, ensuring the application remains available throughout the deployment process.

Rolling Deployment Characteristics:

  • Gradual Update: Replace instances incrementally
  • No Downtime: Application remains available
  • Batch Processing: Update in configurable batch sizes
  • Health Checks: Verify new instances before proceeding
  • Rollback Capability: Can roll back if issues occur

Considerations:

  • Requires multiple instances for availability
  • Slower deployment than blue-green
  • May have mixed versions during deployment
Q33: What is A/B testing?

A/B Testing compares two versions of a feature by showing different versions to different users and measuring performance. It's used to make data-driven decisions about which version performs better based on metrics like conversion rates, user engagement, or other KPIs.

A/B Testing Process:

  • Hypothesis: Define what you're testing and why
  • Split Traffic: Divide users between control (A) and variant (B)
  • Collect Data: Gather metrics on both versions
  • Analyze Results: Compare performance statistically
  • Implement Winner: Roll out the better-performing version

Use Cases:

  • UI/UX improvements
  • Feature validation
  • Pricing strategies
  • Marketing copy optimization
Q34: What is immutable infrastructure?

Immutable Infrastructure replaces servers instead of modifying them. Once deployed, servers are never changed, only replaced. This approach eliminates configuration drift and ensures consistency across environments.

Immutable Infrastructure Principles:

  • Never Modify: Don't change servers after deployment
  • Replace Instead: Deploy new instances for changes
  • Versioned Images: Use versioned machine images or containers
  • Disposable: Treat infrastructure as disposable

Benefits:

  • Consistency: No configuration drift
  • Reproducibility: Easy to recreate environments
  • Rollback: Simple by reverting to previous image
  • Scalability: Easy to scale horizontally
Q35: What is infrastructure as code?

9. CI/CD Best Practices

Q42: What are CI/CD best practices?

CI/CD Best Practices ensure efficient, reliable, and secure software delivery. Following these practices helps teams maximize the benefits of CI/CD while avoiding common pitfalls.

Key Best Practices:

  • Fast Feedback: Keep pipelines fast to provide quick feedback to developers
  • Fail Fast: Run quick tests first to fail early and save time
  • Parallel Execution: Run independent tasks in parallel to reduce overall time
  • Artifact Caching: Cache dependencies and build artifacts to speed up builds
  • Security: Integrate security scanning throughout the pipeline
  • Small Batches: Make small, frequent changes instead of large batches
  • Automate Everything: Automate manual processes to reduce errors
Q42: How do you optimize pipeline performance?

Pipeline Performance Optimization reduces build and deployment time, improving developer productivity and enabling faster releases. Optimization techniques focus on reducing wait times and maximizing resource utilization.

Optimization Techniques:

  • Parallel Stages: Run independent stages concurrently
  • Caching: Cache dependencies, build artifacts, and Docker layers
  • Incremental Builds: Build only changed code instead of full rebuilds
  • Resource Allocation: Use appropriate agent types (faster for build, cheaper for test)
  • Test Parallelization: Split tests across multiple agents
  • Dependency Management: Optimize dependency resolution
Q43: What is pipeline security?

Pipeline Security integrates security practices throughout the CI/CD pipeline to identify and mitigate vulnerabilities early. DevSecOps shifts security left, making it everyone's responsibility.

Security Practices:

  • Secrets Management: Use vaults, never store secrets in code
  • Least Privilege: Grant minimal permissions to pipeline agents
  • Dependency Scanning: Check for known vulnerabilities in dependencies
  • Container Scanning: Scan Docker images for vulnerabilities
  • SAST/DAST: Static and dynamic application security testing
  • Code Signing: Sign artifacts to ensure integrity

Q48: What is the difference between push and pull deployment?

Push Deployment has the CI pipeline push changes to target environments. Pull Deployment has target environments pull changes from Git or a registry. Pull deployment is the foundation of GitOps.

Aspect Push Deployment Pull Deployment
Trigger CI pipeline pushes changes Cluster pulls changes from Git
Control Centralized in CI pipeline Delegated to cluster agents
Use Case Traditional CI/CD GitOps, Kubernetes
Security Requires pipeline credentials Cluster has read-only Git access
Q49: How do you handle rollback in CI/CD?

Rollback is the process of reverting to a previous version when a deployment fails or causes issues. Effective rollback strategies minimize downtime and user impact.

Rollback Strategies:

  • Version Tags: Tag releases for easy rollback to specific versions
  • Blue-Green: Switch back to old environment instantly
  • Database Migrations: Use reversible migrations for rollback
  • Feature Flags: Disable problematic features without redeploying
  • Canary Rollback: Reduce canary traffic to zero
rollback.sh (Bash)
# Rollback script for Kubernetes PREVIOUS_VERSION=$1 # Rollback deployment kubectl rollout undo deployment/myapp --to-revision=$PREVIOUS_VERSION # Or rollback to previous version kubectl rollout undo deployment/myapp # Verify rollback kubectl rollout status deployment/myapp

const { exec } = require('child_process'); async function rollback(version) { // Rollback to specific version await exec(`kubectl rollout undo deployment/myapp --to-revision=${version}`); // Wait for rollback to complete await exec('kubectl rollout status deployment/myapp'); console.log('Rollback completed successfully'); } // Rollback to previous version rollback().catch(console.error);

9. Advanced CI/CD Concepts

Q51: What is a monorepo vs. polyrepo in CI/CD?

Monorepo stores all projects in a single repository, while polyrepo uses separate repositories for each project. Each approach has implications for CI/CD pipeline design and complexity.

Aspect Monorepo Polyrepo
Structure Single repository for all projects Separate repository per project
CI/CD Complexity Complex path filtering, shared pipelines Simpler per-project pipelines
Code Sharing Easy shared libraries and dependencies Requires package management for sharing
Build Time Can be long, requires smart caching Shorter, focused builds
Q52: What is pipeline caching?

Pipeline Caching stores dependencies and build artifacts between pipeline runs to speed up subsequent builds. Caching reduces download times and compilation time by reusing previously computed results.

Cache Types:

  • Dependency Cache: Cache npm, maven, pip dependencies
  • Build Cache: Cache compiled code and build outputs
  • Docker Layer Cache: Cache Docker image layers
  • Custom Cache: Cache any reusable artifacts

Cache Strategies:

  • Key-Based: Cache based on dependency file hash
  • Time-Based: Cache with expiration time
  • Size-Based: Limit cache size
Q53: What is matrix builds?

Matrix Builds run the same pipeline across multiple configurations, such as different operating systems, language versions, or environments. This ensures the application works across all supported platforms.

Matrix Use Cases:

  • Multi-OS Testing: Test on Windows, Linux, macOS
  • Language Versions: Test with Node.js 14, 16, 18
  • Database Versions: Test with PostgreSQL 12, 13, 14
  • Browsers: Test on Chrome, Firefox, Safari
.github/workflows/matrix.yml
strategy: matrix: os: [ubuntu-latest, windows-latest] node: [14, 16, 18]
Q54: What is pipeline chaining?

Pipeline Chaining connects multiple pipelines together, where the output of one pipeline triggers the next. This enables complex workflows across different projects or stages.

Chaining Patterns:

  • Sequential: One pipeline completes before the next starts
  • Parallel: Multiple pipelines run simultaneously
  • Fan-Out/Fan-In: Split into parallel, then merge results
  • Conditional: Next pipeline runs based on conditions
Q55: What is pipeline templates?

Pipeline Templates provide reusable pipeline definitions that can be shared across projects. Templates ensure consistency and reduce duplication in pipeline configuration.

Template Benefits:

  • Consistency: Standardized pipeline structure
  • Reuse: Share common pipeline logic
  • Maintenance: Update once, apply everywhere
  • Compliance: Enforce organizational standards
Q56: What is self-hosted runners?

Self-Hosted Runners are build agents that you host and manage yourself, instead of using cloud-hosted agents. They provide control over the build environment and can access private resources.

Self-Hosted Runner Benefits:

  • Custom Environment: Install specific tools and dependencies
  • Private Network: Access private resources without internet
  • Cost Control: Use existing infrastructure
  • Compliance: Keep builds within specific regions
Q57: What is pipeline artifacts?

Pipeline Artifacts are files generated during a pipeline that need to be passed between stages or stored for later use. Examples include build outputs, test reports, and deployment packages.

Artifact Types:

  • Build Artifacts: Compiled binaries, JARs, WARs
  • Test Reports: JUnit XML, coverage reports
  • Documentation: Generated docs, API specs
  • Deployment Packages: Docker images, installers
Q58: What is pipeline notifications?

Pipeline Notifications alert team members about pipeline status changes. Notifications can be sent via Slack, email, Microsoft Teams, or other communication channels.

Notification Triggers:

  • Success: Pipeline completed successfully
  • Failure: Pipeline failed
  • Started: Pipeline started running
  • Approval Needed: Manual approval required
Q59: What is pipeline approval gates?

Approval Gates require manual approval before a pipeline can proceed to the next stage. They provide human control over critical deployments and ensure proper review.

Approval Use Cases:

  • Production Deployment: Approve before production
  • Security Changes: Review security-related changes
  • Database Migrations: Approve schema changes
  • Compliance: Ensure regulatory compliance
Q60: What is pipeline resource limits?

Resource Limits control the amount of CPU, memory, and disk space available to pipeline runs. They prevent runaway pipelines from consuming excessive resources and ensure fair resource allocation.

Resource Limit Types:

  • CPU Limits: Maximum CPU usage
  • Memory Limits: Maximum memory allocation
  • Timeout Limits: Maximum pipeline duration
  • Storage Limits: Maximum disk usage

10. CI/CD Security

Q61: What is SAST in CI/CD?

Static Application Security Testing (SAST) analyzes source code for security vulnerabilities without executing the code. SAST tools scan code for common security issues like SQL injection, XSS, and insecure coding practices.

SAST Benefits:

  • Early Detection: Find issues during development
  • Code-Level: Identify exact location of vulnerabilities
  • Fast Feedback: Quick results for developers
  • Integration: Easy to integrate into CI pipelines
Q62: What is DAST in CI/CD?

Dynamic Application Security Testing (DAST) tests running applications for security vulnerabilities by simulating attacks. DAST tools interact with the application like a hacker would, identifying runtime security issues.

DAST Benefits:

  • Runtime Issues: Find vulnerabilities in running applications
  • Real-World Testing: Simulate actual attack scenarios
  • Configuration Issues: Identify misconfigurations
  • Compliance: Meet security compliance requirements
Q63: What is dependency scanning?

Dependency Scanning checks project dependencies for known security vulnerabilities. It scans package managers like npm, maven, pip, and cargo for vulnerable libraries.

Dependency Scanning Tools:

  • Snyk: Comprehensive vulnerability scanning
  • OWASP Dependency-Check: Open-source scanner
  • Trivy: Container and file system scanner
  • GitHub Dependabot: Built-in GitHub scanning
Q64: What is container scanning?

Container Scanning scans Docker images for security vulnerabilities, malware, and configuration issues. It ensures that container images are secure before deployment.

Container Scanning Tools:

  • Trivy: Comprehensive container scanner
  • Clair: Open-source vulnerability scanner
  • Aqua Security: Enterprise container security
  • Docker Scout: Built-in Docker scanning
Q65: What is code signing?

Code Signing digitally signs build artifacts to verify their authenticity and integrity. It ensures that artifacts haven't been tampered with and can be traced back to the source.

Code Signing Benefits:

  • Integrity: Verify artifacts haven't been modified
  • Authenticity: Confirm the source of artifacts
  • Trust: Build trust with users
  • Compliance: Meet regulatory requirements
Q66: What is supply chain security?

Supply Chain Security protects the software supply chain from attacks. It includes verifying the integrity of dependencies, build tools, and deployment infrastructure.

Supply Chain Security Practices:

  • SBOM: Software Bill of Materials for transparency
  • Dependency Pinning: Use specific dependency versions
  • Artifact Verification: Verify artifact signatures
  • Build Reproducibility: Ensure reproducible builds
Q67: What is SBOM (Software Bill of Materials)?

Software Bill of Materials (SBOM) is a comprehensive list of all components, libraries, and dependencies in a software product. It provides transparency into the software supply chain and helps identify vulnerabilities.

SBOM Benefits:

  • Transparency: Know all components in your software
  • Vulnerability Management: Track and fix vulnerabilities
  • Compliance: Meet regulatory requirements
  • Licensing: Track license compliance
Q68: What is infrastructure security in CI/CD?

Infrastructure Security protects the CI/CD infrastructure from attacks. It includes securing build agents, artifact repositories, and deployment targets.

Infrastructure Security Practices:

  • Network Isolation: Isolate build environments
  • Least Privilege: Minimal permissions for agents
  • Immutable Agents: Use ephemeral build agents
  • Audit Logging: Log all infrastructure changes
Q69: What is compliance as code?

Compliance as Code defines compliance requirements as code and enforces them through automated checks. It ensures that deployments meet regulatory and organizational standards.

Compliance as Code Benefits:

  • Automation: Automated compliance checks
  • Consistency: Uniform compliance enforcement
  • Documentation: Compliance rules as code
  • Auditability: Track compliance over time
Q70: What is incident response in CI/CD?

Incident Response in CI/CD involves detecting, responding to, and recovering from security incidents in the pipeline. It includes automated rollback, alerting, and post-incident analysis.

Incident Response Steps:

  • Detection: Identify security incidents
  • Containment: Isolate affected systems
  • Eradication: Remove the threat
  • Recovery: Restore normal operations
  • Post-Incident Analysis: Learn and improve
Q71: What is shift-left security?

Shift-Left Security moves security testing earlier in the development lifecycle. Instead of testing security at the end, it's integrated throughout development, catching vulnerabilities early when they're cheaper to fix.

Shift-Left Security Practices:

  • Developer Training: Security training for developers
  • Secure Coding: Security guidelines and linting
  • Pre-Commit Hooks: Security checks before commit
  • CI Integration: Security scanning in CI pipeline
Q72: What is policy as code?

Policy as Code defines organizational policies as code and enforces them through automated checks. It ensures that deployments follow organizational standards and compliance requirements.

Policy as Code Tools:

  • OPA (Open Policy Agent): General-purpose policy engine
  • Conftest: Policy testing for configuration files
  • Gatekeeper: Kubernetes policy controller
  • Policy as Code in CI: Enforce policies in pipelines
Q73: What is zero trust in CI/CD?

Zero Trust in CI/CD assumes no trust by default and requires verification for every access request. It applies zero trust principles to pipeline infrastructure, build agents, and deployment targets.

Zero Trust Principles in CI/CD:

  • Verify Always: Verify every request and access
  • Least Privilege: Minimal permissions for all components
  • Micro-Segmentation: Isolate pipeline components
  • Continuous Monitoring: Monitor all pipeline activity
Q74: What is runtime security?

Runtime Security monitors and protects applications during execution. It detects and prevents security threats in production, such as injection attacks, privilege escalation, and data exfiltration.

Runtime Security Tools:

  • Application Security: WAF, RASP solutions
  • Container Security: Runtime protection for containers
  • Cloud Security: CSPM and CWPP tools
  • Monitoring: Security monitoring and alerting
Q75: What is security posture management?

Security Posture Management continuously assesses and improves the security posture of CI/CD infrastructure and applications. It provides visibility into security risks and recommends remediation actions.

Security Posture Management Benefits:

  • Visibility: Comprehensive security visibility
  • Risk Assessment: Identify and prioritize risks
  • Compliance: Monitor compliance status
  • Remediation: Automated remediation recommendations
