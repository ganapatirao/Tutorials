1. OOPS Fundamentals

Q1: What is Object-Oriented Programming (OOP)?

Object-Oriented Programming (OOP) is a programming paradigm based on the concept of "objects", which can contain data and code. The data is in the form of fields (often known as attributes or properties), and the code is in the form of procedures (often known as methods).

  • Object: An instance of a class with state and behavior
  • Class: A blueprint or template for creating objects
  • State: Represented by attributes/fields of an object
  • Behavior: Represented by methods of an object
// Class definition
class Car {
    // State (attributes)
    private string brand;
    private string model;
    private int year;
    
    // Constructor
    public Car(string brand, string model, int year) {
        this.brand = brand;
        this.model = model;
        this.year = year;
    }
    
    // Behavior (methods)
    public void Start() {
        Console.WriteLine("Car is starting");
    }
    
    public void Accelerate() {
        Console.WriteLine("Car is accelerating");
    }
}

// Object creation
Car myCar = new Car("Toyota", "Camry", 2022);
myCar.Start();
Q2: What are the four main principles of OOP?

1. Encapsulation: Bundling data and methods that operate on the data within a single unit (class), and restricting access to the internal state.

2. Inheritance: Creating new classes based on existing classes, inheriting their properties and methods.

3. Polymorphism: Objects can take many forms. The same method can behave differently based on the object.

4. Abstraction: Hiding implementation details and showing only functionality to the user.

2. Encapsulation

Q4: What is encapsulation and why is it important?

Encapsulation is the bundling of data and methods that operate on that data within a single unit (class). It also involves restricting direct access to some of an object's components.

Benefits:

  • Data Hiding: Protects internal state from external interference
  • Flexibility: Internal implementation can change without affecting external code
  • Maintainability: Easier to understand and modify
  • Reusability: Encapsulated code can be reused in different contexts
class BankAccount {
    // Private fields (encapsulated)
    private decimal balance;
    private string accountNumber;
    
    // Constructor
    public BankAccount(string accountNumber, decimal initialBalance) {
        this.accountNumber = accountNumber;
        this.balance = initialBalance;
    }
    
    // Public methods to access/modify private data
    public decimal GetBalance() {
        return balance;
    }
    
    public void Deposit(decimal amount) {
        if (amount > 0) {
            balance += amount;
        }
    }
    
    public bool Withdraw(decimal amount) {
        if (amount > 0 && balance >= amount) {
            balance -= amount;
            return true;
        }
        return false;
    }
}
Tip: Use access modifiers (private, protected, public) to implement encapsulation. Make fields private and provide public methods (getters/setters) to access them.
Q5: What are access modifiers in OOP?

Access modifiers control the visibility and accessibility of class members.

Modifier Same Class Same Package Subclass World
public ✓ ✓ ✓ ✓
protected ✓ ✓ ✓ ✗
default (package) ✓ ✓ ✗ ✗

class Example {
    public int publicField = 1;        // Accessible everywhere
    protected int protectedField = 2;   // Accessible in class and subclasses
    int packageField = 3;              // Accessible in same package
    private int privateField = 4;       // Accessible only in this class

}

3. Inheritance

Q6: What is inheritance and what are its types?

Inheritance is a mechanism where a new class derives properties and behavior from an existing class. It promotes code reusability and establishes a parent-child relationship between classes.

Types of Inheritance:

  • Single Inheritance: A class inherits from one parent class
  • Multiple Inheritance: A class inherits from multiple parent classes (not supported in C#, supported in C++)
  • Multilevel Inheritance: A class inherits from a derived class (chain inheritance)
  • Hierarchical Inheritance: Multiple classes inherit from a single parent class
  • Hybrid Inheritance: Combination of multiple inheritance types
// Single Inheritance
class Animal {
    public void Eat() {
        Console.WriteLine("Animal is eating");
    }
}

class Dog : Animal {
    public void Bark() {
        Console.WriteLine("Dog is barking");
    }
}

// Multilevel Inheritance
class Vehicle {
    public void Move() {
        Console.WriteLine("Vehicle is moving");
    }
}

class Car : Vehicle {
    public void Drive() {
        Console.WriteLine("Car is driving");
    }
}

class SportsCar : Car {
    public void Race() {
        Console.WriteLine("Sports car is racing");
    }
}
Q7: What is the difference between composition and inheritance?
Aspect Inheritance Composition
Relationship "Is-A" relationship "Has-A" relationship
Tight Coupling High coupling Loose coupling
Flexibility Less flexible More flexible
Runtime Behavior Determined at compile time Can change at runtime
Reuse Code reuse via inheritance Code reuse via delegation 
// Inheritance (Is-A)
class Dog : Animal {
    // Dog inherits from Animal
}

// Composition (Has-A)
class Car {
    private Engine engine; // Car has an Engine
    
    public Car() {
        engine = new Engine();
    }
}

// Composition example with flexibility
class Logger {
    public void Log(string message) {
        Console.WriteLine(message);
    }
}

class UserService {
    private Logger logger; // Composition
    
    public UserService(Logger logger) {
        this.logger = logger; // Dependency injection
    }
    
    public void CreateUser(string name) {
        logger.Log($"Creating user: {name}");
        // User creation logic
    }
}
Important: Favor composition over inheritance when possible. Composition provides more flexibility and loose coupling.

4. Polymorphism

Q8: What is polymorphism and what are its types?

Polymorphism means "many forms". It allows objects of different types to be treated as objects of a common type. The same method can behave differently based on the object it's called on.

Types of Polymorphism:

  • Compile-time Polymorphism (Static Binding): Method overloading, operator overloading
  • Runtime Polymorphism (Dynamic Binding): Method overriding, virtual methods
// Compile-time Polymorphism (Method Overloading)
class Calculator {
    public int Add(int a, int b) {
        return a + b;
    }
    
    public double Add(double a, double b) {
        return a + b;
    }
    
    public int Add(int a, int b, int c) {
        return a + b + c;
    }
}

// Runtime Polymorphism (Method Overriding)
class Shape {
    public virtual void Draw() {
        Console.WriteLine("Drawing shape");
    }
}

class Circle : Shape {
    public override void Draw() {
        Console.WriteLine("Drawing circle");
    }
}

class Rectangle : Shape {
    public override void Draw() {
        Console.WriteLine("Drawing rectangle");
    }
}

// Usage
Shape shape = new Circle();
shape.Draw(); // Output: Drawing circle (runtime polymorphism)
Q9: What is the difference between method overloading and method overriding?
Aspect Method Overloading Method Overriding
Definition Same method name, different parameters Same method name, same parameters in subclass
Occurrence Same class Different classes (inheritance)
Binding Compile-time (static) Runtime (dynamic)
Return Type Can be different or same Must be same or covariant
Access Modifier Can change Cannot be more restrictive
Aspect Method Overloading Method Overriding
Keywords No specific keyword virtual/override keywords 
// Method Overloading
class Printer {
    public void Print(string text) {
        Console.WriteLine(text);
    }
    
    public void Print(int number) {
        Console.WriteLine(number);
    }
    
    public void Print(string text, int count) {
        for (int i = 0; i < count; i++) {
            Console.WriteLine(text);
        }
    }
}

// Method Overriding
class Animal {
    public virtual void MakeSound() {
        Console.WriteLine("Animal makes a sound");
    }
}

class Cat : Animal {
    public override void MakeSound() {
        Console.WriteLine("Meow");
    }
}

5. Abstraction

Q10: What is abstraction and how is it implemented?

Abstraction is the concept of hiding complex implementation details and showing only the essential features of an object. It focuses on what an object does rather than how it does it.

Ways to implement abstraction:

  • Abstract Classes: Can have both abstract and concrete methods
  • Interfaces: Define a contract without implementation
  • Access Modifiers: Hide internal implementation details
// Abstract Class
public abstract class Vehicle {
    public abstract void Start(); // Abstract method
    public abstract void Stop();  // Abstract method
    
    public void Honk() { // Concrete method
        Console.WriteLine("Beep beep!");
    }
}

public class Car : Vehicle {
    public override void Start() {
        Console.WriteLine("Car engine starting");
    }
    
    public override void Stop() {
        Console.WriteLine("Car engine stopping");
    }
}

// Interface
public interface IShape {
    double GetArea();
    double GetPerimeter();
}

public class Circle : IShape {
    private double radius;
    
    public Circle(double radius) {
        this.radius = radius;
    }
    
    public double GetArea() {
        return Math.PI * radius * radius;
    }
    
    public double GetPerimeter() {
        return 2 * Math.PI * radius;
    }
}

6. SOLID Principles

Q12: What are the SOLID principles?

S - Single Responsibility Principle (SRP): A class should have only one reason to change.

// Bad - Multiple responsibilities
class User {
    public string Name { get; set; }
    public string Email { get; set; }
    
    public void SaveToDatabase() { }
    public void SendEmail() { }
    public void Validate() { }
}

// Good - Single responsibility
class User {
    public string Name { get; set; }
    public string Email { get; set; }
}

class UserRepository {
    public void Save(User user) { }
}

class EmailService {
    public void SendEmail(User user) { }
}

class UserValidator {
    public bool Validate(User user) { }
}

O - Open/Closed Principle (OCP): Software entities should be open for extension but closed for modification.

// Bad - Needs modification for new shapes
class AreaCalculator {
    public double CalculateArea(object shape) {
        if (shape is Circle)
            return Math.PI * Math.Pow(((Circle)shape).Radius, 2);
        else if (shape is Rectangle)
            return ((Rectangle)shape).Width * ((Rectangle)shape).Height;
        // Need to add more conditions for new shapes
    }
}

// Good - Open for extension
interface IShape {
    double CalculateArea();
}

class Circle : IShape {
    public double Radius { get; set; }
    public double CalculateArea() {
        return Math.PI * Math.Pow(Radius, 2);
    }
}

class Rectangle : IShape {
    public double Width { get; set; }
    public double Height { get; set; }
    public double CalculateArea() {
        return Width * Height;
    }
}

L - Liskov Substitution Principle (LSP): Subtypes must be substitutable for their base types.

// Bad - Violates LSP
class Bird {
    public virtual void Fly() {
        Console.WriteLine("Flying");
    }
}

class Penguin : Bird {
    public override void Fly() {
        throw new Exception("Penguins can't fly");
    }
}

// Good - Follows LSP
class Bird { }

class FlyingBird : Bird {
    public virtual void Fly() {
        Console.WriteLine("Flying");
    }
}

class Sparrow : FlyingBird {
    public override void Fly() {
        Console.WriteLine("Sparrow flying");
    }
}

class Penguin : Bird {
    public void Swim() {
        Console.WriteLine("Swimming");
    }
}

I - Interface Segregation Principle (ISP): Clients should not be forced to depend on interfaces they don't use.

// Bad - Fat interface
interface IWorker {
    void Work();
    void Eat();
    void Sleep();
}

class Robot : IWorker {
    public void Work() { }
    public void Eat() { throw new Exception(); } // Not needed
    public void Sleep() { throw new Exception(); } // Not needed
}

// Good - Segregated interfaces
interface IWorker {
    void Work();
}

interface IEater {
    void Eat();
}

interface ISleeper {
    void Sleep();
}

class Human : IWorker, IEater, ISleeper {
    public void Work() { }
    public void Eat() { }
    public void Sleep() { }
}

class Robot : IWorker {
    public void Work() { }
}

D - Dependency Inversion Principle (DIP): Depend on abstractions, not concretions.

// Bad - Depends on concrete implementation
class LightBulb {
    public void TurnOn() {
        Console.WriteLine("Light bulb on");
    }
}

class Switch {
    private LightBulb bulb;
    
    public Switch() {
        bulb = new LightBulb(); // Tight coupling
    }
    
    public void Toggle() {
        bulb.TurnOn();
    }
}

// Good - Depends on abstraction
interface ISwitchable {
    void TurnOn();
    void TurnOff();
}

class LightBulb : ISwitchable {
    public void TurnOn() {
        Console.WriteLine("Light bulb on");
    }
    
    public void TurnOff() {
        Console.WriteLine("Light bulb off");
    }
}

class Switch {
    private ISwitchable device;
    
    public Switch(ISwitchable device) {
        this.device = device; // Dependency injection
    }
    
    public void Toggle() {
        device.TurnOn();
    }
}

7. Design Patterns

Q13: What are the common design patterns in OOP?

Creational Patterns: Deal with object creation mechanisms

  • Singleton: Ensures a class has only one instance
  • Factory Method: Creates objects without specifying exact class
  • Builder: Constructs complex objects step by step

Structural Patterns: Deal with composition of classes and objects

  • Adapter: Allows incompatible interfaces to work together
  • Decorator: Adds functionality dynamically
  • Facade: Provides simplified interface to complex system

Behavioral Patterns: Deal with communication between objects

  • Observer: Defines subscription mechanism
  • Strategy: Encapsulates interchangeable algorithms
  • Command: Encapsulates requests as objects
Q14: What is the Singleton pattern?

The Singleton pattern ensures a class has only one instance and provides a global point of access to it.

public sealed class Singleton {
    private static Singleton instance = null;
    private static readonly object padlock = new object();
    
    private Singleton() { }
    
    public static Singleton Instance {
        get {
            if (instance == null) {
                lock (padlock) {
                    if (instance == null) {
                        instance = new Singleton();
                    }
                }
            }
            return instance;
        }
    }
    
    public void DoSomething() {
        Console.WriteLine("Singleton is working");
    }
}

// Usage
Singleton singleton = Singleton.Instance;
singleton.DoSomething();
Tip: Use Singleton when you need exactly one instance of a class (e.g., database connection, logging service). Be careful with Singleton as it can make testing difficult.

8. OOPS vs Procedural Programming

Q15: What is the difference between OOP and procedural programming?
Aspect OOP Procedural Programming
Focus Objects and their interactions Functions and procedures
Data Data and methods bundled together Data separate from functions
Access Controlled via access modifiers Global access to data
Reuse Inheritance and composition Function libraries
Complexity Handles complex systems better Simpler for small programs
Example C#, Java, Python C, Pascal, Fortran 
// Procedural Approach
struct Person {
    char name[50];
    int age;
};

void printPerson(struct Person p) {
    printf("Name: %s, Age: %d\n", p.name, p.age);
}

// OOP Approach
class Person {
    private string name;
    private int age;
    
    public Person(string name, int age) {
        this.name = name;
        this.age = age;
    }
    
    public void Print() {
        Console.WriteLine($"Name: {name}, Age: {age}");
    }
}
