1. Kubernetes Fundamentals

Q1: What is Kubernetes?

Kubernetes (K8s) is an open-source container orchestration platform for automating deployment, scaling, and management of containerized applications. It was originally designed by Google and is now maintained by the Cloud Native Computing Foundation (CNCF).

Key Features:

  • Automated Rollouts and Rollbacks: Gradually roll out changes and roll back if needed
  • Service Discovery: Automatic discovery of services within the cluster
  • Load Balancing: Distribute traffic across container instances
  • Self-Healing: Automatically restart failed containers
  • Horizontal Scaling: Scale applications up and down based on demand
  • Storage Orchestration: Automatically mount storage systems
Q2: What is the difference between Kubernetes and Docker Swarm?

Kubernetes is a more feature-rich, complex orchestration platform, while Docker Swarm is simpler and easier to set up but with fewer features.

Aspect Kubernetes Docker Swarm
Complexity Complex, steep learning curve Simple, easy to learn
Features Advanced (auto-scaling, RBAC, etc.) Basic orchestration
Ecosystem Large, vendor-neutral Docker-focused
Use Case Large, complex deployments Small to medium deployments
Q3: What is a Kubernetes cluster?

Kubernetes Cluster is a set of nodes (machines) that run containerized applications. It consists of a control plane (master node) and worker nodes.

Cluster Components:

  • Control Plane: Manages the cluster (API server, scheduler, controller manager)
  • Worker Nodes: Run containers (kubelet, kube-proxy, container runtime)
  • etcd: Key-value store for cluster data
Q4: What is a Pod?

Pod is the smallest deployable unit in Kubernetes. It represents a single instance of a running process in the cluster and can contain one or more containers.

Pod Characteristics:

  • Ephemeral: Pods are temporary and can be recreated
  • Shared Network: Containers in a pod share the same network namespace
  • Shared Storage: Containers can share volumes
  • IP Address: Each pod gets a unique IP address
Q5: What is a Node?

Node is a worker machine in Kubernetes. It can be a physical or virtual machine that runs pods and is managed by the control plane.

Node Components:

  • Kubelet: Agent that communicates with the control plane
  • Kube-proxy: Network proxy for service networking
  • Container Runtime: Runs containers (Docker, containerd, etc.)

2. Kubernetes Architecture

Q6: What are the main components of Kubernetes control plane?

Kubernetes Control Plane manages the cluster state and makes global decisions about the cluster. It consists of several key components.

Control Plane Components:

  • API Server: REST API endpoint for all cluster operations
  • etcd: Key-value store for cluster configuration and state
  • Scheduler: Assigns pods to nodes based on resource requirements
  • Controller Manager: Runs controller processes (replication, endpoints, etc.)
  • Cloud Controller Manager: Interfaces with cloud provider APIs
Q7: What is the Kubernetes API server?

Kubernetes API Server is the central management entity that receives REST API requests for operations like creating, updating, or deleting resources.

API Server Responsibilities:

  • REST API: Exposes REST endpoints for cluster operations
  • Authentication: Validates user identity
  • Authorization: Checks if user has permission
  • Admission Control: Validates and mutates requests
  • State Storage: Persists state to etcd
Q8: What is etcd in Kubernetes?

etcd is a consistent and highly-available key-value store used as Kubernetes' backing store for all cluster data. It stores the entire state of the cluster.

etcd Characteristics:

  • Consistent: Strong consistency guarantees
  • Highly Available: Can be deployed as a cluster
  • Key-Value Store: Stores configuration and state data
  • Watch API: Allows watching for changes
Q9: What is the Kubernetes scheduler?

Kubernetes Scheduler watches for newly created pods with no assigned node and selects a node for them to run on based on resource requirements, constraints, and policies.

Scheduling Decisions:

  • Resource Requirements: CPU, memory, storage requests
  • Hardware/Software Constraints: Node labels, taints, tolerations
  • Affinity/Anti-Affinity: Pod placement preferences
  • Resource Availability: Available resources on nodes
Q10: What is the Kubernetes controller manager?

Kubernetes Controller Manager runs controller processes that regulate the state of the cluster. It ensures the actual state matches the desired state.

Controller Types:

  • Node Controller: Manages node state
  • Replication Controller: Ensures correct number of pod replicas
  • Endpoints Controller: Populates endpoints for services
  • Service Account Controller: Manages service accounts

3. Pods and Containers

Q11: What is the lifecycle of a Pod?

Pod Lifecycle goes through several phases from creation to termination. Understanding these phases is important for debugging and managing applications.

Pod Phases:

  • Pending: Pod accepted but not yet running
  • Running: Pod is running with at least one container
  • Succeeded: All containers terminated successfully
  • Failed: All containers terminated, at least one failed
  • Unknown: State of pod cannot be determined
Q12: What are the different types of multi-container pods?

Multi-Container Pods can run multiple containers together. There are several patterns for running multiple containers in a single pod.

Multi-Container Patterns:

  • Sidecar: Helper container that extends main container functionality
  • Ambassador: Proxy container for external communication
  • Adapter: Container that transforms output of main container
Q13: What is a Pod specification?

Pod Specification is a YAML or JSON definition that describes the desired state of a pod, including containers, volumes, and other configuration.

pod.yaml
apiVersion: v1 kind: Pod metadata: name: mypod spec: containers: - name: mycontainer image: nginx:latest
Q14: What are init containers?

Init Containers are specialized containers that run before app containers in a pod. They can contain setup scripts or utilities not present in the app image.

Init Container Use Cases:

  • Setup: Run setup scripts before main container starts
  • Validation: Validate dependencies before main container
  • Configuration: Generate configuration files
pod-init.yaml
apiVersion: v1 kind: Pod metadata: name: myapp spec: initContainers: - name: init-db image: busybox command: ['sh', '-c', 'until nslookup db; do echo waiting for db; sleep 2; done;'] containers: - name: app image: myapp:latest
Q15: What is a Pod resource request and limit?

Resource Requests specify the minimum resources a pod needs, while Resource Limits specify the maximum resources a pod can use.

pod-resources.yaml
resources: requests: memory: "64Mi" cpu: "250m" limits: memory: "128Mi" cpu: "500m"
Q16: What are Pod probes?

Pod Probes are diagnostics performed by the kubelet to determine the health of a container. There are three types of probes.

Probe Types:

  • Liveness Probe: Determines if container is running
  • Readiness Probe: Determines if container is ready to serve traffic
  • Startup Probe: Determines if container has started
pod-probes.yaml
spec: containers: - name: myapp livenessProbe: httpGet: path: /health port: 8080 initialDelaySeconds: 30 periodSeconds: 10 readinessProbe: httpGet: path: /ready port: 8080 initialDelaySeconds: 5 periodSeconds: 5 startupProbe: httpGet: path: /startup port: 8080 failureThreshold: 30 periodSeconds: 10
Q17: What is a Pod restart policy?

Pod Restart Policy defines what happens when containers in a pod exit. It can be set to Always, OnFailure, or Never.

Restart Policies:

  • Always: Always restart container on exit
  • OnFailure: Restart only on non-zero exit
  • Never: Never restart container
Q18: What is a Pod disruption budget?

Pod Disruption Budget (PDB) limits the number of pods of a replicated application that are down simultaneously from voluntary disruptions.

PDB Purpose:

  • High Availability: Ensure minimum number of pods running
  • Safe Updates: Allow safe rolling updates
  • Node Maintenance: Allow safe node maintenance
Q19: What is a Pod priority class?

Pod Priority Class assigns priority to pods to influence the scheduler's decision when resources are scarce. Higher priority pods are scheduled before lower priority ones.

Q20: What is a Pod anti-affinity?

Pod Anti-Affinity rules restrict which nodes a pod can be scheduled on based on labels of other pods. It's used to spread pods across nodes for high availability.

pod-anti-affinity.yaml
spec: affinity: podAntiAffinity: requiredDuringSchedulingIgnoredDuringExecution: - labelSelector: matchExpressions: - key: app operator: In values: - myapp topologyKey: kubernetes.io/hostname

4. Services and Networking

Q21: What is a Kubernetes Service?

Kubernetes Service is an abstraction that defines a logical set of pods and a policy to access them. Services provide stable network endpoints for pods.

Service Types:

  • ClusterIP: Exposes service on cluster-internal IP
  • NodePort: Exposes service on each node's IP at a static port
  • LoadBalancer: Exposes service externally using cloud provider's load balancer
  • ExternalName: Maps service to external DNS name
service-clusterip.yaml
apiVersion: v1 kind: Service metadata: name: myservice spec: selector: app: myapp ports: - protocol: TCP port: 80 targetPort: 8080 type: ClusterIP
service-loadbalancer.yaml
apiVersion: v1 kind: Service metadata: name: myservice spec: selector: app: myapp ports: - protocol: TCP port: 80 targetPort: 8080 type: LoadBalancer
Q22: What is the difference between ClusterIP, NodePort, and LoadBalancer?

Service Types determine how services are exposed to the network. Each type serves different use cases.

Type Scope Use Case
ClusterIP Cluster internal Internal communication
NodePort Node IP + port External access (development)
LoadBalancer Cloud load balancer Production external access
Q23: What is a Kubernetes Ingress?

Kubernetes Ingress is an API object that manages external access to services, typically HTTP. Ingress can provide load balancing, SSL termination, and name-based virtual hosting.

Ingress Features:

  • Load Balancing: Distribute traffic across services
  • SSL Termination: Handle SSL/TLS at ingress level
  • Routing: Route based on host/path
  • Rate Limiting: Control traffic rate
ingress.yaml
apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: myingress annotations: nginx.ingress.kubernetes.io/rewrite-target: / spec: ingressClassName: nginx rules: - host: myapp.example.com http: paths: - path: / pathType: Prefix backend: service: name: myservice port: number: 80
Q24: What is a Kubernetes Network Policy?

Network Policy specifies how groups of pods are allowed to communicate with each other and other network endpoints. It provides network segmentation and security.

Network Policy Features:

  • Pod-to-Pod: Control pod communication
  • Ingress Rules: Allow incoming traffic
  • Egress Rules: Allow outgoing traffic
  • Namespace Isolation: Isolate namespaces
network-policy.yaml
apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: allow-web spec: podSelector: matchLabels: app: web policyTypes: - Ingress - Egress ingress: - from: - podSelector: matchLabels: app: api ports: - protocol: TCP port: 80
Q25: What is a Kubernetes Endpoint?

Endpoint is a Kubernetes object that tracks the IP addresses of pods that match a service selector. Services use endpoints to route traffic to pods.

Endpoint Characteristics:

  • Dynamic: Automatically updated as pods change
  • Service Mapping: Maps service to pod IPs
  • Readiness: Only includes ready pods
Q26: What is a Kubernetes Headless Service?

Headless Service is a service that doesn't have a cluster IP. Instead, it returns pod IPs directly to DNS queries. It's used for stateful applications and custom load balancing.

Headless Service Use Cases:

  • Stateful Applications: Database clusters that need direct pod access
  • Custom Load Balancing: Implement custom load balancing logic
  • Service Discovery: Direct pod-to-pod communication
Q27: What is a Kubernetes DNS?

Kubernetes DNS is a DNS server that provides DNS records for Kubernetes services. It enables service discovery within the cluster using DNS names.

DNS Features:

  • Service Discovery: Resolve service names to cluster IPs
  • Pod Discovery: Resolve pod names to pod IPs
  • Automatic: Automatically updates as services/pods change
Q28: What is a Kubernetes Service Mesh?

Service Mesh is a dedicated infrastructure layer for managing service-to-service communication. Popular service meshes include Istio, Linkerd, and Consul.

Service Mesh Features:

  • Traffic Management: Control traffic flow between services
  • Security: mTLS, authentication, authorization
  • Observability: Metrics, logs, tracing
  • Resilience: Retries, circuit breakers, timeouts
Q29: What is a Kubernetes CNI plugin?

CNI (Container Network Interface) Plugin implements network functionality for Kubernetes. It handles pod networking, IP allocation, and network policy enforcement.

Popular CNI Plugins:

  • Calico: Network policy, BGP routing
  • Flannel: Simple overlay network
  • Weave Net: Encrypted mesh network
  • Cilium: eBPF-based networking
Q30: What is a Kubernetes CSI driver?

CSI (Container Storage Interface) Driver enables storage vendors to develop custom storage plugins for Kubernetes. It allows Kubernetes to work with various storage systems.

CSI Features:

  • Standard Interface: Vendor-neutral storage interface
  • Dynamic Provisioning: Automatically provision storage
  • Volume Lifecycle: Manage volume creation, attachment, deletion

5. Deployments and Scaling

Q31: What is a Kubernetes Deployment?

Deployment provides declarative updates for pods and ReplicaSets. It manages the lifecycle of pods, including rolling updates and rollbacks.

Deployment Features:

  • Declarative: Define desired state
  • Rolling Updates: Gradually update pods
  • Rollback: Revert to previous version
  • Scaling: Scale up or down replicas
deployment.yaml
apiVersion: apps/v1 kind: Deployment metadata: name: myapp spec: replicas: 3 selector: matchLabels: app: myapp template: metadata: labels: app: myapp spec: containers: - name: myapp image: myapp:1.0 ports: - containerPort: 8080
Q32: What is a ReplicaSet?

ReplicaSet ensures that a specified number of pod replicas are running at any given time. It's used by Deployments to manage pod replicas.

ReplicaSet Characteristics:

  • Replica Management: Maintain desired replica count
  • Self-Healing: Replace failed pods
  • Selector: Select pods to manage
Q33: What is a StatefulSet?

StatefulSet manages stateful applications, providing stable network identities and persistent storage. It's used for databases and other stateful applications.

StatefulSet Features:

  • Stable Network IDs: Each pod gets a stable hostname
  • Stable Storage: Persistent storage per pod
  • Ordered Deployment: Pods deployed in order
  • Ordered Scaling: Pods scaled in order
statefulset.yaml
apiVersion: apps/v1 kind: StatefulSet metadata: name: postgres spec: serviceName: postgres replicas: 3 selector: matchLabels: app: postgres template: metadata: labels: app: postgres spec: containers: - name: postgres image: postgres:13 volumeMounts: - name: postgres-storage mountPath: /var/lib/postgresql volumeClaimTemplates: - metadata: name: postgres-storage spec: accessModes: ["ReadWriteOnce"] resources: requests: storage: 10Gi
Q34: What is a DaemonSet?

DaemonSet ensures that all (or some) nodes run a copy of a pod. It's used for cluster-wide daemons like log collectors and monitoring agents.

DaemonSet Use Cases:

  • Monitoring: Run monitoring agents on all nodes
  • Logging: Run log collectors on all nodes
  • Network: Run network plugins on all nodes
  • Storage: Run storage drivers on all nodes
daemonset.yaml
apiVersion: apps/v1 kind: DaemonSet metadata: name: fluentd spec: selector: matchLabels: app: fluentd template: metadata: labels: app: fluentd spec: containers: - name: fluentd image: fluentd:v1.14 volumeMounts: - name: varlog mountPath: /var/log volumes: - name: varlog hostPath: path: /var/log
Q35: What is a Job?

Job creates one or more pods and ensures that a specified number of them successfully terminate. It's used for batch processing and one-time tasks.

Job Types:

  • Non-parallel: One pod completes the job
  • Parallel: Multiple pods work in parallel
  • Parallel with fixed completion: Specific number of completions
job.yaml
apiVersion: batch/v1 kind: Job metadata: name: batch-job spec: completions: 5 parallelism: 2 template: metadata: labels: app: batch spec: containers: - name: worker image: busybox command: ['sh', '-c', 'echo "Processing task"; sleep 5'] restartPolicy: OnFailure
Q36: What is a CronJob?

CronJob creates jobs on a time-based schedule. It's used for periodic tasks like backups, reports, and maintenance.

cronjob.yaml
schedule: "*/5 * * * *" jobTemplate: spec: template: spec: containers: - name: myjob image: myimage
Q37: What is Horizontal Pod Autoscaler (HPA)?

Horizontal Pod Autoscaler (HPA) automatically scales the number of pods in a deployment based on CPU utilization or other custom metrics.

HPA Features:

  • Automatic Scaling: Scale based on metrics
  • Custom Metrics: Use custom metrics for scaling
  • Scaling Policies: Control scaling behavior
Q38: What is Vertical Pod Autoscaler (VPA)?

Vertical Pod Autoscaler (VPA) automatically adjusts the CPU and memory resource requests of pods based on historical usage.

VPA Features:

  • Resource Optimization: Optimize resource allocation
  • Historical Analysis: Based on historical usage
  • Automatic Updates: Update resource requests
Q39: What is a rolling update strategy?

Rolling Update Strategy gradually replaces old pods with new ones during a deployment update, ensuring zero downtime.

Rolling Update Parameters:

  • Max Surge: Maximum pods that can be created above desired replicas
  • Max Unavailable: Maximum pods that can be unavailable during update
Q40: What is a recreate strategy?

Recreate Strategy kills all existing pods before creating new ones. This causes downtime but is simpler and faster for some use cases.

Recreate Use Cases:

  • Stateful Applications: Applications that can't run multiple instances
  • Quick Updates: When downtime is acceptable
  • Testing: Testing deployments

6. ConfigMaps and Secrets

Q41: What is a ConfigMap?

ConfigMap is an API object used to store non-confidential data in key-value pairs. Pods can consume ConfigMaps as environment variables, command-line arguments, or configuration files.

ConfigMap Use Cases:

  • Configuration: Store application configuration
  • Environment Variables: Inject as environment variables
  • Config Files: Mount as configuration files
configmap.yaml
apiVersion: v1 kind: ConfigMap metadata: name: app-config data: database.url: postgres://localhost:5432/mydb cache.ttl: "3600" config.json: | { "feature": "enabled", "timeout": 30 }
pod-configmap.yaml
spec: containers: - name: myapp envFrom: - configMapRef: name: app-config volumeMounts: - name: config mountPath: /etc/config volumes: - name: config configMap: name: app-config
Q42: What is a Secret?

Secret is an API object used to store sensitive data such as passwords, OAuth tokens, and SSH keys. Secrets are encoded in base64 and can be mounted as volumes or exposed as environment variables.

Secret Types:

  • Opaque: Arbitrary user-defined data
  • kubernetes.io/dockerconfigjson: Docker registry credentials
  • kubernetes.io/service-account-token: Service account token
  • kubernetes.io/tls: TLS certificate data
secret.yaml
apiVersion: v1 kind: Secret metadata: name: db-secret type: Opaque data: username: YWRtaW4= password: cGFzc3dvcmQxMjM=
pod-secret.yaml
spec: containers: - name: myapp env: - name: DB_USERNAME valueFrom: secretKeyRef: name: db-secret key: username volumeMounts: - name: secret mountPath: /etc/secrets volumes: - name: secret secret: secretName: db-secret
Q43: What is the difference between ConfigMap and Secret?

ConfigMap stores non-sensitive configuration data, while Secret stores sensitive data with additional security features.

Aspect ConfigMap Secret
Data Type Non-sensitive Sensitive
Encoding Plain text Base64 encoded
Security Basic Enhanced (RBAC, encryption)
Use Case Configuration files Passwords, keys
Q44: How do you inject ConfigMap data into a pod?

ConfigMap Injection can be done as environment variables, command-line arguments, or mounted as volumes.

pod-configmap.yaml
env: - name: MY_CONFIG valueFrom: configMapKeyRef: name: myconfig key: config-key
Q45: How do you inject Secret data into a pod?

Secret Injection can be done as environment variables or mounted as volumes. When mounted as a volume, Secret data is automatically updated when the Secret changes.

pod-secret.yaml
env: - name: MY_PASSWORD valueFrom: secretKeyRef: name: mysecret key: password
Q46: What is an immutable ConfigMap?

Immutable ConfigMap is a ConfigMap that cannot be modified after creation. This provides performance benefits and ensures configuration stability.

Immutable ConfigMap Benefits:

  • Performance: Reduces kubelet overhead
  • Stability: Prevents accidental changes
  • Consistency: Ensures all pods use same config
Q47: What is a ServiceAccount?

ServiceAccount provides an identity for processes that run in a pod. It's used for authentication and authorization within the cluster.

ServiceAccount Features:

  • Authentication: Identity for pods
  • Authorization: RBAC permissions
  • Secrets: Automatically mounted service account token
serviceaccount.yaml
apiVersion: v1 kind: ServiceAccount metadata: name: myapp-sa namespace: default
pod-serviceaccount.yaml
apiVersion: v1 kind: Pod metadata: name: mypod spec: serviceAccountName: myapp-sa containers: - name: myapp image: myapp:latest
Q48: What is the default ServiceAccount?

Default ServiceAccount is automatically created in each namespace. Pods that don't specify a service account use the default one.

Default ServiceAccount Characteristics:

  • Auto-created: Created in each namespace
  • Default: Used when no service account specified
  • Limited Permissions: Typically has limited permissions
Q49: What is a PodSecurityPolicy?

PodSecurityPolicy (PSP) is a cluster-level resource that controls security-sensitive aspects of pod specification. Note: PSP is deprecated in favor of Pod Security Standards.

PSP Controls:

  • Privileged Mode: Control privileged containers
  • Host Networking: Control host network access
  • Volume Types: Control volume types
  • Run as User: Control user ID
Q50: What are Pod Security Standards?

Pod Security Standards define security policies for pods. They replace PodSecurityPolicy and provide a more flexible way to enforce security.

Security Levels:

  • Privileged: Unrestricted, most permissive
  • Baseline: Minimal security restrictions
  • Restricted: Strict security policies

7. Storage and Volumes

Q51: What is a Kubernetes Volume?

Kubernetes Volume is a directory in a pod that can be accessed by containers in the pod. Volumes provide data persistence beyond the pod lifecycle.

Volume Types:

  • emptyDir: Empty directory for temporary storage
  • hostPath: Mount a directory from host node
  • PersistentVolumeClaim: Use persistent storage
  • ConfigMap/Secret: Mount configuration data
Q52: What is a PersistentVolume (PV)?

PersistentVolume (PV) is a storage resource in the cluster. It represents a piece of storage in the cluster that has been provisioned by an administrator or dynamically provisioned using StorageClasses.

PV Lifecycle:

  • Provisioning: Static or dynamic provisioning
  • Binding: Bound to a PersistentVolumeClaim
  • Using: Used by a pod
  • Reclaiming: Retained, recycled, or deleted
Q53: What is a PersistentVolumeClaim (PVC)?

PersistentVolumeClaim (PVC) is a request for storage by a user. It's similar to a pod consuming node resources. PVCs consume PV resources.

PVC Characteristics:

  • Request: Specifies storage requirements
  • Access Modes: ReadWriteOnce, ReadOnlyMany, ReadWriteMany
  • Binding: Bound to a matching PV
Q54: What is a StorageClass?

StorageClass provides a way for administrators to describe the "classes" of storage they offer. It enables dynamic provisioning of persistent volumes.

StorageClass Features:

  • Dynamic Provisioning: Automatically provision storage
  • Provisioner: Determines which volume plugin to use
  • Parameters: Provisioner-specific parameters
  • Reclaim Policy: Retain or delete when PVC deleted
storageclass.yaml
apiVersion: storage.k8s.io/v1 kind: StorageClass metadata: name: fast-ssd provisioner: kubernetes.io/aws-ebs parameters: type: gp3 iops: "3000" reclaimPolicy: Delete volumeBindingMode: WaitForFirstConsumer
pvc.yaml
apiVersion: v1 kind: PersistentVolumeClaim metadata: name: mypvc spec: accessModes: - ReadWriteOnce storageClassName: fast-ssd resources: requests: storage: 10Gi
Q55: What is the difference between PV and PVC?

PersistentVolume (PV) is the actual storage resource, while PersistentVolumeClaim (PVC) is a request for storage by a user.

Aspect PV PVC
Type Storage resource Storage request
Created By Admin or dynamic User
Scope Cluster-wide Namespace-scoped
Q56: What is an emptyDir volume?

emptyDir Volume is an empty directory that is created when a pod is created and deleted when the pod is removed. It's used for temporary storage.

emptyDir Use Cases:

  • Temporary Storage: Scratch space for containers
  • Shared Data: Share data between containers in a pod
  • Caching: Cache data during pod lifecycle
Q57: What is a hostPath volume?

hostPath Volume mounts a file or directory from the host node's filesystem into a pod. It's used for accessing host resources.

hostPath Use Cases:

  • System Logs: Access host system logs
  • Docker Socket: Access Docker daemon
  • System Config: Access host configuration
Q58: What is a StatefulSet volume claim template?

StatefulSet Volume Claim Template defines a PersistentVolumeClaim template for each pod in a StatefulSet. Each pod gets its own PVC.

Volume Claim Template Features:

  • Unique PVC per Pod: Each pod gets its own PVC
  • Stable Storage: PVCs persist across pod restarts
  • Ordered Creation: PVCs created in pod order
Q59: What is a dynamic volume provisioning?

Dynamic Volume Provisioning automatically creates PersistentVolumes based on StorageClass when a PersistentVolumeClaim is created.

Dynamic Provisioning Benefits:

  • Automation: Automatically provision storage
  • Simplicity: No manual PV creation
  • Flexibility: Support multiple storage backends
Q60: What is a volume access mode?

Volume Access Mode defines how a volume can be accessed by pods. Common modes include ReadWriteOnce, ReadOnlyMany, and ReadWriteMany.

Access Modes:

  • ReadWriteOnce: Read/write by single node
  • ReadOnlyMany: Read-only by many nodes
  • ReadWriteMany: Read/write by many nodes
  • ReadWriteOncePod: Read/write by single pod

8. Kubernetes Security

Q61: What is RBAC in Kubernetes?

Role-Based Access Control (RBAC) is a method of regulating access to computer or network resources based on the roles of individual users within your organization.

RBAC Components:

  • Role: Defines permissions within a namespace
  • ClusterRole: Defines permissions cluster-wide
  • RoleBinding: Binds a role to users/groups
  • ClusterRoleBinding: Binds a cluster role cluster-wide
role.yaml
apiVersion: rbac.authorization.k8s.io/v1 kind: Role metadata: namespace: default name: pod-reader rules: - apiGroups: [""] resources: ["pods"] verbs: ["get", "list", "watch"]
rolebinding.yaml
apiVersion: rbac.authorization.k8s.io/v1 kind: RoleBinding metadata: name: read-pods namespace: default subjects: - kind: User name: jane apiGroup: rbac.authorization.k8s.io roleRef: kind: Role name: pod-reader apiGroup: rbac.authorization.k8s.io
Q62: What is the difference between Role and ClusterRole?

Role defines permissions within a namespace, while ClusterRole defines permissions cluster-wide.

Aspect Role ClusterRole
Scope Namespace-scoped Cluster-wide
Use Case Namespace-specific permissions Cluster-wide permissions
Q63: What is a ServiceAccount token?

ServiceAccount Token is a JWT token that provides identity for pods. It's automatically mounted into pods and used for authentication.

Token Features:

  • JWT Format: JSON Web Token format
  • Auto-mounted: Automatically mounted in pods
  • Time-limited: Tokens have expiration
Q64: What is Kubernetes admission controller?

Admission Controller is a piece of code that intercepts requests to the Kubernetes API server prior to persistence of the object.

Admission Controller Types:

  • Validating: Validate requests
  • Mutating: Modify requests
  • Built-in: Built-in controllers
  • Custom: Custom webhook controllers
Q65: What is a Kubernetes namespace?

Namespace provides a mechanism for isolating groups of resources within a single cluster. Names of resources need to be unique within a namespace.

Namespace Use Cases:

  • Isolation: Separate environments (dev, test, prod)
  • Resource Quotas: Limit resources per namespace
  • RBAC: Separate permissions per namespace

9. Advanced Kubernetes Concepts

Q66: What is a Custom Resource Definition (CRD)?

Custom Resource Definition (CRD) extends the Kubernetes API by defining custom resources. It allows you to add your own resource types to Kubernetes.

CRD Use Cases:

  • Custom Resources: Define application-specific resources
  • Operators: Build operators for custom resources
  • Extension: Extend Kubernetes functionality
crd.yaml
apiVersion: apiextensions.k8s.io/v1 kind: CustomResourceDefinition metadata: name: crontabs.stable.example.com spec: group: stable.example.com versions: - name: v1 served: true storage: true schema: openAPIV3Schema: type: object properties: spec: type: object properties: cronSpec: type: string image: type: string scope: Namespaced names: plural: crontabs singular: crontab kind: CronTab
custom-resource.yaml
apiVersion: stable.example.com/v1 kind: CronTab metadata: name: my-cron spec: cronSpec: "* * * * *" image: my-cron-image
Q67: What is a Kubernetes Operator?

Kubernetes Operator is a method of packaging, deploying, and managing a Kubernetes application. It uses custom resources to manage application state.

Operator Pattern:

  • Custom Resource: Define application state
  • Controller: Reconcile actual state with desired state
  • Automation: Automate operational tasks
Q68: What is a Kubernetes Helm?

Helm is a package manager for Kubernetes. It helps you manage Kubernetes applications through Helm charts.

Helm Features:

  • Charts: Package Kubernetes manifests
  • Templating: Use templates for configuration
  • Release Management: Manage application versions
  • Rollback: Rollback to previous versions
Chart.yaml

Q69: What is a Kubernetes Operator pattern?

Operator Pattern is a way to extend Kubernetes' capabilities by creating custom controllers that manage application-specific resources.

Operator Components:

  • CRD: Custom resource definition
  • Controller: Reconciliation loop
  • Business Logic: Application-specific logic
Q70: What is a Kubernetes Custom Controller?

Custom Controller is a control loop that watches the state of your cluster and makes changes to move the current state towards the desired state.

Controller Pattern:

  • Watch: Watch for changes to resources
  • Reconcile: Reconcile actual state with desired state
  • Action: Take action to achieve desired state

10. Kubernetes Operations

Q71: What is kubectl?

kubectl is the command-line tool for controlling Kubernetes clusters. It allows you to deploy applications, inspect and manage cluster resources, and view logs.

Common kubectl Commands:

  • get: List resources
  • apply: Apply configuration
  • describe: Show resource details
  • logs: View pod logs
Q72: What is a Kubernetes label?

Label is a key-value pair attached to Kubernetes objects. Labels are used to organize and select subsets of objects.

Label Use Cases:

  • Organization: Organize resources
  • Selection: Select resources for operations
  • Grouping: Group related resources
Q73: What is a Kubernetes selector?

Selector is used to identify objects that match certain criteria. Selectors are used in services, deployments, and other resources to select pods.

Selector Types:

  • Equality-based: Match exact label values
  • Set-based: Match sets of label values
Q74: What is a Kubernetes annotation?

Annotation is a key-value pair attached to Kubernetes objects for storing arbitrary non-identifying metadata. Unlike labels, annotations are not used for selection.

Annotation Use Cases:

  • Metadata: Store arbitrary metadata
  • Documentation: Document resources
  • Integration: Store integration data
Q75: What is a Kubernetes taint?

Taint is a key-value pair applied to a node that repels pods unless they have a matching toleration. Taints are used to control pod placement.

Taint Effects:

  • NoSchedule: Don't schedule unless toleration
  • PreferNoSchedule: Prefer not to schedule
  • NoExecute: Evict if no toleration
