1. RESTful API Fundamentals

Q1: What is a RESTful API?

REST (Representational State Transfer) is an architectural style for designing networked applications. A RESTful API:

  • Uses standard HTTP methods (GET, POST, PUT, DELETE, PATCH)
  • Is stateless - each request contains all information needed
  • Uses resource-based URLs
  • Returns appropriate HTTP status codes
  • Supports multiple data formats (JSON, XML)
  • Follows HATEOAS (Hypermedia as the Engine of Application State)
Q2: What are the REST principles?
  1. Statelessness: No client context stored on server
  2. Client-Server: Separation of concerns
  3. Cacheable: Responses should be cacheable
  4. Uniform Interface: Consistent interface between components
  5. Layered System: Architecture can be layered
  6. Code on Demand (optional): Server can extend client functionality

3. API Controllers and Actions

Q5: What is the difference between Controller and ControllerBase?
  • Controller: Inherits from ControllerBase and adds support for views. Used for MVC applications.
  • ControllerBase: Base class for API controllers without view support. Used for Web API applications.
// API Controller (uses ControllerBase)
[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    // ...
}

// MVC Controller (uses Controller)
public class HomeController : Controller
{
    // ...
}
Q6: What does the [ApiController] attribute do?

The [ApiController] attribute enables several API-specific behaviors:

  • Automatic HTTP 400 response: Invalid model binding returns 400 automatically
  • Automatic body inference: Complex types are inferred from request body
  • Action parameter source inference: Parameters are inferred from route, query, or body
  • Problem details for error responses: Standardized error responses
Tip: Always use [ApiController] on your API controllers for automatic validation and better error handling.

5. Model Binding and Validation

Q9: How does model binding work in Web API?

Model binding in Web API automatically maps HTTP request data to action parameters:

  • [FromBody]: From request body (JSON/XML)
  • [FromForm]: From form data
  • [FromHeader]: From HTTP headers
  • [FromQuery]: From query string
  • [FromRoute]: From route data
  • [FromServices]: From dependency injection
[HttpPost]
public IActionResult Create([FromBody] ProductDto dto)
{
    if (!ModelState.IsValid)
    {
        return BadRequest(ModelState);
    }
    // Process
    return CreatedAtAction(nameof(GetById), new { id = dto.Id }, dto);
}
Q10: How do you implement validation in Web API? 
// DTO with validation attributes
public class ProductDto
{
    public int Id { get; set; }
    
    [Required(ErrorMessage = "Name is required")]
    [StringLength(100, MinimumLength = 2)]
    public string Name { get; set; }
    
    [Range(0.01, 10000)]
    public decimal Price { get; set; }
    
    [EmailAddress]
    public string Email { get; set; }
}

// Controller with [ApiController] - automatic 400 on validation failure
[HttpPost]
public IActionResult Create(ProductDto dto)
{
    // ModelState.IsValid is automatically checked with [ApiController]
    // Returns 400 Bad Request if validation fails
    var product = _mapper.Map<Product>(dto);
    _service.Create(product);
    return CreatedAtAction(nameof(GetById), new { id = product.Id }, dto);
}

6. Content Negotiation

Q11: What is content negotiation?

Content negotiation is the process of selecting the best representation for a given response when multiple representations are available. It considers:

  • Accept header in the request
  • Content-Type header in the request
  • Server configuration
// Add content negotiation services
builder.Services.AddControllers(options =>
{
    options.ReturnHttpNotAcceptable = true;
    options.RespectBrowserAcceptHeader = true;
}).AddXmlDataContractSerializerFormatters();
Q12: How do you return different content types? 
// Automatic content negotiation
[HttpGet("{id}")]
public IActionResult GetById(int id)
{

    return Ok(product); // Returns JSON or XML based on Accept header
}

// Force specific content type

11. Error Handling

Q18: How do you handle errors in Web API?

1. Using Exception Filter:

public class GlobalExceptionFilter : IExceptionFilter
{
  private readonly ILogger<GlobalExceptionFilter> _logger;

  public GlobalExceptionFilter(ILogger<GlobalExceptionFilter> logger)
  {
    _logger = logger;
  }

  public void OnException(ExceptionContext context)
  {
    _logger.LogError(context.Exception, "Unhandled exception");

    context.Result = new ObjectResult(new
    {
      Error = "An error occurred",
      Message = context.Exception.Message
    })
    {
      StatusCode = 500
    };

    context.ExceptionHandled = true;
  }
}

// Register filter
builder.Services.AddControllers(options =>
{
  options.Filters.Add<GlobalExceptionFilter>();
});

2. Using Problem Details:

builder.Services.AddProblemDetails();

if (app.Environment.IsDevelopment())
{
  app.UseExceptionHandler();
}
else
{
  app.UseExceptionHandler(exceptionHandlerApp =>
  {
    exceptionHandlerApp.Run(async context =>
    {
      context.Response.StatusCode = (int)HttpStatusCode.InternalServerError;
      context.Response.ContentType = "application/problem+json";

      var problemDetails = new ProblemDetails
      {
        Status = context.Response.StatusCode,
        Title = "An error occurred",
        Detail = "An unexpected error occurred"
      };

      await context.Response.WriteAsJsonAsync(problemDetails);
    });
  });
}

Q25: How do you implement PATCH in Web API? 
// Using JsonPatch
[HttpPatch("{id}")]
public IActionResult Patch(int id, [FromBody] JsonPatchDocument<ProductDto> patchDoc)
{
    var product = _service.GetById(id);
    if (product == null) return NotFound();
    
    var dto = _mapper.Map<ProductDto>(product);
    patchDoc.ApplyTo(dto, ModelState);
    
    if (!ModelState.IsValid)
        return BadRequest(ModelState);
    
    _mapper.Map(dto, product);
    _service.Update(product);
    
    return NoContent();
}
Q26: What is the difference between PUT and PATCH?
  • PUT: Replaces the entire resource. All fields must be provided. Idempotent.
  • PATCH: Partially updates the resource. Only changed fields are provided. Not idempotent.
Q27: How do you implement rate limiting? 
// Using middleware
public class RateLimitMiddleware
{
    private readonly RequestDelegate _next;
    private readonly IMemoryCache _cache;
    
    public RateLimitMiddleware(RequestDelegate next, IMemoryCache cache)
    {
        _next = next;
        _cache = cache;
    }
    
    public async Task InvokeAsync(HttpContext context)
    {
        var key = context.Connection.RemoteIpAddress.ToString();
        var limit = 100; // 100 requests per minute
        
        var current = _cache.GetOrCreate(key, entry =>
        {
            entry.AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(1);
            return 0;
        });
        
        if (current >= limit)
        {
            context.Response.StatusCode = 429; // Too Many Requests
            return;
        }
        
        _cache.Set(key, current + 1);
        await _next(context);
    }
}

Q33: What is the CQRS pattern in APIs?

CQRS (Command Query Responsibility Segregation) separates read and write operations into different models.

 Query
public class GetProductQuery
{
    public int Id { get; set; }
}

public class GetProductQueryHandler
{
    private readonly AppDbContext _context;

    public GetProductQueryHandler(AppDbContext context)
    {
        _context = context;
    }

    public async Task<ProductDto> Handle(GetProductQuery query)
    {
        var product = await _context.Products.FindAsync(query.Id);
        return new ProductDto { Id = product.Id, Name = product.Name };
    }
}

// Command
public class CreateProductCommand
{
    public string Name { get; set; }
    public decimal Price { get; set; }
}

public class CreateProductCommandHandler
{
    private readonly AppDbContext _context;

    public CreateProductCommandHandler(AppDbContext context)
    {
        _context = context;
    }

    public async Task<int> Handle(CreateProductCommand command)
    {
        var product = new Product { Name = command.Name, Price = command.Price };
        _context.Products.Add(product);
        await _context.SaveChangesAsync();
        return product.Id;
    }
}
Q34: What is the Repository Pattern in APIs?

Repository Pattern mediates between the domain and data mapping layers, acting like an in-memory domain object collection.

 Repository interface
public interface IRepository<T> where T : class
{
    Task<IEnumerable<T>> GetAllAsync();
    Task<T> GetByIdAsync(int id);
    Task<T> AddAsync(T entity);
    Task UpdateAsync(T entity);
    Task DeleteAsync(T entity);
}

// Repository implementation
public class Repository<T> : IRepository<T> where T : class
{
    private readonly AppDbContext _context;
    private readonly DbSet<T> _dbSet;

    public Repository(AppDbContext context)
    {
        _context = context;
        _dbSet = context.Set<T>();
    }

    public async Task<IEnumerable<T>> GetAllAsync()
    {
        return await _dbSet.ToListAsync();
    }

    public async Task<T> GetByIdAsync(int id)
    {
        return await _dbSet.FindAsync(id);
    }

    public async Task<T> AddAsync(T entity)
    {
        await _dbSet.AddAsync(entity);
        await _context.SaveChangesAsync();
        return entity;
    }

    public async Task UpdateAsync(T entity)
    {
        _dbSet.Update(entity);
        await _context.SaveChangesAsync();
    }

    public async Task DeleteAsync(T entity)
    {
        _dbSet.Remove(entity);
        await _context.SaveChangesAsync();
    }
}

// Generate token
public class AuthService
{
    public string GenerateToken(User user)
    {
        var securityKey = new SymmetricSecurityKey(
            Encoding.UTF8.GetBytes(_configuration["Jwt:Key"]));
        var credentials = new SigningCredentials(
            securityKey, SecurityAlgorithms.HmacSha256);

        var claims = new[]
        {
            new Claim(ClaimTypes.NameIdentifier, user.Id.ToString()),
            new Claim(ClaimTypes.Name, user.Username),
            new Claim(ClaimTypes.Role, user.Role)
        };

        var token = new JwtSecurityToken(
            issuer: _configuration["Jwt:Issuer"],
            audience: _configuration["Jwt:Audience"],
            claims: claims,
            expires: DateTime.Now.AddHours(1),
            signingCredentials: credentials);

        return new JwtSecurityTokenHandler().WriteToken(token);
    }
}
Q36: How do you implement API key authentication? 
 Custom API key authentication
public class ApiKeyAuthenticationHandler : AuthenticationHandler<ApiKeyAuthenticationOptions>
{
    private readonly IConfiguration _configuration;

    public ApiKeyAuthenticationHandler(
        IOptionsMonitor<ApiKeyAuthenticationOptions> options,
        ILoggerFactory logger,
        UrlEncoder encoder,
        IConfiguration configuration)
        : base(options, logger, encoder)
    {
        _configuration = configuration;
    }

    protected override async Task<AuthenticateResult> HandleAuthenticateAsync()
    {
        if (!Request.Headers.TryGetValue("X-API-Key", out var apiKeyValues))
        {
            return AuthenticateResult.Fail("API Key missing");
        }

        var apiKey = apiKeyValues.FirstOrDefault();
        if (string.IsNullOrEmpty(apiKey))
        {
            return AuthenticateResult.Fail("API Key missing");
        }

        var validApiKey = _configuration["ApiKey:ValidKey"];
        if (apiKey != validApiKey)
        {
            return AuthenticateResult.Fail("Invalid API Key");
        }

        var claims = new[] { new Claim(ClaimTypes.Name, "APIUser") };
        var identity = new ClaimsIdentity(claims, Scheme.Name);
        var principal = new ClaimsPrincipal(identity);
        var ticket = new AuthenticationTicket(principal, Scheme.Name);

        return AuthenticateResult.Success(ticket);
    }
}

Q38: What is the difference between IActionResult and ActionResult<T>?
  • IActionResult: Interface for action results. Can return any type that implements IActionResult. More flexible.
  • ActionResult<T>: Can return either T or IActionResult. Provides type safety and better API documentation. Recommended for API controllers.
Q39: What is the difference between [FromBody] and [FromForm]?
  • [FromBody]: Binds parameter from request body. Used for JSON/XML content. Good for complex objects.
  • [FromForm]: Binds parameter from form data. Used for multipart/form-data. Good for file uploads.
Q40: What is the difference between [FromRoute] and [FromQuery]?
  • [FromRoute]: Binds parameter from route data (URL path). Used for required identifiers.
  • [FromQuery]: Binds parameter from query string. Used for optional parameters, filters, pagination.
Q41: What is the difference between [FromHeader] and [FromServices]?
  • [FromHeader]: Binds parameter from HTTP header. Used for custom headers.
  • [FromServices]: Binds parameter from DI container. Used to inject services directly into action parameters.
Q42: What is the difference between Ok() and Ok(object value)?
  • Ok(): Returns 200 OK with no body. Used for successful operations with no return data.
  • Ok(object value): Returns 200 OK with body. Used for successful operations with return data.
Q43: What is the difference between Created() and CreatedAtAction()?
  • Created(): Returns 201 Created with location header. Requires URI parameter.
  • CreatedAtAction(): Returns 201 Created with location header. Generates URI from action name. More convenient.
Q44: What is the difference between NotFound() and NotFound(object value)?
  • NotFound(): Returns 404 Not Found with no body. Standard 404 response.
  • NotFound(object value): Returns 404 Not Found with body. Provides error details.
Q45: What is the difference between BadRequest() and BadRequest(object value)?
  • BadRequest(): Returns 400 Bad Request with no body. Standard 400 response.
  • BadRequest(object value): Returns 400 Bad Request with body. Provides validation errors.
Q46: What is the difference between Unauthorized() and Forbid()?
  • Unauthorized(): Returns 401 Unauthorized. User not authenticated.
  • Forbid(): Returns 403 Forbidden. User authenticated but not authorized.
Q47: What is the difference between Problem() and ValidationProblem()?
  • Problem(): Returns RFC 7807 problem details. Used for general errors.
  • ValidationProblem(): Returns RFC 7807 problem details with validation errors. Used for validation failures.
Q48: What are the best practices for API design?
  • Use nouns for resources: /products, /users
  • Use plural nouns: /products not /product
  • Use HTTP methods correctly: GET, POST, PUT, DELETE, PATCH
  • Use proper status codes: 200, 201, 204, 400, 401, 403, 404, 500
  • Use versioning: /api/v1/products
  • Use filtering, sorting, pagination: ?page=1&limit=10&sort=name
  • Use consistent naming: camelCase for JSON
  • Use HATEOAS: Include links to related resources
  • Use proper error responses: RFC 7807 problem details
  • Document your API: Use Swagger/OpenAPI
Q49: What are the best practices for API security?
  • Use HTTPS: Enforce HTTPS in production
  • Use authentication: JWT, OAuth2, API keys
  • Use authorization: Role-based, policy-based
  • Validate input: Data annotations, FluentValidation
  • Rate limiting: Prevent abuse and DDoS
  • CORS: Configure properly
  • Security headers: HSTS, CSP, X-Frame-Options
  • Secure secrets: Azure Key Vault, user secrets
  • Log and monitor: Track API usage and anomalies
  • Keep dependencies updated: Regularly update NuGet packages
Q50: What are the best practices for API performance?
  • Use async/await: Avoid blocking threads
  • Use caching: Response caching, distributed caching
  • Use compression: Enable response compression
  • Optimize database queries: Use EF Core efficiently, avoid N+1
  • Use pagination: Limit result sets
  • Use DTOs: Project only needed data
  • Use HTTP/2: Enable HTTP/2 for better performance
  • Use CDN: Serve static files from CDN
  • Minimize allocations: Use object pooling, span<T>
  • Monitor performance: Use Application Insights, logging
Interview Tip: When discussing Web API, always emphasize REST principles, proper HTTP status codes, and security. Be prepared to discuss real-world scenarios like authentication, versioning, and error handling. For senior positions, demonstrate knowledge of advanced patterns like CQRS, GraphQL, gRPC, and security best practices.

  • Advanced (8-10 years): Implement CQRS pattern, implement Repository pattern, understand GraphQL vs REST, implement gRPC services, implement rate limiting, design scalable API architecture, implement advanced filtering and middleware.
  • Expert (11-13 years): Design enterprise-grade API architecture, implement advanced security patterns (OAuth2, OpenID Connect), design and implement GraphQL APIs, implement gRPC for microservices, optimize API performance at scale, implement advanced monitoring and observability, design API gateway patterns, implement event-driven APIs with webhooks.
