1. Introduction to ASP.NET Core

Q1: What is ASP.NET Core and how does it differ from ASP.NET?

ASP.NET Core is a cross-platform, high-performance, open-source framework for building modern, cloud-based, Internet-connected applications. It is a complete rewrite of ASP.NET 4.x.

Key Differences from ASP.NET:

  • Cross-platform: Runs on Windows, Linux, and macOS
  • High performance: Significantly faster than ASP.NET 4.x
  • Modular: Uses NuGet packages for dependencies
  • Unified: MVC and Web API are combined into a single framework
  • Open-source: Developed and maintained by Microsoft and the community
  • Cloud-ready: Designed for cloud deployment and containerization
  • No System.Web dependency: Lightweight and optimized
Q2: What are the key features of ASP.NET Core?
  • Cross-platform support (Windows, Linux, macOS)
  • Built-in dependency injection
  • Unified MVC and Web API framework
  • Async/await support throughout
  • Self-hosting with Kestrel web server
  • Modular HTTP request pipeline with middleware
  • Cloud-ready configuration system
  • Built-in logging framework
  • Support for multiple hosting models
  • Side-by-side versioning
  • Improved performance and scalability
Q3: What is the difference between .NET Core, .NET Framework, and .NET 5/6/7/8?
Feature .NET Framework .NET Core .NET 5+
Platform Windows only Cross-platform Cross-platform
Open Source Partial Full Full

.NET 5+ is the successor to .NET Core, unifying .NET Core, .NET Framework, and Xamarin into a single platform.

2. ASP.NET Core Architecture

Q4: Explain the ASP.NET Core request processing pipeline.

The ASP.NET Core request processing pipeline consists of the following components:

  1. HTTP Server: Kestrel (default) or IIS receives the HTTP request
  2. Middleware Pipeline: Request passes through configured middleware components
  3. Routing: URL is matched to route templates
  4. Controller/Endpoint Selection: Appropriate controller or endpoint is selected
  5. Action Execution: Controller action or endpoint delegate executes
  6. Response Generation: Response is created and sent back through middleware
// Middleware pipeline example
var app = builder.Build();

app.UseHttpsRedirection();
app.UseStaticFiles();
app.UseRouting();
app.UseAuthentication();
app.UseAuthorization();
app.MapControllers();

app.Run();
Q5: What is the role of Kestrel in ASP.NET Core?

Kestrel is a cross-platform web server for ASP.NET Core. It is:

  • The default web server included in ASP.NET Core
  • Built from scratch for high performance and scalability
  • Can run standalone or behind a reverse proxy (IIS, Nginx, Apache)
  • Supports HTTP/1.1, HTTP/2, and HTTP/3
  • Designed to be secure by default
Important: In production, Kestrel should be used behind a reverse proxy (IIS, Nginx, or Apache) for additional security, SSL termination, and load balancing.
Q6: What are the different hosting models in ASP.NET Core?
  1. In-Process Hosting: Application runs in the same process as IIS (IISHttpServer)
  2. Out-of-Process Hosting: Application runs in a separate process (Kestrel) behind IIS
  3. Self-Contained Deployment: Includes .NET runtime with the application
  4. Framework-Dependent Deployment: Relies on .NET runtime installed on the target machine
  5. Docker Container: Application runs in a containerized environment

3. Dependency Injection

Q7: Explain Dependency Injection in ASP.NET Core.

Dependency Injection (DI) is a design pattern that implements Inversion of Control (IoC) for resolving dependencies between objects. ASP.NET Core includes a built-in DI container.

Service Lifetimes:

  • Transient: Created each time they're requested
  • Scoped: Created once per HTTP request
  • Singleton: Created the first time they're requested and never disposed
// Service registration
builder.Services.AddTransient<IMyService, MyService>();
builder.Services.AddScoped<IRepository, Repository>();
builder.Services.AddSingleton<ILoggerService, LoggerService>();

// Constructor injection
public class MyController : ControllerBase
{
  private readonly IMyService _service;

  public MyController(IMyService service)
  {
    _service = service;
  }
}
Q8: What is the difference between Transient, Scoped, and Singleton lifetimes?
Lifetime Behavior Use Case
Transient New instance for every request Lightweight, stateless services
Scoped Same instance within a request Repositories, DbContext
Singleton Same instance for entire application Caching, configuration, logging
Tip: Never inject a Scoped service into a Singleton service, as it can lead to captive dependency problems.

4. Middleware Pipeline

Q9: What is middleware in ASP.NET Core?

Middleware are software components that are assembled into an application pipeline to handle requests and responses. Each component:

  • Chooses whether to pass the request to the next component
  • Can perform work before and after the next component
  • Is added to the pipeline in the Program.cs file
// Custom middleware example
public class CustomMiddleware
{
  private readonly RequestDelegate _next;

  public CustomMiddleware(RequestDelegate next)
  {
    _next = next;
  }

  public async Task InvokeAsync(HttpContext context)
  {
    // Before next middleware
    await context.Response.WriteAsync("Before next middleware<br>");

    await _next(context);

    // After next middleware
    await context.Response.WriteAsync("After next middleware<br>");
  }
}

// Extension method for registration
public static class CustomMiddlewareExtensions
{
  public static IApplicationBuilder UseCustomMiddleware(this IApplicationBuilder builder)
  {
    return builder.UseMiddleware<CustomMiddleware>();
  }
}
Q10: What is the order of middleware execution?

Middleware executes in the order they are added to the pipeline:

  1. Exception Handler/Developer Exception Page (first)
  2. HTTPS Redirection
  3. Static Files
  4. Routing
  5. CORS
  6. Authentication
  7. Authorization
  8. Custom Middleware
  9. Endpoints (last)
Important: The order matters! Authentication must come before Authorization, and Exception handling should be first.

5. Configuration Management

Q11: How does configuration work in ASP.NET Core?

ASP.NET Core uses a flexible configuration system that supports multiple sources:

  • appsettings.json and appsettings.{Environment}.json
  • Environment variables
  • Command-line arguments
  • User secrets (for development)
  • Key Vault (for production)
  • Custom providers
// Configuration access
var builder = WebApplication.CreateBuilder(args);

// Add configuration providers
builder.Configuration.AddJsonFile("appsettings.json", optional: false);
builder.Configuration.AddEnvironmentVariables();

// Access configuration
var myValue = builder.Configuration["MySetting"];
var connectionString = builder.Configuration.GetConnectionString("DefaultConnection");

// Options pattern
builder.Services.Configure<MyOptions>(
    builder.Configuration.GetSection("MyOptions"));
Q12: What is the Options pattern in configuration?

The Options pattern uses classes to represent groups of related settings:

// Options class
public class MyOptions
{
  public const string SectionName = "MyOptions";

  public string Setting1 { get; set; }
  public int Setting2 { get; set; }
  public bool Setting3 { get; set; }
}

// appsettings.json
{
  "MyOptions": {
    "Setting1": "value1",
    "Setting2": 100,
    "Setting3": true
  }
}

// Registration
builder.Services.Configure<MyOptions>(
  builder.Configuration.GetSection(MyOptions.SectionName));

// Usage
public class MyService
{
  private readonly MyOptions _options;

  public MyService(IOptions<MyOptions> options)
  {
    _options = options.Value;
  }
}

7. Filters and Attributes

Q14: What are filters in ASP.NET Core?

Filters enable running code before or after specific stages in the request processing pipeline:

Filter Types:

  • Authorization filters: Run first to determine if user is authorized
  • Resource filters: Run after authorization, can short-circuit pipeline
  • Action filters: Run before and after action execution
  • Exception filters: Handle exceptions globally
  • Result filters: Run before and after result execution
// Action filter example
public class LogActionFilter : ActionFilterAttribute
{
    private readonly ILogger<LogActionFilter> _logger;
    
    public LogActionFilter(ILogger<LogActionFilter> logger)
    {
        _logger = logger;
    }
    
    public override void OnActionExecuting(ActionExecutingContext context)
    {
        _logger.LogInformation($"Executing: {context.ActionDescriptor.DisplayName}");
    }
    
    public override void OnActionExecuted(ActionExecutedContext context)
    {
        _logger.LogInformation($"Executed: {context.ActionDescriptor.DisplayName}");
    }
}

8. Logging and Monitoring

Q15: How does logging work in ASP.NET Core?

ASP.NET Core includes a built-in logging framework that supports multiple providers:

Log Levels:

  • Trace - Most detailed
  • Debug - Debugging information
  • Information - General information
  • Warning - Warning messages
  • Error - Error messages
  • Critical - Critical failure messages
// Logging in controller
public class ProductsController : ControllerBase
{
    private readonly ILogger<ProductsController> _logger;
    
    public ProductsController(ILogger<ProductsController> logger)
    {
        _logger = logger;
    }
    
    [HttpGet("{id}")]
    public async Task<ActionResult<Product> Get(int id)
    {
        _logger.LogInformation("Getting product with ID: {ProductId}", id);
        
        try
        {
            var product = await _repository.GetByIdAsync(id);
            if (product == null)
            {
                _logger.LogWarning("Product not found: {ProductId}", id);
                return NotFound();
            }
            return Ok(product);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Error getting product: {ProductId}", id);
            throw;
        }
    }
}

9. Performance Optimization

Q16: What are some performance optimization techniques in ASP.NET Core?
  • Response Compression: Use Gzip/Brotli compression
  • Async/Await: Use async for all I/O operations
  • Caching: Implement in-memory and distributed caching
  • Connection Pooling: Optimize database connection pool size
  • Object Pooling: Reuse frequently allocated objects
  • Minimize Allocations: Reduce memory allocations in hot paths
  • Use Span<T>: For string manipulation without allocations
  • Lazy Loading: Load data only when needed
  • Pagination: Implement pagination for large datasets
  • Database Indexing: Proper indexing for queries
// Response compression
builder.Services.AddResponseCompression(options =>
{
    options.EnableForHttps = true;
    options.Providers.Add<GzipCompressionProvider>();
});

app.UseResponseCompression();

// Async best practice
public async Task<IActionResult> Get(int id)
{
    var product = await _repository.GetByIdAsync(id);
    return Ok(product);
}

11. Common Interview Questions

Q18: What is the difference between IApplicationBuilder and IWebHostBuilder?
  • IWebHostBuilder: Used to build and configure the web host. It's responsible for setting up the server, configuration, logging, and dependency injection container.
  • IApplicationBuilder: Used to configure the HTTP request pipeline. It's used to add middleware components to the pipeline.
Q19: What is the purpose of the launchSettings.json file?

launchSettings.json defines different launch profiles for development, including:

  • Application URLs
  • Environment variables
  • Command to run (IIS Express, Kestrel)
  • Browser launch settings
Q20: How do you handle errors in ASP.NET Core?
  • Developer Exception Page: Detailed error information in development
  • Exception Handler Middleware: Custom error handling in production
  • Exception Filters: Handle exceptions at controller/action level
  • Try-Catch: Handle exceptions in code
  • Global Exception Handler: Centralized error handling
// Exception handling middleware
if (app.Environment.IsDevelopment())
{
    app.UseDeveloperExceptionPage();
}
else
{
    app.UseExceptionHandler("/Error");
    app.UseHsts();
}
Q21: What is the difference between AddTransient, AddScoped, and AddSingleton?
  • AddTransient: Services are created each time they're requested from the service container. This is suitable for lightweight, stateless services.
  • AddScoped: Services are created once per client request (within a scope). This is suitable for services that should be shared within a single HTTP request, like DbContext.
  • AddSingleton: Services are created the first time they're requested and remain the same for all subsequent requests. This is suitable for stateless services that are expensive to create.
Q22: What is the purpose of the ConfigureServices method?

The ConfigureServices method is used to configure the dependency injection container. It's where you register services like:

  • Database contexts
  • Application services
  • Authentication and authorization
  • MVC services
  • CORS policies
  • Health checks
  • Any other application services
Q23: How do you implement CORS in ASP.NET Core? 
// Add CORS services
builder.Services.AddCors(options =>
{
    options.AddPolicy("AllowSpecificOrigin",
        builder => builder.WithOrigins("https://example.com")
            .AllowAnyMethod()
            .AllowAnyHeader()
            .AllowCredentials());
});

// Use CORS middleware
app.UseCors("AllowSpecificOrigin");

// Enable CORS on controller
[EnableCors("AllowSpecificOrigin")]
public class MyController : ControllerBase
{
    // ...
}
Q24: What is the difference between app.Use and app.Map in middleware?
  • app.Use: Adds middleware to the pipeline that processes all requests. The middleware can choose whether to pass the request to the next middleware.
  • app.Map: Branches the pipeline based on request path matching. It's used to create sub-pipelines for specific paths.
// Use - processes all requests
app.Use(async (context, next) =>
{
    // Code before next middleware
    await next();
    // Code after next middleware
});

// Map - branches based on path
app.Map("/api", apiApp =>
{
    apiApp.UseRouting();
    apiApp.UseEndpoints(endpoints =>
    {
        endpoints.MapControllers();
    });
});
Q25: How do you implement health checks in ASP.NET Core? 
// Add health check services
builder.Services.AddHealthChecks()
    .AddCheck<DatabaseHealthCheck>("database")
    .AddCheck<ExternalApiHealthCheck>("external-api")
    .AddDbContextCheck<AppDbContext>("database-context");

// Map health check endpoint
app.MapHealthChecks("/health");

// Health check implementation
public class DatabaseHealthCheck : IHealthCheck
{
    public async Task<HealthCheckResult> CheckHealthAsync(
        HealthCheckContext context,
        CancellationToken cancellationToken = default)
    {
        try
        {
            // Check database connectivity
            await using var connection = new SqlConnection(_connectionString);
            await connection.OpenAsync(cancellationToken);
            return HealthCheckResult.Healthy("Database is accessible");
        }
        catch (Exception ex)
        {
            return HealthCheckResult.Unhealthy("Database is not accessible", ex);
        }
    }
}

15. Common Interview Questions (All Levels)

Q26: What is the difference between IApplicationBuilder and IWebHostBuilder?
  • IWebHostBuilder: Used to build and configure the web host. Responsible for setting up the hosting environment, configuration, and services. Used in Program.cs to create the host.
  • IApplicationBuilder: Used to configure the HTTP request pipeline. Responsible for adding middleware components. Used in Startup.cs or Program.cs to configure the application pipeline.
Q27: What is the difference between AddTransient, AddScoped, and AddSingleton?
Lifetime Description Use Case
Transient New instance each time requested Lightweight, stateless services
Scoped New instance per HTTP request Request-specific services, DbContext
Singleton Single instance for app lifetime Stateless services, caching
Q28: What is the difference between async/await and Task.Run?
  • async/await: Used for I/O-bound operations. Doesn't block threads. Efficient for network, file, database operations.
  • Task.Run: Offloads work to ThreadPool. Used for CPU-bound operations. Can cause thread pool starvation if overused.
 Good - async/await for I/O
public async Task<User> GetUserAsync(int id)
{
    return await _context.Users.FindAsync(id);
}

// Good - Task.Run for CPU-bound
public async Task<string> ProcessDataAsync(string data)
{
    return await Task.Run(() => HeavyComputation(data));
}
Q29: What is the difference between IActionResult and ActionResult?
  • IActionResult: Interface for action results. Can return any type that implements IActionResult. More flexible.
  • ActionResult<T>: Can return either T or IActionResult. Provides type safety and better API documentation. Recommended for API controllers.
Q30: What is the difference between appsettings.json and appsettings.Development.json?
  • appsettings.json: Base configuration file. Contains default settings. Used in all environments.
  • appsettings.Development.json: Environment-specific file. Overrides base settings in Development environment. Not deployed to production.
Q31: What is the difference between AddMvc and AddControllers?
  • AddMvc: Adds full MVC support including controllers, views, Razor pages, and API features. Use for web applications with views.
  • AddControllers: Adds only API controller support. Lightweight, no view support. Use for API-only applications.
Q32: What is the difference between [FromRoute] and [FromQuery]?
  • [FromRoute]: Binds parameter from route data (URL path). Used for required identifiers.
  • [FromQuery]: Binds parameter from query string. Used for optional parameters, filters, pagination.
Q33: What is the difference between IFormFile and byte[]?
  • IFormFile: Represents a file sent with the HttpRequest. Provides stream access, file metadata. Efficient for large files.
  • byte[]: Loads entire file into memory. Not recommended for large files. Can cause memory issues.
Q34: What is the difference between OnGet and OnPost in Razor Pages?
  • OnGet: Called when page is requested via GET. Used to initialize page data. Should not modify data.
  • OnPost: Called when form is submitted via POST. Used to process form data. Can modify data.
Q35: What is the difference between ViewComponent and Partial View?
  • ViewComponent: Has its own logic and model. More powerful, can have dependencies. Good for reusable components with logic.
  • Partial View: Simple view fragment. No logic, just rendering. Good for static or simple reusable UI.
Q36: What is the difference between Tag Helpers and HTML Helpers?
  • Tag Helpers: HTML-like syntax, more readable. Work with HTML elements. More modern approach.
  • HTML Helpers: C# method syntax, less readable. Generate HTML strings. Legacy approach.
Q37: What is the difference between [Authorize] and [AllowAnonymous]?
  • [Authorize]: Requires user to be authenticated. Can specify roles/policies. Applied at controller or action level.
  • [AllowAnonymous]: Bypasses authorization. Allows unauthenticated access. Used for login/register pages.
Q38: What is the difference between Authentication and Authorization?
  • Authentication: Verifies who the user is. Process of identifying the user. Examples: login, JWT, cookies.
  • Authorization: Verifies what the user can do. Process of granting permissions. Examples: roles, policies, claims.
Q39: What is the difference between AddIdentity and AddIdentityCore?
  • AddIdentity: Adds full identity system including UI, cookie authentication, role management. Use for web applications with UI.
  • AddIdentityCore: Adds only core identity features without UI. Use for API-only applications.
Q40: What is the difference between AddJwtBearer and AddCookie?
  • AddJwtBearer: JWT token authentication. Stateless, suitable for APIs. Token sent in Authorization header.
  • AddCookie: Cookie-based authentication. Stateful, suitable for web apps. Cookie stored in browser.
Q41: What is the difference between ResponseCache and OutputCache?
  • ResponseCache: ASP.NET Core attribute for HTTP caching. Configures cache headers. Works with client and proxy caches.
  • OutputCache: ASP.NET 4.x feature. Not available in ASP.NET Core. Use ResponseCache instead.
Q42: What is the difference between IMemoryCache and IDistributedCache?
  • IMemoryCache: In-memory cache. Fast, but not distributed. Lost on app restart. Good for single-server apps.
  • IDistributedCache: Distributed cache (Redis, SQL). Slower, but shared across servers. Good for multi-server apps.
Q43: What is the difference between AddMemoryCache and AddDistributedMemoryCache?
  • AddMemoryCache: Adds IMemoryCache service. Local in-memory cache. Not distributed.
  • AddDistributedMemoryCache: Adds IDistributedCache using memory. For development/testing only. Not production-ready.
Q44: What is the difference between BackgroundService and IHostedService?
  • IHostedService: Base interface for background services. Requires implementing StartAsync and StopAsync. More control but more boilerplate.
  • BackgroundService: Abstract base class for long-running tasks. Implements IHostedService. Provides ExecuteAsync method. Simpler to use.
Q45: What is the difference between SignalR and WebSocket?
  • SignalR: High-level abstraction for real-time communication. Handles connection management, fallbacks, scaling. Built on top of WebSockets.
  • WebSocket: Low-level protocol for bidirectional communication. Requires manual connection management. No built-in fallbacks.

16. Advanced Topics (4-13 Years Experience)

Q51: How do you implement Circuit Breaker pattern in ASP.NET Core?

The Circuit Breaker pattern prevents cascading failures by stopping requests to a failing service after a threshold is reached.

// Using Polly for Circuit Breaker
builder.Services.AddHttpClient("ExternalApi")
    .AddTransientHttpErrorPolicy(policy =>
        policy.OrResult(response => response.StatusCode == System.Net.HttpStatusCode.ServiceUnavailable)
              .CircuitBreakerAsync(
                  handledEventsAllowedBeforeBreaking: 3,
                  durationOfBreak: TimeSpan.FromSeconds(30)));

// Usage in service
public class ExternalService
{
    private readonly IHttpClientFactory _httpClientFactory;

    public ExternalService(IHttpClientFactory httpClientFactory)
    {
        _httpClientFactory = httpClientFactory;
    }

    public async Task<string> GetDataAsync()
    {
        var client = _httpClientFactory.CreateClient("ExternalApi");
        return await client.GetStringAsync("/api/data");
    }
}
Important: Combine Circuit Breaker with Retry pattern for resilience. Use different thresholds for different services based on their criticality.
Q52: How do you implement distributed tracing in ASP.NET Core?

Distributed tracing helps track requests across multiple services in microservices architecture.

// Using OpenTelemetry
builder.Services.AddOpenTelemetry()

        resource.AddService("MyService"))
    .AddAspNetCoreInstrumentation()

    })
    .AddJaegerExporter(options =>
    {
        options.AgentHost = "jaeger";
        options.AgentPort = 6831;
    });

// Adding custom spans
using var activity = _activitySource.StartActivity("CustomOperation");
activity?.SetTag("operation.name", "ProcessData");
activity?.SetTag("user.id", userId);

// Process data
await ProcessDataAsync(data);

activity?.SetTag("operation.status", "success");

Q68: How do you implement the Saga pattern for distributed transactions?

Saga pattern manages distributed transactions by breaking them into local transactions with compensating actions.

// Saga orchestration
public class OrderSaga
{
    private readonly IOrderRepository _orderRepository;
    private readonly IInventoryService _inventoryService;
    private readonly IPaymentService _paymentService;
    private readonly IShippingService _shippingService;
    
    public async Task ExecuteAsync(CreateOrderCommand command)
    {
        var order = new Order(command.CustomerId, command.Items);
        await _orderRepository.AddAsync(order);
        
        try
        {
            // Step 1: Reserve inventory
            await _inventoryService.ReserveAsync(order.Id, order.Items);
            
            // Step 2: Process payment
            var paymentResult = await _paymentService.ProcessAsync(
                order.Id, 
                order.TotalAmount);
            
            if (!paymentResult.Success)
            {
                // Compensate: Release inventory
                await _inventoryService.ReleaseAsync(order.Id, order.Items);
                order.MarkAsFailed("Payment failed");
                await _orderRepository.UpdateAsync(order);
                return;
            }
            
            // Step 3: Create shipment
            await _shippingService.CreateShipmentAsync(order.Id, order.ShippingAddress);
            
            order.MarkAsCompleted();
            await _orderRepository.UpdateAsync(order);
        }
        catch (Exception ex)
        {
            // Compensate: Rollback all steps
            await CompensateAsync(order);
            order.MarkAsFailed(ex.Message);
            await _orderRepository.UpdateAsync(order);
        }
    }
    
    private async Task CompensateAsync(Order order)
    {
        await _inventoryService.ReleaseAsync(order.Id, order.Items);
        await _paymentService.RefundAsync(order.Id, order.TotalAmount);
        await _shippingService.CancelShipmentAsync(order.Id);
    }
}

Q69: How do you implement the Strangler Fig pattern for legacy migration?

Strangler Fig pattern gradually replaces legacy systems by building new functionality around the old system.

// API gateway that routes to legacy or new service
public class MigrationGateway
{
    private readonly ILegacyServiceClient _legacyClient;
    private readonly INewServiceClient _newClient;
    private readonly IMigrationConfiguration _config;
    
    public async Task<OrderDto> GetOrderAsync(int orderId)
    {
        // Check if this order is migrated
        if (await _config.IsMigratedAsync("Order", orderId))
        {
            return await _newClient.GetOrderAsync(orderId);
        }
        
        // Fall back to legacy system
        return await _legacyClient.GetOrderAsync(orderId);
    }
    
    public async Task<OrderDto> CreateOrderAsync(CreateOrderRequest request)
    {
        // Always create in new system
        var order = await _newClient.CreateOrderAsync(request);
        
        // Mark as migrated
        await _config.MarkAsMigratedAsync("Order", order.Id);
        
        return order;
    }
}

// Feature flag for gradual rollout
public class FeatureBasedRouter
{
    private readonly IFeatureManager _featureManager;
    private readonly ILegacyService _legacyService;
    private readonly INewService _newService;
    
    public async Task<T> ExecuteAsync<T>(string featureName, Func<ILegacyService, Task<T>> legacy, Func<INewService, Task<T>> @new)
    {
        if (await _featureManager.IsEnabledAsync(featureName))
        {
            return await @new(_newService);
        }
        
        return await legacy(_legacyService);
    }
}

// Data synchronization
public class DataSyncService : BackgroundService
{
    private readonly ILegacyRepository _legacyRepo;
    private readonly INewRepository _newRepo;
    
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            // Sync data from legacy to new system
            var legacyOrders = await _legacyRepo.GetOrdersModifiedSinceAsync(_lastSyncTime);
            
            foreach (var order in legacyOrders)
            {
                if (!await _newRepo.ExistsAsync(order.Id))
                {
                    await _newRepo.AddAsync(MapToNewModel(order));
                }
                else
                {
                    await _newRepo.UpdateAsync(MapToNewModel(order));
                }
            }
            
            _lastSyncTime = DateTime.UtcNow;
            await Task.Delay(TimeSpan.FromMinutes(5), stoppingToken);
        }
    }
}
Q70: How do you implement the Bulkhead pattern for resource isolation?

Bulkhead pattern isolates resources to prevent cascading failures.

// Custom bulkhead implementation
public class BulkheadSemaphore
{
    private readonly SemaphoreSlim _semaphore;
    private readonly ILogger<BulkheadSemaphore> _logger;
    
    public BulkheadSemaphore(int maxConcurrent, ILogger<BulkheadSemaphore> logger)
    {
        _semaphore = new SemaphoreSlim(maxConcurrent, maxConcurrent);
        _logger = logger;
    }
    
    public async Task<T> ExecuteAsync<T>(Func<Task<T>> operation)
    {
        if (!await _semaphore.WaitAsync(TimeSpan.FromSeconds(5)))
        {
            _logger.LogWarning("Bulkhead rejected - timeout waiting for semaphore");
            throw new BulkheadRejectedException("Operation rejected due to bulkhead limit");
        }
        
        try
        {
            return await operation();
        }
        finally
        {
            _semaphore.Release();
        }
    }
}

// Usage in service
public class ExternalApiService
{
    private readonly BulkheadSemaphore _bulkhead;
    
    public ExternalApiService(BulkheadSemaphore bulkhead)
    {
        _bulkhead = bulkhead;
    }
    
    public async Task<Data> GetDataAsync()
    {
        return await _bulkhead.ExecuteAsync(async () =>
        {
            return await _httpClient.GetAsync("/api/data");
        });
    }
